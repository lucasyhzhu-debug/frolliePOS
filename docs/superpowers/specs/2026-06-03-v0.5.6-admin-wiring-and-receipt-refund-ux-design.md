# v0.5.6 — Admin wiring + receipt/refund UX (design spec)

> **Status:** DRAFT — started 2026-06-03 during the production cutover. Sibling to
> the v0.5.5 spec (inventory-SKU + error boundary); kept separate so the two can be
> reviewed/planned independently. Version `v0.5.6` provisional — renumber if it
> collides. Run through `/staffreview` → `/superpowers:writing-plans` →
> `/staffreview` before executing.

## Why this phase exists (context)

Four UX/admin gaps surfaced live while operating the prod deployment. Every one is
a **wiring gap** — the backend already exists; the UI surface was never built. No
new backend functions are required for any part (unlike v0.5.5 Part A).

1. **No self "Change PIN" screen.** `auth.changePin` (self, current-PIN-gated)
   exists but is **not called anywhere in `src/`**. Staff cannot change their own
   PIN in-app. A manager's only other path, `resetStaffPin`, *correctly* refuses
   self-reset (`USE_CHANGE_PIN_FOR_SELF`, `convex/auth/actions.ts:25`) — so a sole
   manager (prod `S-0001` on the default `1111`) is stuck with no in-app way to
   change their own PIN.

2. **No "Generate device setup code" screen.** `staff.public.generateDeviceSetupCode`
   (manager-gated) exists but is **not wired into any UI** — `/mgr/staff` wires
   only create/rename/role/deactivate/reset. The only way to mint a device-
   registration code today is the CLI bootstrap helper
   (`seed/internal:_devMintSetupCode_internal`). Registering a *second* device has
   no in-app path.

3. **No "Print receipt" on transaction history.** v0.5.4 (ADR-043) shipped Bluetooth
   thermal printing — `useThermalPrinter`, `PrinterSheet`, `PrinterProvider`,
   `getReceiptForPrint` (print view-model query) — but it's only surfaced on
   `sale/charge-success.tsx` (right after a sale) and `home.tsx`/`AppHeader`. The
   history detail (`/history/$txnId`) can *view/share* the receipt but has **no
   reprint button**, so a staffer can't reprint an earlier sale's basic receipt.

4. **Refund flow is orphaned (unreachable).** The refund routes exist
   (`router.tsx`: `/refund`, `/refund/:txnId` → `RefundDetail`, a real implemented
   flow from v0.5.1b with `RefundLineSelector` + `computeRefundAmount`), but
   **nothing in the UI links to them** — no button on history, no nav entry. The
   flow is reachable only by typing the URL. "Where is refunds?" → it has no door.

## Goal

- Any logged-in staff can **change their own PIN** in-app.
- A manager can **generate a device setup code** from the UI (no CLI).
- The **history detail screen can reprint** an earlier sale's basic receipt.
- A staffer can **reach the refund flow** from the transaction it refunds.

---

## Part A — self "Change PIN" screen

- Wire the **existing** `auth.changePin` (`sessionId`, `currentPin`, `newPin`,
  `idempotencyKey`) into a UI — **no backend change**.
- Surface: "Change PIN" entry on a profile/account menu (or the lock screen
  overflow), available to **any** logged-in staff (self-service, current-PIN gated
  — not manager-only).
- Form: current + new + confirm via `NumericKeypad`/`PinSheet`. Friendly error
  mapping: `INVALID_PIN` (wrong current), `SAME_PIN`, `NEW_PIN_INVALID` (not 4
  digits); respect the 3-strike lockout (ADR-002).
- Closes the sole-manager dead-end (no 2-manager reset dance needed).

## Part B — "Generate device setup code" screen

- Wire the **existing** `staff.public.generateDeviceSetupCode` (manager-gated) —
  **no backend change**.
- Surface (decided): a **new "Generate setup code" button on the manager Home
  screen** (`src/routes/mgr/home.tsx`), with the minted code **displayed inline**
  on that screen — the 6-digit code + its 1h expiry shown right where the button
  is, ready to read off and type into a new device's `/activate`. No modal hop,
  no CLI, no separate route.
  - UX detail: show the code large/monospace + a countdown to expiry; a
    "regenerate" affordance re-mints (superseding the prior code). Copy button
    optional (booth operator usually reads it aloud to the device holder).
- Optional (scope in planning): list active `registered_devices` with a deactivate
  action. Deactivate may need a new backend mutation — defer if so.
- Replaces the CLI `_devMintSetupCode_internal` bootstrap path for everyday use.

## Part C — "Print receipt" button on history detail

- In `src/routes/history/$txnId.tsx`, add a **"Print receipt"** button that reuses
  the v0.5.4 stack: open `PrinterSheet`, fetch the view-model via the existing
  `getReceiptForPrint` query for this `txnId`, encode with `escpos.encodeReceipt`,
  send via `useThermalPrinter`. **No backend change** — `getReceiptForPrint` and
  the printer hooks already exist; this is the same wiring `charge-success.tsx`
  already does, lifted onto the history detail screen.
- Reuse `PrinterProvider` context so connection state is shared (don't re-pair).
- Works for paid + partially/fully refunded txns (the view-model already encodes
  refund lines per ADR-043).

## Part D — refund flow entry point

- Add a **"Refund"** action on `history/$txnId.tsx` (and/or a refunds list) that
  `navigate(\`/refund/\${txnId}\`)` — giving the existing `RefundDetail` flow a
  door. The flow itself is **already built** (mgr-PIN / Telegram-approval gated);
  this is purely a link.
- Gate visibility: only show "Refund" for paid, not-fully-refunded txns (reuse the
  `refundStatus`/`REFUND_BADGE` the history detail already computes).
- Confirm the `/refund` **index** route's state (list vs stub) in planning — if
  it's a stub, either point the nav at per-txn refunds only, or build a minimal
  "refundable transactions" list. (CLAUDE.md's routes table still calls `refund`
  "Stubbed" — verify against the implemented `refund/detail.tsx`.)

## Testing

- **A:** component test the change-PIN form maps `INVALID_PIN`/`SAME_PIN`/
  `NEW_PIN_INVALID` to copy; happy path calls `changePin` once with the idem key.
- **B:** manager sees a code; non-manager can't reach the screen.
- **C:** history detail renders a print button for a paid txn; clicking opens
  `PrinterSheet` with the right `txnId` view-model (mock `getReceiptForPrint`).
- **D:** "Refund" shows only for paid/not-fully-refunded txns and navigates to
  `/refund/:txnId`; hidden for fully-refunded.

## Success criteria & rollback

- **Done when:** (A) `S-0001` changes `1111`→new PIN in-app; (B) a manager mints a
  setup code and registers a 2nd device with no CLI; (C) reprinting an old sale
  from history produces the same bytes as charge-success; (D) a staffer starts a
  refund from a historical transaction without typing a URL.
- **Rollback:** all four are additive UI wiring of existing backend — revert the
  commits; no schema change, no migration.

## Out of scope

- New backend functions (all four wire existing ones). Exception: a device-
  *deactivate* mutation if Part B's optional device list is included.
- Refund *logic* changes (the flow is v0.5.1b; this only adds the entrance).
- Receipt *template* changes (Part C reuses the existing view-model).
- A general settings/profile information architecture (Part A just needs one entry
  point, not a full settings section).
