## Why this phase exists (context)

Four UX/admin gaps surfaced live while operating the prod deployment. Every one is
a **wiring gap** — the backend already exists; the UI surface was never built. No
new backend functions are required for any part (unlike v0.5.5 Part A).

1. **No self "Change PIN" screen.** `auth.changePin` (self, current-PIN-gated)
   exists but is **not called anywhere in `src/`**. Staff cannot change their own
   PIN in-app. A manager's only other path, `resetStaffPin`, *correctly* refuses
   self-reset (`USE_CHANGE_PIN_FOR_SELF`, inside `resetStaffPin`, `convex/auth/actions.ts`) — so a sole
   manager (prod `S-0001` on the default `1111`) is stuck with no in-app way to
   change their own PIN.

2. **No "Generate device setup code" screen.** `staff.public.generateDeviceSetupCode`
   (manager-gated) exists but is **not wired into any UI** — `/mgr/staff` wires
   only create/rename/role/deactivate/reset. The only way to mint a device-
   registration code today is the CLI bootstrap helper
   (`seed/internal:_devMintSetupCode_internal`). Registering a *second* device has
   no in-app path.

3. **No "Print receipt" on transaction history.** v0.5.4 (ADR-043) shipped Bluetooth
   thermal printing — `useThermalPrinter`, `PrinterSheet`, `PrinterProvider`,
   `getReceiptForPrint` (print view-model query) — but it's only surfaced on
   `sale/charge-success.tsx` (right after a sale) and `home.tsx`/`AppHeader`. The
   history detail (`/history/$txnId`) can *view/share* the receipt but has **no
   reprint button**, so a staffer can't reprint an earlier sale's basic receipt.

4. **Refund flow is orphaned (unreachable).** The refund routes exist
   (`router.tsx`: `/refund`, `/refund/:txnId` → `RefundDetail`, a real implemented
   flow from v0.5.1b with `RefundLineSelector` + `computeRefundAmount`), but
   **nothing in the UI links to them** — no button on history, no nav entry. The
   flow is reachable only by typing the URL. "Where is refunds?" → it has no door.

## Goal

- Any logged-in staff can **change their own PIN** in-app.
- A manager can **generate a device setup code** from the UI (no CLI).
- The **history detail screen can reprint** an earlier sale's basic receipt.
- A staffer can **reach the refund flow** from the transaction it refunds.

## Part A — self "Change PIN" screen

- Wire the **existing** `auth.changePin` (`sessionId`, `currentPin`, `newPin`,
  `idempotencyKey`) into a UI — **no backend change**. Defined at
  `convex/auth/actions.ts:177` (verified).
- **Surface (decided):** a **"Change PIN" tile in the `you` group of the staff home
  launcher** (`src/routes/home.tsx` `TILES`, alongside Settlements / Lock+handoff) →
  a new **`/account`** spoke route hosting the form. Available to **any** logged-in
  staff (`mgrOnly: false`). This matches the tile-launcher → spoke idiom used
  everywhere; avoids the de-scoped "settings IA." (Resolves Open Item #1.)
- Form: current + new + confirm via `NumericKeypad`/`PinSheet`.
- **Idempotency (pinned):** `changePin` is an **action with its own action-level
  cache** (`_lookup_internal` / `_writeCache_internal`) — **not** a `withIdempotency`
  mutation. The FE mints a **one-shot `crypto.randomUUID()`** at submit (the
  `shareReceipt` / `sale/drafts.tsx` convention, e.g. `history/$txnId.tsx:57`) —
  **do NOT use `useIdempotency`** (that's for replayed mutations like login/payment).
- **Full error map** (the action throws all of these, `actions.ts:184-230`):
  - `NEW_PIN_INVALID` — new PIN not 4 digits.
  - `SAME_PIN` — new === current.
  - `INVALID_PIN` — wrong current PIN (or deactivated staff).
  - `LOCKED_OUT` — 3-strike lockout (ADR-002); show the 60s-lockout state, reuse the
    login screen's lockout copy if present.
  - `SESSION_INVALID` — session force-ended; toast + redirect to `/login`.
- Closes the sole-manager dead-end (no 2-manager reset dance needed).

## Part B — "Generate device setup code" screen

- Wire the **existing** `staff.public.generateDeviceSetupCode` (manager-gated,
  `convex/staff/public.ts:113`, returns `{ code, expiresAt }`) — **no backend change**.
- **Surface (revised by staffreview):** a **new `/mgr/device-setup` spoke route** +
  a **10th `NAV_CARD`** on `mgr/home.tsx`. Rationale: `mgr/home.tsx` is a **pure
  `NAV_CARDS` launcher** (grid of `<Link>`s) — every manager function (products,
  staff, vouchers, spoilage, receipt, refunds-pending, stock) is its own spoke.
  Embedding a stateful mint→display→countdown widget *inside* the grid breaks that
  uniformity. A spoke route honors the spec's intent (no modal, no CLI) while
  matching the established idiom; a spoke is **not** a "modal hop."
  - On the `/mgr/device-setup` spoke: a "Generate setup code" button → mints, then
    shows the 6-digit code **large/monospace** + a **countdown to expiry** (driven
    off `expiresAt`) ready to read off and type into a new device's `/activate`. A
    "regenerate" affordance re-mints (server supersedes the prior code). Copy button
    optional (booth operator usually reads it aloud). Expired (countdown → 0) ⇒
    prompt regenerate; never present a dead code as usable.
  - Manager-gate the spoke (mirror `mgr/home.tsx`'s `role !== "manager"` →
    `<Navigate to="/" replace />`).
- Optional (scope in planning): list active `registered_devices` with a deactivate
  action. Deactivate may need a new backend mutation — **defer** (keeps phase
  backend-free).
- Replaces the CLI `_devMintSetupCode_internal` bootstrap path for everyday use.

## Part C — "Print receipt" button on history detail

- In `src/routes/history/$txnId.tsx`, add a **connect-or-print button** that **lifts
  the exact `charge-success.tsx` pattern** (lines 38-57, 159-163) — **not**
  `PrinterSheet`. `PrinterSheet` is the standalone connection-management sheet (home
  header / `AppHeader`); the per-receipt print trigger is the `usePrinter()` button:
  - `const { status, connect, print } = usePrinter();` (context from `PrinterProvider`,
    which already wraps `<Outlet>` in `RootLayout.tsx:56` — history detail has it, no
    re-pair).
  - `const printData = useQuery(api.receipts.public.getReceiptForPrint, sessionId && txnId ? { sessionId, txnId } : "skip");`
  - `onPrint`: `encodeReceipt(printData.viewModel, printData.status, printData.statusLabel)` → `print(bytes)` → Sonner toast.
  - Button: `onClick={status === "connected" ? onPrint : connect}`; `disabled` when
    `printing` / `unsupported` / `!printData` (mirror `charge-success.tsx:159-163`).
- **No backend change** — `getReceiptForPrint` + printer hooks already exist.
- Works for paid + partially/fully refunded txns (the view-model already encodes
  refund lines per ADR-043). `getReceiptForPrint` returns `null` for non-paid /
  out-of-scope → button stays disabled (the `!printData` guard).

## Part D — refund flow entry point (LINKING ONLY — list already exists)

- **Resolved:** `/refund` index is **already a complete, tested list**
  (`src/routes/refund/index.tsx` = `RefundList` → `refunds.public.listTodaysRefundable`
  (`convex/refunds/public.ts:21`) → navigates to `/refund/:txnId`). Routed at
  `router.tsx:96`. **Do NOT rebuild it.** The only gap is *nothing links in.* So
  Part D is two link additions:
  - **(a) Per-txn button** on `history/$txnId.tsx`: a **"Refund"** button →
    `navigate(\`/refund/\${txnId}\`)`. The `RefundDetail` flow is already built
    (mgr-PIN / Telegram-approval gated); this is purely a link.
  - **(b) Home tile** on `src/routes/home.tsx` `TILES` → `/refund` (the existing
    list). Staff-visible (the list + detail self-gate; the ACT is PIN/Telegram-gated
    inside `RefundDetail`).
- **Gate (per-txn button):** show only for **paid AND `refundStatus !== "full"`**.
  `history/$txnId.tsx` already computes `refundStatus` (`$txnId.tsx:116`); keys are
  `none | partial | full` (`src/lib/pos-labels.ts:9`). Show for `none`/`partial`,
  hide for `full`. Confirm `getTransactionDetail`'s `txn` exposes a paid signal
  (`paid_at` is present, `$txnId.tsx:118`).
- **Docs:** update CLAUDE.md's routes table — `refund` is **live** (list + detail),
  not "Stubbed."

## Testing

- **A:** component test the change-PIN form maps the **full** error set —
  `INVALID_PIN` / `SAME_PIN` / `NEW_PIN_INVALID` / **`LOCKED_OUT`** (60s lockout
  state) / **`SESSION_INVALID`** (redirect to `/login`) — to copy; happy path calls
  `changePin` **once** with a `crypto.randomUUID()` key (assert the key shape — guards
  against a `useIdempotency` regression).
- **B:** manager reaches `/mgr/device-setup` and sees a code + countdown; **non-manager
  is redirected** (`<Navigate to="/" />`, mirror `mgr/home` gate). Regenerate re-mints.
- **C:** history detail renders the connect-or-print button for a paid txn; clicking
  `onPrint` calls `encodeReceipt(viewModel, status, statusLabel)` from the mocked
  `getReceiptForPrint` (asserts it consumes the query, not a re-derived view-model);
  disabled when `getReceiptForPrint` → `null`.
- **D:** "Refund" button shows for `refundStatus` `none`/`partial`, **hidden for
  `full`**, and navigates to `/refund/:txnId`. (Don't modify the existing
  `refund/__tests__/index.test.tsx` — the list is untouched.)

## Success criteria & rollback

- **Done when:** (A) `S-0001` changes `1111`→new PIN in-app; (B) a manager mints a
  setup code and registers a 2nd device with no CLI; (C) reprinting an old sale
  from history produces the same bytes as charge-success; (D) a staffer starts a
  refund from a historical transaction without typing a URL.
- **Rollback:** all four are additive UI wiring of existing backend — revert the
  commits; no schema change, no migration.

## Out of scope

- New backend functions (all four wire existing ones). Exception: a device-
  *deactivate* mutation if Part B's optional device list is included.
- Refund *logic* changes (the flow is v0.5.1b; this only adds the entrance).
- Receipt *template* changes (Part C reuses the existing view-model).
- A general settings/profile information architecture (Part A just needs one entry
  point, not a full settings section).
