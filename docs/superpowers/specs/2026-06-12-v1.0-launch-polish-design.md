## Part 1 — Polish slice (code, ~half day)

### 1a. Audit sweep (~1h, no fixes yet)

On dev (`helpful-grasshopper-46`, seeded), walk the staff-critical loop as a **staff** session. Exact live routes in scope (verified against `src/routes/`):

`/sale`, `/sale/charge` (QRIS + BCA VA), `/sale/charge-success`, `/sale/drafts`, `/sale/voucher`, `/history`, `/history/$txnId`, `/refund`, `/refund/$txnId`, `/stock`, `/stock/in`, `/stock/recount`, `/stock/$skuId`.

For each screen, record three states in a findings table (screen × state × verdict):

1. **Empty state** — no data (no txns today, no drafts, no refundable txns, empty stock). Verdict: friendly copy + CTA, or blank/raw?
2. **Error state** — query failure and mutation failure. Verdict: surfaced (Sonner toast / inline error), or silent?
3. **Offline behavior** — vs ADR-025: catalog/cart/drafts/stock-in queue must work offline; payments/auth/refunds must **block with clear UI**, never fail cryptically.

The audit output is the fix list. No fix is in scope unless it appears in the findings table.

### 1b. Fix bucket (bounded)

Only three categories of fix are allowed:

- **Missing empty states** — friendly copy + CTA. No blank screens.
- **Silent mutation failures** — surface via Sonner (existing toast pattern).
- **Missing/unclear offline blocks** — payment/refund paths must show an explicit offline-blocked state per ADR-025.

**Explicitly out of scope:** layout changes, animation, new features, `mgr/*` screens, settlements/account/approve routes, pixel polish, any change to the money path (commit funnel, webhook, refund commit funnel are untouched).

### 1c. Gate

`npm run typecheck` + `npm run lint` + `npm run test` (full vitest) + `npm run test:e2e` (Playwright/chromium; `npm run test:e2e:install` first if browsers absent) — all green before Part 2 starts. Standard review pipeline applies to the slice.

## Part 2 — Launch checklist (ops, zero code)

Ordered; each step gates the next.

### 2-0. Deploy current `main` to prod (gates everything below)

Prod was last deployed at the 2026-06-03 cutover — v0.6.1 (auth hardening), v0.7 (settlements + `settlement-sync` cron), and the `e8e85a5` RFC3339 fix are **not in prod**. After the polish slice merges to `main`:

1. `npx convex deploy` — backend to `savory-zebra-800`.
2. Vercel prod deploy off the merged `main` (`npx vercel deploy --prod --yes`).
3. Verify in the prod Convex dashboard: **both** crons registered (`founders-shift-summary`, `settlement-sync`); `/payments/webhook` answers (401 on a wrong-token probe).

### 2a. Telegram final check (prod)

- Verify chats bound on the prod bot for all three roles: `managers`, `founders`, `inventory` (admin surface `/mgr/telegram-chats`).
- **Managers role:** send `/activatepos` in the managers group — a real, harmless round-trip (mints a 6-digit setup code, 1h TTL) proving webhook → role lookup → reply end-to-end.
- **Founders role:** on-demand summary (`npx convex run --prod telegram/foundersSummary:sendFoundersSummary`). The 22:00 WIB cron also fires tonight — a same-day double-send is acceptable.
- **Inventory role:** passive check (binding listed) — low-stock/drift alerts fire from real conditions only.
- Any missing binding → fix via `/register` in the target group + role assignment in-app.

### 2b. Prod data seeding (Lucas, manual, in-app admin)

Ordered checklist — order matters (products reference SKUs; stock-in needs both):

1. Inventory SKUs on `/mgr/products` (manager-PIN `createInventorySku`) — finished-goods SKUs only (e.g. `dubai`).
2. Products + component mappings (1pc, 3pcs, 8pcs, Mixed Box 4pcs → SKU decrements), same screen.
3. Opening stock-in via `/stock/in` — logged movements with `source`, never number edits (business rule #8).
4. Staff accounts (manager-PIN `createStaff` — **the manager sets each initial PIN at creation**; there is no set-on-first-login flow); staff rotate their PIN via `/account` at first login.
5. Optional: launch vouchers (manager-PIN `createVoucher` on `/mgr/vouchers`).

This step is performed by Lucas; the deliverable from the slice is the checklist itself (it becomes part of the RUNBOOK prod section, §2d).

### 2c. Live smoke test (one transaction validates the money loop)

1. Create temp product "Test" at Rp 1.000 (integer rupiah).
2. Real QRIS sale on the booth device; watch prod Convex logs for `qr.payment` → webhook confirm → status `paid`.
3. Receipt: share `/r/<token>` link + thermal print.
4. Refund the Rp 1.000 txn per ADR-038 semantics: create the refund (manager-PIN, booth path) → no real bank transfer needed (own money) → `markRefundSettled` ack (manager-session) → refund badge visible on `/history/$txnId`. Exercises the full refund funnel including the settle ack.
5. Archive the test product (`archiveProduct`, manager-session).

Pass = all five steps clean. Any failure stops the launch and gets diagnosed before staff use.

### 2d. Booth runbook (prod section in `docs/RUNBOOK.md`)

New section "Booth operations (prod)" — the v0.3 bootstrap runbook explicitly reserved this for the v1.0 cutover. One page:

- **Payment didn't confirm** → wait for webhook; if stuck, manager manual-override flow (booth PIN or Telegram `/approve` link).
- **Device dead** → activate a staff/manager phone as a backup device via `/activatepos` in the managers Telegram group (v0.5.7) and keep selling; paper notebook is the last resort (stays at the booth for week 1), with manager recount to reconcile stock afterwards.
- **Telegram down** → approvals fall back to booth manager-PIN; summaries skip (audited, no retry storm).
- **Xendit outage** → no digital payment possible; paper IOU protocol + manager notified.
- **Escalation** — who to contact (Lucas), where logs live (Convex dashboard prod), link to `RUNBOOK-telegram.md` for bot ops.
- **Prod data seeding order** (from §2b) for future re-setup.

## Risks

- **Fresh UI code hours before first real sale.** Mitigated by scope: fixes are additive presentation states only; the money path is untouched, and the full test suite + e2e gate sits between the slice and the launch.
- **Smoke-test failure on prod.** The checklist is ordered so this is discovered before staff onboarding; the refund step also burns the test Rp 1.000 cleanly.
- **Seeding mistakes (wrong component mappings).** Mitigated by the ordered checklist + the smoke test exercising a real decrement.

## Success criteria

1. Audit findings table produced; every staff-critical screen has non-blank empty states, surfaced errors, and ADR-025-conformant offline behavior.
2. All gates green (typecheck, lint, vitest, Playwright).
3. Prod runs the launch commit: backend deployed, Vercel deployed, both crons registered, webhook probe answers.
4. Telegram: managers + founders verified with a real prod send; inventory binding confirmed.
5. Smoke test: one real Rp 1.000 QRIS sale confirmed paid via webhook, receipt shared + printed, refunded + settled, product archived.
6. `RUNBOOK.md` has the booth-operations prod section.
7. `v1.0.0` tagged; PROGRESS.md reconciled; v1.0.1 phase created with the deferred list.
8. Staff ring real sales today.
