## 1. Problem

Two coupled defects on the payment path:

1. **BCA VA error storm.** `createBcaVaCharge` (`convex/payments/xendit.ts:96`) is LIVE-UNVERIFIED and throws `XENDIT_VA_FAILED` â†’ `BANK_NOT_ACTIVATED_ERROR`. The charge screen's auto-create effect (`src/routes/sale/charge.tsx:173-233`) re-fires `requestPayment({method:"BCA_VA"})` on every tab/invoice change; each failure raises a `toast.error` (`charge.tsx:222`) â†’ a storm of error toasts. BCA VA never completes a sale.
2. **No working non-QRIS tender.** Booth staff need a fallback when a customer can't pay via QRIS. The real-world fallback already in use is a **manual bank transfer** to the company BCA account, with proof posted to the Block M staff WhatsApp group â€” but the POS has no way to record that sale, so those transactions are invisible to the ledger, the receipt, and reconciliation.

## 2. Goal

- Kill the storm **by construction**: QRIS becomes the sole Xendit method; the BCA VA tab disappears, so the only auto-created invoice is QRIS (live-proven).
- Add a **manual bank-transfer tender**: staff see the static company account, tap a self-attestation confirm, and the sale commits â€” marked `manual_bca` so it is fully tracked.
- Make the attestation **safe without a photo gate**: a daily + clock-out **itemized reconciliation summary** lets operators validate every manual-BCA deposit against the real BCA account they already control. The per-sale Telegram ticker flags manual-BCA sales distinctly.

## 3. Non-goals (explicit out of scope)

- **Dynamic BCA VA revival** â€” backlogged (BCA-only, non-universal). Keep the code dormant.
- **In-app photo capture / `_storage` upload / `navigator.share` / `.webp`** â€” the proof flow stays fully out-of-band in WhatsApp; the POS records only the attestation tap. (Photo work is #3.)
- **The #6 clock-out screen** â€” #10 ships only the backing reconciliation **query** + the EOD cron hook. #6 builds the clock-out UI and its own staff-gated read wrapper.
- **Real Xendit refunds** â€” #9, spike-gated, lands after #10.
- **Removing the `BCA_VA` schema literal** â€” deploy-skew-fatal; breaks historical `pos_xendit_invoices` rows. Hide UI only.

### 6.1 Settings (`convex/settings/`)

**Schema** (`settings/schema.ts`) â€” add to `pos_settings` (all optional, additive; no migration):

**Defaults** (`settings/internal._getSettings_internal`) â€” add a `manual_bca` sub-object mirroring the existing `receipt` sub-object, single-sourced via a `MANUAL_BCA_DEFAULTS` const:


**Manager CRUD** (`settings/public.ts`, mirror `getReceiptConfig` / `updateReceiptConfig`):

- `updateManualBcaConfig({ idempotencyKey, sessionId, enabled, bank_name, account_name, account_number })` â†’ withIdempotency + `authCheck: requireManagerSession`; 120-char bound on each string (reuse the receipt-config bound); audit `payment.confirmed`-style â†’ emit `settings.manual_bca_updated` (new audit string â€” `audit_log.action` is `v.string()`, no enum migration). **Staff read** (`settings/public.ts`):

> **Account-number validation:** `account_number` is digits only in practice but stored as a string (leading-zero safe). Bound at 120 chars; no numeric coercion (ADR-015 is about money, not account ids).

### 6.2 Confirm path (`convex/payments/` + `convex/transactions/internal.ts`)

**`_confirmPaid_internal`** (`transactions/internal.ts:142`) â€” THE FUNNEL. Changes:
- `source` arg union: add `v.literal("manual_bca")`.
- New optional arg `confirm_staff_id: v.optional(v.id("staff"))` â€” the attesting cashier (audit actor for the manual-BCA path).
- Confirmed-via patch (step 7): `confirmed_via: args.source` already stores the source verbatim â†’ `"manual_bca"` flows through once the schema union accepts it.
- Audit (step 8) + the `confirmed_on_terminal` branch: for `source === "manual_bca"`, set `actor_id: args.confirm_staff_id ?? "system"` and `source: "booth_inline"`. Keep `action: "payment.confirmed"`, `metadata.source: "manual_bca"`. (Webhook/polling keep `source:"system"`; manager `manual` keeps its `approvalSource` logic.)
- Ticker scheduling (step 9) unchanged â€” fires once per paid txn for all sources.

**`pos_transactions.confirmed_via`** (`transactions/schema.ts:38`) â€” add `v.literal("manual_bca")` to the union.

**`_getTxnForTicker_internal`** (`transactions/internal.ts:752`) â€” widen the returned `confirmed_via` union type to include `"manual_bca"`.

**New mutation `confirmManualBcaPayment`** (`convex/payments/public.ts`):
```ts
args: { idempotencyKey: v.string(), sessionId: v.id("staff_sessions"), txnId: v.id("pos_transactions") }
returns: { confirmed: true, receiptNumber: string }
```
- withIdempotency-wrapped (cache key `args.idempotencyKey`); `authCheck: requireSession` (ADR-013 â€” gate the cache lookup itself so a replay can't read back `{confirmed}` without a session).
- Handler: resolve `{ staffId } = requireSession`; `runMutation(_confirmPaid_internal, { txnId, source:"manual_bca", confirm_staff_id: staffId })`; **then `runMutation(_cancelActiveInvoiceForTxn_internal, { txnId, cancel_reason:"manual_bca_confirmed", actor_id: staffId, source:"booth_inline" })`** (I1 â€” supersede the live QRIS invoice; see below); read back `txn`, assert `status === "paid" && receipt_number` else throw `RECEIPT_UNCONFIRMED` (mirrors `_onPaidManual_internal`'s C4 guard); return `{ confirmed: true, receiptNumber }`.
- **No Xendit call, no manager PIN.** The funnel's status-guard makes this idempotent and safe against a webhook/cancel race (a non-`awaiting_payment` txn no-ops). Reuses snapshot pricing + the `by_line_and_sku` double-decrement guard (business rule #17) because it routes through the same funnel.
- **Does NOT gate on `manual_bca_enabled`.** The toggle controls UI visibility only â€” a sale already in flight when a manager flips the toggle off must still commit. (Server-side enabled-gating would fail a legitimate mid-flow confirm.)
- **Idempotency key (I2):** the FE passes a **stable** `confirm-manual:${txnId}` (via `useIdempotency`), matching the `pay:${txnId}:${method}` convention â€” an accidental double-tap is a cache hit, not a second funnel attempt.

> **I1 â€” live-QR double-pay safety.** The charge screen defaults to QRIS, so a live QR is auto-minted on mount before staff switch to the manual tab. Confirming manual flips the txn to `paid` but leaves that QR **live on Xendit** (ADR-036 Decision E: QR has no expire API). A customer scanning it afterward pays twice; the webhook re-fire is ledger-safe (funnel status-guards â†’ silent no-op) but cash is double-collected with no alert. Mitigation: the confirm calls `_cancelActiveInvoiceForTxn_internal` (local supersede + audit) so the charge screen stops re-rendering the QR and the cancel is traceable. **Residual** (documented in the ADR-036 amendment): the local cancel cannot expire the QR on Xendit's side, so the reconciliation summary â€” which surfaces every manual-BCA deposit â€” remains the backstop for catching a stray double-pay.

> **Instrument badge in history (accepted, not new):** a `manual_bca` sale carries the auto-created (now-cancelled) QRIS invoice, so `_getPaidInvoiceForTxn_internal` â†’ `instrumentFromInvoice` reports `"qris"` for the sales-history instrument badge. This is **identical to how existing manager-override `manual` sales already display** (they too keep their QRIS invoice), so #10 does **not** expand scope to re-derive the history badge from `confirmed_via`. Reconciliation â€” the surface that actually matters for manual-BCA â€” keys off `confirmed_via === "manual_bca"`, never the invoice, so it stays accurate. (If a future phase wants the badge to reflect the confirmation path, it applies uniformly to `manual` + `manual_bca`, out of scope here.)

> **Why a new mutation, not `_onPaidManual_internal`:** that internal mutation mandates `mgr_approver_id: v.id("staff")` (non-optional) and stamps `confirmed_mgr_approver_id`. Manual-BCA has no manager approver, so it funnels to `_confirmPaid_internal` directly via this new public mutation rather than threading a fake manager through the manager path.

### 6.3 Charge screen (`src/routes/sale/charge.tsx`, `src/hooks/useXenditPayment.ts`)

- `Method` type â†’ `"QRIS" | "MANUAL_BCA"` (drop `"BCA_VA"` from the FE union; the backend `requestPayment` union keeps `BCA_VA` valid-but-unused).
- Tabs: `QRIS` | `Bank transfer (manual)`. Render the manual tab only when `getManualBcaAccount().enabled`.
- **QRIS tab:** unchanged (existing auto-create + QR render, driven by the `phase` machine).
- **MANUAL_BCA tab (C1 â€” renders independent of `phase`):** the manual tender has **no invoice**, so it must **not** be routed through the `phase.kind` switch (`computePhase` returns `"loading"` whenever `!invoice` â€” `useXenditPayment.ts:25` â€” which would otherwise leave the manual tab stuck on the "Preparing paymentâ€¦" spinner forever). Instead, the render branches first on `selectedMethod === "MANUAL_BCA"` and shows its own block: amount-due + the static account (bank / name / number, copy-to-clipboard on the number) + a required-checkbox attestation *"I confirm payment proof has been sent to the Block M staff WhatsApp group"* + a confirm button (disabled until checked). The `phase` machine governs the QRIS tab only.
- **Confirm + navigation:** on confirm â†’ `await confirmManualBcaPayment({ sessionId, txnId, idempotencyKey })`; **navigate to `/sale/charge/:txnId/success` on the awaited resolve** (NOT via the reactive `phase.kind === "paid"` effect â€” that effect can't fire for a manual sale because there's no invoice, so `computePhase` short-circuits to `"loading"` even after `txn.status` flips to `paid`). Inline `FieldMessage` (error tone) on failure â€” **no toast**.
- The auto-create effect (`charge.tsx:173-233`) must **skip** when `selectedMethod === "MANUAL_BCA"` (no invoice for the manual tab). Its catch branch (`charge.tsx:216-223`) converts `toast.error(msg)` â†’ an inline `FieldMessage` (L7) and keeps the re-fire guard (`requestedMethods`).
- `useXenditPayment` still drives the QRIS subscription; the manual tab doesn't subscribe to an invoice (there is none).
- The `usePathChangeBlocker(txn?.status === "awaiting_payment")` guard stays active on the manual tab (txn is awaiting_payment until confirm), so leaving without confirming/cancelling still prompts â€” correct behavior.
