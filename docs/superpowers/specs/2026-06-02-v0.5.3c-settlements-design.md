# v0.5.3c — Settlements: Xendit payout reconciliation (design spec)

**Date:** 2026-06-02
**Status:** Approved (brainstorm complete; ready for `/superpowers:writing-plans`)
**Phase:** v0.5.3c — third and final slice of v0.5.3 (per `docs/PROGRESS.md` line 1020; decomposition table in the v0.5.3a spec)
**Approach:** New deep `settlements/` module ingesting Xendit settlement data via a **nightly poll-cron** (no webhook exists) + a **manager manual-entry fallback**, surfaced as a read-only `/settlements` view with a per-day reconciliation cross-check. Ships code-complete / live-unverified pending KYB.
**Predecessor:** v0.5.3a (reporting), v0.5.3b (in-app admin), v0.5.2 (inventory), v0.5.1 (refunds + receipts) — all merged to `main` (this branch is cut from `origin/main` @ 908549e). Runs in parallel with the in-flight `dev-device-seed` branch; no code dependency on it.

## Decomposition context — why this is "3c"

v0.5.3 as scoped in PROGRESS.md is six independent subsystems. The v0.5.3a spec clustered them into three slices by *nature of work*:

| Slice | Contents | Risk |
| -- | -- | -- |
| v0.5.3a — Reporting *(shipped)* | Transaction history + manager dashboard | Low |
| v0.5.3b — In-app admin *(shipped)* | Staff CRUD + product CRUD + receipt config | Low–medium |
| **v0.5.3c — Settlements** *(this spec)* | `pos_settlements` + Xendit ingest + nightly recon | High |

3c was deliberately specced separately and last given its risk weight: it is the only net-new subsystem (new table, new external-event ingress, new cron) and it closes the load-bearing v1.0 **"Xendit settlement timing" risk-register item** (foundations §7, kept under watch).

## Goal

Give the booth a trustworthy, read-only window into **what Xendit has actually paid out to the merchant's bank** — per payout day, gross / MDR / net / count — and a reconciliation cross-check that catches money collected but not yet settled. Settlement data is ingested automatically by a nightly cron polling Xendit's Transactions API, with a manager manual-entry fallback so the view is usable before KYB clears. Closes the v1.0 settlement-timing risk item.

**"You'll be able to" outcomes** (carried from PROGRESS.md line 1031, scoped to 3c):
- Reconcile Xendit settlements — see, per payout day, what Xendit owes vs what it has paid out (gross, MDR, net, transaction count, destination bank last-4).
- See which of the booth's paid transactions have **settled** vs are still **pending** vs are **unmatched** — surfacing collected-but-unpaid-out money.
- (Manager) Record a settlement by hand when reading it from the Xendit dashboard, so `/settlements` is populated before live API settlement is available.

**"Still not yet" out-of-scope reminders** (the design enforces these — not on the build list):
- No automated **refund payouts** (Disbursements API) — refund money-movement stays manual per [ADR-038](../ADR/038-refund-settlement-manual-v1.md). (Different ledger — see Naming guard.)
- No multi-currency, no MDR *prediction* (we record Xendit's actual `fee`, never compute it).
- No CSV/report-file import path (rejected in favour of the API poll — see Q1).
- No Frollie Pro sync (the `synced_to_frollie_pro_at` field is a dormant v1.1 seam only).
- No editing/deleting of settlement rows once written (append/upsert-by-date only; corrections are a new manual row or a re-poll).

## Locked brainstorm answers (the source of truth for design decisions)

| Q | Decision | Rationale |
| -- | -- | -- |
| **Q1 — ingress** | **Nightly cron polls the List Transactions API**, aggregating SETTLED transactions by `settlement_date` into `pos_settlements`. *Not* a webhook. | **Verified 2026-06-02: Xendit has no settlement webhook** (`docs/xendit-reference/settlement-reconciliation.md`). Settlement is per-transaction (`settlement_status` + `settlement_date` + `fee`), knowable only via `GET /transactions` or report CSVs. The phase already bundled a nightly cron, so this is a mechanism pivot, not a scope change. Rejected: CSV import (adds an upload UI + parser + a manual step Xendit's own API makes unnecessary). |
| **Q2 — KYB fallback** | **Manual-entry fallback + automatic poll, both.** A manager can record a settlement by hand; the cron backfills/auto-creates once live. | KYB is not cleared → the integration runs on **TEST keys where real payouts/settlements don't occur**, so the poll yields nothing live yet. The manual path keeps `/settlements` usable now and mirrors ADR-036's webhook-primary + manager-override-fallback duality. The two paths converge on one writer + one `source` discriminator. |
| **Q3 — manual-entry gate** | **Manager-session, NOT manager-PIN.** | Recording a settlement is a *"tally what already happened"* bookkeeping write — the exact precedent is `markRefundSettled` ([ADR-038](../ADR/038-refund-settlement-manual-v1.md), business rule #21/#22: session-not-PIN because it moves no money). The cron writes the **identical row with no auth at all**, so a PIN gate on the human-entered twin would be ceremony without a security gain. Still fully audited (who/when). |
| **Q4 — visibility** | **Read-only, any authenticated session (staff + manager).** Per [ADR-012](../ADR/012-settlements-visible-to-staff-and-managers.md), unchanged. | Staff care that their shift's sales hit the bank; hiding it erodes trust in the POS as a financial record. Net + MDR shown; bank destination last-4 only. The *only* manager-gated writes are the manual entry + the on-demand "sync now" trigger. |
| **Q5 — settlement key / granularity** | **One `pos_settlements` row per `settlement_date`**, dedup/upsert by date. `xendit_settlement_id` synthesised from the date when no batch id is exposed. | Xendit exposes no clean "settlement batch" object (foundations §7's many-payments-per-settlement is real, but the grouping is *ours* to do, by `settlement_date`). Upsert-by-date + a multi-day lookback makes the cron self-healing: a re-run re-aggregates the day's now-larger settled set. |
| **Q6 — new ADR** | **Write ADR-043** superseding the "settlement webhook" *mechanism* in foundations §7 + ADR-012; keep ADR-012's *visibility* decision intact. | Same spike-corrects-a-speculative-ADR pattern as ADR-036→ADR-011. The webhook claim was written 2026-05-21 before verification; the ADR records the corrected poll-based mechanism + the live-unverified-pending-KYB stance, so it isn't re-litigated. |
| **Q7 — protocol isolation** | **All Xendit protocol detail stays in the deep adapter** `convex/payments/xendit.ts` (ADR-034 Decision D). | The new `fetchSettledTransactions(window)` joins `createQrisCharge`/`parseXenditWebhook` behind the narrow adapter surface — no Xendit URL/header/field name leaks into the cron, module, or view. Reviewers recognise the idiom. |

## Architecture overview — one deep module, two ingress paths, one read surface

```
  INGRESS (two paths, one writer)
  ├─ nightly cron ─▶ _reconcileSettlements_internal (action, "use node")
  │                     └─ xendit.fetchSettledTransactions(window)   [deep adapter]
  │                     └─ lib.aggregateBySettlementDate(...)         [pure, V8-safe]
  │                     └─ _upsertSettlement_internal(date, agg, source:"xendit_sync")
  └─ manager manual ─▶ recordSettlement (mutation, manager-session)
                          └─ _upsertSettlement_internal(date, fields, source:"manual")

  READ SURFACE (narrow — all session-gated per ADR-012)
  ├─ listSettlements(range?)        per-day rows (any session)
  └─ getSettlementDetail(id|date)   reconciliation cross-check: which paid txns
                                     SETTLED / PENDING / unmatched (any session)

  WRITE TRIGGERS (manager-gated)
  ├─ recordSettlement               manual fallback (session)
  └─ syncSettlementsNow             on-demand re-poll (session) → same cron action
```

The surface hides *how* settlements are ingested (poll vs manual), *how* days are aggregated, and *how* the reconciliation join is computed. Consumers (one view) just call queries.

```
  convex/settlements/
  ├─ schema.ts     pos_settlements (table + 2 indexes)
  ├─ lib.ts        pure aggregators (aggregateBySettlementDate, reconcileDay) — V8-safe, unit-tested
  ├─ internal.ts   _upsertSettlement_internal (single writer) · _reconcileSettlements_internal body · join helpers
  ├─ actions.ts    "use node": sendable cron action (resilient wrapper) + syncSettlementsNow
  └─ public.ts     listSettlements · getSettlementDetail · recordSettlement
```

## Backend (`convex/`)

### A. Deep adapter extension — `convex/payments/xendit.ts`

- **`fetchSettledTransactions(window): Promise<XenditSettledTxn[]>`** — `GET /transactions` filtered to `settlement_status=SETTLED` over a `[from, to]` window, following pagination to completion. Reuses `authHeader()` (Buffer-based; runs under the `"use node"` action). Returns a normalized array `{ referenceId, settlementStatus, settlementDate, amount, fee, channelCode, currency, createdAt }` — **no raw Xendit field names escape the adapter** (Q7). Pure body-builders (`buildTransactionsQuery(window)`) exported for tests, mirroring `buildQrisHeaders`/`buildQrisBody`.
  - ⚠️ **Field/param names asserted-not-verified** (`docs/xendit-reference/settlement-reconciliation.md` §Option A): the exact query params (`settlement_status`, date filters, pagination cursor) and response fields are confirmed against a real `GET /transactions` response (or `docs/xendit-reference` re-verification) **at plan/impl time**, exactly as ADR-036 treats the BCA VA FVA callback shape. The adapter's normalize step is the single place that absorbs any shape correction.

### B. Settlements module — `convex/settlements/`

- **`lib.ts`** *(pure, V8-safe — no `"use node"`, the v0.5.1a lesson)*:
  - `aggregateBySettlementDate(txns) → Map<date, { gross, mdr, net, count }>` where `gross = Σ amount`, `mdr = Σ fee`, `net = gross − mdr`, `count = n`. Integer rupiah throughout (rule #14 — no floats).
  - `reconcileDay(settledRefIds, posPaidTxnsForWindow) → { settled, pending, unmatched }` — the cross-check classifier (pure; takes plain arrays).
- **`internal.ts`**:
  - **`_upsertSettlement_internal({ settlementDate, gross, mdr, net, count, source, bankLast4?, payload?, actorId? })`** — the **single writer** for both ingress paths. Upsert by `settlement_date` via `by_settlement_date`: insert if absent, else patch the aggregates. Stamps `created_at`/`updated_at` server-side (rule #15). Emits `logAudit` `settlement.synced` (cron) / `settlement.recorded` (manual) — *(verb-collision check at plan time; `settlement.synced` already in SCHEMA.md:623, `settlement.recorded` is new)*. `source` discriminates the row.
  - **`_reconcileSettlements_internal` body** — orchestration called by the action: window → `fetchSettledTransactions` → `aggregateBySettlementDate` → upsert each date. Idempotent (upsert-by-date); a re-run over an overlapping window is a no-op-or-refine, never a duplicate.
  - **reconciliation join helper** — given a settlement row's date window, read the booth's `paid` `pos_transactions` for the corresponding sale window and classify via `reconcileDay`. Matches Xendit txns back to ours on `reference_id` (`pos-${txnId}`) / `pos_xendit_invoices.receipt_id` — the join key v0.3 captured for exactly this (foundations §7, ADR-036 consequence). Cross-module reads via `_internal` (ADR-034), never direct foreign-table queries.
- **`actions.ts`** *(`"use node"` — `authHeader()` uses Buffer)*:
  - **`sendSettlementReconResilient`** (internal action, the cron target) — resilient wrapper mirroring `telegram/foundersSummary.sendFoundersSummaryResilient`: retries transient overload via `cronRetry`; non-transient errors surface to the cron dashboard with an **audited skip** (`settlement.sync_skipped` or reuse the foundersSummary skip idiom), no retry storm. Polls a **multi-day lookback window** (default 7 days) so a missed night self-heals.
  - **`syncSettlementsNow`** (public action, manager-session) — on-demand trigger that runs the same recon body for a manager-requested/default window. Action-level idempotency pre-check (the `resetStaffPin`/refund pattern) so a double-tap doesn't double-run.
- **`public.ts`**:
  - **`listSettlements(range?)`** — any session (ADR-012). Returns per-day rows over `by_settlement_date`, newest first, bounded to a default window (e.g. last 30 days) — the table is tiny (≤1 row/day) but the query stays bounded on principle (the v0.5.3a unbounded-`.collect()` lesson).
  - **`getSettlementDetail(idOrDate)`** — any session. Returns the row + the reconciliation classification (`settled` / `pending` / `unmatched` txn lists with receipt #s + amounts).
  - **`recordSettlement({ idempotencyKey, sessionId, settlementDate, gross, mdr, net, count, bankLast4? })`** — **manager-session** (Q3). `withIdempotency` + `authCheck: requireManagerSession` (rules #20/#21). Calls `_upsertSettlement_internal(..., source:"manual", actorId)`. Validates: integer rupiah, `net = gross − mdr` consistency, a sane date label. Audited `settlement.recorded`.

### Schema — `pos_settlements` (`convex/settlements/schema.ts`)

Table is **already documented** in `docs/SCHEMA.md:507` (xendit_settlement_id, settlement_date, gross_amount, mdr_amount, net_amount, transaction_count, bca_account_destination, payload, synced_to_frollie_pro_at?, created_at) with indexes `by_settlement_date` + `by_xendit_id`. Implement that, **plus**:

- **`source: v.union(v.literal("xendit_sync"), v.literal("manual"))`** — discriminates auto vs manual rows; powers the view trust badge and reconciliation confidence. Has a real consumer (avoids the "schema field without a consumer is a dead write" anti-pattern, v0.5.2 lesson).
- **`updated_at: v.number()`** + **`created_by: v.optional(v.id("staff"))`** — `created_by` set for manual rows (audit parity), absent for cron rows; `updated_at` re-stamped on every upsert.
- **`bca_account_destination` is `v.optional(v.string())`** — the payout bank last-4 is **not** in the transaction object. Resolution (Q-open → plan): sourced from a `pos_settings` field a manager sets once, or entered on the manual row, or left `undefined` for cron rows until configured. Optional keeps cron rows valid without it.
- **`xendit_settlement_id`** — synthesised as the `settlement_date` label (or `xnd-settle-${date}`) when Xendit exposes no batch id; the field stays a stable dedup handle for `by_xendit_id`. Document the synthesis in SCHEMA.md.
- All `_at` fields server-stamped (rule #15); money as integer rupiah (rule #14).

> **No other schema changes.** No edits to `pos_transactions`/`pos_xendit_invoices` — the `reference_id`/`receipt_id` join keys already exist (v0.3).

### Crons — `convex/crons.ts`

- **`settlement-recon`**, daily at **~09:00 WIB (02:00 UTC)** — after overnight T+1 posting, clear of the 22:00 founders-summary window and the 03:00 WIB purges. Target: `internal.settlements.actions.sendSettlementReconResilient` with `{ attempt: 0 }`. Lookback window default 7 days. *(Exact hour confirmable at plan time; the resilient wrapper + upsert-by-date make timing non-critical.)*

## Frontend (`src/`)

Replace the `src/routes/settlements.tsx` **Stub** with the real read-only view. Per ADR-012 this lives at `/settlements` for **any** authenticated session (not a `mgr/*` route) — the existing route + `SpokeLayout` are already wired (`src/router.tsx`).

- **`routes/settlements.tsx`** — reverse-chronological per-day list driven by `listSettlements`: each row shows `settlement_date`, gross, **MDR**, **net** (what hit BCA), `transaction_count`, bank last-4, and a **`source` badge** (Auto / Manual).   - **Sync now** — `syncSettlementsNow`; toast with rows-updated count.
- Staff see the list + detail, no write actions.

## Data flow

```
  nightly cron (09:00 WIB)
    └▶ sendSettlementReconResilient (action) ─▶ xendit.fetchSettledTransactions([t-7d, t])
            └─ aggregateBySettlementDate ─▶ _upsertSettlement_internal(date, agg, "xendit_sync")  [+audit settlement.synced]

  manager "Record settlement"
    UI ──recordSettlement(idempotencyKey, sessionId, …)──▶ withIdempotency{authCheck: requireManagerSession}
            └─ _upsertSettlement_internal(date, fields, "manual", actorId)  [+audit settlement.recorded]

  /settlements view  ──listSettlements(range?)──▶  per-day rows (any session)
  row tap            ──getSettlementDetail(id)──▶  reconcileDay: settled / pending / unmatched
                                                      (Xendit refs ↔ pos_transactions via reference_id)
```

Live-updating via Convex subscriptions: a cron upsert or manual entry reactively updates the list. At ≤1 row/day the recompute cost is negligible.

## Access & correctness

- **Visibility (ADR-012):** `listSettlements` / `getSettlementDetail` resolve the session (`_resolveSession_internal`, ADR-034) but do **not** role-gate reads — staff and managers both see settlements. Writes (`recordSettlement`, `syncSettlementsNow`) require a **manager** session.
- **Gate tier (Q3):** manual entry is **manager-session**, mechanical `authCheck: requireManagerSession` (rule #21 — not conditional on payload). No PIN. Audited.
- **Server time wins (rule #15):** `settlement_date` is a WIB **date label** derived from Xendit's `settlement_date` (cron) or the manager's input (manual); all `_at` stamps are `Date.now()` inside the function. No client timestamp is authoritative.
- **Integer rupiah (rule #14):** gross/MDR/net are integer rupiah; `recordSettlement` validates `net = gross − mdr`.
- **Idempotency:** the cron writer is upsert-by-date (re-run safe); `recordSettlement` carries `idempotencyKey` + `withIdempotency`; `syncSettlementsNow` has an action-level pre-check.
- **No money moves.** This entire phase is a **recording/reconciliation ledger** — it never calls a Xendit payout/refund/disbursement endpoint. `fetchSettledTransactions` is read-only (`GET`).
- **Append/upsert-only:** settlement rows are never deleted; corrections are a re-poll (cron) or a fresh manual row. (Mirrors the audit-log append-only discipline in spirit.)

## Testing

Per "How to add a feature" #7, settlements is a **money/reconciliation surface**, so the gated paths + the aggregation/reconciliation math warrant tests; the pure `lib.ts` units are cheap and high-value. **No live Xendit dependency in CI** (KYB-blocked + good hygiene).

- **Unit (`lib.ts`):** `aggregateBySettlementDate` over fixture txn arrays — gross/MDR/net/count, multi-date grouping, empty input, integer-rupiah exactness. `reconcileDay` — settled/pending/unmatched classification incl. an unmatched Xendit ref and a POS txn Xendit never returned.
- **Unit (adapter, pure):** `buildTransactionsQuery(window)` emits the expected `settlement_status=SETTLED` + date filters (the `buildQrisHeaders`-style assertion); `fetchSettledTransactions` normalize step maps a **fixture** `GET /transactions` JSON payload (sampled from docs) to the normalized shape — pins the asserted field mapping so a shape correction is a one-line fixture+normalize change.
- **Integration (writer):** `_upsertSettlement_internal` inserts on first date, **patches (not duplicates)** on re-run for the same date; `source` set correctly per path.
- **Integration (gate):** `recordSettlement` succeeds with a manager session, **rejects a staff session** and an absent/invalid session (authCheck); `syncSettlementsNow` likewise manager-only. `listSettlements` succeeds for **both** staff and manager sessions (ADR-012).
- **Integration (idempotency):** `recordSettlement` replay with the same `idempotencyKey` is a no-op; cron recon over an overlapping window doesn't duplicate rows.
- **Behavioural:** with a manual entry, `/settlements` shows the row with a Manual badge; detail shows the reconciliation breakdown; (once on live keys) the cron auto-creates a row and reconciliation matches the day's paid txns.

## Success criteria & rollback

**Success criteria (minimum bar):**
- `npm run typecheck` + `npm run build` + `npm run lint` clean.
- New unit tests green: `lib.ts` aggregators + `reconcileDay`, adapter query-builder + normalize.
- New integration tests green: upsert-by-date (no dup), gate tiers (manual + sync manager-only; list staff+manager), idempotency.
- Behavioural: a manager records a settlement and it renders read-only for staff + manager; the reconciliation detail classifies a day's txns; the cron is registered and a manual `syncSettlementsNow` runs without error against TEST keys (returns zero/empty live, by design).
- ADR-043 written + foundations §7 / ADR-012 annotated as mechanism-superseded; SCHEMA.md `pos_settlements` updated (`source`, `updated_at`, `created_by`, synthesised-id note); CHANGELOG updated.

**Rollback:** additive new table + new module + new cron → no destructive migration. Rollback = revert the slice's commits + remove the cron registration (a dangling `pos_settlements` table with no readers is inert). Deployment order: backend (`npx convex deploy`) before frontend (Vercel). The cron must be deployed with `XENDIT_SECRET_KEY` present on the target deployment (already set for payments) — no new env var.
