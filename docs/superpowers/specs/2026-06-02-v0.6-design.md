# v0.6 ‚Äî Vouchers + spoilage + nightly stock-recon + Playwright E2E (design spec)

**Date:** 2026-06-02
**Status:** Approved (brainstorm complete; ready for `/superpowers:writing-plans`)
**Phase:** v0.6 ‚Äî single consolidated slice (no a/b/c/d split)
**Approach:** Four work-streams bundled into one phase because (a) the voucher *engine* is already shipped, leaving a thin CRUD+UX delta, (b) spoilage piggybacks on existing infrastructure (`pos_stock_movements` enum slot + the v0.4 `APPROVAL_KINDS` registry), (c) nightly stock-recon has a ready template in `telegram/foundersSummary.sendFoundersSummaryResilient`, and (d) Playwright E2E lands **last** so it covers the surfaces this phase ships in the same PR ‚Äî instead of chasing them in a follow-up.
**Predecessor:** v0.5.3c settlements (different ledger; see Naming guard). Cuts from `origin/main` once v0.5.3c lands; no code dependency.

---

## What the user gets when v0.6 ships

**"You'll be able to"** (carried from PROGRESS line 1145, scoped to full v0.6):
- (Manager) Create a percentage or fixed-amount voucher from `/mgr/vouchers` ‚Äî name the code, set type+value at birth, optionally cap by `expires_at` / `min_cart_value` / `max_redemptions`. Toggle active, extend expiry, archive. See `used_count` and the redemption history per code.
- (Staff) Apply a voucher offline against the cached list and have the sale commit cleanly, **or** see a clear "voucher expired during offline" banner that bumps back to the picker (ADR-009).
- (Manager) Log spoilage from `/mgr/spoilage` ‚Äî pick SKU(s), enter qty + reason, enter manager PIN at the booth, OR request off-booth via Telegram (single-use URL button ‚Üí PIN). Real margin numbers stop drifting from reported margins.
- (Operationally) Trust that nightly stock reconciliation rebuilds `pos_stock_levels.on_hand` from the immutable `pos_stock_movements` ledger and posts a drift alert to the inventory Telegram chat if cache ‚âÝ ledger ‚Äî catches any silent corruption (theoretical: a partial mutation, a manual dashboard edit, a missed audit).
- (DevX/CI) Trust that a Playwright E2E suite has proven the **golden path on real Chrome**: sign-in ‚Üí sale ‚Üí QRIS payment (Xendit test-mode simulate) ‚Üí refund-with-Telegram-approval ‚Üí voucher apply (online + offline) ‚Üí spoilage entry. The suite runs in CI on every push.

**"Still not yet"** (this phase enforces ‚Äî not on the build list):
- No voucher **stacking** ([ADR-010](../ADR/010-no-voucher-stacking.md)) or line-level vouchers ([ADR-024](../ADR/024-discount-ordering-line-voucher-tax.md)).
- No voucher-value mutation post-create (Q1 below) ‚Äî archive + recreate to change.
- No voucher analytics dashboard; no bulk-import; no auto-generated codes.
- No **adjustment** mutation (CLAUDE.md rule #9 lists adjustments as PIN-gated ‚Äî implementing them isn't in v0.6's redirect). Same for **stock_in** ‚Äî both stay deferred. Spoilage is the only new `pos_stock_movements` source this phase plumbs.
- No automated **drift correction** ‚Äî the recon cron *reports* drift, it does not silently rewrite `pos_stock_levels` to match the ledger. A manager re-runs the cron after intervening on the audit signal. (Decision Q11.)
- No multi-stall, no settlement reconciliation (v0.5.3c ‚Äî different ledger).
- No Playwright suite covering manager admin (`mgr/staff`, `mgr/products`, `mgr/receipt`) ‚Äî those existing surfaces stay vitest-only. Playwright covers the **transactional** golden path + the new v0.6 surfaces.

## Locked brainstorm answers (the source of truth for design decisions)

### Vouchers

| Q | Decision | Rationale |
| -- | -- | -- |
| **Q1 ‚Äî value mutability** | **Birth-immutable.** No value mutation post-create. Archive + create to change. | Mutating mid-life breaks the snapshot invariant and confuses cashiers. Costs one extra row; matches operational reality. |
| **Q2 ‚Äî manager tier** | **`createVoucher` = manager-PIN** (birth of money instrument, mirrors `createProduct`/`updateProductPricing`). **`updateVoucherMeta` / `archiveVoucher` / `listAllVouchers` / `getVoucherRedemptions` = manager-session** (config tuning, mirrors `updateProductMeta`/`archiveProduct`). | CLAUDE.md rule #22 / v0.5.3b precedent. PIN-gates the *birth* of the discount instrument; tunes it without PIN. |
| **Q3 ‚Äî offline-validate fallback** | **`sale/voucher.tsx` falls back to the cached `vouchers` snapshot** when `validateVoucher` is `undefined`. Same reason-code logic via a shared `lib/voucher-validate.ts`. | The set is already in IDB. One selector + one helper. |
| **Q4 ‚Äî offline-apply reject UX** | **`commitCart` returns `voucher_rejected?: { code, reason }`** additively. Charge screen renders the ADR-009 banner. No throw ‚Äî sale never blocks. | Today the drop is silent; the customer sees an unexplained total. ADR-009 explicitly calls for this banner. |
| **Q5 ‚Äî `validateVoucher` `used_count` check** | **Unchanged** ‚Äî race resolves at redemption only; "valid at validate, rejected at commit" is exactly the rejection signal Q4 surfaces. | Race is irreducibly atomic at redemption. |

### Spoilage

| Q | Decision | Rationale |
| -- | -- | -- |
| **Q6 ‚Äî manager tier** | **Booth path = manager-PIN inline** (CLAUDE.md rule #9 lists "spoilage" with stock adjustments ‚Äî PIN-gated). **Off-booth path = Telegram approval** via new `APPROVAL_KINDS` kind `spoilage`. Both write the same `pos_stock_movements{source:"spoilage", qty:negative}` row through a single internal funnel. | Mirrors the refund-approval architecture (booth-inline + Telegram-off-booth converging on one writer per CLAUDE.md rule #18 spirit). |
| **Q7 ‚Äî schema additions** | **None to `pos_stock_movements`** (`spoilage` literal already exists). **One additive field on `pos_stock_movements`:** `spoilage_reason?: v.string()` (free-form, ‚â§200 chars; manager-supplied; nullable on non-spoilage rows). No new `pos_spoilage` table ‚Äî spoilage is a movement, not a domain. | Source of truth stays the immutable ledger. ADR-007 append-only honored. |
| **Q8 ‚Äî multi-SKU per event** | **One movement row per `(inventory_sku_id, qty)` pair.** Manager UI groups them as a single "spoilage event" via `spoilage_event_id?: v.string()` (UUID minted client-side, ‚â§1 per movement). Audit emits one parent row + one per movement. | Lets a manager log "2 chocolate, 3 mango spoiled together" as one event; reporting can group by event_id. No `pos_spoilage_events` table. |
| **Q9 ‚Äî booth-inline retains rule #9 PIN** | The inline mutation funnels through **`verifyManagerPinOrThrow`** (v0.5.3b funnel). Off-booth path skips it because the `/approve/:token` flow handles PIN verification (ADR-029: token authorizes VIEW, PIN authorizes ACT). | Single PIN funnel; off-booth path matches refund precedent. |

### Nightly stock-reconciliation

| Q | Decision | Rationale |
| -- | -- | -- |
| **Q10 ‚Äî cron mechanics** | **`inventory/actions.ts:sendStockReconResilient`** mirroring `telegram/foundersSummary.sendFoundersSummaryResilient` exactly: `cronRetry` 3-attempt linear back-off, transient-vs-non-transient discrimination, audited skip. Cron `stock-recon` at **02:00 WIB (19:00 UTC)** ‚Äî after midnight day-roll, before the 03:00 receipt-cache purge window. | House-style; well-understood operationally. |
| **Q11 ‚Äî drift action** | **Report-only.** Drift = `cached.on_hand !== reconstructed.on_hand`. On drift: audit `stock.recon_drift` with `{sku_id, sku_code, cached, reconstructed, delta}`, append to a per-SKU `pos_stock_drift_log` (small append-only table), and Telegram-alert the **`inventory`** role with a compact summary ("3 SKUs drifted overnight ‚Äî open `/mgr/stock`"). **Do NOT silently auto-correct `pos_stock_levels`.** A manager investigates and re-runs the cron after fixing root cause. | Auto-correction masks the corruption source. Report-only matches ADR-018's "flag, don't block" philosophy applied to inventory hygiene. |
| **Q12 ‚Äî reconstruction definition** | **`reconstructed.on_hand = Œ£ pos_stock_movements.qty WHERE inventory_sku_id = X`** (the ledger is the truth; cache is derived). For each *active* SKU. | Movements are signed; the sum IS the true on_hand by ledger definition. |
| **Q13 ‚Äî efficiency** | Stream movements via the `by_sku_created` index (one query per SKU). Active SKU count ‚â§ 20 today (catalog seed has 5; v0.6+ adds gradually) ‚Äî full scan-per-SKU is cheap. **Defer pagination/cursors to v1.0+** when SKU count or movement volume warrants. | YAGNI; the scale doesn't justify complexity. |
| **Q14 ‚Äî drift log retention** | `pos_stock_drift_log` rows kept indefinitely (small ‚Äî typically 0 rows/night; auditable). | Append-only audit-of-audit. ADR-007 spirit. |
| **Q15 ‚Äî new ADR** | **Write ADR-043: nightly stock reconciliation = report-only, ledger-is-truth.** Records Q11 + Q12 decisions and the "we do NOT silently correct cache" stance so a future contributor doesn't re-litigate it. | Mirrors ADR-018's flag-don't-block stance applied to recon. (Note: v0.5.3c's spec also references ADR-043 for settlement ‚Äî confirm the numbering at plan time; if 043 is taken, this becomes 044.) |

### Playwright E2E

| Q | Decision | Rationale |
| -- | -- | -- |
| **Q16 ‚Äî scope** | **Golden transactional path + new v0.6 surfaces.** Specifically: (1) sign-in (PIN), (2) sale full cart, (3) QRIS payment via Xendit test-mode simulate, (4) BCA VA payment via simulate, (5) refund (booth-inline path), (6) voucher apply online + offline (with reject banner), (7) spoilage entry (booth-inline PIN). Manager-admin happy paths are already integration-tested cheaply via vitest + RTL. |
| **Q17 ‚Äî runtime target** | **Chromium via Playwright** (Chrome is the Android target per `frollie-pos design files`). Mobile viewport (`isMobile: true`, e.g. Pixel 7). No iOS Safari, no Firefox in v1. | Single-device POS per CLAUDE.md scope; testing the deployment surface. |
| **Q18 ‚Äî backend under test** | **Dev Convex deployment (`helpful-grasshopper-46`) + real Vite dev server.** Tests seed via `npx convex run seed/actions:reset` (existing ‚Äî pre-registers `dev-booth-device`). The dev deployment is exercised this way already. |
| **Q19 ‚Äî CI** | **GitHub Actions matrix: Chromium + 2 retries.** Add `.github/workflows/e2e.yml` (or extend existing ‚Äî check at plan time). Runs on every push to `main` and on PRs. Times out at 15 min total. | Standard Playwright CI shape; visibility into flakes via retries. |
| **Q20 ‚Äî fixtures** | **Reset Convex dev between specs** via `npx convex run seed/actions:reset`. Each spec gets a fresh deployment. No test-pollution. | Slowest part of the suite, but the only way to keep tests independent against a real backend. ~5s/spec acceptable for ‚â§10 specs. |
| **Q21 ‚Äî secrets** | `XENDIT_SECRET_KEY` + `XENDIT_CALLBACK_TOKEN` + `XENDIT_WEBHOOK_SIMULATE_TOKEN` (if needed for programmatic simulate) come from GitHub Actions repo secrets. Local dev reads from `.env.local`. | Standard CI secret handling. |

## Data flow

```
  MANAGER creates voucher
    UI ‚Üí createVoucher(idempotencyKey, sessionId, code, type, value, ..., managerPin)
      ‚Üí verifyManagerPinOrThrow ‚Üí _createVoucher_internal     [+audit voucher.created]
      ‚Üí pos_vouchers visible to next catalog query; offline cache refreshed on next online window

  STAFF applies voucher offline
    UI ‚Üê useCatalogCache().snapshot.vouchers ‚Üí validateLocally(code, subtotal)
    ‚Üí useCart.setVoucher(code) ‚Üí cart-charge calls commitCart(..., voucherCode)
        commitCart server re-check:
          ‚úì snapshot voucher onto pos_transactions
          ‚úó return voucher_rejected:{code, reason}; sale still commits at full price
        FE charge ‚Üí ADR-009 banner ‚Üí "Pick a different voucher"

  PAID confirmation (webhook|manual)
    _confirmPaid_internal ‚Üí _redeemVoucher_internal
        winner: insert pos_voucher_redemptions + used_count++ + audit voucher.redeemed
        loser:  VOUCHER_OVER_REDEEMED flag + audit voucher.over_redeemed

  MANAGER records spoilage (booth)
    UI ‚Üí recordSpoilage(idempotencyKey, sessionId, lines, reason, managerPin)
      ‚Üí verifyManagerPinOrThrow ‚Üí _recordSpoilage_internal(source:"booth_inline")
          for each line: insert pos_stock_movements{source:"spoilage", qty:-N, event_id, reason}
                          patch pos_stock_levels.on_hand -= N
          audit stock.spoilage_recorded + stock.spoilage_line per line

  MANAGER requests spoilage (off-booth)
    UI ‚Üí requestSpoilageApproval(...)
      ‚Üí _createRequest_internal({kind:"spoilage", context})
      ‚Üí sendTemplate({kind:"spoilage", role:"managers"}) ‚Üí Telegram URL button
    Manager opens /approve/:token ‚Üí /approve/pin ‚Üí approveSpoilage(token, managerPin)
      ‚Üí PIN verify ‚Üí _recordSpoilage_internal(source:"telegram_approval") [SAME WRITER]
      ‚Üí _markResolved_internal under top-level idempotency key

  NIGHTLY 02:00 WIB
    sendStockReconResilient (action)
      ‚Üí _runStockRecon_internal:
          for each active SKU: reconstructed = Œ£ movements
          compare to pos_stock_levels.on_hand
          drift ‚Üí insert pos_stock_drift_log + audit stock.recon_drift
      ‚Üí if drifted.length > 0:
          sendTemplate({kind:"stock_drift_alert", role:"inventory"})

          audit skip (no telegram)
      ‚Üí return {scanned, drifted}

  MANAGER resolves drift
    UI (drift log row) ‚Üí resolveDrift(driftId, note)
      ‚Üí patch row + audit stock.recon_drift_resolved

  CI on every push
    GH Actions ‚Üí Convex preview deploy ‚Üí vite preview ‚Üí playwright test (Chromium mobile)
      ‚Ü≥ runs 7 specs against real dev backend, ~5 min, artifacts on failure
```

The cardinal property across all subsystems: **the ledger is the truth, the cache is derived, the user is never blocked, and the system reports its own corruption rather than silently correcting it.**

## Access & correctness

- **Tiers (rule #22):**
  - **Manager-PIN:** `createVoucher`, `recordSpoilage` (booth path), `approveSpoilage` (off-booth PIN ‚Äî via `/approve` flow).
  - **Manager-session:** `updateVoucherMeta`, `archiveVoucher`, `listAllVouchers`, `getVoucherRedemptions`, `resolveDrift`, `listStockDrift`.
  - **Any session:** `getActiveVouchers`, `validateVoucher` (cart-side), the existing read surfaces.
- **PIN funnel:** all PIN-gated paths go through `verifyManagerPinOrThrow` (`convex/auth/verifyPin.ts`) ‚Äî never directly call `auth.verifyPin` from a new module.
- **Idempotency (rule #20):** every public mutation accepts `idempotencyKey`, wraps with `withIdempotency` + `authCheck`. PIN-gated actions use the action-level cache pattern (`auth/actions:resetStaffPin`) ‚Äî `ctx.runQuery(internal.idempotency.internal._lookup_internal, ...)`.
- **Server time wins (rule #15):** all `_at` set inside the function via `Date.now()`.
- **Integer rupiah (rule #14):** voucher value (when type=amount), `min_cart_value`, all spoilage line qty ‚Äî integer; existing `computeVoucherDiscount` floors per ADR-024.
- **Snapshot invariant (rule #1):** redemption rows snapshot `code_snapshot` + `discount_amount`; drift log snapshots `sku_code`; spoilage movements snapshot `spoilage_reason`.
- **Append-only audit (rule #2):** every state-changing path emits audit. Drift log is itself append-only audit-of-audit.
- **Cross-module reads (ADR-034):** vouchers‚Üîtransactions, inventory‚Üîcatalog (active SKU list) ‚Äî always via `_internal` queries on the owning module.
- **No money moves.** Vouchers config + redemption (a row insert), spoilage = a stock ledger write, recon = read-only diff + log. No payment/refund/disbursement endpoints called.

| Term | Subsystem | Direction | Source |

They share none of the same helpers, none of the same `source` enums, none of the same Telegram templates. Keep module boundaries strict.

## Testing

Per CLAUDE.md "How to add a feature" #7: **vouchers + spoilage + recon are money/inventory surfaces ‚Äî tests required**; Playwright IS the cross-cutting test phase.

### Voucher tests (vitest)

- **Unit (`lib/voucher-validate.ts`):** each reason path, happy path, edge: voucher expiring exactly at `Date.now()` (strict ‚â§ = expired), value:0 percentage (refuse at create per Q-spec-leans).
- **Unit (createVoucher validators):** code format, uniqueness, value bounds per type, `max_redemptions >= 1`, `expires_at` in future, `min_cart_value >= 0`.
- **Integration (CRUD):** create happy path + audit + idempotency replay; non-manager rejected; wrong PIN counts toward lockout; updateVoucherMeta refuses `max_redemptions < used_count`; archive double-no-op; archive preserves redemptions; list/redemptions reject non-manager + bounded by `limit`.
- **Integration (commit reject signal):** 3 reason paths return `voucher_rejected`; happy path absent; existing redemption race (regress) ‚Äî confirm test exists or add.

### Spoilage tests (vitest)

- **Unit (validateContext spoilage):** per-line shape; `total_qty === Œ£ lines.qty` cross-check (the v0.5.1b lesson ‚Äî assert this rejects a forged total).
- **Integration (booth-inline):** `recordSpoilage` happy path with 2 lines ‚Üí 2 movements inserted with correct `event_id` linkage + `spoilage_reason` + cached on_hand decremented; non-manager rejected; wrong PIN rejected + lockout count.
- **Integration (off-booth):** `requestSpoilageApproval` creates request + Telegram send (mocked); `approveSpoilage` happy path commits same writer with `source:"telegram_approval"` + audit + idempotency replay; deny path.
- **Integration (idempotency replay):** `recordSpoilage` replay = cached success, no duplicate movements.

### Nightly recon tests (vitest)

- **Unit (`inventory/lib.ts`):** `reconstructOnHand` over fixture arrays (positive/negative/empty); `computeDrift` (drift / no-drift / boundary integer cases).
- **Integration (`_runStockRecon_internal`):** seed N movements summing to X; set `pos_stock_levels.on_hand` to X ‚Üí no drift, no log row; set to X+1 ‚Üí drift row + audit + return shape correct.
- **Integration (`sendStockReconResilient`):** zero-drift path = audit skip + no Telegram; drift path = Telegram template send invoked with correct payload (mock); transient send error ‚Üí retried; non-transient ‚Üí audit skip + rethrow.
- **Integration (drift resolution):** `resolveDrift` patches row + audits; non-manager rejected.

Coverage per the spec list above. **Each spec is full end-to-end** against the real dev Convex + a real browser. Specs are independent (`resetDevConvex()` in `beforeEach`).
