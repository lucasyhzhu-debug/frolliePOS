# v0.6 — Vouchers + spoilage + nightly stock-recon + Playwright E2E (design spec)

**Date:** 2026-06-02
**Status:** Approved post-staffreview (brainstorm + staffreview complete; ready for `/superpowers:writing-plans`). Staffreview report: `docs/reviews/staffreview-v0.6-spec-2026-06-02.md` — 2 Critical + 4 Improvement findings all addressed inline.
**Phase:** v0.6 — single consolidated slice (no a/b/c/d split)
**Approach:** Four work-streams bundled into one phase because (a) the voucher *engine* is already shipped, leaving a thin CRUD+UX delta, (b) spoilage piggybacks on existing infrastructure (`pos_stock_movements` enum slot + the v0.4 `APPROVAL_KINDS` registry), (c) nightly stock-recon has a ready template in `telegram/foundersSummary.sendFoundersSummaryResilient`, and (d) Playwright E2E lands **last** so it covers the surfaces this phase ships in the same PR — instead of chasing them in a follow-up.
**Predecessor:** v0.5.3c settlements (different ledger; see Naming guard). Cuts from `origin/main` once v0.5.3c lands; no code dependency.

## What the user gets when v0.6 ships

**"You'll be able to"** (carried from PROGRESS line 1145, scoped to full v0.6):
- (Manager) Create a percentage or fixed-amount voucher from `/mgr/vouchers` — name the code, set type+value at birth, optionally cap by `expires_at` / `min_cart_value` / `max_redemptions`. Toggle active, extend expiry, archive. See `used_count` and the redemption history per code.
- (Staff) Apply a voucher offline against the cached list and have the sale commit cleanly, **or** see a clear "voucher expired during offline" banner that bumps back to the picker (ADR-009).
- (Manager) Log spoilage from `/mgr/spoilage` — pick SKU(s), enter qty + reason, enter manager PIN at the booth, OR request off-booth via Telegram (single-use URL button → PIN). Real margin numbers stop drifting from reported margins.
- (Operationally) Trust that nightly stock reconciliation rebuilds `pos_stock_levels.on_hand` from the immutable `pos_stock_movements` ledger and posts a drift alert to the inventory Telegram chat if cache ≠ ledger — catches any silent corruption (theoretical: a partial mutation, a manual dashboard edit, a missed audit).
- (DevX/CI) Trust that a Playwright E2E suite has proven the **golden path on real Chrome**: sign-in → sale → QRIS payment (Xendit test-mode simulate) → refund-with-Telegram-approval → voucher apply (online + offline) → spoilage entry. The suite runs in CI on every push.

**"Still not yet"** (this phase enforces — not on the build list):
- No voucher **stacking** ([ADR-010](../ADR/010-no-voucher-stacking.md)) or line-level vouchers ([ADR-024](../ADR/024-discount-ordering-line-voucher-tax.md)).
- No voucher-value mutation post-create (Q1 below) — archive + recreate to change.
- No voucher analytics dashboard; no bulk-import; no auto-generated codes.
- No **adjustment** mutation (CLAUDE.md rule #9 lists adjustments as PIN-gated — implementing them isn't in v0.6's redirect). Same for **stock_in** — both stay deferred. Spoilage is the only new `pos_stock_movements` source this phase plumbs.
- No automated **drift correction** — the recon cron *reports* drift, it does not silently rewrite `pos_stock_levels` to match the ledger. A manager re-runs the cron after intervening on the audit signal. (Decision Q11.)
- No multi-stall, no settlement reconciliation (v0.5.3c — different ledger).
- No Playwright suite covering manager admin (`mgr/staff`, `mgr/products`, `mgr/receipt`) — those existing surfaces stay vitest-only. Playwright covers the **transactional** golden path + the new v0.6 surfaces.

## Locked brainstorm answers (the source of truth for design decisions)

### Vouchers

| Q | Decision | Rationale |
| -- | -- | -- |
| **Q1 — value mutability** | **Birth-immutable.** No value mutation post-create. Archive + create to change. | Mutating mid-life breaks the snapshot invariant and confuses cashiers. Costs one extra row; matches operational reality. |
| **Q2 — manager tier** | **`createVoucher` = manager-PIN** (birth of money instrument, mirrors `createProduct`/`updateProductPricing`). **`updateVoucherMeta` / `archiveVoucher` / `listAllVouchers` / `getVoucherRedemptions` = manager-session** (config tuning, mirrors `updateProductMeta`/`archiveProduct`). | CLAUDE.md rule #22 / v0.5.3b precedent. PIN-gates the *birth* of the discount instrument; tunes it without PIN. |
| **Q3 — offline-validate fallback** | **`sale/voucher.tsx` falls back to the cached `vouchers` snapshot** when `validateVoucher` is `undefined`. Same reason-code logic via a shared `lib/voucher-validate.ts`. | The set is already in IDB. One selector + one helper. |
| **Q4 — offline-apply reject UX** | **`commitCart` returns `voucher_rejected?: { code, reason }`** additively. Charge screen renders the ADR-009 banner. No throw — sale never blocks. | Today the drop is silent; the customer sees an unexplained total. ADR-009 explicitly calls for this banner. |
| **Q5 — `validateVoucher` `used_count` check** | **Unchanged** — race resolves at redemption only; "valid at validate, rejected at commit" is exactly the rejection signal Q4 surfaces. | Race is irreducibly atomic at redemption. |

### Spoilage

| Q | Decision | Rationale |
| -- | -- | -- |
| **Q6 — manager tier** | **Booth path = manager-PIN inline** (CLAUDE.md rule #9 lists "spoilage" with stock adjustments — PIN-gated). **Off-booth path = Telegram approval** via new `APPROVAL_KINDS` kind `spoilage`. Both write the same `pos_stock_movements{source:"spoilage", qty:negative}` row through a single internal funnel. | Mirrors the refund-approval architecture (booth-inline + Telegram-off-booth converging on one writer per CLAUDE.md rule #18 spirit). |
| **Q7 — schema additions** | **None to `pos_stock_movements`** (`spoilage` literal already exists). **One additive field on `pos_stock_movements`:** `spoilage_reason?: v.string()` (free-form, ≤200 chars; manager-supplied; nullable on non-spoilage rows). No new `pos_spoilage` table — spoilage is a movement, not a domain. | Source of truth stays the immutable ledger. ADR-007 append-only honored. |
| **Q8 — multi-SKU per event** | **One movement row per `(inventory_sku_id, qty)` pair.** Manager UI groups them as a single "spoilage event" via `spoilage_event_id?: v.string()` (UUID minted client-side, ≤1 per movement). Audit emits one parent row + one per movement. | Lets a manager log "2 chocolate, 3 mango spoiled together" as one event; reporting can group by event_id. No `pos_spoilage_events` table. |
| **Q9 — booth-inline retains rule #9 PIN** | The inline mutation funnels through **`verifyManagerPinOrThrow`** (v0.5.3b funnel). Off-booth path skips it because the `/approve/:token` flow handles PIN verification (ADR-029: token authorizes VIEW, PIN authorizes ACT). | Single PIN funnel; off-booth path matches refund precedent. |

### Nightly stock-reconciliation

| Q | Decision | Rationale |
| -- | -- | -- |
| **Q10 — cron mechanics** | **`inventory/actions.ts:sendStockReconResilient`** mirroring `telegram/foundersSummary.sendFoundersSummaryResilient` exactly: `cronRetry` 3-attempt linear back-off, transient-vs-non-transient discrimination, audited skip. Cron `stock-recon` at **02:00 WIB (19:00 UTC)** — after midnight day-roll, before the 03:00 receipt-cache purge window. | House-style; well-understood operationally. |
| **Q11 — drift action** | **Report-only.** Drift = `cached.on_hand !== reconstructed.on_hand`. On drift: audit `stock.recon_drift` with `{sku_id, sku_code, cached, reconstructed, delta}`, append to a per-SKU `pos_stock_drift_log` (small append-only table), and Telegram-alert the **`inventory`** role with a compact summary ("3 SKUs drifted overnight — open `/mgr/stock`"). **Do NOT silently auto-correct `pos_stock_levels`.** A manager investigates and re-runs the cron after fixing root cause. | Auto-correction masks the corruption source. Report-only matches ADR-018's "flag, don't block" philosophy applied to inventory hygiene. |
| **Q12 — reconstruction definition** | **`reconstructed.on_hand = Σ pos_stock_movements.qty WHERE inventory_sku_id = X`** (the ledger is the truth; cache is derived). For each *active* SKU. | Movements are signed; the sum IS the true on_hand by ledger definition. |
| **Q13 — efficiency** | Stream movements via the `by_sku_created` index (one query per SKU). Active SKU count ≤ 20 today (catalog seed has 5; v0.6+ adds gradually) — full scan-per-SKU is cheap. **Defer pagination/cursors to v1.0+** when SKU count or movement volume warrants. | YAGNI; the scale doesn't justify complexity. |
| **Q14 — drift log retention** | `pos_stock_drift_log` rows kept indefinitely (small — typically 0 rows/night; auditable). | Append-only audit-of-audit. ADR-007 spirit. |
| **Q15 — new ADR** | **Write ADR-044: nightly stock reconciliation = report-only, ledger-is-truth.** Records Q11 + Q12 decisions and the "we do NOT silently correct cache" stance so a future contributor doesn't re-litigate it. | Mirrors ADR-018's flag-don't-block stance applied to recon. **Numbering locked at staffreview:** v0.5.3c gets ADR-043 (lands first; settlements is plan-stage on `main`); v0.6 gets ADR-044. |

## Backend (`convex/`) — by subsystem

### Vouchers

**`convex/vouchers/actions.ts`** *(new, `"use node"`)*:

- **`createVoucher`** *(action, manager-PIN)* — args: `{ idempotencyKey, sessionId, code, type:"percentage"|"amount", value, min_cart_value?, max_redemptions?, expires_at?, managerPin }`. Funnels through **`verifyManagerPinOrThrow(ctx, { sessionId, managerPin, idempotencyKey })`** (real signature per `convex/auth/verifyPin.ts:67-93` — single params object including the **idempotencyKey** used to derive the `${key}:failed` attempt record); destructures the returned `{ managerId, deviceId }` for audit attribution; then `runMutation(_createVoucher_internal, ..., createdBy: managerId, deviceId)`. Validations:
  - `code` matches `/^[A-Z0-9_-]+$/`, length 3–32, uppercased.
  - Uniqueness via `_getVoucherByCode_internal` → else `"CODE_EXISTS"`.
  - `type==="percentage"` → `0 < value <= 100` integer; `type==="amount"` → `value > 0` integer rupiah (rule #14).
  - `min_cart_value`, `max_redemptions`, `expires_at` (epoch ms) — if present, positive integers; `expires_at > Date.now()`.
  - Action-level idempotency cache via `internal.idempotency.internal._lookup_internal` / `_writeCache_internal` (same as `auth/actions:resetStaffPin`). Cache key includes monotonic component (the idempotency key) — v0.5.2 lesson.
  - `_createVoucher_internal` inserts with `used_count:0, active:true, created_at:Date.now(), created_by_staff_id: <managerId>`; audits `voucher.created` with `actor_id: managerId, device_id: deviceId, source: "booth_inline"`.

**`convex/vouchers/public.ts`** *(extended)*:

- **`updateVoucherMeta`** *(mutation, manager-session)* — patches present fields only. `withIdempotency` + `authCheck:requireManagerSession` (rule #20). Refuses `max_redemptions < used_count`. Audits `voucher.edited` with `metadata.fields_changed` = actual diffs.
- **`archiveVoucher`** *(mutation, manager-session)* — `active:false`; double-archive no-op; redemptions unaffected (rule #1). Audits `voucher.deactivated`.
- **`listAllVouchers(sessionId)`** *(query, manager-session)* — all rows incl. inactive/expired, manager-only via `requireManagerSession`.
- **`getVoucherRedemptions(sessionId, voucherId, limit?)`** *(query, manager-session)* — reads `pos_voucher_redemptions` via `by_voucher`, newest first, `limit` default 50, max 500 (v0.5.3a unbounded-`.collect()` lesson). Receipt-number annotation via **a new `transactions/internal._fetchReceiptByTxnIds_internal` helper** — staffreview confirmed this does not exist today; the plan **must add it** in `convex/transactions/internal.ts` as a small internalQuery that takes `Array<Id<"pos_transactions">>` and returns `Record<Id, receipt_number|null>` (ADR-034 cross-module access).

**`convex/vouchers/internal.ts`** *(extended)*:

- **`_createVoucher_internal`** — pure write + audit; called only from `createVoucher` action.

**Change V2 — shared validation helper**:

- **`convex/lib/voucher-validate.ts`** *(new — V8-safe, importable by FE)*: pure reason-code logic used by both the Convex `validateVoucher` query handler and the FE offline-cache fallback. Imports `computeVoucherDiscount` (already in `convex/lib/voucher.ts`; **confirm V8-safe at plan time** — v0.5.1a lesson). Guarantees online/offline parity.

**Change V3 — `commitCart` rejection signal**:

- `convex/transactions/public.ts:commitCart` captures the reason on server-side voucher-reject (`NOT_FOUND`/`INACTIVE`/`EXPIRED`/`MIN_CART_VALUE`) and returns `voucher_rejected?: { code, reason }` additively. **No throw** — sale never blocks (matches the existing silent-drop rationale in the code comment at line 230-233). `VOUCHER_OVER_REDEEMED` continues to flow via `_confirmPaid_internal` (Q5).
- **Existing test `convex/transactions/__tests__/commitCart.test.ts` needs updating** — confirm at plan time.

**Audit verbs:** new `voucher.created`, `voucher.edited`, `voucher.deactivated`. Free `v.string()` (rule #4 / v0.5.3b lesson). Confirm no SCHEMA.md collision at plan time.

### Spoilage

**`convex/approvals/kinds.ts`** *(extended — adds `spoilage` to the registry; all four touchpoints per CLAUDE.md #8)*:

- **`ApprovalKind`** add `"spoilage"`.
- **`SpoilageContext`** type: `{ lines: Array<{ inventory_sku_id, sku_code, qty }>, total_qty, reason, spoilage_event_id }`. (`qty` positive — represents units spoiled; the writer negates for the ledger.)

## Data flow

```
  MANAGER creates voucher
    UI → createVoucher(idempotencyKey, sessionId, code, type, value, ..., managerPin)
      → verifyManagerPinOrThrow → _createVoucher_internal     [+audit voucher.created]
      → pos_vouchers visible to next catalog query; offline cache refreshed on next online window

  STAFF applies voucher offline
    UI ← useCatalogCache().snapshot.vouchers → validateLocally(code, subtotal)
    → useCart.setVoucher(code) → cart-charge calls commitCart(..., voucherCode)
        commitCart server re-check:
          ✓ snapshot voucher onto pos_transactions
          ✗ return voucher_rejected:{code, reason}; sale still commits at full price
        FE charge → ADR-009 banner → "Pick a different voucher"

  PAID confirmation (webhook|manual)
    _confirmPaid_internal → _redeemVoucher_internal
        winner: insert pos_voucher_redemptions + used_count++ + audit voucher.redeemed
        loser:  VOUCHER_OVER_REDEEMED flag + audit voucher.over_redeemed

  MANAGER records spoilage (booth)
    UI → recordSpoilage(idempotencyKey, sessionId, lines, reason, managerPin)
      → verifyManagerPinOrThrow → _recordSpoilage_internal(source:"booth_inline")
          for each line: insert pos_stock_movements{source:"spoilage", qty:-N, event_id, reason}
                          patch pos_stock_levels.on_hand -= N
          audit stock.spoilage (one per event; lines in metadata)

  MANAGER requests spoilage (off-booth)
    UI → requestSpoilageApproval(...)
      → _createRequest_internal({kind:"spoilage", context})
      → sendTemplate({kind:"spoilage", role:"managers"}) → Telegram URL button
    Manager opens /approve/:token → /approve/pin → approveSpoilage(token, managerPin)
      → PIN verify → _recordSpoilage_internal(source:"telegram_approval") [SAME WRITER]
      → _markResolved_internal under top-level idempotency key

  NIGHTLY 02:00 WIB
    sendStockReconResilient (action)
      → _runStockRecon_internal:
          for each active SKU: reconstructed = Σ movements
          compare to pos_stock_levels.on_hand
          drift → insert pos_stock_drift_log + audit stock.recon_drift
      → if drifted.length > 0:
          sendTemplate({kind:"stock_drift_alert", role:"inventory"})
        else:
          audit skip (no telegram)
      → return {scanned, drifted}

  MANAGER resolves drift
    UI (drift log row) → resolveDrift(driftId, note)
      → patch row + audit stock.recon_drift_resolved

  CI on every push
    GH Actions → Convex preview deploy → vite preview → playwright test (Chromium mobile)
      ↳ runs 7 specs against real dev backend, ~5 min, artifacts on failure
```

The cardinal property across all subsystems: **the ledger is the truth, the cache is derived, the user is never blocked, and the system reports its own corruption rather than silently correcting it.**

## Access & correctness

- **Tiers (rule #22):**
  - **Manager-PIN:** `createVoucher`, `recordSpoilage` (booth path), `approveSpoilage` (off-booth PIN — via `/approve` flow).
  - **Manager-session:** `updateVoucherMeta`, `archiveVoucher`, `listAllVouchers`, `getVoucherRedemptions`, `resolveDrift`, `listStockDrift`.
  - **Any session:** `getActiveVouchers`, `validateVoucher` (cart-side), the existing read surfaces.
- **PIN funnel:** all PIN-gated paths go through `verifyManagerPinOrThrow` (`convex/auth/verifyPin.ts`) — never directly call `auth.verifyPin` from a new module.
- **Idempotency (rule #20):** every public mutation accepts `idempotencyKey`, wraps with `withIdempotency` + `authCheck`. PIN-gated actions use the action-level cache pattern (`auth/actions:resetStaffPin`) — `ctx.runQuery(internal.idempotency.internal._lookup_internal, ...)`.
- **Server time wins (rule #15):** all `_at` set inside the function via `Date.now()`.
- **Integer rupiah (rule #14):** voucher value (when type=amount), `min_cart_value`, all spoilage line qty — integer; existing `computeVoucherDiscount` floors per ADR-024.
- **Snapshot invariant (rule #1):** redemption rows snapshot `code_snapshot` + `discount_amount`; drift log snapshots `sku_code`; spoilage movements snapshot `spoilage_reason`.
- **Append-only audit (rule #2):** every state-changing path emits audit. Drift log is itself append-only audit-of-audit.
- **Cross-module reads (ADR-034):** vouchers↔transactions, inventory↔catalog (active SKU list) — always via `_internal` queries on the owning module.
- **No money moves.** Vouchers config + redemption (a row insert), spoilage = a stock ledger write, recon = read-only diff + log. No payment/refund/disbursement endpoints called.
