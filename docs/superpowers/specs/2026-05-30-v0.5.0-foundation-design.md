v0.5.0 is the **foundation slice** of the v0.5 cycle. No new user features beyond a nav shell, a lock + handoff UX, and a small stack of v0.4 follow-up bug-class fixes. Every screen v0.5.1, v0.5.2, and v0.5.3 ship will sit on what lands here — nav chrome, the `effectiveStatus` helper, the strict idempotency rule.

Single PR `feat/v0.5.0-foundation`, squash-merge. ~26 atomic commits, one per Task ID, in wave order. This is a hair over the staffreview's ~20-task guideline; the overage comes from (a) the authCheck migration (one task umbrella, ~10–15 actual commits, each mechanical) and (b) the new `cancelAwaitingPayment` mutation added in response to staffreview Critical #2. ## 3. Workstream A — Nav shell

**Goal:** Every spoke route — `/sale`, `/sale/drafts`, `/sale/voucher`, `/sale/charge`, `/sale/charge-success`, `/refund/*`, `/history`, `/stock`, `/stock/in`, `/settlements`, `/lock`, `/mgr/*` — has consistent header chrome. Hub route (`/`) and unauthenticated/public routes (`/login`, `/activate`, `/approve/*`, `/r/*`) keep their bespoke layouts.

### New components

**`src/components/layout/AppHeader.tsx`** — sticky top chrome.

```tsx

  title: string;                     // screen-local, set by route
  backTo?: string;                   // default "/"; overridable for nested spokes
  onBack?: () => void | Promise<void>; // override for cart-abandon dialog
  rightSlot?: React.ReactNode;       // optional per-screen affordance

```

Layout: `[← Home] [title] [staff name · ConnDot]`. Sticky, 48px tall, divider below. Uses existing `ConnDot` and `useSession` for the staff badge. Tappable Home arrow always wins over browser-back semantics — `useNavigate("/")` not `navigate(-1)`, so a mid-flow reload never lands on a stale prior screen.

**`src/components/layout/SpokeLayout.tsx`** — thin wrapper.

```tsx

</SpokeLayout>
```

Renders `<AppHeader />` over `<main className="flex-1">{children}</main>`. Spoke routes import and wrap. No router-level magic — explicit per route so each screen owns its title and back behaviour.

### Cart-abandon dialog — two variants

The abandon flow exists on **two routes** with different state-machine implications. The dialog component is shared; the wiring + button set differs.

**`src/components/pos/AbandonCartDialog.tsx`** — built on shadcn `Dialog`. Accepts a `variant: "cart" | "payment"` prop that controls button set and copy.

#### Navigation interception — `useBlocker` is the load-bearing primitive

Header back goes through `useNavigate`; Android hardware/gesture back and browser back fire `popstate`. The new chrome must catch BOTH. React Router v7's `useBlocker` intercepts every kind of intra-app navigation (header `Link`, `navigate()` call, `popstate`) uniformly. `beforeunload` is the secondary guard for hard page-leave only (tab close, refresh, navigate to external URL).

```tsx
// inside /sale or /sale/charge
const blocker = useBlocker(({ currentLocation, nextLocation }) =>
  shouldBlock(currentLocation, nextLocation),
);

useEffect(() => {
  if (blocker.state === "blocked") setAbandonOpen(true);
}, [blocker.state]);

// dialog handlers:
const onConfirm = async () => { await save_or_discard(); blocker.proceed?.(); };
const onCancel  = () => { blocker.reset?.(); setAbandonOpen(false); };

// secondary hard-leave guard
useEffect(() => {
  if (!shouldBlock()) return;
  const handler = (e: BeforeUnloadEvent) => { e.preventDefault(); };
  window.addEventListener("beforeunload", handler);
  return () => window.removeEventListener("beforeunload", handler);
}, [shouldBlock()]);
```

#### Variant: `cart` — on `/sale` (cart-edit step)

Three buttons: `Save as draft` · `Discard` · `Cancel`. `shouldBlock` returns true when `cart.lines.length > 0`.
- **Save as draft** → calls existing `commitCart({ status: "draft" })` → clears Zustand cart → `blocker.proceed()`.
- **Discard** → `cart.clear()` → `blocker.proceed()`. No server round-trip.
- **Cancel** → `blocker.reset()`, close dialog.

#### Variant: `payment` — on `/sale/charge` (awaiting-payment step)

Two buttons: `Cancel payment` · `Keep waiting`. `shouldBlock` returns true when the txn is in `awaiting_payment` AND no successful webhook has landed.

- **Cancel payment** → calls **new mutation `transactions.public.cancelAwaitingPayment({ sessionId, txnId, idempotencyKey })`** (see §3.1 below). The mutation transitions the txn to `cancelled`, marks the active `pos_xendit_invoices` row as locally superseded (no Xendit cancel API call per [ADR-036](../../ADR/036-xendit-dedicated-apis-inline.md)), fires the `cancel-cancels-approval` side-effect against any pending `manual_payment_override` request. Then clears Zustand cart → `blocker.proceed()` → navigate to `/`.
- **Keep waiting** → `blocker.reset()`, close dialog.

**Empty-cart fast path:** on either route, if `shouldBlock()` returns false (empty cart on `/sale`; webhook-paid on `/sale/charge`), the blocker is inactive — navigation proceeds silently. The header back-button does not open the dialog in those cases.

### §3.1 — New mutation: `cancelAwaitingPayment`

**`convex/transactions/public.ts`** — new public mutation, **born under the strict ESLint rule** (must ship with `authCheck` wired from day one).

```ts
export const cancelAwaitingPayment = mutation({
  args: {
    sessionId: v.id("staff_sessions"),
    txnId: v.id("pos_transactions"),
    idempotencyKey: v.string(),
  },
  handler: withIdempotency<...>("transactions.cancelAwaitingPayment", async (ctx, args) => {
    const session = await requireSession(ctx, args.sessionId);
    const txn = await ctx.db.get(args.txnId);
    if (!txn) throw new Error("TXN_NOT_FOUND");
    if (txn.status !== "awaiting_payment") throw new Error("TXN_NOT_AWAITING");
    // belt-and-braces: prevent racing webhook from being clobbered
    // (webhook would have flipped to paid; this guard yields cleanly)

    // 1. Transition txn
    await ctx.db.patch(args.txnId, { status: "cancelled", cancelled_at: Date.now() });

    // 2. Local-supersede the active invoice (no Xendit call per ADR-036)
    const activeInvoice = await ctx.db.query("pos_xendit_invoices")
      .withIndex("by_txn", q => q.eq("source_transaction_id", args.txnId))
      .filter(q => q.eq(q.field("status"), "active"))
      .first();
    if (activeInvoice) {
      await ctx.db.patch(activeInvoice._id, { status: "cancelled", cancelled_at: Date.now() });
    }

    // 3. Fire the cancel-cancels-approval side-effect (shared with the v050-be-cancel-cancels-approval task)
    await ctx.runMutation(internal.approvals.internal._cancelPendingApprovalsForTxn_internal, {
      txnId: args.txnId,
      reason: "txn_cancelled",
    });

    // 4. Audit
    await logAudit(ctx, {
      actor_id: session.staffId,
      action: "transaction.cancelled",
      entity_type: "pos_transactions",
      entity_id: args.txnId,
      source: "booth_inline",
      reason: "user_cancelled_at_payment",
    });
  }, {
    authCheck: async (ctx, args) => { await requireSession(ctx, args.sessionId); },
  }),
});
```

This mutation is the consumer of `v050-be-cancel-cancels-approval`'s internal helper (`_cancelPendingApprovalsForTxn_internal`) — they ship together.

## 4. Workstream B — Lock + handoff

**Goal:** End-of-shift lock is one tap from home; resuming on the same device drops the next staffer into PIN entry for the previous person (one PIN tap to switch, no auth bypass). When the previous staffer is no longer active, silently fall back to the staff list.

### Last-staff persistence

**`src/lib/storage-keys.ts`** — new module centralising every localStorage key in the app (per Refinement R1). Two keys today; future hooks add here, not inline.

```ts
export const SESSION_KEY    = "frollie-session-id";
export const LAST_STAFF_KEY = "frollie-last-staff";
```

`useSession.ts` migrates its inline `STORAGE_KEY` constant to import from here (one-line refactor) so the namespace is grep-able.

**`src/hooks/useLastStaff.ts`** — new hook, mirrors the `useSession` localStorage pattern.

```ts
import { LAST_STAFF_KEY } from "@/lib/storage-keys";

export function rememberLastStaff(staffId: Id<"staff">): void {
  localStorage.setItem(LAST_STAFF_KEY, staffId);
}

export function getLastStaff(): Id<"staff"> | null {
  return (localStorage.getItem(LAST_STAFF_KEY) ?? null) as Id<"staff"> | null;
}

/**
 * Forward-looking API. Not called from any v0.5.0 site — wired in v0.5.3 manager-portal
 * "forget this device" flow (or whenever a manager explicitly deactivates a staff record
 * and we want the bound device to fall back to the staff list immediately).
 */
export function forgetLastStaff(): void {
  localStorage.removeItem(LAST_STAFF_KEY);
}
```

**Single write site — co-located in `storeSession`** (Refinement R3): extend `storeSession`'s signature to `storeSession(sessionId: string, staffId: Id<"staff">)` and have it write BOTH keys atomically:

```ts
// in useSession.ts (refactor)
export function storeSession(sessionId: string, staffId: Id<"staff">): void {
  localStorage.setItem(SESSION_KEY, sessionId);
  localStorage.setItem(LAST_STAFF_KEY, staffId);
  notify(sessionId);
}
```

Callers (`login.tsx` `onPinSubmit`) pass both arguments in one call — harder to forget than two separate calls. Lock does NOT touch either key beyond `clearSession()` (which clears SESSION_KEY only — LAST_STAFF_KEY stays sticky). The key is overwritten by the next successful login.

**Read site:** `login.tsx` initial-stage derivation (replaces the hardcoded `useState<Stage>({ kind: "list" })`). Note: the success path also changes — `storeSession(sessionId, stage.staff._id)` replaces `storeSession(sessionId)`:

```ts
const staff = useQuery(api.auth.public.getActiveStaff, {});
const [stage, setStage] = useState<Stage>({ kind: "list" });

useEffect(() => {
  if (staff === undefined) return;             // still loading
  if (stage.kind !== "list") return;           // already advanced past list
  const lastId = getLastStaff();
  if (!lastId) return;
  const match = staff.find((s) => s._id === lastId);
  if (match) setStage({ kind: "pin", staff: match });
  // else silently stay on list — deactivated/removed staff falls through
}, [staff]);
```

### Lock route

**`src/routes/lock.tsx`** — replaces the current `Stub`.

Layout (under `SpokeLayout` title "Lock + handoff"):
- Big confirm card: *"End ‹staff name›'s shift?"* with two buttons: `Lock` (primary) · `Cancel` (back to `/`)
- Below: muted hint *"The next person taps their name and PIN to sign in."*

Behaviour on `Lock`:
1. Call existing `api.auth.public.logout({ sessionId, idempotencyKey })`.
2. `clearSession()` — clears `frollie-session-id`.
3. **Keep** `frollie-last-staff` set (the whole point of the resume UX).
4. `navigate("/login", { replace: true })`.

The existing home-screen "Lock" button at the bottom of `routes/home.tsx` is **kept** as a fast path.

### Deactivation edge cases

- **Staffer deactivated server-side between lock and resume:** `getActiveStaff` won't include them → `match` undefined → silent fallback to list.
- **Staffer's role changed (staff → manager or vice versa):** still in `getActiveStaff` → still pre-stages. Role is read fresh from the query result.
- **localStorage cleared (cache wipe, new device):** `getLastStaff()` returns null → list view.
- **Multiple tabs:** `rememberLastStaff` doesn't broadcast — the key is read on login mount, not subscribed.
- **Last staffer is currently locked-out (ADR-002):** still pre-stage; PIN entry will fail with the standard `LOCKED_OUT:Ns` message and existing toast. v0.4 Telegram lockout-notification fires as normal.

### Tests

- `useLastStaff.test.ts` — round-trip set/get/forget; tolerates missing localStorage
- `login.test.tsx` extension — three cases:
  1. No last-staff key → list shown
  2. Last-staff key set + matches active list → pre-stages to PIN
  3. Last-staff key set + NOT in active list → silently shows list
- `lock.test.tsx` — logout called with idempotency key; navigate to `/login` after; `frollie-last-staff` NOT cleared

## 5. Workstream C — Per-token PIN cap

**Goal:** A stolen `/approve/:token` URL can't be brute-forced against every manager's PIN. The token's lifetime (60 min) AND its action surface (`approveStaffPinReset`, `approveManualPayment`, `denyRequest`) are capped at **5 total failed PIN attempts** across all manager codes before the request auto-denies and the token dies.

### Threat model

- **Today's defence:** per-staff lockout (ADR-002) — 3 wrong PINs against any one manager triggers a 60s lockout for that manager. But the attacker can iterate: 3 attempts against M1, 3 against M2, 3 against M3 → 9 attempts in seconds across 3 managers, every attempt against a fresh PIN with no per-token throttle.
- **Today's gap:** Nothing in `approveStaffPinReset` / `approveManualPayment` / `denyRequest` increments anything on the request row when PIN verify fails. The token is reusable until it expires (60 min) regardless of attempts.
- **What this fix closes:** Bind the attempt counter to the *token*, not the *manager*. After 5 misses across any combination of manager codes, the token is dead.

### Schema change

**`convex/approvals/schema.ts`** — add one optional field to `pos_approval_requests`:

```ts
failed_pin_attempts: v.optional(v.number()),  // count; absent = 0; auto-deny at 5
```

Additive, no migration. Existing rows read as `undefined` → treated as 0.

**Validator widen:** `denied_by_manager_id` widens from `v.optional(v.id("staff"))` to `v.optional(v.union(v.id("staff"), v.literal("system")))`. Readers branch on `=== "system"` for display copy. The `actor_id` field already permits the `"system"` literal — same convention.

**Cap constant** lives in `convex/approvals/lib.ts` (same file as `effectiveStatus`):

```ts
export const TOKEN_PIN_ATTEMPT_CAP = 5;
```

### Auto-deny lifecycle reuses existing `denied` state

The 5th miss transitions the request:
- `status` → `"denied"`
- `denied_at` → `Date.now()`
- `denied_by_manager_id` → `"system"`
- `deny_reason` → `"too_many_pin_attempts"`

### Shared helper: `_markDeniedBySystem_internal`

Three v0.5.0 callers need to write a "system-denied" row: (1) the PIN-cap trip below, (2) `v050-be-cancel-pending-approval`'s manager-initiated cancel, (3) `v050-be-cancel-cancels-approval`'s cascade from `cancelAwaitingPayment`. Extract one helper rather than inline at each site.

**`_markDeniedBySystem_internal`** in `convex/approvals/internal.ts`:

```ts
export const _markDeniedBySystem_internal = internalMutation({
  args: {
    requestId: v.id("pos_approval_requests"),
    deny_reason: v.string(),              // "too_many_pin_attempts" | "txn_cancelled" | "manager_cancelled" | custom
    cancelled_by_manager_id: v.optional(v.id("staff")),  // present for path #2; absent for paths #1/#3
  },
  handler: async (ctx, args) => {
    const req = await ctx.db.get(args.requestId);
    if (!req) throw new Error("REQUEST_NOT_FOUND");
    if (req.status !== "pending") return { denied: false };  // already terminal — no-op

    await ctx.db.patch(args.requestId, {
      status: "denied",
      denied_at: Date.now(),
      denied_by_manager_id: args.cancelled_by_manager_id ?? "system",
      deny_reason: args.deny_reason,
    });
    await logAudit(ctx, {
      actor_id: args.cancelled_by_manager_id ?? "system",
      action: KIND_AUDIT[req.kind].denied,
      entity_type: "pos_approval_requests",
      entity_id: args.requestId,
      source: args.cancelled_by_manager_id ? "booth_inline" : "system",
      reason: args.deny_reason,
      metadata: { approval_request_id: args.requestId, kind: req.kind },
    });
    return { denied: true };
  },
});
```

### `_cancelPendingApprovalsForTxn_internal` — for the txn-cancel cascade

**`convex/approvals/internal.ts`** — looks up live pending requests for a given txn and runs `_markDeniedBySystem_internal` against each. Used by `cancelAwaitingPayment` (§3.1).

```ts
export const _cancelPendingApprovalsForTxn_internal = internalMutation({
  args: { txnId: v.id("pos_transactions"), reason: v.string() },
  handler: async (ctx, args) => {
    const pending = await ctx.runQuery(internal.approvals.internal._listPendingByKind_internal, {
      kind: "manual_payment_override",
      entityId: args.txnId,
    });
    for (const req of pending) {
      await ctx.db.patch(req._id, {
        status: "denied",
        denied_at: Date.now(),
        denied_by_manager_id: "system",
        deny_reason: args.reason,
      });
      await logAudit(ctx, {
        actor_id: "system",
        action: KIND_AUDIT[req.kind].denied,
        entity_type: "pos_approval_requests",
        entity_id: req._id,
        source: "system",
        reason: args.reason,
        metadata: { approval_request_id: req._id, kind: req.kind, cascaded_from_txn: args.txnId },
      });
    }
  },
});
```

### Increment-and-maybe-cap mutation

**`_recordTokenPinFailure_internal`** in `convex/approvals/internal.ts` — uses `_markDeniedBySystem_internal` for the cap-trip case:

```ts
export const _recordTokenPinFailure_internal = internalMutation({
  args: { requestId: v.id("pos_approval_requests") },
  handler: async (ctx, args) => {
    const req = await ctx.db.get(args.requestId);
    if (!req) throw new Error("REQUEST_NOT_FOUND");
    if (req.status !== "pending") return { capped: false };  // already terminal — no-op

    const next = (req.failed_pin_attempts ?? 0) + 1;
    await ctx.db.patch(args.requestId, { failed_pin_attempts: next });

    if (next >= TOKEN_PIN_ATTEMPT_CAP) {
      // delegate to shared system-deny helper; pass forensic count via separate audit
      await ctx.db.patch(args.requestId, {
        status: "denied",
        denied_at: Date.now(),
        denied_by_manager_id: "system",
        deny_reason: "too_many_pin_attempts",
      });
      await logAudit(ctx, {
        actor_id: "system",
        action: KIND_AUDIT[req.kind].denied,
        entity_type: "pos_approval_requests",
        entity_id: args.requestId,
        source: "system",
        reason: "too_many_pin_attempts",
        metadata: { approval_request_id: args.requestId, kind: req.kind, failed_pin_attempts: next },
      });
      return { capped: true };
    }
    return { capped: false };
  },
});
```

Why the cap-trip case inlines the patch + audit rather than calling `_markDeniedBySystem_internal`: it needs to capture `failed_pin_attempts: next` in audit metadata for forensic detail, which the shared helper doesn't take. The shared helper covers the two simpler cases (manual cancel + txn cascade). Acceptable duplication: ~10 lines, one extra metadata field, clearly motivated.

### Wire into the three action sites

**`convex/approvals/actions.ts`** — three places where argon2id PIN verify currently throws on failure:

1. `approveStaffPinReset`
2. `approveManualPayment`
3. `denyRequest`

Each gets the same hook AFTER argon2 verify fails, BEFORE throwing:

```ts
const result = await ctx.runMutation(internal.approvals.internal._recordTokenPinFailure_internal, {
  requestId: req._id,
});
if (result.capped) throw new Error("REQUEST_REVOKED");
throw new Error("INVALID_PIN");
```

`REQUEST_REVOKED` is a new error code so the `/approve` UI can show the "this approval was revoked" copy immediately on the cap-trip request itself, not just on the next visit.

### Frontend wiring — TWO display paths, identical copy

The revoke message reaches the user via two code paths that must render identical copy:

**Path A — same-visit (the cap-trip request):** the action throws `REQUEST_REVOKED`. The existing `mapError` at `src/routes/approve/index.tsx:31` has branches for `REQUEST_RESOLVED` and `INVALID_PIN` — **add a `REQUEST_REVOKED` branch** returning the revoke copy.

```ts
function mapError(err: unknown): string {
  const msg = err instanceof Error ? err.message : String(err);
  if (msg.includes("REQUEST_RESOLVED")) return "Already resolved";
  if (msg.includes("REQUEST_REVOKED")) return "Approval revoked — too many wrong PIN attempts. Ask the staffer to retry.";
  if (msg.includes("INVALID_PIN")) return "Wrong manager PIN";
  // ... existing branches
}
```

**Path B — next-visit (any subsequent open of the same token):** `getByToken` returns a row with `effectiveStatus(req) === "denied"`. The `/approve` UI branches on `deny_reason === "too_many_pin_attempts"` to render the same copy (vs. a manager-initiated deny, which shows the manager's name + reason). The PIN input is hidden in this branch.

Both paths land the user in the same end-state: clear message, no PIN input, suggestion to ask the staffer to retry. Failing to wire Path A means the cap-trip user sees a generic error and only learns the truth on retry.

### `effectiveStatus` plumbing

A capped request is `status: "denied"` → `effectiveStatus` returns `"denied"` → drives Path B. No special-case in the helper.

### Per-token cap is sticky across legitimate AND attacker attempts

The 5-attempt counter does not distinguish between an attacker iterating manager codes and a legitimate manager fumbling. If an attacker burns 4 wrong attempts and walks away, a legitimate manager later trying once and fumbling once auto-revokes the approval. **This is intended.** Operators retrying a revoked approval mint a fresh token via the normal flow (the dedup guard skips `denied` rows). Document in CHANGELOG (§8) and UAT (§11) so operators aren't surprised.

### Token-invalidation strategy

Once `status === "denied"`, the existing `/approve/:token` UI path is dead via `effectiveStatus`. No separate token revocation table. The staffer retrying the original lockout / manual payment mints a fresh request with a fresh token via the normal flow — `_listPendingByKind_internal` / `_listPendingForStaff_internal` dedup guards skip only `status === "pending"` rows, so a `denied` row doesn't block a retry.

### Audit trail end-state

A revoked request leaves this fingerprint:
```
approval.created           (system)
approval.notified          (system)
manual_payment.denied      (system, reason: too_many_pin_attempts, metadata: failed_pin_attempts: 5)
```

We deliberately **don't audit each individual PIN failure** — the per-staff lockout already audits `auth.failed_attempt` for the verifying manager, so we're not losing the forensic trail. The terminal deny captures the count in metadata.

## 6. Workstream D — ESLint idempotency rule + authCheck migration

**Goal:** No public mutation can ship without (a) `idempotencyKey: v.string()` in args, (b) handler wrapped in `withIdempotency(...)`, AND (c) `authCheck` arg wired into the wrapper. Mechanically enforced by ESLint, ratcheted in by refactoring every existing public mutation in one pass.

### The rule

**`tools/eslint-rules/idempotency-required.js`** — sibling to `no-cross-module-db-access.js`.

**Scope:** files matching `convex/**/public.ts`. Other files (internal.ts, actions.ts, root `convex/*.ts`) are unconditionally exempt — actions use the action-level pattern, internals are caller-controlled.

**Three assertions per `export const X = mutation({...})`:**

1. **`args` has `idempotencyKey: v.string()`** — walk the `args` ObjectExpression, find a Property named `idempotencyKey`, assert its value is a CallExpression matching `v.string()`.
2. **`handler` is a call to `withIdempotency(...)`** — the `handler` property's value must be a CallExpression whose callee is `Identifier("withIdempotency")`. Tolerates the `withIdempotency<...>(...)` TS generic form.
3. **`withIdempotency`'s third arg is an ObjectExpression with an `authCheck` property** — must exist and its value must be a function expression (Arrow or Function). The rule does not inspect the body; that's a code-review concern.

**Error messages:**
- `missingIdempotencyKey`: "Public mutation '{{name}}' must accept `idempotencyKey: v.string()` (ADR-013)."
- `missingWithIdempotency`: "Public mutation '{{name}}' handler must be wrapped in `withIdempotency(...)` (ADR-013)."
- `missingAuthCheck`: "Public mutation '{{name}}' must wire an `authCheck` in the `withIdempotency` options object (v0.5.0 strict rule)."

**Fixture tests** in `tools/eslint-rules/__tests__/idempotency-required.test.js`:
- Positive: passing public.ts file → 0 errors
- Negative × 3: one fixture file per missing-piece → exactly 1 error each, correct message id
- Exempt: a `convex/foo/internal.ts` and a `convex/foo/actions.ts` with no idempotency → 0 errors

**Wiring into `eslint.config.js`:** add as sibling to `no-cross-module-db-access`. Phased severity — Wave-1 commit lands the rule as `warn` so CI stays green during the migration; the final commit of `v050-be-authcheck-migrate` flips it to `error`.

**CI gate for the severity flip** (required — closes the staffreview Improvement #1 fragility concern): add a one-line CI assertion (in the lint job, runs after `npm run lint`) that grep-checks `eslint.config.js` for the explicit `"error"` severity on `idempotency-required`. Two-line script in `tools/ci/assert-strict-idempotency-rule.sh`:

```bash
#!/usr/bin/env bash
grep -qE '"idempotency-required"\s*:\s*"error"' eslint.config.js \
  || { echo "ESLint rule idempotency-required must be severity 'error' before merge"; exit 1; }
```

Wired into the existing CI lint step. Without this, the warn→error flip lives only in the migration task's final commit; a regression that reverts severity to `warn` would ship invisibly.

### authCheck migration — the dual-call pattern

Every existing public mutation currently does auth inline (`const session = await requireSession(...)` as the first handler line). The strict rule expects `authCheck` to be in the wrapper options. The `withIdempotency` API does NOT thread the auth result into the handler — widening it would explode the type-parameter surface across every callsite.

**Resolution:** the `authCheck` slot is the **canonical security boundary** (runs before cache lookup). The handler RE-CALLS the same `require*Session` helper to get the typed session object it needs. The re-call is intentional and cheap (one indexed query against `staff_sessions`); the duplication is what makes the ESLint rule mechanical.

**Per-mutation diff shape** (each is one commit):

```ts
// BEFORE
export const cancelPendingRequest = mutation({
  args: { sessionId: v.id("staff_sessions"), requestId: v.id("pos_approval_requests"),
          reason: v.optional(v.string()), idempotencyKey: v.string() },
  handler: withIdempotency<...>("approvals.cancelPendingRequest", async (ctx, args) => {
    const session = await requireManagerSession(ctx, args.sessionId);  // inline auth
    // ... work
  }),
});

// AFTER
export const cancelPendingRequest = mutation({
  args: { sessionId: v.id("staff_sessions"), requestId: v.id("pos_approval_requests"),
          reason: v.optional(v.string()), idempotencyKey: v.string() },
  handler: withIdempotency<...>("approvals.cancelPendingRequest", async (ctx, args) => {
    const session = await requireManagerSession(ctx, args.sessionId);  // re-call for typed session
    // ... work
  }, {
    authCheck: async (ctx, args) => { await requireManagerSession(ctx, args.sessionId); },
  }),
});
```

`withIdempotency`'s JSDoc gets a paragraph describing the dual-call convention so a reader doesn't mistake the duplication for an oversight.

## 7. Workstream E — `effectiveStatus` helper + 9 backend stabilizers

### 7.1 `effectiveStatus` helper

**`convex/approvals/lib.ts`** — new file, pure TypeScript, zero runtime imports beyond `Doc` type.

```ts
import type { Doc } from "../_generated/dataModel";

export type EffectiveStatus = "pending" | "resolved" | "denied" | "expired";

/**
 * Single canonical derivation of an approval request's user-visible status.
 *
 * The schema only tracks three terminal states (resolved, denied) plus the
 * in-flight "pending". The fourth — "expired" — is a virtual state derived
 * from token_expires_at vs. now. This helper centralises that derivation so
 * the 5 reader sites (getByToken, getRequestStatus, getRecentPinResetForStaff,
 * /approve UI, ApprovalPending overlay) can never drift.
 *
 * The `now` parameter is for test determinism. Default is Date.now().
 */
export function effectiveStatus(
  row: Pick<Doc<"pos_approval_requests">, "status" | "token_expires_at">,
  now: number = Date.now(),
): EffectiveStatus {
  if (row.status !== "pending") return row.status;
  return row.token_expires_at <= now ? "expired" : "pending";
}

/** Hard cap on failed PIN attempts per approval token. See §5. */
export const TOKEN_PIN_ATTEMPT_CAP = 5;
```

**Reader migration** — 5 sites, each a one-line change:

| Site | File | Change |
|------|------|--------|
| 1 | `convex/approvals/public.ts` `getByToken` | inline expiry check → `effectiveStatus(req)` |
| 2 | `convex/approvals/public.ts` `getRequestStatus` | reads `req.status` today (latent expiry bug) → `effectiveStatus(req)` (closes the bug) |
| 3 | `convex/approvals/public.ts` `getRecentPinResetForStaff` | inline expiry check → `effectiveStatus(req)` |
| 4 | `src/routes/approve/index.tsx` | branches on `EffectiveStatus` from the typed query result |
| 5 | `src/components/pos/ApprovalPending.tsx` | branches on `EffectiveStatus` from the typed query result |

**`_listPendingForStaff_internal` / `_listPendingByKind_internal` stay inline.** Their purpose is dedup ("is there ANY pending+live row"), not display. Add a one-line comment referencing `effectiveStatus` so a future reader knows the centralised helper exists; don't refactor the dedup path.

**Tests** in `convex/approvals/__tests__/effectiveStatus.test.ts`:
- `pending + not expired` → `"pending"`
- `pending + token_expires_at === now` → `"expired"` (boundary; helper uses `<=`)
- `pending + token_expires_at < now` → `"expired"`
- `resolved + expired-by-time` → `"resolved"` (terminal wins)
- `denied + expired-by-time` → `"denied"` (terminal wins)
- Default `now` arg matches explicit `Date.now()` within 100ms

The one observable behaviour change: `getRequestStatus` against a `pending` row whose token has expired now returns `"expired"` instead of `"pending"` — the reader-site `ApprovalPending` test needs a one-line update to assert the new (correct) state.

## 10. Risks & mitigations

| Risk | Mitigation |
|------|-----------|
| ESLint rule lands in Wave 1 but the codebase doesn't comply until Wave 2 — CI red between commits | Wave 1's ESLint task ships the rule as `warn`. Wave 2's `v050-be-authcheck-migrate` final commit flips it to `error` after all mutations comply. **CI assertion (`tools/ci/assert-strict-idempotency-rule.sh`) hard-fails the build if severity ever drops back to `warn`** — closes the fragility flagged in staffreview Improvement #1. Each interim commit stays CI-green. |
| Reader migration to `effectiveStatus` changes `getRequestStatus` observable behaviour for expired-pending rows | Update the affected reader-site test in the same commit (one-line assertion change — new state is the correct state); the surprise is the bug, the new behaviour is the fix. |
| Stuck Telegram messages after auto-revoke (the original "Open approval →" button stays in the chat) | Out of scope for v0.5.0. Audit row + `/approve` UI revoke-copy is sufficient signal. Flag in CHANGELOG; v0.5.3+ concern. |
| `useBlocker` + `beforeunload` interception for the abandon dialog conflicts with PWA service worker controls | `useBlocker` is the primary mechanism (covers popstate, header back, `useNavigate`); `beforeunload` is secondary for hard page-leave. Test on installed Android PWA before merge as part of the mandatory manual smoke (§3 Tests). |
| `manuallyConfirmPayment` signature change with `managerStaffCode` breaks the current callsite on `/sale/charge` | Same commit (`v050-be-mgr-picker-override`) covers backend + booth-picker UI + UI test. Atomic. |
| Dual-call pattern (`authCheck` + inline `requireSession`) looks like a redundant call to future readers | `withIdempotency`'s JSDoc paragraph explains the convention. CLAUDE.md business rule #21 (extended per Improvement #3) describes the convention in two sentences. New pattern doc `docs/PATTERNS/idempotency-dual-call-authcheck.md` linked from rule #21. |
| Per-token PIN cap counts legitimate-manager fumbles toward the 5 — could surprise operators | Documented in CHANGELOG (per Improvement #4), UAT (§11), and surfaced in the revoke copy itself: "Ask the staffer to retry" (which mints a fresh token via the normal flow). |
| `cancelAwaitingPayment` races a successful webhook landing | Mutation guards on `txn.status !== "awaiting_payment"` and yields cleanly with `TXN_NOT_AWAITING` if webhook won. Frontend treats this as the success case (txn paid; nothing to cancel; navigate away). Test included in `cancelAwaitingPayment.test.ts`. |

## 11. Acceptance criteria (UAT)

- Every spoke route shows the new header with working back-to-home
- `/sale` with a non-empty cart prompts the abandon dialog on header back, browser back, AND Android gesture-back (Critical #1)
- `/sale` with an empty cart leaves silently in all three back-paths
- `/sale/charge` mid-payment prompts the payment-variant dialog; "Cancel payment" calls `cancelAwaitingPayment` end-to-end (txn cancelled, invoice locally superseded, pending approval cascade-cancelled) (Critical #2)
- `/sale/charge` after a successful webhook → blocker inactive → back-button navigates silently
- Lock from home → `/login` pre-stages to the previous staffer; their PIN logs them straight in
- Deactivating a staffer server-side → next login on that device silently falls back to staff list
- `/approve/:token` with 5 wrong PINs (any combination of attacker iteration + legitimate fumble) → 6th visit shows the revoke message; no PIN input (Improvement #4 — operator-visible behaviour)
- `/approve/:token` on the cap-trip request itself → user sees the revoke copy (not a generic error) via the new `REQUEST_REVOKED` `mapError` branch (Critical #3)
- `npm run lint` fails if a public mutation is added without `idempotencyKey` + `withIdempotency` + `authCheck`
- CI build fails if `idempotency-required` rule severity is anything other than `"error"` (Improvement #1 — `tools/ci/assert-strict-idempotency-rule.sh`)
- An expired pending request shown in `ApprovalPending` overlay reports `expired` not `pending`
- Cancel-sale on `/sale/charge` with a pending Telegram approval → approval transitions to `denied: "txn_cancelled"` via the cascade helper
- Cron `founders-shift-summary` continues to fire green under a deliberate role-rebind mid-window (test against dev)
- Booth manager-picker on `/sale/charge` shows all active managers, accepts any one's PIN (not just the logged-in session's) (Improvement #5)
- Test suite grows materially over the post-v0.4 baseline (new tests across §3–§7); exact count derived during plan-phase
