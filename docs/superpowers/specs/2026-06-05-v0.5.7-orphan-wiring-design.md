# v0.5.7 — Orphaned-function wiring (design spec)

**Date:** 2026-06-05
**Phase:** v0.5.7
**Source handoff:** `.claude/handoff/execute_v0.5.7-orphan-wiring.md`
**Scope:** Wire three public Convex functions that exist + are tested but have **zero UI callers** — the same "backend exists, no door" gap v0.5.6 closed. Mostly frontend; **one small backend change** (Part A actor-label pre-derivation, decided below).

## Problem

A 2026-06-03 audit of every `convex/**/{public,actions}.ts` export against `src/` `api.` callers found three genuinely orphaned public functions:

| Part | Function | Location | Today |
|---|---|---|---|
| A | `audit.public.list` | `convex/audit/public.ts:7` | manager-gated audit query, no viewer |
| B | `transactions.public.listRecentAwaitingPayment` | `convex/transactions/public.ts:392` | recovery query, no caller (hook never built) |
| C | `approvals.public.cancelPendingRequest` | `convex/approvals/public.ts:509` | cancel mutation, no UI affordance |

(`inventory.public.getStockLevels` + `vouchers.public.getActiveVouchers` looked orphaned but are consumed cross-module by `catalog.public.catalog` per ADR-034 — **not gaps, left alone.**)

## Verified facts (read against real code, 2026-06-05)

1. **`audit.public.list` returns raw `audit_log` rows** via `auditListHandler` (`convex/audit/internal.ts:57`). Rows carry `actor_id` (`Id<"staff"> | "system"`), `action`, `entity_type`, `entity_id?`, `source`, `reason?`, `mgr_approver_id?`, `created_at` — **no actor name**. Args `{ sessionId, limit?, action? }`; `limit` clamped to `min(limit ?? 100, 500)`; `action` filter uses `by_action_date` index; default order `desc`.
2. **`_listStaffNames_internal` exists** at `convex/auth/internal.ts:469` and is **already used this exact way** by `convex/transactions/internal.ts:593` to build a staff-id→name map for reporting labels. This is the established "BE pre-derives labels" pattern (v0.5.3a lesson).
3. **`listRecentAwaitingPayment`** (`convex/transactions/public.ts:392`) — args `{ sessionId }`, returns `Doc<"pos_transactions">[]` in `awaiting_payment` from the last 5 min via `by_status_created`, `[]` on invalid session. Booth-wide (not per-staff), correct for a single shared device. Rows come back ascending `created_at` (index order) — "latest" = max `created_at`.
4. **`/sale/charge/:txnId` resumes cleanly.** `charge.tsx` reads `:txnId` and subscribes reactively via `useXenditPayment`; an existing awaiting txn with a live invoice lands in `phase==="showing"` and re-renders its QR. The init-create effect skips creation when an invoice already exists for the selected method (`charge.tsx:175`). **No resume guard needed.** (Edge: charge defaults to the QRIS tab; if the awaiting invoice was BCA_VA the existing tab-swap logic re-mints a QRIS invoice — pre-existing single-active-invoice behavior, ADR-014, acceptable.)
5. **`cancelPendingRequest`** (`convex/approvals/public.ts:509`) — `withIdempotency` mutation, args `{ sessionId, requestId, reason?, idempotencyKey }` → `{ denied: boolean }`, `requireManagerSession`-gated, audited `source: "booth_inline"`, records `cancelled_by_manager_id`. **Manager-session gated** — caller must be a manager session.
6. **`ApprovalPending`** (`src/components/pos/ApprovalPending.tsx`) — props `{ requestId, successMessage?, onResolved?, onDenied?, onExpired? }`. The `pending` branch (lines 49-58) renders a spinner + "Waiting…" text with **no button**. `onDenied`/`onExpired` already render symmetric "Retry" buttons.
7. **Part C applies to only TWO hosts, not three.** `sale/charge.tsx` (`approvalRequestId` state, renders `<ApprovalPending>` at ~`:591`) and `refund/detail.tsx` (`approvalRequestId` state, renders `<ApprovalPending>` at ~`:284`) both hold a pending request. **`mgr/spoilage.tsx` does NOT** — its Telegram path (`handleRequest`, `:195`) is fire-and-forget: it toasts "Sent to managers via Telegram" and calls `resetForm()`, never rendering `ApprovalPending` nor holding a `requestId`. **The handoff's claim that spoilage needs `onCancel` is wrong** — there is no pending UI there to cancel.
8. **Manager-gate pattern** (`mgr/home.tsx:38`, `mgr/spoilage.tsx:79`): `session.status !== "active" || session.staff.role !== "manager"` → `<Navigate to="/" replace />`. `refunds-pending.tsx` uses an inline "Manager access required" card instead — both are in use; the `<Navigate>` form is the v0.5.6 convention for new spokes.
9. **Router** (`src/router.tsx`): `lazy()` import + a child route object under the `/` `RootLayout` group. `NAV_CARDS` on `mgr/home.tsx:13` is a flat array of `{ to, label, hint, glyph }`.

## Decision: Part A actor labels — BE pre-derive (user-approved 2026-06-05)

`audit.public.list` will be **extended to attach `actor_name` server-side** via `_listStaffNames_internal`, mapping each row's `actor_id` (`"system"` → `"System"`, unknown id → fallback to the raw id string). Rationale: matches the v0.5.3a "BE pre-derives labels when it has all inputs" lesson and the existing `transactions/internal.ts` precedent; avoids a client-side second query + id→name join. **Consequence: this phase is not strictly frontend-only — it includes one backend query change + its test.** `mgr_approver_id` is left as-is (not surfaced in v0.5.7).

## Design

### Part A — Audit-log viewer (`/mgr/audit`)

- **Backend:** extend `audit.public.list` handler to map rows → `{ ...row, actor_name }`. Fetch the staff-name map once via `ctx.runQuery(internal.auth.internal._listStaffNames_internal, {})` (cross-module read per ADR-034), then label. Return type changes from `Doc<"audit_log">[]` to `(Doc<"audit_log"> & { actor_name: string })[]`. Keep `_list_internal` untouched (no label — tests/server callers don't need it). Add a test asserting `actor_name` resolves for a real staff actor and `"System"` for `"system"`.
- **Frontend:** new `src/routes/mgr/audit.tsx` — manager-gated spoke (outer role-guard → `<Navigate to="/" replace />`, inner data component taking `sessionId`), mirroring `mgr/refunds-pending.tsx` + `mgr/stock.tsx`. `SpokeLayout title="Audit log" backTo="/mgr"`. Reverse-chron `Card` rows: `fmtDate`/`fmtTime` timestamp, action verb (mono), `actor_name`, `entity_type` + `entity_id`, a `source` badge, and `reason` (line-clamped) when present. ADR-007 — audit is append-only/read-only, **no row actions.**
  - **Action filter:** a controlled text `Input` (or `Select` over distinct verbs from the loaded rows); when non-empty, passes `action` to the query. Keep it simple — text input feeding `action` arg, debounced is optional.
  - **Load more:** local `limit` state (default 100), a "Load more" button bumps it by 100 up to the server cap (500). Reactive query re-runs on `limit` change.
- **Nav:** add a 10th `NAV_CARD` to `mgr/home.tsx` (`{ to: "/mgr/audit", label: "Audit log", hint: "Append-only activity trail", glyph: "❡" }` or similar) + lazy route `mgr/audit` in `src/router.tsx`.

### Part B — Awaiting-payment recovery banner (`home.tsx`)

- **Hook:** new `src/hooks/useAwaitingPaymentRecovery.ts`, shaped after `useRecountNudge.ts`. Runs `useQuery(api.transactions.public.listRecentAwaitingPayment, sessionId ? { sessionId } : "skip")`. Returns `{ count: number; latest: Doc<"pos_transactions"> | null }` — `count = list.length`, `latest =` the row with max `created_at` (or `null` while loading / empty). Returns `{ count: 0, latest: null }` while `undefined` (loading) so the banner never flashes.
- **Banner:** in `home.tsx`, next to the existing recount nudge (`home.tsx:86-93`), render an amber `Link`-style banner when `count > 0`: e.g. `"{count} pembayaran belum selesai — ketuk untuk lanjutkan"`. Tapping `navigate`s to `/sale/charge/${latest._id}`. Read-only, **no backend change.**

### Part C — Cancel a pending approval (`ApprovalPending` + 2 hosts)

- **Component:** add optional `onCancel?: () => void` to `ApprovalPending`. When provided, render a "Batalkan permintaan" ghost/outline button **inside the `pending` branch only** (it makes no sense after a terminal status). Symmetric to the existing `onDenied`/`onExpired` Retry buttons.
- **Hosts (2):**
  - `sale/charge.tsx` — add `onCancel` to its `<ApprovalPending>` (~`:591`): call `cancelPendingRequest({ sessionId, requestId: approvalRequestId, idempotencyKey: crypto.randomUUID() })`, then `setApprovalRequestId(null)` to return to the pre-request CTAs. Map errors via `toast.error`.
  - `refund/detail.tsx` — add `onCancel` to its `<ApprovalPending>` (~`:284`): same call; on success `setApprovalRequestId(null)` and `clearIntent("refund-telegram:" + txnId)` (so a retry mints a fresh request, consistent with the existing denied/expired handlers).
- **`cancelPendingRequest` is manager-session-gated.** In both hosts the cancelling user may be a non-manager staff (the original requester). **Open question for the plan staffreview:** confirm whether a non-manager session calling `cancelPendingRequest` throws (`requireManagerSession`). If it does, the cancel button must be **gated to manager sessions** (or the affordance only shown when `session.staff.role === "manager"`), and the copy/empty-state adjusted. Resolve against `requireManagerSession` before coding the hosts. *(Hypothesis: it throws for non-managers; most-likely outcome is the button is manager-only, OR we accept that the staff requester can't self-cancel and instead the button is only useful on the manager's own device. The plan must pin this down.)*
- **`/wait/:requestId`** (`src/routes/wait.tsx`) is a Stub; the real pending UX is the inline `ApprovalPending`, which is why cancel belongs there. Leave the dead `/wait` route as-is (out of scope).

## Out of scope / non-goals

- No schema, migration, or new index.
- No changes to `mgr/spoilage.tsx` (fire-and-forget, no pending UI — fact #7).
- No retiring of the `/wait` stub.
- `mgr_approver_id` not surfaced in the audit viewer.
- No new approval KIND, no Telegram template change.

## Testing

- **Part A:** BE test — `actor_name` resolves for a staff actor + `"System"` for `"system"`. FE tests — manager renders rows; non-manager redirects (`<Navigate>`); action filter passes `action` arg; "Load more" bumps `limit`.
- **Part B:** banner shows when list non-empty + navigates to latest `_id`; hidden when empty/loading.
- **Part C:** `ApprovalPending` shows the cancel button only in `pending` and only when `onCancel` provided; calls `cancelPendingRequest` once with a UUID key; host resets `approvalRequestId` on success. (Plus the manager-gate behavior once resolved.)
- Harness: `vi.mock("convex/react")` dispatching `useQuery`/`useMutation`/`useAction` by `getFunctionName` + `MemoryRouter` (see `src/routes/history/__tests__/$txnId.test.tsx`).
