## Goal

Ship the operational core of the booth: staff take a sale end-to-end on the Android device, build a cart, apply a voucher, charge via QRIS or BCA VA through Xendit, and confirm payment via three coordinated paths (webhook primary, polling fallback, manager-PIN manual override). Drafts save and resume. Bootstrap a fresh deployment with a single manager (PIN 1111) that can immediately rotate its own PIN. Managers can reset other staff PINs at the booth — or, if a staff member locks themselves out, reset remotely via a Telegram link.

## Scope summary

**Ships in v0.3:**
- Cart-to-payment-to-receipt sale flow (QRIS + BCA VA)
- Three-path payment confirmation (webhook + polling + manager-PIN manual override)
- Saved drafts (commit, list, resume, delete)
- Voucher apply (cached read, server-revalidate, race-tolerant redemption)
- Negative-stock allowed at sale, flagged per ADR-018
- Reconciliation-on-reload per ADR-026
- Receipt number allocation (R-YYYY-NNNN per ADR-023)
- PIN management trio: bootstrap action, self-change PIN, manager-reset-other-staff PIN
- Telegram-driven off-booth PIN reset when staff lock themselves out (first use of the v0.4 approval pattern, generalized to support more kinds later)
- ADR-035: Telegram as internal comms channel (supersedes ADR-027 + ADR-033)

**Deferred:**
- Refunds → v0.5
- Receipt PDF/sharing/print → v0.5
- Transaction history list → v0.5
- Stock-in / stock-check / manager stock tools → v0.5
- Manager dashboard → v0.5

### Module dependency graph (one-way, no cycles)

```
       ┌──── approvals ────► auth (via _changePinCommit_internal)
       │
       │    payments ────► transactions ────► inventory
       │       │              │                  │
       │       │              ▼                  │
       │       │           vouchers              │
       │       │              │                  │
       └───────┴─── audit ◄───┴──────────────────┘   (audit is the foundational allow-list module
                                                      that everyone can import from)
       telegram ◄── approvals (action calls Node telegram.send)
```

**Allowed cross-module imports:** the ADR-034 allow-list — `audit/internal.logAudit` is callable from any module; same for `idempotency/internal.withIdempotency`. Module-boundary lint rule rejects everything else (DB access across module boundaries).

**Cross-module dispatch via `ctx.scheduler`:** the lint catches cross-module *table* access, not scheduler dispatch. `auth/internal._recordFailedAttempt` calls `ctx.scheduler.runAfter(0, internal.approvals.actions.notifyStaffLockout, ...)` — this is a runtime reference through the generated `internal.*` API, not a cross-module import or DB access, so it passes the lint. The two transactions are independent (scheduler runs the action in a separate transaction), so the auth ↔ approvals logical bidirectionality is broken at runtime. This pattern is the canonical way one module fires "events" for another to handle without static dependency cycles.

**Telegram delivery:** `telegram/send.ts` is consumed by `approvals/actions.notifyStaffLockout` in v0.3. Per ADR-035 (drafted in this phase), Telegram is the internal comms infrastructure — added to the foundational allow-list alongside `audit/` and `idempotency/`. Future modules can call `telegram.send` for system notifications without violating the boundary.

```
┌─────────────────────────────────────────────────────────────────────┐
│  STAFF DEVICE                                                       │
│                                                                     │

│  Returns to frontend: { invoiceId, qrString?, vaNumber? }           │
└─────────┬───────────────────────────────────────────────────────────┘
          ▼
┌─────────────────────────────────────────────────────────────────────┐
│  routes/sale/charge.tsx                                             │
│  ├─ useXenditPayment(txnId):                                        │
│  │   ├─ useQuery(transactions.public.getById, {txnId})  ── reactive │
│  │   ├─ useQuery(payments.public.getCurrentInvoice, {txnId})        │
│  │   ├─ effect: wait 2s, then every 2s up to 60s ceiling:           │
│  │   │     callAction payments.actions.checkInvoiceStatus(invId)    │
│  │   └─ effect (ceiling reached): show "Manager override?" CTA      │
│  └─ reactive: when txn.status flips to "paid" → navigate(success)   │
└─────────────────────────────────────────────────────────────────────┘

THREE CONFIRMATION PATHS (all idempotent, all converge on one funnel):

(1) Xendit webhook ──► payments/webhook.ts (httpAction)
    └─ sig verify, dedupe by xendit_invoice_id
    └─ runMutation payments.internal._onPaidWebhook
       └─ runMutation transactions.internal._confirmPaid(source="webhook")

(2) Polling ──► payments.actions.checkInvoiceStatus(invoiceId)
    └─ HTTP GET Xendit /v2/invoices/{id}
    └─ if PAID: runMutation payments.internal._onPaidPolling
       └─ runMutation transactions.internal._confirmPaid(source="polling")

(3) Manual override ──► payments.actions.manuallyConfirmPayment
    └─ argon2id verify managerPin
    └─ runMutation payments.internal._onPaidManual
       └─ runMutation transactions.internal._confirmPaid(source="manual",
                                                          mgr_approver_id,
                                                          manual_reason)

THE FUNNEL — transactions.internal._confirmPaid:
  Guard: if txn.status !== "awaiting_payment" → no-op (idempotent re-fire)
  1. allocate receipt_number from pos_receipt_counters (atomic increment)
  2. inventory.internal._recordSaleMovement (writes movements, decrements on_hand,
                                              re-checks NEG_STOCK and OR-s in the flag)
  3. if voucher_code_snapshot present: vouchers.internal._redeemVoucher
     (atomic used_count < max_redemptions check; sets VOUCHER_OVER_REDEEMED if loser)
  4. patch pos_transactions: status=paid, receipt_number, paid_at, confirmed_via, flags
  5. logAudit({ action: "payment.confirmed", source })
```

**Why this is "deep" (ADR-034):** all three confirmation paths converge on `_confirmPaid_internal`. Adding a v0.4 path (Telegram-approved manual override) means a fourth caller, not a fourth copy of the confirmation logic. Adding a future v1.1 path (e.g., direct-bank confirmation) is the same.

### WIB time helper (new file)

```ts
// convex/lib/time.ts (NEW)
export function wibYear(timestamp: number = Date.now()): number {
  // WIB is UTC+7, no DST. Add 7h to the UTC timestamp and read the UTC year of the shifted moment.
  const wibMs = timestamp + 7 * 60 * 60 * 1000;
  return new Date(wibMs).getUTCFullYear();
}
```

Used inside `_allocateReceiptNumber_internal`. The function lives in `convex/lib/time.ts` because it's runtime-neutral (no Convex API), and future code (receipt PDF rendering in v0.5, settlement reports in v0.5) will reuse it.

### Convex emulated UNIQUE constraints

Convex has no native UNIQUE — emulated via index + pre-insert query:
- `pos_stock_movements (source_transaction_line_id, sku_id)` — enforced inside `_recordSaleMovement_internal`. If lookup returns a row, insert is a no-op (idempotent for ADR-026 reconciliation).
- `pos_voucher_redemptions (transaction_id)` — enforced inside `_redeemVoucher_internal`. Returns `{ alreadyRedeemed: true }` on collision (idempotent re-fire).
- `pos_approval_requests (token_hash)` — argon2id collisions effectively impossible; lookup-by-index inside `getByToken` returns single row.

## State machine — `pos_transactions.status`

```
                  ┌──[commitCart intent="draft"]────────────────────► draft
       (none) ────┤                                                    │
                  └──[commitCart intent="charge"]────► awaiting_payment│
                                                              │        │
                                                              │        └─[resumeDraft]─►(row deleted,
                                                              │                          lines returned)
                                                              ▼
                                                            ┌────────────────────────────┐
                                                            │ paid (terminal in v0.3)    │
                                                            ├──── via "webhook"          │
                                                            ├──── via "polling"          │
                                                            └──── via "manual"           │
                                                                                         │
                                              awaiting_payment ──[_cancelTransaction]──► cancelled
                                              (user cancel OR ADR-026 reconciliation finds Xendit expired)
                                                              (terminal in v0.3)
```

**Invariants:**
- `draft` rows: only `resumeDraft` (deletes them) or `deleteDraft` (explicit) or v0.5+ purge job (ADR-032 territory) modify them.
- `awaiting_payment → paid`: the ONLY path that allocates a receipt number, writes stock movements, redeems vouchers — all inside `_confirmPaid_internal`.
- `awaiting_payment → cancelled`: does NOT write stock movements (stock was never decremented). Calls Xendit cancel-invoice as a safety pass.
- Once `paid` or `cancelled`, status is immutable in v0.3.

## Backend module surface APIs

### transactions/

**public.ts (queries + mutations, V8 runtime):**
- `getById(txnId)` → row | null
- `listDrafts(sessionId)` → rows[] where `status="draft"`, ordered by `created_at` desc
- `commitCart({ sessionId, idempotencyKey, intent: "draft" | "charge", lines, voucherCode? })` → `{ transactionId, totals, flags }`
- `resumeDraft({ sessionId, draftId, idempotencyKey })` → `{ lines, voucherCode? }` (row deleted in same transaction)
- `deleteDraft({ sessionId, draftId, idempotencyKey })` → `{ deleted: true }`

**actions.ts (Node runtime — needs Xendit HTTP for cancel):**
- `cancelTransaction({ sessionId, txnId, reason, idempotencyKey })` → `{ cancelled: true }`
  - Guard: txn.status must be "awaiting_payment" (cannot cancel draft — use deleteDraft; cannot cancel paid — refunds are v0.5)
  - If `xendit_invoice_id_current` is set: HTTP POST Xendit cancel-invoice — **best-effort**. On Xendit error: log `payment.invoice_cancelled` with metadata `{ success: false, error }` and continue. The 5-min Xendit invoice expiry is the backstop; reconciliation-on-reload will not re-poll a cancelled txn.
  - If no current invoice (e.g., requestPayment failed earlier): skip Xendit call.
  - `runMutation` internal cancel-commit (flips status to cancelled, sets cancelled_at + cancelled_reason)
  - logAudit({ action: "transaction.cancelled", reason })

**internal.ts:**
- `_confirmPaid_internal({ txnId, source, mgr_approver_id?, manual_reason? })` — **THE FUNNEL**
- `_projectedNegStockFlag_internal(lines)` — pure helper, used at commit + re-check at confirm
- `_allocateReceiptNumber_internal()` → string `"R-YYYY-NNNN"` — atomic counter increment using **WIB year** (UTC+7) for the prefix. Implementation: read counter for current WIB year via `by_year` index → if exists, patch `next_number = n + 1` and return `R-${wibYear}-${pad4(n)}`; if absent, insert new counter row with `next_number = 2` and return `R-${wibYear}-0001`. Convex's optimistic concurrency retries the mutation on conflict (two concurrent _confirmPaid calls in the same WIB year serialize automatically — no manual locking needed).

### payments/

**public.ts (queries only):**

- `_replaceInvoiceCommit_internal({ txnId, prev_invoice_id, new_xendit_id, new_xendit_idempotency_key, method, ... })`

### vouchers/

**public.ts:**
- `getActiveVouchers()` → row[] (active=true, not expired) — included in offline cache snapshot
- `validateVoucher({ code, cartSubtotal })` → `{ valid, discountAmount, reason? }` — called from `commitCart` AND from `routes/sale/voucher.tsx` for live feedback

**Discount calculation (per ADR-024 ordering: line → voucher → tax; tax=0 in v1):**
```
voucherDiscount =
  type === "percentage" ? Math.floor(subtotal * value / 100)
                        : Math.min(value, subtotal)   // "amount" type, capped at subtotal
total = subtotal - voucherDiscount
```
All integer rupiah per ADR-015. Floor on percentage (never round up customer-favourably or against them — floor is consistent and auditable).

**internal.ts:**
- `_redeemVoucher_internal({ voucher_id, transaction_id, code_snapshot, discount_amount })` → `{ overRedeemed: boolean }`
  - Re-check `used_count < max_redemptions` (single-mutation read + write — Convex serializes via optimistic concurrency; on conflicting writes between two concurrent _confirmPaid mutations, one of them retries and reads the updated `used_count`).
  - On race loss (after retry, still `used_count >= max_redemptions`): returns `{ overRedeemed: true }` — caller sets `VOUCHER_OVER_REDEEMED` flag on txn.
  - Pre-insert uniqueness check on `pos_voucher_redemptions(transaction_id)` via `by_transaction` index (idempotent re-fire returns `{ alreadyRedeemed: true }`).

**Concurrency model note (applies to receipt counters + voucher `used_count` + any other contended row):** Convex mutations are serializable transactions. Two concurrent mutations that both read+write the same row will have one of them re-executed by Convex's runtime (optimistic concurrency). No manual locking, no explicit transaction blocks needed. Tests should exercise this directly via `convex-test`'s concurrent-invocation primitive.

### approvals/

**public.ts (queries only):**
- `getByToken(rawToken)` → `{ kind, subject_staff_name, subject_staff_code, status, locked_at, token_expires_at }` — reactive (drives `/approve/:token` UI; sha256-hashes the raw token to look up by `by_token_hash`). Returns **only the projection shown above** — never includes `token_hash` itself in the response. Defense-in-depth: don't ship even the hashed form back to the client.

**actions.ts (Node runtime):**
- `notifyStaffLockout({ staffId })` — internal action, fired by `auth/_recordFailedAttempt` scheduler
  - **Dedup check first**: query `pos_approval_requests.by_subject_staff(staffId)` for any row with `kind="staff_pin_reset"`, `status="pending"`, `token_expires_at > Date.now()`. If found → return `{ skipped: true, reason: "pending_request_exists" }` and do nothing else. Prevents multiple Telegram messages stacking up when the same staff hits the lockout repeatedly within the 60-minute token TTL.
  - Generate raw token (32 bytes URL-safe base64); sha256-hash → `token_hash`
  - `runMutation _createRequest_internal`
  - Build URL: `${POS_BASE_URL}/approve/${rawToken}`
  - Call `internal.telegram.send.sendTemplate({ kind: "staff_pin_reset", params: {...} })`
  - `runMutation _markNotified_internal`
- `approveStaffPinReset({ token, managerPin, newPin, idempotencyKey })` → `{ resolved: true }`
  - sha256-hash the incoming token, look up by `by_token_hash` (single row). After retrieval, do an extra `crypto.timingSafeEqual(computedHashBuf, storedHashBuf)` — constant-time compare for consistency with ADR-034's API-token pattern (cheap defense-in-depth even though SHA-256 collision-resistance already makes leakage unlikely).
  - Verify `status === "pending"` and `token_expires_at > Date.now()`.
  - argon2id-verify managerPin against MANAGER's `pin_hash` (PIN is low-entropy → argon2id is correct here)
  - Failed verify → `_recordFailedAttempt` against manager's counter; throw
  - On success → `runMutation auth.internal._changePinCommit_internal({ staffId: subject, newPinHash, actor: { kind: "manager_reset", mgr_approver_id: manager.id } })` + `_markResolved_internal`

**internal.ts:**
- `_createRequest_internal({ kind, subject_staff_id, triggered_by_event, triggered_at, token_hash, token_expires_at })` → `{ requestId }`
- `_markNotified_internal(requestId)`

## Telegram approval flow — staff_pin_reset

```
Staff Lucy fails PIN 3 times on booth device
                 │
                 ▼
auth/internal._recordFailedAttempt (mutation)
  counter = 3, locked_until = now + 60s
  ctx.scheduler.runAfter(0, internal.approvals.actions.notifyStaffLockout, { staffId: Lucy })
                 │
                 ▼
approvals/actions.notifyStaffLockout(staffId)
  ├─ rawToken = random(32 bytes, URL-safe base64)
  ├─ token_hash = sha256(rawToken)  // deterministic, allows index lookup
  ├─ runMutation _createRequest_internal({ kind: "staff_pin_reset", ... })
  ├─ url = `${POS_BASE_URL}/approve/${rawToken}`
  ├─ telegram.send.sendTemplate({
  │      chat_id: "Frollie · Managers" group,
  │      kind: "staff_pin_reset",
  │      params: { staff_name: "Lucy", staff_code: "S-0042",
  │                locked_at_iso, request_url: url }
  │  })
  └─ runMutation _markNotified_internal({ requestId })

Telegram message (rendered HTML in managers group):
  🚨 Staff locked out
  Lucy (S-0042) failed PIN entry 3 times.
  Locked at 14:23 WIB.
  [Tap to reset PIN →]   ← URL = https://pos.frolliepos.com/approve/<rawToken>
  This link expires in 60 minutes.

Manager (anywhere) taps link → browser opens /approve/<rawToken>
                 │
                 ▼
routes/approve/[token].tsx (PUBLIC — no session gate)
  useQuery(approvals.public.getByToken, { rawToken })
     → reactive — shows live status (pending / resolved / expired)
  Form: [Your manager PIN] [New PIN for Lucy] [Reset Lucy's PIN]
                 │
                 ▼ on submit
approvals/actions.approveStaffPinReset({ token, managerPin, newPin, idempotencyKey })
  ├─ verify token (status=pending, not expired)
  ├─ argon2id verify managerPin against caller-manager.pin_hash
  ├─ (failed → _recordFailedAttempt on MANAGER's counter; throw)
  └─ runMutation auth.internal._changePinCommit_internal({
        staffId: Lucy.id,
        newPinHash,
        actor: { kind: "manager_reset", mgr_approver_id: manager.id }
     })
     + _markResolved_internal({ requestId, resolved_by_manager_id: manager.id })
                 │
                 ▼
Approve page reactive sub fires → shows "✓ Lucy's PIN has been reset"
Lucy can now log in at booth with the new PIN.
```

**Why this is "deep":**
- One mutation funnel (`_changePinCommit_internal`) backs both the booth-inline `resetStaffPin` action AND the off-booth `approveStaffPinReset` action. Identical audit log shape (action="staff.pin_reset", mgr_approver_id set). Two callers, one truth.
- The approval request lifecycle is fully reactive: the manager seeing "expired" or "already resolved" on the approve page is driven by Convex's reactive subscription on `pos_approval_requests`, not polling.

**Locked-out manager self-reset (by design, document explicitly):**
- In v0.3 bootstrap with a single manager (Lucas), Lucas can lock himself out at the booth, receive the Telegram message in the Managers group, tap the link, enter his own PIN, and set himself a new PIN.
- This works because `approveStaffPinReset` argon2id-verifies `managerPin` against `staff.pin_hash` directly — it does NOT consult `pos_auth_attempts` for lockout state. Lockout only restricts `loginWithPin` at the booth.
- Future readers: **do not "fix" this oversight**. The Telegram path is explicitly designed to be an off-booth escape hatch from booth-side lockout. This is intentional per ADR-029 (token authorizes VIEW, PIN authorizes ACT) and ADR-035 (Telegram as internal comms channel).
- If a security policy ever requires blocking locked-out-manager self-reset, add the lockout check inside `approveStaffPinReset` action and document the decision in a new ADR.

**v0.4 graduation:** the same `pos_approval_requests` table and `/approve/:token` route handle additional `kind` values (refund, manual_payment, etc.). Adding a kind = add a row to the telegram template union + add an action that resolves it + add a UI variant in `approve.tsx`. No new module, no new route.

### Action-level idempotency (resolves staffreview Critical #1)

Convex actions are NOT transactional with their internal-mutation calls. A naive `withIdempotency` wrap on an action would write the cache row AFTER the side-effecting HTTP call, leaving a window where the HTTP succeeded but the cache wasn't recorded — retry would re-fire the HTTP. For Xendit-touching actions, this means duplicate invoices on the Xendit side.

**Pattern for all Xendit-touching actions (`requestPayment`, `retryWithFreshInvoice`, `cancelTransaction`, `manuallyConfirmPayment`):**

1. **At action start**: call `internal.idempotency._lookup_internal({ key: args.idempotencyKey })`. If hit → return cached response (skip everything below).
2. **Side-effecting HTTP**: pass `args.idempotencyKey` as the `X-IDEMPOTENCY-KEY` header to Xendit. Xendit's Invoice API supports this header — same key → same invoice (or same response for cancel). This makes Xendit's side idempotent too.
3. **Commit + cache atomically**: the internal mutation that records the Convex-side effect (e.g., `_persistInvoiceCommit_internal`) also writes the `pos_idempotency` row with the response blob in the **same transaction**. If the mutation rolls back, the cache doesn't get a stale entry; if it succeeds, the cache is committed atomically with the state change.

This guarantees:
- First call succeeds end-to-end → cache row exists → retry hits cache → returns cached response.
- First call's HTTP succeeds but action throws before mutation runs → no Convex state, no cache → retry re-fires HTTP, but Xendit dedupes via the header → same invoice returned → mutation commits + cache atomically.
- First call's mutation throws → no state, no cache → retry re-fires HTTP (Xendit dedupes) + mutation (fresh) → consistent state.

**Schema implication:** `withIdempotency` mutation helper signature gains an optional `cacheInSameMutation: boolean` flag (default false for backward-compat with v0.2 callers). Internal mutations called from actions set it true. Tests cover both paths.

**Webhook handler dedupe is separate:** the webhook uses `pos_xendit_invoices.by_xendit_invoice_id` index for dedup. The action-level idempotency-key prevents *the action layer* from double-acting; the webhook dedup prevents *Xendit's retry* from double-acting.

### Additional tests required (from staffreview findings)

- **`tokenLifecycle.test.ts` boundary** — token at `token_expires_at - 1ms` accepts; at `token_expires_at` rejects. Guards against off-by-one in date math.
- **`cancelTransaction.test.ts` no-invoice** — commit charge intent, do NOT call requestPayment, then call cancelTransaction. Asserts: status flips to cancelled, no Xendit HTTP attempted, audit logs `transaction.cancelled`.
- **`cancelTransaction.test.ts` Xendit-cancel-fails** — mock Xendit `cancelInvoice` to throw 5xx. Asserts: status STILL flips to cancelled (best-effort behavior), audit logs `payment.invoice_cancelled` with `success: false` + `error` metadata.
