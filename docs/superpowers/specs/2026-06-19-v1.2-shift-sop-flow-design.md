## 1. Goal

Turn the two laminated paper SOPs (open booth / close booth) into guided in-app journeys so the POS *is* the SOP, attendance is captured by signing in, stock counts land in inventory, and every shift boundary produces an attributed, audited record. Replace the logout-only "lock" with a real shift lifecycle.

**You'll be able to:**
- Sign in to a closed booth and be walked through the **start-of-day** SOP (count stock → power on → stock the display → tidy + photo) before the booth opens.
- Tap **End shift** and either **close the booth for the day** (full closing SOP → sign-off) or **hand over to the next staff** (booth stays open).
- See your own **hours** at sign-off; founders/managers get the full financial summary in Telegram.
- Lock the device for a break and resume as yourself without re-selecting an account; only a manager can take over a locked booth.

**Not in scope:** i18n toggle (#1 — copy ships Bahasa now); product photos (#3); changing the recount math (ADR-041 reused as-is); a separate stock-in/delivery form (recount remains the staff stock-write); replacing the 22:00 founders cron.

## 2. The booth state machine (single source of truth)

The booth runs on one device. State is **derived from the latest `pos_shift_events` row for that device** (no separate status flag to drift). `staff_sessions` tells you *who is signed in right now* (via `by_device_active`), but it goes **stale across a lunch-lock** (§3E) — so it is never the state source. Three states:

| State | Meaning | Allowed actions |
|---|---|---|
| **CLOSED** | No active shift (fresh day, or after an end-of-day sign-off) | Sign in → account picker → **start-of-day wizard** → OPEN |
| **OPEN** | A shift is active for one staff | **Lock** → LOCKED · **End shift → Close** → CLOSED · **End shift → Handover** → OPEN (new staff) |
| **LOCKED** | Locked mid-shift — **session ended** (ADR-003), booth flagged locked-to-staff-A | Staff A's fresh login → OPEN · **Manager-PIN unlock** → takeover → OPEN (manager) |

**Transitions:**
```
CLOSED ──sign in (pick account) + start-of-day wizard──────────────▶ OPEN(staff A)
OPEN(A) ──Lock────────────────────────────────────────────────────▶ LOCKED(A)
LOCKED(A) ──A's PIN───────────────────────────────────────────────▶ OPEN(A)        (no wizard, no picker)
LOCKED(A) ──Manager M's PIN (takeover)────────────────────────────▶ OPEN(M)        (A force-ended; M counts)
OPEN(A) ──End shift→Close: end-of-day wizard + sign-off────────────▶ CLOSED
OPEN(A) ──End shift→Handover: A's wizard + pick B + B's PIN + B counts──▶ OPEN(B)
```

**Account selection appears on exactly two transitions** — open-from-CLOSED and handover-incoming. Every other entry (lock resume) is same-staff-only. This is the core invariant: **changing who is responsible always crosses a counted, attributed boundary.**

**Stale-shift edge (forgot to close):** if the latest event is OPEN/LOCKED but from a **prior WIB day**, the next sign-in is treated as open-from-CLOSED — the stale shift is auto-closed with a `stale_autoclose: true` flag and its summary still fires to Founders. (Prevents a forgotten close from suppressing the next morning's start-of-day wizard.)

**Lock & session lifecycle (ADR-003):** Lock still **ends the `staff_sessions` row** (current `lock.tsx` behavior — a held session on an unattended device is replayable; ADR-003 closed that). LOCKED is therefore a **booth-state layer**, not a live session: a `lock` event names the locked staff, the login gate enforces same-staff fresh-login (or manager-unlock), and a successful resume logs a `resume` event → OPEN. The session is recreated on resume — which is exactly why hours anchor to the shift-start **event**, not a session row (§4).

## 3. The wizards

Shared pattern (all wizards): **one step per screen · vertical teal step-rail showing done ✓ / current (citrus) / pending · Back allowed · "Next" = "I've done this" → step banks green.** Two step types:
- **Instruction step** — physical-world action; Next is a tap-to-confirm attestation (POS records only the attestation, like #10's manual-BCA pattern).
- **Count step** — embeds the existing recount UI (`/stock/recount`'s component: SKU rows, system value, input, live delta); writes a `recount` movement via `api.inventory.public.recordRecount` on advance. Reused verbatim — no new ledger path (ADR-041).

### A. Start of day (`Mulai hari`) — fires on open-from-CLOSED
Login steps from the paper SOP drop (signing in *is* the attendance record). Steps:
1. **Count Dubai Chewy Cookies** (count step → inventory)
2. Power on: WiFi · printer · Frollie phone · confirm **GoFood OPEN** (instruction)
3. Put **5** cookies into the display (instruction)
4. Tidy booth · banner clear from escalator · **photo front+inside → WA group** (instruction)
→ **"Mulai hari"** → home. (Paper step "collect delivery from Lalamove/Gosend" folds into a step-1 note; out-of-band, reconciled by the count.)

### B. End of day (`Tutup booth`) — End shift → Close — closing SOP steps 2–6
1. *Reminder:* everything goes in the lockers by close; losses are the closing staff's responsibility — you lock up at the last step (instruction/acknowledge). **Reworded from the paper's "lock all items now"** per Lucas — staff keep using items until close.
2. **Final stock count** (count step → inventory)
3. Check supplies: stickers · seal stickers · paperbags · onion-bags · cable ties; if <3 days left → report in WA group (instruction)
4. Coil cables · WiFi + printer off · all devices on the locker's bottom shelf · cover with cloth (instruction)
5. Lock **both** lockers · take your key home (instruction) → **"Sign off — selesai hari ini"**
→ Staff sign-off summary (hours) → Founders Telegram summary fires → logout → CLOSED.

### C. Handover (`Serah terima`) — End shift → Handover — booth stays OPEN
Close-only steps (power-off, locker-lock, lock-up reminder) are omitted. Outgoing:
1. **Stock count** (count step → inventory)
2. Check supplies (instruction)
→ **"Serah terima"** → account picker → **incoming staff PIN** → **incoming staff does their own count** (count step → inventory; **no accept/confirm screen** — keep it simple, per Lucas) → outgoing session ends + outgoing summary to Founders → incoming OPEN, shift started.

**Handover stock = two writes** (outgoing count + incoming count). The two figures bracket the boundary; a discrepancy is the audit signal. No review/accept UI.

### D. Manager takeover (from LOCKED)
Lock screen offers same-staff PIN **and** a "Ganti staff? Manajer buka kunci →" path. Manager enters PIN (verified via `verifyManagerPinOrThrow`, ADR-005). This is **structurally a handover with the manager as incoming** — *same code path as C* with a `takeover: true` flag:
- Original staff is **force-ended** (session closed; cannot resume without a fresh sign-in).
- Outgoing wizard steps are **skipped** (the absent staff can't self-count) → the **manager's incoming count is authoritative**, and the boundary is flagged `outgoing_uncounted: true` so a manager can later see the gap.
- Original staff still gets a **system-computed** shift summary to Founders, flagged **"diakhiri oleh manajer"** (ended-by-manager) vs. a clean self-sign-off.
- Manager's shift begins (booth OPEN). The manager may later **hand over to a staff** via a normal End shift → Handover.

A non-manager **cannot** take over a locked booth — manager-PIN is the only escape, enforcing the counted/attributed boundary.

*(Impl: `verifyManagerPinOrThrow` (`auth/verifyPin.ts:98`) is `ActionCtx` — argon2. The takeover runs through an **action** like `loginWithPin` (force-end A, then a committed mutation starts M); a throwing mutation rolls back all its writes, so this cannot be a single mutation.)*

### E. Lock (ends session; tightened resume)
**Lock ≠ End shift.** Lock **ends the session** (honoring ADR-003 — unchanged from today's `lock.tsx`) and records a `lock` event naming staff A. The login gate then shows **only staff A's PIN** (no picker, no wizard); A's fresh login logs `resume` → OPEN, same shift (hours unaffected — they anchor to the shift-start event, §4). To change staff you must End shift → Handover, or a manager unlock. A non-manager cannot pass the lock.

## 4. Data model

### New table: `pos_shift_events`
Source of truth for booth state, attendance, the step-completion ledger, and shift summaries.

| Field | Type | Notes |
|---|---|---|
| `device_id` | string | which booth device |
| `type` | union | `"start_of_day" \| "lock" \| "resume" \| "signoff_close" \| "handover_out" \| "handover_in" \| "manager_takeover"` |
| `staff_id` | Id<"staff"> | the actor for this event |
| `shift_started_at` | number | server time of the **shift-start event** (`start_of_day`/`handover_in`/`manager_takeover`) — the hours anchor, copied onto the sign-off event. NOT a `staff_sessions` row (lock fragments sessions). |
| `shift_ended_at` | number? | set on the closing/handover-out/takeover-displaced event |
| `steps` | array | `{ key, label, type: "instruction"\|"count", confirmed_at }[]` — the banked rail |
| `count_movement_ids` | array? | links to the `pos_stock_movements` recount rows for this event |
| `summary` | object? | computed snapshot: `{ duration_ms, sales_total?, txn_count?, manual_bca?: {count,total,items[]} }` (financials present only on the Founders payload, see §5) |
| `takeover` | boolean? | manager unlock takeover |
| `outgoing_uncounted` | boolean? | takeover where the displaced staff never self-counted |
| `stale_autoclose` | boolean? | forgot-to-close auto-resolution |
| `linked_event_id` | Id? | handover_out ↔ handover_in pairing |
| `created_at` | number | server time (ADR-031) |

Indexes: `by_device_created` `(device_id, created_at)` for "latest event = current state"; `by_staff_started` `(staff_id, shift_started_at)` for per-staff history.

**Hours** = `shift_ended_at − shift_started_at` (server time), where `shift_started_at` is the timestamp of the staff's **shift-start event** (`start_of_day`/`handover_in`/`manager_takeover`), **NOT** a `staff_sessions` row — because lock ends and recreates sessions, so a shift spans multiple sessions (§3E). The roadmap's `by_device_started` index is dropped; state and hours both live in `pos_shift_events`.

**No `audit_log` schema change** — every state-changing mutation still emits `logAudit` (new free-string verbs: `shift.start_of_day`, `shift.signoff`, `shift.handover`, `shift.manager_takeover`). `audit_log.action` is `v.string()` (no enum to edit).

## 5. Shift summary — split by audience

- **Staff sign-off screen (on-device):** **hours only** — shift span → duration headline — plus the stock counts just entered as confirmation. **No sales total, no txn count, no manual-BCA.** (Lucas: the staff screen is not a performance scoreboard.)
- **Founders Telegram summary:** the full picture — staff · span · hours · sales total · txn count · **itemized manual-BCA tally** (count + total + per-txn times, reusing `_manualBcaReconciliation_internal` from #10) · stock deltas. Fires on **every** sign-off (close, handover-out, and manager-takeover-displacement).
- The existing **22:00 WIB founders cron stays** as the end-of-day rollup; per-shift summaries are additive.

### Telegram template (add-a-template, CLAUDE.md §how-to #8 / v0.6+v1.0.1 pattern)
⚠️ A `shift_summary` kind **already exists** — the **daily** founders rollup (`send.ts:37` union + `foundersSummary.ts:127`, payload `{dateLabel, totalSalesIdr, txnCount, flaggedCount, manualBca}`; `telegramHtml.ts` already renders the `manualBca` itemization). The **per-shift** message must use a **distinct** kind (`shift_signoff`) and **reuse the existing `manualBca` payload shape + its `telegramHtml` renderer fragment** — do not write a second manual-BCA renderer or overload the daily payload.

New **informational** template `shift_signoff` (no URL button) → **founders** role:
- literal added to `sendTemplate`'s `kind` union (`convex/telegram/send.ts:37`)
- `renderShiftSignoff` in `convex/lib/telegramHtml.ts` (V8-safe), composed from the shared `manualBca` fragment
- routing via founders role (`convex/telegram/config.ts` — `founders` is a `KNOWN_TELEGRAM_ROLES` member ✓)
- flag the manager-ended / `outgoing_uncounted` variants in the header.

## 6. Login-gate changes (`src/routes/login.tsx`, `useSession`, `useLastStaff`, `lock.tsx`)

The login screen forks on booth state (a new query, e.g. `shifts.public.boothState({ deviceId })`):
- **CLOSED** → account picker (full staff list) → on PIN success route into **start-of-day wizard**, not home.
- **LOCKED** → pre-staged single-staff PIN (the locked staff); **no list**; plus the "Manajer buka kunci" affordance → manager-PIN → takeover handover.
- **Handover-incoming** → account picker (excluding the outgoing staff) → on PIN success route into the **incoming count step**, then home.

Resume-as-prev-staff (`useLastStaff`) is **tightened to LOCKED-only** — it must not silently allow account switching outside handover.

New routes (under the authed shell): `/shift/start` (start-of-day wizard), `/shift/end` (End shift choice → close/handover wizards), `/shift/handover` (incoming pick + count). The home top-left keeps 🔒 **Lock** (→ `/lock`) and gains 🏁 **End shift** (→ `/shift/end`).

## 7. Locked decisions (from brainstorm)

1. Mid-day handovers happen → **End shift branches** (Close vs Handover).
2. Handover stock = **outgoing counts + incoming counts (2 writes)**; incoming has no accept screen, just the count input.
3. **Start-of-day = full wizard** (count + physical ticks).
4. **Lock and End shift coexist**; Lock = same-staff resume only.
5. Manager-PIN unlock = **manager-as-incoming takeover** (one code path with C); displaced staff force-ended; their count skipped (manager's is authoritative, `outgoing_uncounted` flag); their summary still fires to Founders flagged manager-ended.
6. **Hours = per-staff span**, anchored to the **shift-start event** → sign-off (NOT a session row — lock fragments sessions, §4).
7. **Staff screen = hours + stock**; **manual-BCA + financials = Founders Telegram only**.
8. **New `pos_shift_events` table** = booth-state source of truth + attendance + step ledger.
9. Renames: wizard A = "Mulai hari / Start of day", wizard B = "Tutup booth / End of day".
10. Copy ships **Bahasa** (staff-facing); EN toggle deferred to #1.

## 8. Open decisions for the plan / staffreview

- **Booth-state query shape** — derive purely from latest `pos_shift_events` row, vs. a denormalized current-state marker on `registered_devices` for a cheaper login-gate read. (Lean: derive; one indexed read.)
- **Step-confirmation granularity** — persist each step's `confirmed_at` (full rail audit) vs. only a terminal "wizard completed" record. (Lean: full rail — it's the visible ledger and cheap.)
- **Hard-gate strength** — can a staffer dismiss the start-of-day wizard to take an emergency sale before counting? (Lean: hard-gate the count; allow the physical ticks to be completed but block home until count submitted.)
- **Supplies check** — pure instruction tick vs. a free-text "what's low" note that posts to the inventory Telegram role. (Lean: tick now; note is a fast-follow.)
- **Manual-BCA tally window** — per-shift span vs. WIB-day to date, for the Founders per-shift summary. (Lean: per-shift span, matching the hours window.)

## 9. Constraints & collision warnings

- **ADR-041 recount reuse** — count steps must call the existing `recordRecount` (staff-allowed `recount` source); do not invent a new stock-write. Each handover = two legitimate recount movements.
- **ADR-005 manager-PIN** — takeover unlock funnels through `verifyManagerPinOrThrow`; never log PIN values.
- **ADR-031 server-time-wins** — all `_at` fields set inside the mutation via `Date.now()`.
- **Idempotency (ADR-013 / rule #20)** — the wizard's state-changing mutations (start-of-day complete, sign-off, handover, takeover) are public mutations → `idempotencyKey` + `withIdempotency` + `authCheck`.
- **`convex/lib/telegramHtml.ts` V8-safe** — no `"use node"` in the renderer.
- **Single active session invariant** — `staff_sessions.by_device_active` [`device_id`,`ended_at`] enforces one active session; handover/takeover end A before starting B (no interleave → never two active or zero).
- **ADR-003 amendment** — lock continues to END the session; this spec layers a `lock`/`resume` booth-state on top (§3E). The new ADR records this amendment (lock-as-session-end preserved; LOCKED is booth-state, not a held session).
- **Founders noise** — per-shift Telegram summaries add traffic on handover-heavy days; the silent/disable-notification pattern (v1.0.1 ticker) is available if it's too loud.
- **i18n** — these screens hardcode Bahasa; when #1 lands, the copy must be extraction-ready (no concatenated strings).
- **Currency** — IDR integer formatting via `src/lib/format.ts` (rule #14), untouched.
