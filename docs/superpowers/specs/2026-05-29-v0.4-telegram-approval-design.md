# v0.4 — Telegram approval graduation + self-registration + founders share — Design Spec

> **Status:** Draft — pre-staffreview.
> **Date:** 2026-05-29
> **Phase:** v0.4 (currently 🗂️ BACKLOG → entering planning)
> **Preceded by:** [v0.3 sale + Xendit](./2026-05-27-v0.3-sale-xendit-design.md) (shipped 29 May 2026, PR #3), [Xendit dedicated-APIs](./2026-05-28-xendit-dedicated-apis-design.md) (ADR-036), [Telegram POC](./2026-05-25-telegram-poc-design.md) (shipped v0.2)
> **Canonical pattern source:** [`convex-telegram-bot-starter`](https://github.com/lucasyhzhu-debug/convex-telegram-bot-starter) — the authoritative Convex+Telegram patterns (role-indirection registry, self-registration, command-registry webhook, cron resilience, chunking). Where the in-repo POC and the starter differ, **the starter wins**; deltas are noted inline.
> **Brainstorm record:** This document captures the 2026-05-29 brainstorming session (superpowers brainstorming skill). Four scoping decisions were made:
> 1. **Kind-set:** land ONE complete graduated off-booth approval kind (`manual_payment_override` — its commit funnel already exists) and invest the bulk of v0.4 in a **generalized approval framework + kind-registry scaffold** so refund/void/discount/stock/spoilage/settings drop in cleanly later.
> 2. **Telegram routing:** adopt the starter's **full self-registration** (role-indirection `telegramChats` registry + `/register` command-registry webhook + manager-gated admin), replacing the single hardcoded `TELEGRAM_CHAT_ID`.
> 3. **Founders trigger:** **cron** (with the starter's resilience wrapper) + a **manager opt-out toggle** (minimal `pos_settings` singleton, default ON per ADR-033).
> 4. **Manager mobile:** **defer `/mgr/home`** dashboard entirely to v0.5. The only manager frontend in v0.4 is the Telegram-chats admin route.

## Goal

Graduate the v0.3 approval scaffolding (one kind, `staff_pin_reset`) into a **production approval framework** that any manager-PIN gate can plug into, and ship the first graduated off-booth kind: managers confirm a payment from anywhere via Telegram, no booth presence required. Replace the single hardcoded Telegram destination with a managed role-routing registry so the bot reliably reaches the right group (managers vs founders) without redeploys or chat-id surgery. Auto-post an end-of-shift summary to the founders' Telegram group on a resilient daily schedule. Harden the POC-grade send path to production (idempotency, typed payloads, message↔record linking, audited failures).

The success test: a booth staff member with no manager present can request a manual payment confirmation, a manager approves it from their phone via a single Telegram link + their PIN, and the sale completes — with the whole exchange auditable and the approval framework shaped so adding the next kind (refund, in v0.5) is a registry entry plus four small wirings, not a re-architecture.

## Scope summary

**Ships in v0.4:**
- Generalized `pos_approval_requests` schema + an `APPROVAL_KINDS` **kind-registry** (the elegant-additions scaffold).
- First graduated off-booth kind: **`manual_payment_override`** (booth path already shipped in v0.3; v0.4 adds the off-booth Telegram path end-to-end).
- A **`denied`** lifecycle state + `_markDenied_internal` funnel; deny affordance on `/approve`.
- Reusable requester-side waiting UX: `useApproval` hook + `ApprovalPending` component (consumed inline by the charge screen).
- **Telegram self-registration** (full starter port): `telegramChats` registry, `getChatIdByRole`, `/register` + `/start` command-registry webhook, roles `"managers"` + `"founders"`, manager-session-gated admin route.
- **Founders shift-summary** via resilient cron + manager opt-out toggle (`pos_settings` singleton).
- **POC→prod hardening:** `withIdempotency`-wrapped `sendTemplate`, per-kind discriminated payload validators (drop `v.any()`), Telegram message ↔ approval-request linking, send failures audited (not `console.warn`), `source` rename `wa_approval` → `telegram_approval`.
- ADR-037 (self-registration & role-indirection); amendments to ADR-030 (generalized approval schema) + ADR-035 (role routing).

**Deferred:**
- Refund / void / ad-hoc discount / stock-adjustment / spoilage / settings approval KINDS → v0.5 (refund) and beyond. v0.4 proves the framework with `manual_payment_override` only; these kinds have no booth-side action yet, so an approval would approve nothing.
- `/mgr/home` mobile manager dashboard (live tape) → v0.5.
- Full `pos_settings` module (v0.4 ships a minimal singleton with one field) → v0.5.
- Production Telegram cutover (separate prod bot/group/env on `savory-zebra-800`, `setWebhook`) → v1.0 prod cutover.
- Inline approve/deny *from the manager mobile screen* → there is no manager mobile screen in v0.4; approval is via the Telegram link.

### Module dependency graph (one-way, no cycles)

```
   approvals ──► auth (via _changePinCommit_internal; pin_reset kind)
   approvals ──► payments (via _onPaidManual_internal; manual_payment_override kind)
   approvals ──► telegram (send.sendTemplate; role-routed)
   payments  ──► transactions ──► inventory
   settings  ──► (none; read by the founders cron action)
   crons ──► telegram.send (founders summary, role "founders")
   telegram, audit, idempotency  =  foundational allow-list (importable anywhere, ADR-034/035)
```

**Cross-module dispatch stays scheduler-based** (the v0.3 pattern): `payments` fires `ctx.scheduler.runAfter(0, internal.approvals.actions.requestManualPaymentApproval, …)` rather than importing across the boundary — the module-boundary lint catches DB access, not scheduler dispatch. The two transactions are independent, breaking any logical cycle at runtime.

**Telegram is foundational infra (ADR-035):** kept at `convex/telegram/` (shared), NOT nested under `convex/approvals/telegram/`. *Delta vs ADR-035, which anticipated the nested path:* the starter treats Telegram as a shared registry that multiple feeds (approvals AND the founders cron) route through, so a shared location is correct. ADR-035 is amended accordingly.

## A. Approval framework generalization — the scaffold

The v0.3 row is purpose-built for `staff_pin_reset`: `subject_staff_id: v.id("staff")`, `kind` a single-literal union, no requester, no entity, no reason, no `denied` state. v0.4 generalizes it **once**, additively, and mechanizes the four-touchpoint pattern (CLAUDE.md §"How to add a feature" #8) into a registry.

### A.1 Generalized `pos_approval_requests` schema

```ts
pos_approval_requests: defineTable({
  kind: v.union(
    v.literal("staff_pin_reset"),
    v.literal("manual_payment_override"),   // NEW in v0.4
  ),

  // WHO asked (optional — pin_reset is system-triggered, no human requester)
  requester_staff_id: v.optional(v.id("staff")),

  // WHAT is being approved — generic entity pointer (ADR-030 shape)
  entity_type: v.optional(v.string()),       // e.g. "pos_transactions"
  entity_id: v.optional(v.string()),         // stringified id (kept generic per ADR-030)
  // back-compat: pin_reset keeps using subject_staff_id (now optional)
  subject_staff_id: v.optional(v.id("staff")),

  // Per-kind validated context (NO v.any()). Discriminated on `kind` in app code
  // via APPROVAL_KINDS[kind].contextValidator. Carries display + commit inputs:
  //   manual_payment_override → { txn_id, amount_idr, receipt_preview?, reason }
  //   staff_pin_reset         → { } (subject_staff_id carries everything)
  context: v.optional(v.any()),              // SCHEMA-level v.any() is unavoidable for a
                                             // shared column; the GUARANTEE is that every
                                             // WRITER validates via the kind registry first.
                                             // (Documented invariant — staffreview seed #1.)
  reason: v.optional(v.string()),

  triggered_by_event: v.string(),
  triggered_at: v.number(),

  // Token (authorises VIEW per ADR-029) — unchanged from v0.3
  token_hash: v.string(),
  token_expires_at: v.number(),

  // Lifecycle — adds "denied"
  status: v.union(
    v.literal("pending"),
    v.literal("resolved"),
    v.literal("denied"),                     // NEW
    v.literal("expired"),
  ),
  notified_at: v.optional(v.number()),
  resolved_at: v.optional(v.number()),
  resolved_by_manager_id: v.optional(v.id("staff")),
  denied_at: v.optional(v.number()),         // NEW
  denied_by_manager_id: v.optional(v.id("staff")),  // NEW
  deny_reason: v.optional(v.string()),       // NEW

  // Telegram linkage (POC→prod hardening §E)
  notification_channel: v.optional(v.literal("telegram")),
  telegram_message_id: v.optional(v.number()),  // best-effort, patched AFTER notify
  telegram_chat_id: v.optional(v.string()),
  // NO audit_log_id column (staffreview Improvement #2): logAudit returns void
  // (audit/internal.ts:38), so a stored FK would mean editing the foundational
  // append-only helper. ADR-030 "linked back" is satisfied by the shipped code:
  //   audit → approval : audit_log.metadata.approval_request_id (logAudit accepts metadata)
  //   approval → audit : the by_entity index on (entity_type, entity_id)
})
  .index("by_token_hash", ["token_hash"])
  .index("by_status_triggered", ["status", "triggered_at"])
  .index("by_subject_staff", ["subject_staff_id"])
  .index("by_kind_status", ["kind", "status"]),   // NEW — _listPendingByKind_internal + dedup guards
```

> **Decision: discriminated `context` + kind-registry** (vs per-kind tables / fully-generic payload).
> - *Per-kind tables* — rejected: forks the unified `/approve` flow, the token model, and the lifecycle funnel per kind; the scaffold goal is the opposite.
> - *Fully-generic `v.any()` everywhere* — rejected: it's exactly the anti-pattern §E removes from `sendTemplate`. The compromise is honest: the schema column is `v.any()` (Convex can't express a per-row discriminated union as a stored type), but **every writer must pass through `APPROVAL_KINDS[kind].contextValidator`** before insert. That invariant is enforced in `_createRequest_internal` and is a staffreview seed.

### A.2 The `APPROVAL_KINDS` registry (`convex/approvals/kinds.ts`)

A single source-of-truth map. Each entry declares the four touchpoints so adding a kind is local:

```ts
type ApprovalKindConfig = {
  contextValidator: (raw: unknown) => KindContext;   // (1) validate context before insert
  telegramTemplateKind: TemplateKind;                // (2) which send.ts template + renderer
  approveRouteVariant: "pin_reset" | "manual_payment";// (3) which /approve UI variant renders
  commit: CommitRef;                                 // (4) internal funnel the approve action calls
  audit: { requested: string; resolved: string; denied: string };  // audit action strings
};

export const APPROVAL_KINDS: Record<ApprovalKind, ApprovalKindConfig> = {
  staff_pin_reset:        { /* existing behavior, refactored behind the registry */ },
  manual_payment_override:{ /* new — see §B */ },
};
```

**`_createRequest_internal` generalization:** its args widen to accept `kind`, `requester_staff_id?`, `entity_type?`, `entity_id?`, `subject_staff_id?`, `context?`, `reason?`. It calls `APPROVAL_KINDS[kind].contextValidator(context)` and refuses to insert an unvalidated/mismatched context. The audit row uses `APPROVAL_KINDS[kind].audit.requested`. The existing `staff_pin_reset` callers are migrated to pass `subject_staff_id` + empty context — behavior-preserving.

`_markResolved_internal` / `_markDenied_internal` are kind-agnostic — they patch lifecycle + write the kind's audit action; they do **not** run the commit. The **action layer** (per kind) runs the commit funnel *then* calls `_markResolved_internal` in the same idempotency envelope (the v0.3 I6 pattern).

### A.3 `getByToken` generalization

Today it returns staff-pin-reset display fields. v0.4 returns a discriminated shape: `{ kind, status, token_expires_at, display }` where `display` is built per kind (pin_reset → subject staff name/code; manual_payment → amount + receipt preview + reason). The `/approve` page switches UI variant on `kind` (§A.2 touchpoint 3). Cross-module reads (staff name, txn summary) go via the owning module's internal query, respecting ADR-034.

## B. `manual_payment_override` — first graduated off-booth kind

### B.1 Why this kind first
Its commit funnel **already exists and is tested**: `payments/actions.manuallyConfirmPayment` → `_onPaidManual_internal` → `_confirmPaid_internal` (source `manual`). v0.3 ships the *booth-inline* path (manager present, enters PIN on-device). v0.4 adds the *off-booth* path through the approval framework. Nothing else in the candidate kind-set has a funnel yet, so this is the only end-to-end-testable graduation — exactly what proves the scaffold.

### B.2 Flow

```
[charge screen, payment not auto-confirmed, no manager at booth]
  └─ staff taps "Request manager approval", enters reason
     └─ approvals.requestManualPaymentApproval(sessionId, txnId, reason, idempotencyKey)
        • resolves requester (booth staff) from session
        • mints token (32B URL-safe; only sha256 hash persisted — ADR-029)
        • _createRequest_internal(kind=manual_payment_override,
            requester_staff_id, entity_type="pos_transactions", entity_id=txnId,
            context={txn_id, amount_idr, receipt_preview, reason},
            token_hash, token_expires_at = now+60min)
        • telegram.send.sendTemplate(kind=manual_payment_override, role="managers",
            payload={amount, reason, staff_name, approve_url=/approve/:rawToken})
            → URL-button card (NOT a callback button — PIN can't ride a callback)
        • _markNotified_internal  (or _deleteRequest_internal + rethrow on send failure — v0.3 recovery pattern)
        • _linkTelegramMessage_internal(requestId, message_id, chat_id) — BEST-EFFORT,
            ordered AFTER notify; a failed link patch does NOT fail the request
            (the notification already went out). (staffreview Improvement #3)
  └─ charge screen shows <ApprovalPending requestId> (reactive on getRequestStatus)

[manager, anywhere, taps the Telegram link]
  └─ GET /approve/:token  → getByToken → kind=manual_payment_override variant
     renders: amount, reason, requester, [enter manager PIN] [Approve] [Deny]
  └─ Approve: approvals.approveManualPayment(token, managerStaffCode, managerPin, idempotencyKey)
        • token sha256 + constant-time compare; status===pending; not expired
        • resolve manager by code; argon2-verify manager PIN (lockout NOT consulted — ADR-029 §self-approve)
        • runMutation _onPaidManual_internal(txnId, reason, mgr_approver_id)   ← existing funnel
        • _markResolved_internal(requestId, resolved_by_manager_id, idempotencyKey)  ← same envelope (I6)
  └─ Deny: approvals.denyRequest(token, managerStaffCode, managerPin, denyReason, idempotencyKey)
        • verifies manager PIN → _markDenied_internal
  └─ charge screen flips to paid (resolved) / "declined by <mgr>" (denied), reactively
```

### B.3 Edge cases
- **Cart edited / superseded after request:** the request is bound to `entity_id=txnId`; if the txn is no longer `awaiting_payment` when the manager approves, `_onPaidManual_internal`'s existing C4 guard rejects the false-success and the approve action surfaces a clear error; the request is left `pending` to expire (or staff cancels). *(staffreview seed #2 — confirm the desired terminal state.)*
- **Dedup:** `_listPendingByKind_internal(kind, entity_id)` prevents minting a second live request for the same txn while one is pending (mirrors the v0.3 `_listPendingForStaff_internal` dedup guard).
- **Token reuse across kinds:** none — one token per request row, single-use, 60-min TTL (ADR-028/029) unchanged.
- **Locked-out manager self-approve:** intentional and unchanged from v0.3 — token + correct PIN are sufficient authority; lockout state is not consulted on the approve path.

### B.4 Requester waiting UX (reusable)
- `useApproval(requestId)` hook — subscribes to `approvals.public.getRequestStatus`, returns `"pending" | "resolved" | "denied" | "expired"` + display.
- `<ApprovalPending>` component — spinner + "waiting for a manager to approve in Telegram", flips on resolution. Rendered **inline on the existing charge screen** (no new route). *Supersedes the board's `routes/wait/[requestId].tsx`.* Both are kind-agnostic so v0.5 refund reuses them.

## C. Telegram self-registration (full starter port)

### C.1 What ports (≈ verbatim from the starter)
- **`telegramChats` table** + `by_chatId` and the compound `by_role_archived` index (LESSON 1 + 13 — never a bare `by_role` with a post-scan `archivedAt === undefined` filter).
- **`chatRegistry.ts`** — `registerChat`, `assignRole`, `archiveChat` (clears role atomically — LESSON 12), `restoreChat`, `listChats`, `getChatIdByRole` (active-row → env-fallback → throw), `touchChatLastSeen` (update-only, skips archived). Each as an `…Impl` core.
- **`registryCommands.ts`** — `buildRegistryCommands(scheduler)` → `/register`, `/start`.
- **`commands.ts`** — `CommandRegistration`, `MessageContext` (LESSON 15: one ctx object, zero-arg dispatch still assignable).
- **`webhook.ts`** — `buildHandleTelegramWebhook(commandsFactory, { trackLastSeen: true })`: secret-token verify (constant-time, LESSON via `constantTimeEqual.ts`), parse, match command, **always-200 after any dedupe commit** (LESSON 2), non-command text → `touchChatLastSeen`.
- **`lib/`** ports: `chunking.ts`, `constantTimeEqual.ts`, `cronRetry.ts`, `dateAnchors.ts`, and `sendTelegramHtml` added to `lib/telegramHtml.ts`.

### C.2 What Frollie adapts (the embed points)
- **`config.ts`:** `KNOWN_TELEGRAM_ROLES = ["managers", "founders"] as const`. `TELEGRAM_ADMIN_URL` → the Frollie manager admin route. `TELEGRAM_BOT_USERNAME` from env.
- **Admin auth:** the starter's `requireAdminKey`/`ADMIN_KEY` shim is **replaced** by the EXISTING `requireManagerSession` (`convex/auth/sessions.ts:24`, already used by `staff/public.ts:32`, `audit/public.ts:14`) — LESSON 14 ("when you embed this in an app with real users, replace `requireAdminKey` with your auth"). `ADMIN_KEY` is never introduced to this repo. The `admin*` twins become `mgr*` functions in `telegram/mgrAdmin.ts` gated by a manager session; the `internal*` twins (dashboard / `npx convex run`) port unchanged.
- **`http.ts`:** `/telegram-webhook` → `buildHandleTelegramWebhook((scheduler) => [...buildRegistryCommands(scheduler)], { trackLastSeen: true })`. (No feature `/commands` beyond register/start in v0.4 — approvals use URL links, not chat commands.)
- **Migrate existing sends to roles:** `notifyStaffLockout` (PIN reset) → role `"managers"`. The single `TELEGRAM_CHAT_ID` becomes the `TELEGRAM_FALLBACK_ROLE`/`TELEGRAM_CHAT_ID` migration shim (starter step-2 fallback) so the dev group keeps working during the transition, then is retired once `"managers"` is bound.

### C.3 Manager admin surface (`src/routes/mgr/telegram-chats.tsx`)
The **only** manager frontend in v0.4 (distinct from the deferred `/mgr/home` dashboard). Manager-session-gated. Lists registered chats, assigns/reassigns/archives roles, test-send. Mirrors the starter's React admin (`TelegramChatsManager.tsx`) but wired to the `mgr*` session-gated functions. Mobile-friendly (single device), shadcn/Tailwind tokens, pragmatic — no animation polish.

### C.4 Operator flow (RUNBOOK delta)
Add bot to a group → group sends `/register@<bot>` → row appears in admin → manager assigns role `managers` / `founders`. Privacy-mode command registration via BotFather `/setcommands` (LESSON 6) documented. Supergroup-migration recovery (LESSON 9) and negative-chat-id `env set key=value` (LESSON 8) folded into `docs/RUNBOOK-telegram.md`.

## D. Founders shift-summary

### D.1 Trigger
A daily Convex cron (`crons.ts`) at booth-close WIB (configurable; default e.g. 22:00 WIB = 15:00 UTC). No Lock-screen dependency (Lock is v0.5). The cron points at a **`*Resilient` wrapper** `internalAction` (`sendFoundersSummaryResilient`) per the starter's `cronRetry.ts`: on a *transient* Convex-overload error before the send loop it self-reschedules (60s/120s, ≤3 attempts); anything else rethrows to surface in the cron dashboard (LESSON 10). The raw `sendFoundersSummary` stays callable on-demand (admin test-send / `npx convex run`).

### D.2 Content & aggregation
`sendFoundersSummary` reads today's WIB window (via ported `dateAnchors.ts`) over `pos_transactions` (paid only): total sales (integer rupiah), txn count, top SKUs, flagged-txn count (`NEG_STOCK`, payment-mismatch). Rendered via a `renderFoundersSummary` template (extends the existing `renderShiftSummary` shape — but keyed on the day, not a single staff "shift", since v1 has no shift open/close per ADR-006). Routed to role `"founders"` via `getChatIdByRole`. Chunked through `chunking.ts` (LESSON 3) in case the SKU list grows.

### D.3 Opt-out toggle
A minimal **`pos_settings` singleton** (new `settings/` module): one field `founders_summary_enabled: boolean`, default **ON** (ADR-033). `sendFoundersSummary` reads it via `_getSettings_internal` and no-ops (no send, audit a `skipped` row) when disabled. A manager toggles it via `settings.public.setFoundersSummaryEnabled` (manager-session-gated), surfaced as a switch on the Telegram-chats admin route.
> *Alternative considered:* a `summaryEnabled` flag on the founders `telegramChats` binding row. Rejected to keep the ported registry table **starter-faithful** (Frollie config lives in Frollie tables, not in the generic registry). The singleton is the semantically-correct home and v0.5's full settings work extends it. *(staffreview seed #3 — confirm singleton vs binding-flag.)*

## E. POC → production hardening

| # | POC state (v0.2/v0.3) | v0.4 hardening |
|---|------------------------|----------------|
| 1 | `sendTemplate` has no idempotency | wrap in `withIdempotency` (ADR-013) keyed on `(kind, request_id)` so a scheduler re-fire / retry doesn't double-post |
| 2 | `payload: v.any()` | per-kind discriminated validators; `sendTemplate` args become a `kind`-tagged union; renderers receive typed payloads |
| 3 | `telegram_log` is a sandbox table, unlinked | `_linkTelegramMessage_internal` patches `telegram_message_id` + `telegram_chat_id` onto the request row (best-effort, after notify — §B.2); ADR-030 linkage via `metadata.approval_request_id` + `by_entity`, **no `audit_log_id` column** (Improvement #2); repurpose `telegram_log` to debug-trail only (or retire) |
| 4 | `editMessageText`/send failures `console.warn` | surface as `audit_log` rows (`telegram.send_failed`) so a nightly query can alert (PROGRESS "Telegram bot SPOF" risk) |
| 5 | webhook dedupe-then-maybe-throw | always-200 after dedupe commit (LESSON 2) — already partially true; the rewritten registry webhook formalizes it |
