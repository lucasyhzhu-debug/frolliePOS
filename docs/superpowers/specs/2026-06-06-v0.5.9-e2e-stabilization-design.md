## 1. Background & motivation

Issue #44 ("e2e session-on-hard-nav: signedIn fixture spec lands on /login on second `page.goto`") has been the subject of four planning/implementation cycles, all built on a misdiagnosis:

| PR / Plan | Hypothesis | Status |
|---|---|---|
| PR #41 | Convex client warm-up needed → added 1500ms sleep in `e2e/fixtures.ts:37-41` | Shipped, not actually addressing the bug |
| PR #43 | session-loss-on-hard-nav → `test.skip` 6 PIN-gated specs | Shipped, never traced |
| v0.5.7.1 Option A (PR #45) | 1500ms debounce in `useSession` | Replanned as Option B before execute |
| v0.5.7.1 Option B (PR #47) | Evidence-based null trust via `useRef` in `useSession` | **REFUTED** by instrumented CI on PR #48 |
| PR #48 (pivot) | a11y aria-label drift on catalog "Add" buttons | Partial fix shipped to draft; 6 specs still red on CI |

**Evidence that refuted Options A and B:** PR #48 ran instrumentation (`console.warn` + sessionStorage ring buffer + Playwright `page.on("console")` bridge) across `auth.spec.ts` and `sale-qris.spec.ts` for 3 retries, 17+ captured renders. `validation === null` with `stored=Y` **never appears** post-login. The transient on hard-nav is `validation === undefined`, which already maps to `status: "loading"` (current `src/hooks/useSession.ts:58`) and does NOT wipe localStorage. The `isDead` effect (`useSession.ts:49-55`) is unreachable in the scenarios the SKIP comments cited.

**What is actually broken:**
1. The catalog UI's `aria-label="Add ${p.name}"` (`src/routes/sale/index.tsx:183`, pre-PR-#48) omitted `pack_label`, so three Dubai SKUs (`1 pc`, `3 pcs`, `8 pcs`) rendered three buttons all named "Add Dubai". E2e specs' `/Dubai 1pc/i` selector never matched. PR #48 partially fixed this (`Add ${p.name} ${p.pack_label}`) but did not handle the empty-`pack_label` edge or sweep similar gaps elsewhere.
2. Charge screen's payment method picker uses Radix `TabsTrigger` (`src/routes/sale/charge.tsx:494-497`), which renders as `role="tab"`. E2e specs use `getByRole("button", { name: /QRIS|BCA/i })` — wrong role. Affects `refund`, `sale-qris`, `sale-bca-va`.
3. Spoilage form's `Qty` Label (`src/routes/mgr/spoilage.tsx:264`) has no `htmlFor` and its sibling `Input` has no `id`. Playwright's `getByLabel(/Qty/i)` cannot resolve. Same pattern in `vouchers.tsx:599` for the `Type` Label. Affects `spoilage`, `voucher-offline`, `voucher-online`.
4. `voucher-offline.spec.ts` has `<TBD>` placeholders inside `execSync(...)` calls, wrapped in a silent `catch` and a `Promise.race(...).catch(() => {})` — un-skipping this spec produces a false-green (it reports passing while no assertion runs). Pre-existing in the spec body, surfaced by un-skip.
5. Two planning artifacts (`docs/superpowers/plans/2026-06-05-issue-44-usesession-transient-null-fix.md`, `docs/reviews/staffreview-issue-44-architectural-options-2026-06-05.md`) sit on `main` recommending the now-refuted Option B. Future readers will land on them and take their conclusions as live.

**What enabled the misdiagnosis chain:** the original PR #41 + PR #43 author never instrumented to confirm the inferred mechanism. Two architectural reviews (Option A → Option B replan) compared four ways to fix a bug nobody had observed. The user's memory bullet "always distinguish root cause from symptom" applies, but the discipline lives at the prose level — there's no review-template gate that forces evidence citation before a plan can be approved.

## 7. Risks & open questions

### Q1 — voucher-offline.spec's `<TBD>` tokens
The spec body has `execSync("npx convex run inventory:adjust { sessionId: <TBD>... }")` inside silent catch blocks, predating issue #44. Fixing the tokens requires sourcing real IDs from `seed/actions:reset` output. Two options for Slice 1.C:
- **Attempt fix** (cleaner ship): trace seed output, inject real IDs, verify locally. If the fix shape is non-trivial (more than ~5 lines), fall back to honest re-skip.
- **Honest re-skip** (safer ship): leave the `<TBD>` tokens in place; replace `test(` → `test.skip(`; replace the misdiagnosed SKIP block with a true one citing "voucher-offline spec body has unresolved `<TBD>` tokens, blocked on seed-ID sourcing — see #<n>".

**Default proposal**: attempt the fix; fall back to honest re-skip if not ≤ 5 lines.

### Q2 — Slice 2 audit hit-bound
If the audit finds >5 gaps, fix all 5 + file follow-up vs fix all of them in this phase? Default proposal: scope-cap at "blocks an e2e selector OR matches the same htmlFor pattern"; everything else gets a follow-up issue. Plan will name a concrete budget (e.g., max 10 files modified across Slice 2).

### Q3 — Postmortem dir vs `docs/reviews/`
Reviews are pre-merge artifacts. Postmortems are post-incident retrospectives. Different genres; new dir justified. Risk: dir proliferation. Mitigation: index README explicitly contrasts the two genres.

### Q4 — Staffreview skill insertion point (Slice 3.A)
The 557-line skill needs a thoughtful insertion location. Plan-stage decision: read the skill, propose insertion point (likely adjacent to existing "Findings format" or "Spec review" section). Plan will name the exact section.

### Q5 — Slice 3 in this PR vs separate
The personal-global skill change is cross-project. Risk: if the change breaks any other project's staffreview invocation, that's a regression with blast-radius outside Frollie. Default proposal: include in this PR (one coherent ship), but call out in PR body that the skill change touches a non-Frollie file. Alternative (cleaner blast-radius): separate `staffreview-skill` PR; deferred from this phase.
