# v0.7 — Xendit settlement reconciliation (design spec)

**Date:** 2026-06-08
**Status:** Draft (pre-staffreview)
**Origin:** The last load-bearing feature gap before v1.0. Closes the "Xendit
settlement timing" risk under watch (`docs/PROGRESS.md` §Risks) and the long-open
`settlements.ts` / `pos_settlements` / `/settlements` items carried since the
v0.5.3 omnibus. Design is pre-grounded by
[`docs/xendit-reference/settlement-reconciliation.md`](../../xendit-reference/settlement-reconciliation.md)
(researched 2026-06-02 for exactly this phase) and [ADR-012](../../ADR/012-settlements-visible-to-staff-and-managers.md).

> **Manual-first, auto-poll-gated.** Xendit runs on TEST keys until KYB clears,
> and TEST keys produce no real settlements — so the auto-poll parser **cannot be
> live-verified yet**. This is the same constraint [ADR-036](../../ADR/036-xendit-dedicated-apis-inline.md)
> Decision C hit for BCA VA. We therefore ship a **verifiable manual-entry path**
> as the launch primary, and build the nightly auto-poll **shape-tested only**,
> with live-verification filed as a KYB-gated follow-up.

---

## 1. Motivation

Staff and managers have no way to see whether the QRIS/BCA-VA sales they collected
actually landed in the booth's BCA account. Per [ADR-012](../../ADR/012-settlements-visible-to-staff-and-managers.md),
this is a trust requirement, not an admin nicety: staff ask "did Friday's sales
arrive on Saturday?" every week. Today the answer is only in the Xendit dashboard,
invisible inside the POS — which is supposed to be the booth's financial record.

The 2026-06-02 research established the load-bearing facts (full citations in the
knowledge base):

1. **There is NO settlement webhook.** Xendit never pushes a callback when
   collected funds are paid out to your bank. Settlement is knowable only by
   **polling the List Transactions API** (`GET /transactions`) or downloading
   reports.
2. **Settlement is per-transaction, not per-batch.** Each transaction carries a
   `settlement_status` (`PENDING` / `SETTLED` / `EARLY_SETTLED` / `null`), a
   `settlement_date`, and a `fee` (the Xendit MDR). There is **no "settlement
   object"** with its own id. A payout day = the aggregate of SETTLED transactions
   sharing a `settlement_date`. `net = amount − fee`.
3. **The join key already exists.** `pos_xendit_invoices.receipt_id` (bank RRN) and
   our `reference_id` (`pos-${txnId}`) were captured at payment time precisely as
   the settlement-report join key (ADR-036 consequence).
4. **TEST-mode caveat.** Until KYB clears, real payouts/settlements do not occur on
   TEST keys, so settlement figures cannot be live-verified — mirroring ADR-036
   Decision C for BCA VA.
5. **The List Transactions field shapes are asserted-not-verified.** The doc fetcher
   intermittently 404'd on the `apidocs/list-transactions` page during research;
   exact query-param names, pagination cursor, and the response field list were not
   fully confirmed. They MUST be confirmed against a real `GET /transactions`
   response (or the OpenAPI at `https://docs.xendit.co/llms.txt`) before the parser
   is trusted — exactly as ADR-036 treated the FVA callback shape.

These facts force the architecture: **nightly poll + aggregate-by-date**, not a
webhook; and a **manual fallback** so `/settlements` is useful at launch despite
the unverifiable auto path.

## 2. Goals / Non-goals

**Goals**
- G1: `/settlements` shows per-day settlement rows (date, gross, MDR, net, txn
  count, source, BCA last-4) to **any authenticated session** — staff and
  managers both (ADR-012).
- G2: A manager can **enter a settlement day manually** (PIN-gated, audited) from
  figures read off the Xendit dashboard — the verified, launch-day path.
- G3: A nightly cron polls `GET /transactions`, aggregates SETTLED rows by
  `settlement_date`, and **upserts** `pos_settlements` rows — built and
  shape-tested, with live-verification gated behind KYB.
- G4: The `GET /transactions` response parser and the date-aggregator are **pure,
  isolated, golden-tested** functions, so a wrong-field-shape surprise is contained
  to one unit and discovered by test, not in production.
- G5: A missed cron run **self-heals** via a multi-day lookback + idempotent upsert
  (no duplicate day rows, late-settling transactions backfill correctly).
- G6: Correct the documented `pos_settlements` schema to the verified
  per-transaction-aggregate model; amend ADR-012 + SCHEMA.md to match.

**Non-goals**
- N1: **Per-transaction match-back** (mapping each settled Xendit row to a specific
  `pos_transaction`) — deferred to the v1.0 "Settlement reconciliation polish" item.
  We store the raw aggregated rows in `payload` so match-back can be added later
  with no migration.
- N2: **Variance detection / alerts** ("collected X but only Y settled") — same
  v1.0 polish item, not this phase.
- N3: **Live verification** of the auto-poll path against real settled funds —
  KYB-gated follow-up, filed at PR-open (parallels ADR-036 Decision C).
- N4: **Frollie Pro sync** of settlement data — `synced_to_frollie_pro_at` stays a
  dormant v1.1 hook; no cross-deployment work here.
- N5: No change to the payment/refund paths, or to `pos_refunds.settlement_status`
  (the merchant→customer ledger — orthogonal to `pos_settlements`, despite the
  shared word; see §4.4 naming guard).

## 3. Schema correction (amends ADR-012 + SCHEMA.md)

The documented `pos_settlements` (SCHEMA.md:545, written 2026-05-21) predates the
2026-06-02 research and assumes a Xendit "settlement object" with an
`xendit_settlement_id`. **That object does not exist.** A `pos_settlements` row is
*our own daily aggregate*, keyed by date, not a mirror of a Xendit entity. The row
can also originate from a manager's manual entry, so it needs a provenance
discriminant. Corrected shape:

| Field | Type | Notes |
|---|---|---|
| `_id` | `Id<"pos_settlements">` | |
| `settlement_key` | `string` | **Replaces `xendit_settlement_id`.** Synthesized `settle-YYYY-MM-DD` (WIB date). The unique upsert key — one row per settlement day. |
| `settlement_date` | `string` | ISO date (`YYYY-MM-DD`, WIB). |
| `gross_amount` | `number` | Integer rupiah (ADR-015). Sum of settled txn amounts for the date. |
| `mdr_amount` | `number` | Sum of Xendit `fee` for the date. |
| `net_amount` | `number` | `gross_amount − mdr_amount` — what hits BCA. |
| `transaction_count` | `number` | Settled txns aggregated into this day. |
| `source` | `"xendit_poll" \| "manual"` | **New.** Provenance: nightly cron vs manager hand-entry. |
| `entered_by` | `Id<"staff">?` | **New.** Set for `source="manual"` rows (the entering manager); absent for poll rows. |
| `last_synced_at` | `number?` | **New.** Server-set on each poll upsert for this date; drives self-heal observability. Absent for manual rows. |
| `bca_account_destination` | `string?` | **Now optional.** Last 4 digits for verification (ADR-012). Manual entry fills it; auto-poll may leave it `null` if `GET /transactions` rows don't carry the destination. |
| `payload` | `string?` | Raw aggregated Xendit rows as JSON (poll) — debug + future match-back (N1). Optional/absent for manual rows. |
| `synced_to_frollie_pro_at` | `number?` | Dormant v1.1 hook (unchanged). |
| `created_at` | `number` | Server-set (ADR-031). |

**Indexes:** `by_settlement_date` on `settlement_date`; `by_settlement_key` on
`settlement_key` (**unique** — enforces one-row-per-day for the upsert).

**Conflict-resolution between sources.** If a manager manually entered a day and the
poll later produces real figures for the same date, the **poll wins** (it's the
authoritative source once KYB clears) — the upsert overwrites the manual row's
amounts and flips `source` to `"xendit_poll"`, preserving `created_at`. This is
audited as `settlement.poll_superseded_manual`. Before KYB, polls produce nothing,
so manual rows stand. (Stated explicitly so the upsert is not silently lossy.)

ADR-012 is amended: drop the "settlement object" framing and the speculative
"settlement webhook" wording; document the verified per-transaction-aggregate model
+ the manual/poll dual-source. The read-access decision (staff + managers, last-4
only) is **unchanged**.

## 4. Components (new `convex/settlements/` module, ADR-034)

A new deep module. Module-boundary lint already forbids cross-module `db` access, so
all reads of `pos_xendit_invoices` / `pos_transactions` for future match-back go via
`_internal` helpers in their owning modules (N1 keeps this out of scope for now).

```
convex/settlements/
  schema.ts        pos_settlements table fragment (§3) → composed into root schema
  lib.ts           PURE, V8-safe: parseListTransactions + aggregateSettledByDate
  internal.ts      _upsertSettlementDay_internal (single writer), audit helpers
  actions.ts       "use node": enterSettlementManually (PIN-gated)  [+ syncSettlements lives in cronActions]
  cronActions.ts   V8: syncSettlements + syncSettlementsResilient (cron entry)
  public.ts        listSettlements (role-agnostic read)
  __tests__/       lib.test.ts (golden), manualEntry.test.ts, sync.test.ts, upsert.test.ts
```

### 4.1 `lib.ts` — pure aggregation + parse (the de-risk surface, must be V8-safe)

Two pure functions, no `ctx`, no Node — testable in isolation, and the *only* place
the asserted-not-verified Xendit shape is interpreted:

```ts
// Narrow shape we depend on; tolerant of extra fields.
export type XenditTxnRow = {
  reference_id: string;
  settlement_status: "PENDING" | "SETTLED" | "EARLY_SETTLED" | null;
  settlement_date: string | null;   // ISO date
  amount: number;
  fee: number;
  // ...other fields ignored
};

/** Parse a raw GET /transactions response body into XenditTxnRow[].
 *  Isolated so a wrong field path is a localized, golden-tested failure. */
export function parseListTransactions(body: unknown): XenditTxnRow[];

/** Aggregate SETTLED + EARLY_SETTLED rows by settlement_date into daily totals.
 *  Filters out PENDING and null-settlement rows. Pure; deterministic. */
export function aggregateSettledByDate(rows: XenditTxnRow[]): Array<{
  settlement_date: string;
  gross_amount: number;
  mdr_amount: number;
  net_amount: number;
  transaction_count: number;
}>;
```

Both treated as **asserted-not-verified** until a real response confirms field
paths (G4, R1). `parseListTransactions` MUST throw a clear, identifiable error on a
shape it doesn't recognize rather than silently returning `[]` — a silent empty
parse would look identical to "no settlements" and mask the bug (memory:
`v051a-receipts` lesson — silent stubs throw, not return).

### 4.2 `payments/xendit.ts` — add the List Transactions reader

The single fetch surface (the adapter is already the only Node/HTTP boundary for
Xendit). Add `listTransactions({ settledAfter, settledBefore })` building the
authenticated request with the same key/header discipline as `buildQrisHeaders`
(asserted by a header test, per the ADR-036 pattern). Pagination cursor handled here
and confirmed against a real response before trust (R1). The action calls this and
passes the raw body straight into `parseListTransactions`.

### 4.3 `internal.ts` — `_upsertSettlementDay_internal` (single writer)

The **one writer** for `pos_settlements`, used by both the cron and manual entry
(single-writer invariant — memory: `v055-sku-admin`, multi-writer drift → canonical
insert helper). Looks up the existing row by `settlement_key`; inserts or patches
per the §3 conflict-resolution rule; sets server time (ADR-031); emits the audit
verb. Both callers pass `source` + the day totals; only the manual path passes
`entered_by`.

### 4.4 `actions.ts` — `enterSettlementManually` (PIN-gated, the verified path)

Per business-rule #22, manual settlement entry is **money bookkeeping → manager-PIN**
(not manager-session). Shape mirrors the existing PIN-gated admin actions
(`recordSpoilage` / `createVoucher`):

```ts
enterSettlementManually(
  sessionId, settlementDate, grossAmount, mdrAmount, transactionCount,
  bcaAccountLast4, managerPin, idempotencyKey,
)
```

- `withActionCache` + **required `authCheck`** = `assertManagerSessionInAction`
  (pre-cache auth, ADR-046 — the v0.6.1 invariant).
- `verifyManagerPinOrThrow` inside the cached body.
- `net = gross − mdr` computed server-side (never client-supplied).
- Validates integer rupiah, `net ≥ 0`, valid ISO date, last-4 is 4 digits.
- Calls `_upsertSettlementDay_internal` with `source: "manual"`, `entered_by`.
- Audited `settlement.entered`.

> **Naming guard.** `pos_settlements` (Xendit → merchant payout) is a *different*
> ledger from `pos_refunds.settlement_status` (merchant → customer refund money-back,
> ADR-038). Same word, orthogonal concepts — do not cross the audit verbs or the
> tables.

### 4.5 `cronActions.ts` — `syncSettlements` + `syncSettlementsResilient`

Copied structurally from the `stock-recon` resilient-cron pattern
(`convex/inventory/cronActions.ts`) — V8 file (no `"use node"`; only
`ctx.runQuery/runMutation/runAction`), inner worker + resilient wrapper using the
shared `cronRetry` policy (linear back-off, `RESILIENT_MAX_ATTEMPTS`, audited skip
on non-transient, no retry storm).

`syncSettlements` flow:
1. Compute a **multi-day lookback window** (e.g. last 7 days, WIB) — `settledAfter`.
2. `ctx.runAction` → `payments/xendit.listTransactions(window)`.
3. `parseListTransactions` → `aggregateSettledByDate` (pure).
4. For each day: `_upsertSettlementDay_internal({ source: "xendit_poll", ... })`.
5. Audited summary skip when the window yields zero settled rows (reason
   `"no_settlements"`) — expected on TEST keys pre-KYB, so this is the normal path
   today, not an error.

Cron registered in `convex/crons.ts` at a pre-dawn WIB slot — **03:30 WIB =
20:30 UTC** — outside booth hours and not colliding with the existing 19:00/20:00/
20:05 UTC jobs. Daily idempotency key `settlement-sync:YYYY-MM-DD`.

### 4.6 `public.ts` — `listSettlements`

Role-agnostic read (ADR-012: staff + managers). Optional date-range filter via
`by_settlement_date`. Returns the view fields incl. `source` (so the UI can badge
manual vs poll rows) and `bca_account_destination` (last-4 only — the row never
stores more). Newest-day-first.

## 5. Frontend

`src/routes/settlements/index.tsx` (+ route registration in `src/router.tsx`, + a
home tile). Wireframe intent: `frollie-pos design files/.../wireframes/` if a
settlements artboard exists; otherwise follow the `/history` list conventions.

- **Read-only list** for everyone: per-day cards/rows — date, net (primary, "hit
  BCA"), gross, MDR, txn count, a **source badge** (`Otomatis` / `Manual`), BCA
  last-4. ADR-012 framing copy: clearly "this is when Xendit paid out" — staff have
  no control over settlement.
- **Manager-only manual-entry**: a "Catat settlement" button → form (date picker,
  gross, MDR, count, BCA last-4) → `PinAction` sheet wiring `enterSettlementManually`
  (reuses the v0.5.5 `PinAction` pattern + one-shot idempotency key). Net is shown
  computed, not entered. Toasts on success/failure (Sonner).
- Empty state ("Belum ada settlement") — expected pre-KYB, so it must read as
  normal, not broken.
- Loading + error states per the v1.0 route-error conventions.

## 6. Testing strategy

Payment-adjacent feature → tests are **required** (CLAUDE.md "How to add a feature"
#7).

- **`lib.test.ts` (golden, the core de-risk):** `aggregateSettledByDate` over fixture
  rows — multi-day grouping, `EARLY_SETTLED` included, `PENDING` + `null`-settlement
  excluded, fee summation, `net = gross − mdr`, empty input → `[]`.
  `parseListTransactions` — a known-good fixture body parses; an unrecognized shape
  **throws** (not silent `[]`); extra/unknown fields tolerated.
- **`manualEntry.test.ts` (convex-test):** happy path writes a `source="manual"` row
  + audit; non-manager session rejected; wrong PIN rejected + counts toward lockout;
  `net` computed server-side ignores any client value; idempotent replay returns the
  same row without a second write; pre-cache authCheck rejects a cached-key replay
  with a dead session (ADR-046 parity).
- **`upsert.test.ts`:** re-running the same date updates in place (no dupe — unique
  `by_settlement_key`); poll-over-manual supersede flips `source` + audits
  `settlement.poll_superseded_manual` + preserves `created_at`; multi-day lookback
  backfills a late-settling earlier date.
- **`sync.test.ts`:** `syncSettlements` end-to-end with a mocked `listTransactions`
  body → correct per-day upserts; zero-rows window → audited `no_settlements` skip;
  resilient wrapper transient-retry + non-transient audited-skip parity with
  stock-recon.
- **Pre-KYB live caveat:** auto-poll is **not** asserted against real settled funds;
  `sync.test.ts` proves the wiring against a fixture body only. Live verification is
  the N3 / R1 follow-up.

## 7. Risks & open questions

- **R1 — `GET /transactions` field shapes are asserted-not-verified.** The research
  pass couldn't fully confirm query-params, pagination cursor, or response field
  names. **Mitigation:** isolate parse + fetch (§4.1/§4.2); golden-test the parser;
  confirm against a real TEST-key `GET /transactions` response (or `llms.txt`
  OpenAPI) during implementation *before* trusting `listTransactions`; parser throws
  on unrecognized shape. Record confirmed field paths back into
  `docs/xendit-reference/settlement-reconciliation.md` (memory: external-API facts
  live there).
- **R2 — TEST-mode produces no settlements (KYB pending).** The auto-poll cannot be
  live-verified at launch. **Mitigation:** the whole manual-first design — manual
  entry is the verified launch path; auto-poll ships shape-tested + dormant-correct
  (zero-rows is the normal pre-KYB result); N3 follow-up flips it on post-KYB.
- **R3 — WIB vs UTC date boundary.** `settlement_date` from Xendit may be UTC; the
  booth + accounting think in WIB. **Mitigation:** decide and document one canonical
  interpretation in the plan; reuse `convex/lib/time.ts` (`WIB_OFFSET_MS`) — the same
  calendar the receipt counter + reporting already use. (Open: does Xendit's
  `settlement_date` need WIB re-bucketing, or is it already a calendar date? Confirm
  with R1.)
- **OQ1 — Module home for the List Transactions reader.** Proposed: extend
  `payments/xendit.ts` (the existing single Xendit adapter) rather than a new file —
  keeps the Xendit HTTP surface in one place.
- **OQ2 — Lookback window length.** Proposed: 7 days. Long enough to self-heal a
  multi-day cron outage and catch late `settlement_date` posting; short enough to
  keep the poll cheap. Revisit if pagination cost is high.
- **OQ3 — Settlement `_id` exposure.** `listSettlements` returns rows to staff;
  follow the reporting convention of not leaking raw `Id`s where a stable key serves
  (memory: `staff_code over _id for external surfaces`). `settlement_key` is the
  stable external handle.

## 8. Out of scope

- Per-transaction match-back of settled rows → `pos_transaction` (N1; v1.0 polish).
- Variance detection + settlement alerts (N2; v1.0 polish).
- Live verification of the auto-poll against real settled funds (N3; KYB-gated
  follow-up).
- Frollie Pro settlement sync (N4; v1.1 — `synced_to_frollie_pro_at` stays dormant).
- Any change to payment confirmation, refunds, or `pos_refunds.settlement_status`
  (N5).
- Balance/Transactions **report CSV** ingestion (Option B in the knowledge base) —
  the List Transactions API is the chosen primitive; CSV is a manual cross-check
  managers can still do in the Xendit dashboard.
