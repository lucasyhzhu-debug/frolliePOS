# v0.5.5 ‚Äî Inventory-SKU admin + route error boundary (design spec)

> **Status:** DRAFT ‚Äî started 2026-06-03 during the production cutover smoke test.
> Two small, independent slices bundled into one phase. Run through
> `/staffreview` ‚Üí `/superpowers:writing-plans` ‚Üí `/staffreview` before executing.
> Version `v0.5.5` is provisional (slots after v0.5.4 printing; v0.6 is already
> spec'd separately) ‚Äî renumber if it collides.

## Why this phase exists (context)

Both items were discovered live while bringing the prod deployment (`savory-zebra-800`) online:

1. **No way to create an inventory SKU in-app.** v0.5.3b shipped product CRUD, but
   `pos_inventory_skus` rows can only be created by `seed`/tests ‚Äî there is no
   `createInventorySku` mutation, action, or UI anywhere in the codebase. The
   components editor in `/mgr/products` can only map products to the **5 seeded
   SKUs** (`dubai`, `choco`, `matcha`, `lotus`, `brownie`). The moment the booth
   sells anything outside that set, the manager is stuck. This is a v1 scope gap:
   the catalog assumed a fixed SKU line and only wired up *product* creation.

2. **Stale-chunk crash has no error boundary.** After a redeploy, a client holding
   a cached `index.html` (PWA service worker, `registerType: "autoUpdate"`) lazy-
   imports a route chunk whose hash no longer exists ‚Üí "Failed to fetch dynamically
   imported module" ‚Üí React Router's bare default error screen (the one that says
   "you can provide your own error boundary"). For a booth POS where staff can't
   debug, this must degrade gracefully (auto-recover, not a stack trace).

These are unrelated in code (catalog backend vs. frontend resilience) but both are
small, both surfaced during cutover, and bundling avoids two near-empty phases.

---

## Goal

- A manager can **create a new inventory SKU** from the admin UI (PIN-gated,
  audited), and it immediately becomes selectable in the product components editor
  and the `/stock` surfaces.
- A stale-chunk / dynamic-import failure **auto-recovers** (one guarded reload)
  instead of showing staff a raw error, with a friendly fallback if recovery fails.

## Part A ‚Äî `createInventorySku` + admin UI

### A.1 Backend ‚Äî `convex/catalog/`

Mirror the **`createProduct`** pattern exactly (CLAUDE.md #22 ‚Äî this is an
identity/structure write, so **manager-PIN**, not session-only):

- **`convex/catalog/actions.ts` ‚Üí `createInventorySku`** (`action`):
  - args: `idempotencyKey`, `sessionId`, `managerPin`, `sku`, `name`,
    `low_threshold`, `code?`, `initials?`, `hue?`.
    (`unit` is always the literal `"piece"` ‚Äî set in the commit, not an arg.
    `active: true`, `created_at` server time ‚Äî set in commit. `photo_storage_id`
    out of scope.)
  - `withActionCache({ key, mutationName: "catalog.createInventorySku" })`.
  - `verifyManagerPinOrThrow(ctx, { sessionId, managerPin, idempotencyKey })`
    (`convex/auth/verifyPin.ts`) ‚Üí `{ managerId }`.
  - delegates to `_createInventorySkuCommit_internal` with derived `:commit` key.

- **`convex/catalog/internal.ts` ‚Üí `_createInventorySkuCommit_internal`**
  (`internalMutation`, `withIdempotency`):
  - **Validation:** `sku` non-empty, trimmed, lowercase-slug shape; `name`
    non-empty; `low_threshold` a non-negative integer.
  - **Uniqueness:** query `pos_inventory_skus` `by_sku` ‚Üí throw `SKU_EXISTS` if a
    row with that `sku` already exists. If `code` provided, also check `by_code`
    ‚Üí `CODE_EXISTS`. (Mirror the `PRICE_INVALID` guard style in
    `_createProductCommit_internal`.)
  - `ctx.db.insert("pos_inventory_skus", { sku, code, name, unit: "piece",
    low_threshold, initials, hue, active: true, created_at: now })`.
  - **Seed the stock-level cache:** insert a `pos_stock_levels` row for the new SKU
    at qty 0 (so `/stock` and recount don't choke on a missing cache row) ‚Äî CONFIRM
    during planning whether the read paths tolerate an absent level row or require
    one (see Open items).
  - `logAudit(ctx, { actor_id: mgrId, action: "inventory_sku.created",
    entity_type: "inventory_sku", entity_id: skuId, source: "manager_pin",
    metadata: { sku, name } })` ‚Äî **new audit verb** `inventory_sku.created`
    (document it in `docs/SCHEMA.md`; `audit_log.action` is a free `v.string()`,
    no enum to edit ‚Äî CLAUDE.md #4).
  - returns `{ skuId }`.

- **Schema:** no change ‚Äî `pos_inventory_skus` already exists
  (`convex/catalog/schema.ts`). Confirm the admin read query (`listAllProducts`
  equivalent for SKUs, or the existing components-editor query) returns the new
  SKU.

### A.2 Frontend ‚Äî `src/routes/mgr/`

Place the create surface where SKUs are most naturally managed. Two options to
resolve in planning:
- **(preferred) `/mgr/stock` admin** ‚Äî an "Add SKU" button on the stock admin
  screen, since SKUs are an inventory concept; OR
- **`/mgr/products`** ‚Äî reuse the existing admin page that already holds the
  components editor and `PinSheet`.

Either way, reuse the established admin form pattern from `products.tsx`:
- "Add SKU" dialog (`Dialog`) with fields: `sku` (slug), `name`, `low_threshold`
  (integer-only input ‚Äî reuse `parseIntStrict`), optional `code`, `initials`,
  `hue`.
- `useIdempotency("catalog.createInventorySku")` + `useAction(api.catalog.actions.createInventorySku)`.
- `PinSheet` for the manager PIN (same as `createProduct`).
- Toast on success/error; map `SKU_EXISTS` / `CODE_EXISTS` / `PRICE_INVALID`-style
  errors to friendly messages (mirror `friendlyError` in `products.tsx`).
- On success the components dropdown / stock list re-renders reactively (Convex
  subscription) ‚Äî no manual refetch.

### A.3 Access & correctness

- **Manager-PIN gated** via `verifyManagerPinOrThrow` (CLAUDE.md #22 tier:
  identity/structure). Off-booth path is **not** required for v0.5.5 (creating a
  SKU is a booth action; no Telegram-approval KIND needed).
- **Idempotency:** public action takes `idempotencyKey` + `withActionCache`
  wrapping an idempotent commit (CLAUDE.md #20, dual-call authCheck pattern).
- **Snapshot rule unaffected** ‚Äî SKUs aren't price-bearing; no ADR-001 concern.

## Part B ‚Äî route error boundary + chunk-reload recovery

### B.1 Frontend ‚Äî router config + `src/components/layout/`

- Add an **`errorElement`** (React Router) at the root route in the
  `createBrowserRouter` config (CONFIRM file ‚Äî `RouterProvider` is wired in
  `src/main.tsx`; the router definition is imported, locate it in planning).
- New `src/components/layout/RouteErrorBoundary.tsx`:
  - `const error = useRouteError()`.
  - **Chunk-load detection:** if `String(error?.message ?? error)` matches
    `/dynamically imported module|Importing a module script failed|Failed to fetch dynamically/i`,
    treat as a stale-deploy and **reload once**, guarded by a
    `sessionStorage` flag (`"chunk-reload-attempted"`) so a genuinely-missing
    chunk can't infinite-loop. Clear the flag on a successful normal load.
  - **Otherwise:** render a minimal branded fallback ("Something went wrong" +
    a "Reload" button), using the shadcn/Frollie tokens. No stack trace to staff.
- Keep it dependency-free and V8-safe-irrelevant (pure client component).

### B.2 Correctness

- The `sessionStorage` guard is the load-bearing bit: reload-on-chunk-error is
  correct exactly once; a second failure means the chunk is truly gone (bad
  deploy / offline), so fall through to the friendly fallback instead of looping.
- Does **not** replace the PWA update flow (`autoUpdate` stays) ‚Äî it's the
  catch-net for the gap between a new SW landing and the running tab's cached
  `index.html`.

## Testing

**Part A (write tests ‚Äî catalog/identity write):**
- `_createInventorySkuCommit_internal`: happy path inserts row + audit; duplicate
  `sku` ‚Üí `SKU_EXISTS`; duplicate `code` ‚Üí `CODE_EXISTS`; bad `low_threshold` ‚Üí
  validation throw; idempotency replay returns the same `skuId` (no double insert).
- PIN gate: wrong/expired PIN ‚Üí throws before insert (reuse `productAdmin.test.ts`
  harness + `staff_sessions` seed shape).
- New SKU appears in the components-editor read query.

**Part B (optional but encouraged):**
- Unit-test the error-classification helper (extract the regex match into a pure
  `isChunkLoadError(err): boolean` so it's testable without a router).

## Success criteria & rollback

- **A done when:** a manager creates "Test SKU" from the UI on prod, it appears in
  the components dropdown, an `inventory_sku.created` audit row exists, and a
  duplicate `sku` is rejected with a friendly toast.
- **B done when:** forcing a stale-chunk import (or deleting a chunk + reload)
  auto-recovers on first reload; a hard-missing chunk shows the friendly fallback,
  not a loop or a raw stack.
- **Rollback:** both slices are additive ‚Äî revert the commits; no schema migration
  (Part A uses the existing table), no data backfill.

## Out of scope

- Editing / deactivating / archiving inventory SKUs (create-only this phase;
  follow the v0.5.3b product pattern later if needed).
- `photo_storage_id` upload for SKUs.
- Off-booth Telegram-approval KIND for SKU creation.
- Replacing the PWA update strategy or `esc-pos-encoder`/canvas work (separate).
- Generalising the error boundary into per-route boundaries (root-level is enough).
