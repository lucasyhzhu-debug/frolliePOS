# v0.5.5 — Inventory-SKU admin + route error boundary (design spec)

> **Status:** STAFFREVIEW-APPLIED — started 2026-06-03 during the production cutover
> smoke test. Staffreview findings folded in (see
> `docs/reviews/staffreview-spec-v0.5.5-inventory-sku-admin-and-error-boundary-2026-06-03.md`),
> design extended after user input to add a checkbox in "Add Product" that
> atomically creates + links a 1:1 inventory SKU (see [A.0 Decisions](#a0-decisions)).
> Next: `/superpowers:writing-plans` → second `/staffreview` → land plan on main.
> Two small, independent slices bundled into one phase. Version `v0.5.5` slots
> after v0.5.4 printing; v0.6 is spec'd separately — no collision.

## Why this phase exists (context)

Both items were discovered live while bringing the prod deployment (`savory-zebra-800`) online:

1. **No way to create an inventory SKU in-app.** v0.5.3b shipped product CRUD, but
   `pos_inventory_skus` rows can only be created by `seed`/tests — there is no
   `createInventorySku` mutation, action, or UI anywhere in the codebase. The
   components editor in `/mgr/products` can only map products to the **5 seeded
   SKUs** (`dubai`, `choco`, `matcha`, `lotus`, `brownie`). The moment the booth
   sells anything outside that set, the manager is stuck. This is a v1 scope gap:
   the catalog assumed a fixed SKU line and only wired up *product* creation.

2. **Stale-chunk crash has no error boundary.** After a redeploy, a client holding
   a cached `index.html` (PWA service worker, `registerType: "autoUpdate"`) lazy-
   imports a route chunk whose hash no longer exists → "Failed to fetch dynamically
   imported module" → React Router's bare default error screen (the one that says
   "you can provide your own error boundary"). For a booth POS where staff can't
   debug, this must degrade gracefully (auto-recover, not a stack trace).

These are unrelated in code (catalog backend vs. frontend resilience) but both are
small, both surfaced during cutover, and bundling avoids two near-empty phases.

---

## Goal

- A manager can **create a new inventory SKU** standalone from `/mgr/products`
  (PIN-gated, audited), and it immediately becomes selectable in the product
  components editor and the `/stock` surfaces.
- The **"Add Product" dialog gains a checkbox** to atomically create-or-link a
  matching inventory SKU in a single PIN entry — covering both the "new flavor
  1pc" case (qty 1, new SKU) and the "Dubai 3pcs" multi-pack case (qty 3,
  existing SKU reused). Slug derives from `sku_family` (lowercased). Component
  qty is an editable input (defaults to 1). If the SKU already exists, it's
  reused (no duplicate error) and only the component link is created.
- A stale-chunk / dynamic-import failure **auto-recovers** (one guarded reload)
  instead of showing staff a raw error, with a friendly fallback if recovery
  fails — covering both the app shell and the three public sibling routes.

---

## Part A — SKU creation (standalone + product-bundled)

**Two entry points** for SKU creation, by design (see [Decisions](#a0-decisions)):

1. **Standalone `Add SKU`** in `/mgr/products` — for SKUs that don't yet have a
   sellable product (rare: ingredient-style or pre-staged items). Pure
   PIN-gated SKU insert.
2. **Checkbox in `Add Product`** — for the common "new flavor" case where a
   product IS its own SKU. Atomic single-mutation: creates the product, ensures
   a matching SKU exists (creates if absent, reuses if present), and inserts a
   `pos_product_components` link at `qty 1` — all in one PIN entry, one
   transaction.

### A.0 Decisions

| Question | Decision | Why |
|---|---|---|
| Keep standalone Add SKU? | Yes | Manager needs an escape hatch for SKU-without-product (ingredient, prep item). Rare but inevitable. |
| Server-side flow when checkbox is on? | Atomic single mutation | One PIN entry, all-or-nothing. Uniqueness collision on SKU rolls back the product too. |
| SKU slug source when checkbox is on? | Auto-derive from `sku_family` (lowercased, trimmed) | Manager already types it; convention is lowercase (seed values: `dubai`, `choco`, `matcha`, `lotus`, `brownie`). Show a read-only preview under the checkbox. |
| Auto-link `pos_product_components`? | Yes, with an editable qty (defaults to 1) | The point of the checkbox is "this product is made of N of this SKU." Default qty 1 covers the "new flavor 1pc" case; an editable qty covers the "Dubai 3pcs" / "Dubai 8pcs" multi-pack case — manager picks `3` or `8` at the moment they're adding the product, instead of creating at 1 then opening the components editor. Auto-linking means sale-decrement, low-stock, recount all work immediately. |
| If slug collides with existing SKU? | Reuse + link (no `SKU_EXISTS` throw) | Manager wants the 1:1 link to succeed; failing because the SKU already exists is hostile to the common "I already added the Dubai SKU last week, now I'm adding the Dubai 3pcs product" workflow. Audit emits `inventory_sku.created` ONLY when newly inserted. |

### A.1 Backend — standalone `createInventorySku` (`convex/catalog/`)

### A.1 Backend — `convex/catalog/`

Mirror the **`createProduct`** pattern exactly (CLAUDE.md #22 — this is an
identity/structure write, so **manager-PIN**, not session-only):

- **`convex/catalog/actions.ts` → `createInventorySku`** (`action`):
  - args: `idempotencyKey`, `sessionId`, `managerPin`, `sku`, `name`,
    `low_threshold`, `code?`, `initials?`, `hue?`.
    (`unit` is always the literal `"piece"` — set in the commit, not an arg.
    `active: true`, `created_at` server time — set in commit. `photo_storage_id`
    out of scope.)
  - `withActionCache({ key, mutationName: "catalog.createInventorySku" })`.
  - `verifyManagerPinOrThrow(ctx, { sessionId, managerPin, idempotencyKey })`
    (`convex/auth/verifyPin.ts`) → `{ managerId, deviceId }`. Capture both —
    `deviceId` threads into the audit row's `device_id` field for forensics.
  - Delegates to `_createInventorySkuCommit_internal` passing
    `idempotencyKey: \`${args.idempotencyKey}:commit\`` (NOT the bare key) so
    an action retry crashed between commit and the action-level cache write
    is absorbed by the inner mutation's `withIdempotency` wrap. Mirrors
    `catalog.actions.createProduct:50-52` and the `withActionCache` docblock
    at `convex/idempotency/action.ts:17-32`. Carry the verbatim 4-line "why
    the dual cache" docblock from `createProduct` so future readers don't
    have to spelunk the pattern doc.

- **`convex/catalog/internal.ts` → `_createInventorySkuCommit_internal`**
  (`internalMutation`, `withIdempotency`):
  - **Validation:** `sku` matches `/^[a-z0-9-]{1,32}$/` after trim (resolve
    open-item #5 with this shape; mirrors seed values `dubai`, `choco`,
    `matcha`, `lotus`, `brownie`) — throw `SKU_INVALID`; `name` non-empty
    (1-80 chars to match product name discipline) — throw `NAME_INVALID`;
    `low_threshold` a non-negative integer — throw `LOW_THRESHOLD_INVALID`.
    Treat `code` provided but empty-after-trim as not-provided (don't insert
    empty string).
  - **Uniqueness:** query `pos_inventory_skus` `by_sku` → throw `SKU_EXISTS` if a
    row with that `sku` already exists. If `code` provided, also check `by_code`
    → `CODE_EXISTS`. (Mirror the `PRICE_INVALID` guard style in
    `_createProductCommit_internal`.) Uniqueness ignores `active` — a
    deactivated row still collides (v0.5.5 has no SKU deactivate UI anyway).
  - `ctx.db.insert("pos_inventory_skus", { sku, code, name, unit: "piece",
    low_threshold, initials, hue, active: true, created_at: now })`.
  - **No `pos_stock_levels` seed:** the canonical `upsertStockLevel` helper
    (`convex/inventory/internal.ts:20-42`) lazy-inits the row on first
    movement; every read path defaults absent rows to 0
    (`convex/inventory/public.ts:16-34`, plus all `level?.on_hand ?? 0`
    consumers). Seeding a zero-qty row would be churn with no reader.
    Resolves open-item #1.
  - `logAudit(ctx, { actor_id: mgrId, action: "inventory_sku.created",
    entity_type: "inventory_sku", entity_id: skuId, source: "booth_inline",
    device_id: deviceId, metadata: { sku, name } })` — **new audit verb**
    `inventory_sku.created` (document it in `docs/SCHEMA.md`;
    `audit_log.action` is a free `v.string()`, no enum to edit — CLAUDE.md
    #4). `source: "booth_inline"` — `"manager_pin"` is **not** a valid
    `audit_log.source` literal (the enum is
    `booth_inline | wa_approval | telegram_approval | system | reaper`,
    see `convex/audit/schema.ts:14-19` / `convex/audit/internal.ts:6-12`).
    Matches `_createProductCommit_internal:227` and the v0.5.3b convention.
  - returns `{ skuId }`.

- **Schema:** no change — `pos_inventory_skus` already exists
  (`convex/catalog/schema.ts`). `listAllProducts`
  (`convex/catalog/public.ts:77-98`) already returns `skus` filtered by
  `by_active`, so a newly-inserted `active: true` SKU appears immediately
  in `/mgr/products` without a new admin query. A regression test should
  pin this — see §Testing.

### A.1b Backend — extend `createProduct` for atomic product+SKU+link

Extend the existing `catalog.actions.createProduct` and
`_createProductCommit_internal` instead of adding a third action — this keeps
"Add Product" as one entry point and lets the checkbox flow through the
already-PIN-gated, already-cached action.

- **`convex/catalog/actions.ts` → `createProduct` (extended args):**
  - New optional args:
    - `withInventorySku?: v.boolean()` — when true, ensures a matching SKU +
      component link.
    - `inventorySkuLowThreshold?: v.number()` — required when
      `withInventorySku` is true; otherwise ignored. Validated server-side
      (non-negative integer); FE pre-validates with `parseIntStrict`.
    - `inventorySkuComponentQty?: v.number()` — required when
      `withInventorySku` is true; otherwise ignored. Positive integer
      (≥ 1). Represents how many SKU units the product consumes per sale
      ("Dubai 3pcs" → 3; "Dubai 1pc" → 1; "Mixed Box 4pcs" stays at the
      standalone components-editor flow because it links multiple
      different SKUs). FE pre-validates with `parseIntStrict` and a `> 0`
      check; throws `QTY_INVALID` server-side on violation (same error
      code the existing `setProductComponents` flow uses).
  - **No new auth surface, no new cache entry** — same `withActionCache`
    wrapper, same `verifyManagerPinOrThrow`, same `${key}:commit` derived
    key to the inner mutation.
  - Defaults: when `withInventorySku` is undefined or false, behaviour is
    identical to today (back-compat for `MgrProducts` tests + any external
    consumer).

- **`convex/catalog/internal.ts` → `_createProductCommit_internal` (extended):**
  - Existing args + new optional `withInventorySku`, `inventorySkuLowThreshold`,
    `inventorySkuComponentQty`.
  - Same `PRICE_INVALID` / name-trim guards as today.
  - **If `withInventorySku === true`:**
    1. **Derive slug:** `skuSlug = args.sku_family.trim().toLowerCase()`.
       Validate against the same `/^[a-z0-9-]{1,32}$/` regex shared with
       the standalone `_createInventorySkuCommit_internal`; on mismatch
       throw `SKU_FAMILY_NOT_SLUGGABLE` (distinct error — manager needs to
       know the cause is the family field, not a separate SKU input).
    2. **Validate `inventorySkuLowThreshold`** — required, non-negative
       integer; throw `LOW_THRESHOLD_INVALID` (same code as standalone).
       Validate `inventorySkuComponentQty` — required, positive integer
       (≥ 1); throw `QTY_INVALID` (same code `setProductComponents` uses).
    3. **Lookup-or-create the SKU:** query `pos_inventory_skus` `by_sku`
       with `skuSlug`. If a row exists, reuse its `_id` (no insert, no
       `inventory_sku.created` audit). If absent, insert the row
       (`unit: "piece"`, `active: true`, `created_at: now`, no `code` —
       the bundled checkbox flow doesn't surface the optional code field;
       manager uses standalone Add SKU if they need a code) and emit
       `inventory_sku.created` audit (verb + source as standalone).
    4. **Insert the component link:** `ctx.db.insert("pos_product_components",
       { product_id: productId, inventory_sku_id: skuId, qty:
       args.inventorySkuComponentQty })`. No pre-check needed — the
       product is brand new, so it has no existing components.
    5. **Emit `product.components_set` audit** (matches the verb the
       standalone `setProductComponents` flow uses, so audit forensics
       group cleanly) with `metadata: { product_id, sku_id, qty,
       source: "create_product_bundled" }`.
    6. Returns `{ productId, inventorySkuId, skuCreated: boolean,
       componentQty: number }` so the FE can render an accurate success
       toast ("Dubai 3pcs added — linked to existing dubai SKU at qty 3"
       vs "Matcha 1pc added — matcha SKU created, linked at qty 1").
  - **Transaction discipline:** all writes are in one Convex mutation
    transaction → any throw rolls back everything (product insert + SKU
    insert + component insert + all 1-3 audit rows). This is the
    all-or-nothing guarantee the atomic decision promises.

- **Schema:** no change. `pos_product_components` schema already exists
  (`convex/catalog/schema.ts:40-46`).

### A.2 Frontend — `src/routes/mgr/products.tsx`

**Home decided: `/mgr/products`.** `/mgr/stock` is the v0.6 (R9 / ADR-044)
drift-log triage screen — manager-session-gated, no `PinSheet` wiring; mixing
a PIN-gated create dialog into a session-only drift UI is a UX clash.
`/mgr/products` already carries every primitive both flows need: `PinSheet`,
`data.skus` from `listAllProducts` (the read seam that has to show new SKUs),
the "Add product" dialog shape to clone, the `PinAction` discriminated
union, the `humanizeCatalogError` mapper, and `parseIntStrict` for
integer-only inputs. Resolves open-item #2.

Two UI changes:

#### A.2.1 — Extend "Add Product" with the bundled-SKU checkbox

- Below the existing form fields (`src/routes/mgr/products.tsx:671-786`),
  add a checkbox row:

- Toast text branches on the action return shape:
  - `skuCreated === true`: "<name> added — <slug> SKU created, linked at qty N"
  - `skuCreated === false`: "<name> added — linked to existing <slug> SKU at qty N"
- The existing idempotency intent `"catalog.createProduct"` covers both
  bundled and non-bundled cases (same action, same cache key).
- `humanizeCatalogError` gets `SKU_FAMILY_NOT_SLUGGABLE` ("SKU family must
  be lowercase letters, numbers, or hyphens (max 32) when creating a
  matching SKU."). `QTY_INVALID` is already in the error map (line 82) —

#### A.2.2 — Add the standalone "Add SKU" button

- Add an **"Add SKU"** button in the page header next to "Add product"

- New **Add SKU** Dialog (clone the Add Product dialog shape at

#### A.2.3 — Reactivity (applies to both flows)

- **No manual refetch** — the components-editor dropdown and `data.skus`
  refresh reactively via the live Convex subscription on
  `listAllProducts`. The new product appears in `sortedProducts`; the new
  SKU (whether from A.2.1 or A.2.2) appears in the components dropdown.

  its level row (per A.1 §"No `pos_stock_levels` seed"). Until then it
  shows in the manager components-editor dropdown via `data.skus`.

### A.3 Access & correctness

- **Manager-PIN gated** via `verifyManagerPinOrThrow` (CLAUDE.md #22 tier:
  identity/structure). Off-booth path is **not** required for v0.5.5 (creating a
  SKU is a booth action; no Telegram-approval KIND needed).
- **Idempotency:** public action takes `idempotencyKey` + `withActionCache`
  wrapping an idempotent commit (CLAUDE.md #20, dual-call authCheck pattern).
- **Snapshot rule unaffected** — SKUs aren't price-bearing; no ADR-001 concern.

## Part B — route error boundary + chunk-reload recovery

### B.1 Frontend — router config + `src/components/layout/` + `src/lib/`

Router config lives at `src/router.tsx` (`createBrowserRouter(routes)` at
line 118), imported by `src/main.tsx:6`. The route table has two tiers:
**public siblings** (`/activate`, `/approve/:token`, `/r/:receiptNumber` —
lines 75-77) and the **app shell** (`path: "/"` with `<RootLayout />` — line
80). The boundary must cover BOTH tiers — `/r/:receiptNumber` is the
customer-facing receipt link (served via Telegram) and `/approve/:token` is
the manager off-booth landing; both are exactly the prod-stale-chunk failure
surfaces we cannot show a raw stack trace on. Attaching `errorElement` only
to the app-shell parent would leave the three public siblings on React
Router's default screen.

- New **`src/lib/chunkLoadError.ts`** — pure helper:
  ```ts
  export function isChunkLoadError(err: unknown): boolean {
    const msg = String((err as { message?: string } | null)?.message ?? err);
    return /dynamically imported module|Importing a module script failed|Failed to fetch dynamically/i.test(msg);
  }
  ```
  Required deliverable (promoted from optional). Lets the boundary stay a
  thin React component and the regex live in one unit-tested place.

- New **`src/components/layout/RouteErrorBoundary.tsx`**:
  - `const error = useRouteError()`.
  - **Chunk-load detection:** if `isChunkLoadError(error)` returns true,
    treat as a stale-deploy and **reload once**, guarded by a
    `sessionStorage` flag (`"chunk-reload-attempted"`) so a genuinely-missing
    chunk can't infinite-loop. Clear the flag in a `useEffect` on the
    boundary's NORMAL (non-error) subtree mount — putting the clear inside
    the boundary itself means a never-recovers chunk repeatedly clears its
    own guard. (Implementation note: the simplest place is a top-level
    `useEffect(() => sessionStorage.removeItem("chunk-reload-attempted"),
    [])` on `RootLayout` plus equivalent in the public-route components, OR
    in a tiny `ChunkReloadGuard` mounted alongside `RouterProvider` in
    `main.tsx`. Plan picks one.)
  - **Otherwise:** render a minimal branded fallback ("Something went wrong"
    + a "Reload" button), using the shadcn/Frollie tokens. **Never** a stack
    trace, even in dev — staff debug at the booth too.
  - Customer-facing copy (mounted on `/r/:receiptNumber`): default copy is
    Indonesian — "Halaman tidak bisa dimuat. Buka ulang link dari Telegram."
    — since receipts are seen by customers, not staff. Plan picks
    where/how to vary copy by route (a simple `useLocation()` check, or
    a `variant` prop on the boundary).

- **Router wiring** (`src/router.tsx`): attach `errorElement: <RouteErrorBoundary />`
  to:
  1. The root app-shell route (line 80 `path: "/"`).
  2. Each public sibling — lines 75 (`/activate`), 76 (`/approve/:token`),
     77 (`/r/:receiptNumber`).

  Alternative (cleaner if planning agrees): wrap the three public siblings
  under a `PublicShell` layout component that carries the same
  `errorElement`. Trade-off: one extra component vs DRY-ing the
  `errorElement` declaration. Plan picks.

- **Tiny ADR** capturing the policy: "Route-level chunk-reload error
  boundary — auto-reload once via sessionStorage, then friendly fallback;
  covers app-shell + public routes." Not a foundations-level change.

- Keep it dependency-free and V8-safe-irrelevant (pure client component +
  pure helper).

### B.2 Correctness

- The `sessionStorage` guard is the load-bearing bit: reload-on-chunk-error is
  correct exactly once; a second failure means the chunk is truly gone (bad
  deploy / offline), so fall through to the friendly fallback instead of looping.
- Does **not** replace the PWA update flow (`autoUpdate` stays) — it's the
  catch-net for the gap between a new SW landing and the running tab's cached
  `index.html`.

## Testing

**Part A.1 — standalone `_createInventorySkuCommit_internal` (REQUIRED):**
- Happy path inserts row + audit row with `source: "booth_inline"` and the new
  `inventory_sku.created` verb.
- Duplicate `sku` → `SKU_EXISTS`.
- Duplicate `code` → `CODE_EXISTS`.
- Bad `sku` shape (uppercase, space, >32 chars) → `SKU_INVALID`.
- Bad `low_threshold` (negative or non-integer) → `LOW_THRESHOLD_INVALID`.
- Bad `name` (empty or >80) → `NAME_INVALID`.
- `code` passed as `"  "` (whitespace) → inserted as undefined, not empty string.
- Idempotency replay (same bare key, simulated successful round-trip) returns
  the same `{ skuId }` with no double insert and no second audit row.
- **Crash-retry test** (the `${key}:commit` dual-cache guarantee): simulate
  an action that committed the inner mutation but crashed before the
  action-level cache write; the retry must absorb at the inner
  `withIdempotency` and leave exactly one `pos_inventory_skus` row + one
  audit row. Mirrors `convex/catalog/__tests__/productAdmin.test.ts`
  patterns.
- PIN gate: wrong PIN → throws `INVALID_PIN` before insert; expired session
  → `SESSION_INVALID`; non-manager session → `NOT_MANAGER`. Reuse the
  `productAdmin.test.ts` harness + `staff_sessions` seed shape.
- **Read-seam test:** after a successful create, `listAllProducts({ sessionId })`
  returns the new SKU in `result.skus` (confirms ADR-034 module-boundary
  read-seam still surfaces fresh rows; closes the I4 finding from spec
  staffreview).

**Part A.1b — extended `_createProductCommit_internal` (REQUIRED):**
- Back-compat: when `withInventorySku` is undefined/false, behaviour is
  identical to today; existing `productAdmin.test.ts` cases still pass with
  no test changes.
- Bundled path, fresh SKU at qty 1: product + SKU + component-link
  inserted in one transaction; three audit rows emitted (`product.created`,
  `inventory_sku.created`, `product.components_set`); return shape includes
  `inventorySkuId`, `skuCreated: true`, `componentQty: 1`.
- Bundled path, fresh SKU at qty 3 (multi-pack): same as above with
  `componentQty: 3`; `pos_product_components.qty` row is 3.
- Bundled path, existing SKU at qty 3 ("Dubai 3pcs" linking to existing
  `dubai`): SKU is reused (no second insert, no `inventory_sku.created`
  audit); component link inserted at qty 3; return shape includes
  `inventorySkuId` (existing), `skuCreated: false`, `componentQty: 3`.
- Bundled path, `sku_family` fails slug regex (e.g. `"Dubai Mall"` with a
  space) → throws `SKU_FAMILY_NOT_SLUGGABLE` and rolls back the product.
- Bundled path, missing/invalid `inventorySkuLowThreshold` → throws
  `LOW_THRESHOLD_INVALID` and rolls back.
- Bundled path, invalid `inventorySkuComponentQty` (0, negative, non-integer)

## Success criteria & rollback

- **A done when:** a manager creates "Test SKU" from the UI on prod, it appears in
  the components dropdown, an `inventory_sku.created` audit row exists, and a
  duplicate `sku` is rejected with a friendly toast.
- **B done when:** forcing a stale-chunk import (or deleting a chunk + reload)
  auto-recovers on first reload; a hard-missing chunk shows the friendly fallback,
  not a loop or a raw stack.
- **Rollback:** both slices are additive — revert the commits; no schema migration
  (Part A uses the existing table), no data backfill.
