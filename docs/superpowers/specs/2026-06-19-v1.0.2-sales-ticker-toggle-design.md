# v1.0.2 — In-App Manager Sales-Ticker Toggle

**Date:** 2026-06-19
**Status:** Design (spec gate)
**Slug:** `v1.0.2-sales-ticker-toggle`

## Problem

v1.0.1 shipped the live sales ticker — a silent `txn_ticker` Telegram message to the
Managers channel on every paid sale (`convex/telegram/txnTicker.ts`), gated by
`pos_settings.txn_ticker_enabled`. But it shipped **read-only**: the field exists in the
schema and is read at send time, yet **no mutation writes it**. The documented off-switch is
a manual Convex-dashboard edit (`docs/RUNBOOK.md:248`, `docs/SCHEMA.md:563`). A manager who
finds the feed noisy — or who is generating test transactions — has no in-app way to silence
it.

This phase adds the missing write path: a manager-session in-app toggle, so the ticker can be
turned off (and back on) from `/mgr/telegram-chats` without dashboard access.

## Why this is small

The ticker toggle is a near-verbatim sibling of an **already-shipped, already-reviewed**
toggle for the founders summary:

- **Backend:** `convex/settings/public.ts:21` `setFoundersSummaryEnabled` — a
  manager-session-gated, `withIdempotency`-wrapped, `authCheck`-guarded boolean mutation that
  patches `pos_settings` (insert when absent) and emits an audit row. The ticker mutation is a
  line-for-line clone targeting a different field + audit verb.
- **Frontend:** `src/routes/mgr/telegram-chats.tsx:73` `FoundersSummaryToggle` — a shadcn
  `Switch` + `Label`, reading `api.settings.public.getSettings` and calling the mutation with a
  throwaway `crypto.randomUUID()` idempotency key, with sonner success/error toasts.
- **Schema:** `txn_ticker_enabled: v.optional(v.boolean())` **already exists**
  (`convex/settings/schema.ts:23`). **No schema migration.**

The whole feature is additive: one query field, one mutation, one UI component, tests, docs.

## Design

### 1. Backend — `convex/settings/public.ts`

**1a. Extend `getSettings`** (currently returns only `founders_summary_enabled`) to also
surface the ticker flag, keeping the read-time default single-sourced with the `?? true`
convention used in `_getSettings_internal`:

```ts
export const getSettings = query({
  args: {},
  handler: async (ctx) => {
    const row = await ctx.db.query("pos_settings").first();
    return {
      founders_summary_enabled: row?.founders_summary_enabled ?? true,
      txn_ticker_enabled: row?.txn_ticker_enabled ?? true, // NEW
    };
  },
});
```

This is back-compat: existing `FoundersSummaryToggle` only destructures
`founders_summary_enabled`, so adding a field to the return is non-breaking.

**1b. Add `setTxnTickerEnabled`** — a clone of `setFoundersSummaryEnabled` with three
differences: the patched field, the inserted-row shape, and the audit verb.

```ts
export const setTxnTickerEnabled = mutation({
  args: {
    idempotencyKey: v.string(),
    sessionId: v.id("staff_sessions"),
    enabled: v.boolean(),
  },
  handler: withIdempotency<
    { idempotencyKey: string; sessionId: Id<"staff_sessions">; enabled: boolean },
    ToggleResult
  >(
    "settings.setTxnTickerEnabled",
    async (ctx, args) => {
      const { staffId } = await requireManagerSession(ctx, args.sessionId);
      const row = await ctx.db.query("pos_settings").first();
      if (row) {
        await ctx.db.patch(row._id, {
          txn_ticker_enabled: args.enabled,
          updated_at: Date.now(),
          updated_by: staffId,
        });
      } else {
        // Schema requires founders_summary_enabled (non-optional) + updated_at on
        // insert. Default founders to true so creating the row via THIS toggle does
        // not silently disable the founders summary. Mirrors updateReceiptConfig's
        // insert branch (settings/public.ts).
        await ctx.db.insert("pos_settings", {
          founders_summary_enabled: true,
          txn_ticker_enabled: args.enabled,
          updated_at: Date.now(),
          updated_by: staffId,
        });
      }
      await logAudit(ctx, {
        actor_id: staffId,
        action: "settings.txn_ticker_toggled", // NEW verb (free v.string(), no enum)
        entity_type: "pos_settings",
        source: "booth_inline",
        metadata: { enabled: args.enabled },
      });
      return { ok: true as const };
    },
    {
      authCheck: async (ctx, args) => {
        await requireManagerSession(ctx, args.sessionId);
      },
    },
  ),
});
```

**Auth tier:** manager-**session**, not manager-PIN. Per CLAUDE.md rule #22, a notification
opt-out is low-stakes config (it moves no money and changes no identity), so it sits with
`updateProductMeta` / receipt-config CRUD, not the PIN-gated identity/money writes. Same tier
as the sibling founders toggle.

**Rule #20 compliance:** public mutation → carries `idempotencyKey`, wrapped in
`withIdempotency`, with the `authCheck` that re-runs `requireManagerSession` BEFORE the cache
lookup (the dual-call pattern — see `docs/PATTERNS/idempotency-dual-call-authcheck.md`). The
duplication of `requireManagerSession` in both `authCheck` and handler is intentional.

### 2. Frontend — `src/routes/mgr/telegram-chats.tsx`

Add a `TxnTickerToggle` component, a clone of `FoundersSummaryToggle`, rendered **directly
beneath** it (above the existing `<Separator />` at line ~382). It reuses the single
`getSettings` subscription (React/Convex dedupes the two `useQuery` calls to one subscription),
reads `settings?.txn_ticker_enabled ?? true`, and calls `setTxnTickerEnabled`.

```tsx
function TxnTickerToggle({ sessionId }: { sessionId: string }) {
  const settings = useQuery(api.settings.public.getSettings, {});
  const setEnabled = useMutation(api.settings.public.setTxnTickerEnabled);
  const [busy, setBusy] = useState(false);
  const enabled = settings?.txn_ticker_enabled ?? true;

  async function handleToggle(next: boolean) {
    setBusy(true);
    try {
      await setEnabled({
        idempotencyKey: crypto.randomUUID(),
        sessionId: sessionId as Doc<"staff_sessions">["_id"],
        enabled: next,
      });
      toast.success(next ? "Sales ticker enabled" : "Sales ticker disabled");
    } catch (err) {
      toast.error(errorMessage(err));
    } finally {
      setBusy(false);
    }
  }

  return (
    <div className="flex items-center gap-3 rounded-md border p-3">
      <Switch
        id="txn-ticker-toggle"
        checked={enabled}
        onCheckedChange={handleToggle}
        disabled={busy || settings === undefined}
        aria-label="sales ticker toggle"
      />
      <Label htmlFor="txn-ticker-toggle" className="cursor-pointer text-sm">
        Post each paid sale to the Managers channel
        <span className="ml-1 text-xs text-muted-foreground">(silent)</span>
      </Label>
    </div>
  );
}
```

Render site (inside the page body, right after `<FoundersSummaryToggle sessionId={sessionId} />`):

```tsx
<FoundersSummaryToggle sessionId={sessionId} />
<TxnTickerToggle sessionId={sessionId} />
```

No new imports required — `Switch`, `Label`, `useQuery`, `useMutation`, `toast`,
`errorMessage`, `Doc` are all already imported by the founders toggle on this page.

**⚠️ Test-harness impact (call-order-indexed mocks).** `telegram-chats.test.tsx` mocks
`useQuery`/`useMutation` by **depth-first call order**, not by name (see its header comment, lines
~12-20, and `setupMutationMock`, lines ~127-131). Today:

```
useQuery:    0 = settings (Founders)             1 = chats
useMutation: 0 = setFoundersSummaryEnabled       1 = assignRole   2 = archiveChat
```

Rendering `<TxnTickerToggle>` right after `<FoundersSummaryToggle>` inserts one `useQuery` and one
`useMutation` call **mid-order**, shifting every later stub:

```
useQuery:    0 = settings (Founders)   1 = settings (Ticker)   2 = chats
useMutation: 0 = setFoundersSummaryEnabled   1 = setTxnTickerEnabled   2 = assignRole   3 = archiveChat
```

Convex dedupes the two `getSettings` subscriptions at runtime, but the **mock counts literal hook
calls** — both slots must be stubbed. The implementation MUST therefore (before adding new tests):
1. Update `setupQueryMock` so `settings` is returned for query slots **0 AND 1**, `chats` at slot **2**.
2. Update `setupMutationMock` to add `setTxnTickerEnabled` at mutation slot **1**, shifting
   `assignRole`→**2**, `archiveChat`→**3**.
3. Update both header comment blocks (lines ~12-20 and ~127-131) to the new ordering.

Skipping this regresses the existing founders-toggle AND chat-card tests.

### 3. Tests

## Out of scope (YAGNI)

- **No new route or page** — co-located with the founders toggle on `/mgr/telegram-chats`
  (decision: both are Telegram-notification toggles; a dedicated `/mgr/notifications` page was
  considered and rejected as premature).
- **No schema change** — the field already exists.
- **No PIN gate** — low-stakes config (rule #22).
- **No dev/prod env plumbing** — the toggle writes per-deployment `pos_settings`, exactly like
  the founders toggle.
- **No founders-toggle changes** — left as-is.

## Deployment & rollback

- **Additive, no skew hazard.** `setTxnTickerEnabled` is a brand-new public mutation (no
  mutation↔action rename at an existing name), so there is no deploy-skew fatality — it ships
  atomically with the FE via the normal `npm run build` Convex+FE deploy (CLAUDE.md "Convex
  deployment"). Old FE never references it; new FE references the new name.
- **No migration / backfill.** The `txn_ticker_enabled` field already exists; existing rows simply
  start returning the `?? true` default until first written.
- **Rollback** = revert the PR. Even reverted, the Convex-dashboard edit of `txn_ticker_enabled`
  remains the break-glass kill-switch (unchanged from v1.0.1). Per-deployment: flipping dev does not
  affect prod and vice-versa (each writes its own `pos_settings`).

## Acceptance

1. A manager on `/mgr/telegram-chats` sees a "Post each paid sale to the Managers channel"
   switch, defaulting ON (matching the read default), beneath the founders-summary switch.
2. Flipping it OFF persists `txn_ticker_enabled: false`; the next paid sale produces NO
   `txn_ticker` Telegram message (verified by `txnTicker.ts:39`'s `skipped: "disabled"` path).
3. Flipping it back ON resumes the ticker.
4. A non-manager session cannot call `setTxnTickerEnabled`.
5. Each flip writes a `settings.txn_ticker_toggled` audit row.
6. `npm run typecheck` + the settings and telegram-chats test suites pass.
