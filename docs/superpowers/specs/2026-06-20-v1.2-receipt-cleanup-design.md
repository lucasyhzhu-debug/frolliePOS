# Frollie POS — v1.2 #13 Receipt Cleanup (design spec)

**Date:** 2026-06-20
**Phase:** v1.2 #13 · `M · deps: #2 (shipped), #10 (shipped)`
**Status:** Design — feeds `/spec-plan-pipeline`. One PR.
**Source roadmap:** `docs/superpowers/specs/2026-06-18-pos-backlog-roadmap-design.md` (#13).

## Problem

The printed/digital receipt has three wording warts the booth flagged:

1. **`LUNAS` / `[ LUNAS ]` badge on paid receipts** — noise. A receipt the customer is
   handed is paid by definition; the status badge only earns its place for the **refund**
   states (`SEBAGIAN DIKEMBALIKAN`, `DIKEMBALIKAN`).
2. **Two-line payment block** — `Dibayar via QRIS` then `RRN: <id>` on a second line.
   Verbose; reads better as one compact line.
3. **Stale config strings on the prod row** — the business name renders **"Dapur Raya"**
   (a leftover) and the footer is **"Terima kasih! 💛"**. The booth wants **"FROLLIE"** +
   an English **"Thank you!"**.

A fourth, adjacent correctness bug surfaced while grounding: **manual-BCA sales (#10) render
`QRIS` as their payment method**, because the cancelled QRIS invoice still resolves through
`_getPaidInvoiceForTxn_internal`. Fixed here (decision locked: fix in #13).

### Architecture note — two renderers, one view-model

`ReceiptViewModel` (`convex/receipts/template.ts`) is the single contract. Two **independent**
consumers render it:

- **HTML** — `template.ts::renderReceipt` (pure, V8/server; cached 24h in `pos_receipt_html_cache`).
- **Thermal** — `src/lib/escpos.ts::encodeReceipt` (ESC/POS bytes, client-side Web Bluetooth).

Both must change in lockstep, and **neither may branch on `payment_method` string values**
(method-agnostic, so #10's tender needs no rework). The method *label* is decided once, upstream,
in `convex/receipts/internal.ts::buildVmFromTxnWithLines`.

## Decisions (locked)

1. **Suppress the paid badge only when `status === "paid"`.** Keep the refund-state badges
   (`SEBAGIAN DIKEMBALIKAN` / `DIKEMBALIKAN`) verbatim — they carry real signal.
2. **Collapse the payment block to one line, middot-separated:** `"{payment_method} · {rrn}"`.
   - No RRN → just `"{payment_method}"`.
   - `payment_method === "—"` (no invoice) → `"—"`.
   - Drop the `Dibayar via` prefix and the `RRN:` label entirely.
   - Middot matches existing house style (the VM type comment already cites `"QRIS · GoPay"`).
3. **Manual-BCA method label.** In `buildVmFromTxnWithLines`, when the txn was confirmed via
   `manual_bca`, set `payment_method = "Transfer Bank (manual)"` and `rrn = undefined`
   (a manual transfer has no Xendit RRN). This overrides the leaked cancelled-QRIS invoice.
   Requires the txn aggregate to expose `confirmed_via` (verify in plan; add to the projection
   if absent).
4. **Footer + header:**
   - **Code default:** change `RECEIPT_DEFAULTS.footer_text` → `"Thank you!"` and the
     `SAMPLE_RECEIPT` fixture to match, so fresh/dev deploys and golden tests reflect the
     English direction. `business_name` default is already `"FROLLIE"` — no code change.
   - **Prod data:** the live `pos_settings` row currently overrides both
     (`receipt_business_name:"Dapur Raya"`, a custom footer). The code default does **not**
     reach an already-set field — so the prod fix is a **manual `/mgr/receipt` edit**
     (business name → `FROLLIE`, footer → `Thank you!`). **Owner-owned post-merge step**, not code.
5. **i18n boundary.** Receipts are explicitly **out of scope for v1.2 i18n** (ADR-049). The receipt
   *body* stays Indonesian (`Subtotal`, `Pengembalian`, `Simpan struk ini…`, line annotations).
   Only the configurable footer moves to English, and only the two warts above are touched. No
   body-string translation in this phase.

## Changes

### A. HTML renderer — `convex/receipts/template.ts`

- **Status badge:** render the badge `<div>` only when `status !== "paid"`. Paid receipts emit
  no badge. Keep `STATUS_LABELS` exported (the thermal print query still reads `.label` for the
  refund states).
- **Payment block:** replace the two-line `Dibayar via … / RRN: …` block with a single centered
  line: `payment_method` then `· {rrn}` only if `rrn` is present.

### B. Thermal renderer — `src/lib/escpos.ts`

- **Status line:** emit `[ {statusLabel} ]` only when `_status !== "paid"`. (The `_status` arg is
  already passed in — it's currently unused-prefixed; this gives it a job. Keep `statusLabel` for
  the refund states.)
- **Payment line:** replace lines 78-79 with one line: `payment_method` + ` · {rrn}` when `rrn`
  is present.
- **`SAMPLE_RECEIPT`:** update `footer_text` → `"Thank you!"`. (`statusLabel:"LUNAS"` stays as the
  fixture value — it's still the correct label *string*; the renderer just won't print it for the
  paid sample. The thermal golden test asserts the absence of `[ LUNAS ]` for paid.)

### C. Method label — `convex/receipts/internal.ts`

- In `buildVmFromTxnWithLines`, branch the method/rrn derivation on the txn's `confirmed_via`:
  `manual_bca` → `payment_method = "Transfer Bank (manual)"`, `rrn = undefined`; otherwise the
  existing invoice-derived label.
- **Verified:** `_getPaidTxnWithLinesForReceipt_internal` (and the by-token variant) return the
  **full** txn doc via `ctx.db.get`, so `confirmed_via` is already present at runtime — **no
  `transactions/internal.ts` change needed.** The only edit is widening
  `buildVmFromTxnWithLines`'s `txn` param type (`convex/receipts/internal.ts:27-36`) to include
  `confirmed_via?: "webhook" | "polling" | "manual" | "manual_bca" | null` (it currently narrows
  the doc to a 6-field subset). When `manual_bca`, skip the invoice read entirely (no RRN to fetch).

### D. Footer default — `convex/settings/internal.ts`

- `RECEIPT_DEFAULTS.footer_text: "Terima kasih! 💛"` → `"Thank you!"`.

### E. Tests (update assertions; renderers stay TDD-covered)

- `convex/receipts/__tests__/template.test.ts` — paid receipt asserts **no** `LUNAS` badge;
  refund states still assert their badges; payment line asserts `"{method} · {rrn}"` (and method-only
  when no RRN).
- `src/lib/__tests__/escpos.test.ts` — paid golden asserts **no** `[ LUNAS ]`; refund goldens keep
  their `[ … ]`; one-line payment assertion; footer `"Thank you!"`.
- `convex/receipts/__tests__/getReceiptForPrint.test.ts` — `statusLabel` projection unchanged
  (still returns the label string); adjust any assertion that conflated "label exists" with
  "label printed".
- New/extended test: a `manual_bca`-confirmed txn renders `payment_method = "Transfer Bank (manual)"`
  with no RRN, on **both** renderers — guards the #10 correctness fix.
- **Scoping regression test:** a `manual` (manager-PIN override, `source:"manual"`) confirmed txn
  still renders its **real invoice method** (e.g. `QRIS`), NOT `"Transfer Bank (manual)"` — locks
  the branch to `manual_bca` only and blocks a future over-broad widening.
- **Payment-line shape tests (both renderers):** (a) method **with** rrn → `"{method} · {rrn}"`;
  (b) method **without** rrn → method only, no trailing ` · `; (c) no-invoice `"—"` → `"—"` alone.

## Risks / collisions

- **#2 + #13 both touch receipt renderers** — #2 (phthalo-dark) shipped; confirm no in-flight
  branch is mid-edit on `template.ts` styling at merge time (rebase + revalidate per ship-it memory).
- **Method-agnostic invariant** — the renderers must not switch on method strings; the only
  method-aware code is the single upstream label derivation in `internal.ts`.
- **Cache staleness** — changing the renderer means existing `pos_receipt_html_cache` rows
  (24h TTL) still hold old HTML, including already-cached `manual_bca` receipts that show `QRIS`.
  The fix is **not retroactive** to cached rows; they self-heal within 24h as the TTL expires.
  Acceptable for a wording/label change — no forced purge (refund commits already purge; a deploy
  doesn't have to). Note in CHANGELOG.
- **`_status` arg in `encodeReceipt`** — currently prefixed `_status` (unused). The suppression
  branch (`_status !== "paid"`) gives it a job; rename to `status`. Callers pass positionally
  (`useThermalPrinter`, tests) so no call-site change. Cosmetic.
