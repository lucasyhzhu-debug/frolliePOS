## Decisions (locked)

1. **Scope:** all 6 files enter the ESLint registry; the 4 with real validation convert.
2. **Fence:** ban `toast.error(t(...))` / `toast.warning(t(...))` in registered files (the
   post-i18n shape of an escaped sync validation), keeping the existing literal ban.
3. **No new message type.** Every case maps onto the existing `FieldMessage`:
   - per-field error → `FieldMessage` under the input (slice-1 pattern);
   - settlements **net < 0** → an **MDR-field** error ("MDR can't exceed gross"), not a
     cross-field banner — it points at the fixable input;
   - receipt **logo** validations → `FieldMessage` under the logo upload control (new
     anchor, same primitive);
   - **preconditions** (settlements missing-intent guard 160; device-not-ready guard 43)
     are not validation — they stay the rare defensive `toast` (the device submit is
     already `disabled={!deviceId}`).
4. **Reuse verbatim:** `FieldMessage` (`error`/`success` tones, dark-lifted
   `--color-error`/`--color-success`, dotted-id convention), `useFieldErrors`
   (`applyErrors`/`clearFieldError`/`clearErrors`, `prefix.field` keys). Server errors keep
   routing through `humanize*Error(err, t)` / `friendly*(err, t)`.

## Message placement (the taste call)

Validation messages anchor to **what the person controls** (frontend-design principle):
the message lives under the offending control, one job per message, specific + actionable
in the interface voice. Collect-all + focus-first (slice-1) shows every field's error at
once and focuses/`scrollIntoView`s the first errored field on submit.

| settlements form | `entry.date`, `entry.gross`, `entry.mdr`, `entry.count`, `entry.last4` | net<0 → `entry.mdr` |
| staff add | `add.name`, `add.pin` | — |
| staff rename (inline row) | `rename.name` | — |
| device activate | `act.code`, `act.label` | — |
| receipt logo | `logo.file` (under the upload control) | — |

## Fence redesign (ADR-048 amendment)

ADR-048's fence keys on `arguments.0.type='Literal'`. Post-i18n, a sync validation is
`toast.error(t("..."))` — a `CallExpression` the heuristic treats as legal server/async.
So the fence is toothless for i18n-converted files. New rule, appended to the registry
block in `eslint.config.js`, for the slice-2 files:

```js
// sync validation, post-i18n shape — BANNED in registered files
{ selector: "CallExpression[callee.object.name='toast'][callee.property.name='error'][arguments.0.type='CallExpression'][arguments.0.callee.name='t']",
  message: "Sync form-validation must use <FieldMessage>, not toast.error(t(...)). Server errors route through humanize*Error(err, t). See ADR-048." },
{ selector: "CallExpression[callee.object.name='toast'][callee.property.name='warning'][arguments.0.type='CallExpression'][arguments.0.callee.name='t']",
  message: "Sync form-validation must use <FieldMessage>, not toast.warning(t(...)). See ADR-048." },
```

The existing literal-arg bans stay (they catch `staff.tsx`'s raw literals). Net convention,
now crisp and machine-checkable: **server errors always go through a humanizer
(`humanize*`/`friendly*`); a bare `toast.error(t(...))` always means a sync validation that
escaped inline.**

**Tension this resolves:** `stock/$skuId.tsx` shows *server* errors as
`toast.error(t("stockDetail.errManagerOnly"))` (catch block, 42–45) — legitimately
bucket-C, but it'd trip the new `t()` ban. Fix: extract the catch-block classification into
`humanizeThresholdError(err, t)` (the established `humanize*Error(err, t)` shape) so the
call becomes `toast.error(humanizeThresholdError(err, t))`. Behavior identical; fence
passes; file joins the registry. `mgr/stock.tsx` already uses a humanizer → register as-is.

## Per-file conversion recipe

Each converting file follows slice 1:
1. `const { errors, applyErrors, clearFieldError, clearErrors } = useFieldErrors();`
2. Replace early-return `toast.error(...)` validation with **collect-all**: build a
   `Record<field,msg>` of every failure, `applyErrors(prefix, next, focusMap)`, bail if it
   returned `true`.
3. Render `<FieldMessage>` under each field, gated on `errors["prefix.field"]`; add
   `aria-invalid` + `aria-describedby="prefix.field-error"` on the input and
   `id="prefix.field-error"` on the message.
4. `clearFieldError("prefix.field")` in each input's `onChange`; `clearErrors(prefix)` on
   dialog open/close.
5. Keep `toast.success` and `humanize*`/`friendly*` server toasts untouched.
6. Add the file to the ESLint registry `files:` array.

Copy for the converted messages reuses the **existing i18n keys** (`settlements.errorXxx`,
`mgrReceipt.logoXxx`, `deviceActivation.toastXxx`) via `t(...)` — no new keys except
`mgrStaff.*` for `staff.tsx`'s two raw literals (currently untranslated — converting fixes
that i18n gap) and one MDR-exceeds-gross key. The net<0 message gets a clear key
(e.g. `settlements.errorMdrExceedsGross`).

## Testing

Per converting file, an inline-behavior test on the slice-1 `renderRoute` harness (mock
`useIdempotency` → string to dodge the jsdom-undefined-disables-submit trap noted in
slice-1 lessons):
- invalid submit → expected `FieldMessage`(s) present with `role="alert"`, **no toast
  fired** (assert the `sonner` mock not called);
- valid submit → proceeds (PIN sheet opens / action dispatched);
- a field-specific case per form (e.g. settlements net<0 surfaces under MDR).

Plus a fence smoke-test (manual, documented in the plan): a scratch
`toast.error(t("x"))` in a registered file fails `npm run lint`;
`toast.error(humanizeX(e))` passes; remove scratch.

`humanizeThresholdError` gets a small pure unit test (each error-code branch → key).
