# v0.5.1 — Refunds + customer receipts (design spec)

**Date:** 2026-05-31
**Status:** Approved (brainstorm complete; ready for `/superpowers:writing-plans`)
**Phase:** v0.5.1 — Refunds + customer receipts (per `docs/PROGRESS.md` line 821)
**Approach:** Two PRs, receipts-first, refunds-second (locked in brainstorm Approach 2)
**Predecessor:** v0.5.0.1 housekeeping hotfix (merged to main, PR #7, retroactively renumbered from `v0.5.1 — Housekeeping`)

## Goal

Ship the full refund ledger + a customer-facing signed-URL receipt that correctly projects refund state, without ever mutating the original sale. Money movement for refunds stays manual in v1 per ADR-038; the POS is the system-of-record + audit + settlement-tracker.

**"You'll be able to" outcomes** (carried verbatim from PROGRESS.md):
- Issue refunds end-to-end — staff initiate, manager approves (inline at booth, or via Telegram off-booth), refund logged as a new row
- Share signed-URL receipts — customer scans/taps, gets an itemised receipt
- See refunded lines clearly on the public receipt, with the original sale never mutated
- Audit every refund with manager-approver, reason, and timestamp
- Track which approved refunds still owe the customer money — `settlement_status: pending → settled` (manager moves cash manually in Xendit/BCA dashboard per ADR-038)

**"Still not yet" out-of-scope reminders** (the design enforces these — they're not on the build list):
- POS does NOT call Xendit refund or Disbursements APIs (manual transfer, mgr settles in dashboard then marks settled — ADR-038)
- No stock-in / stock-check / spoilage flow in-app (v0.5.2)
- No transaction history view (v0.5.3) — refunds reached via "today's recent" list only
- No settlements dashboard with filters/history (v0.5.3) — only `/mgr/refunds-pending` for the active queue
- No receipt template configuration UI (v0.5.3) — template hardcoded for v0.5.1
- No staff/product management surfaces (v0.5.3)

## Architecture overview

Two PRs, sequenced, each shippable on its own.

### PR A — Receipt subsystem (ships standalone)

Adds the capability for every paid sale to have a shareable signed-URL receipt. No refund affordances yet — the receipt template is the paid-only variant.

- **New convex module:** `convex/receipts/{schema,public,internal,http}.ts`
- **New table:** `pos_receipt_html_cache { token, html, expires_at }` with `by_token` index. 24h TTL per ADR-022; **lazy regenerate on read** (no reaper cron — storage is cheap; the cron would be ceremony).
- **Modified table:** `pos_transactions.receipt_token: v.optional(v.string())` (optional so pre-v0.5.1 rows don't break schema validation). Unique index on `receipt_token`.
- **Modified mutation:** `convex/transactions/internal._confirmPaid` mints a 32-byte URL-safe random token (reusing the `randomBytes(32).toString("base64url")` pattern from `convex/approvals/actions.ts:71`) and patches the txn row. **Helper extraction:** if `mintUrlSafeToken(bytes = 32)` ends up used in 2+ places after PR A lands (currently 1 in approvals + 1 in receipts = 2), extract to `convex/lib/tokens.ts` in PR A. Otherwise inline the call and refactor on the third caller.
- **New httpAction:** `GET /r/:token` registered in `convex/http.ts`. Returns `text/html`, sets `Cache-Control: private, max-age=300` so refreshes feel instant.
- **New frontend lib:** `src/lib/receipt-template.ts` (hardcoded, ADR-039 §4 layout, Indonesian copy). `src/lib/format.ts` gains `rp()` negative-amount handling.
- **Time formatting:** the receipt template renders datetimes in WIB regardless of server timezone using the existing `convex/lib/time.ts` helpers (`wibDayWindow`, etc.). If a `formatWibDateTime(epochMs)` helper doesn't exist there yet, extend `time.ts` rather than hand-roll the offset arithmetic in the template (which is how timezone bugs ship).

**Ship deliverable:** every paid sale produces a shareable `/r/<token>` URL that returns a clean Indonesian-language receipt.

### PR B — Refund subsystem + settlement (builds on A)

Adds the full refund ledger, both approval paths, and the manager settlement surface. Receipt template gains the refund block + the cache-purge wiring fires here.

- **New convex module:** `convex/refunds/{schema,public,internal,actions}.ts` (actions hosts the argon2-verifying entry points `commitRefundInline` and `requestRefundApproval` per CLAUDE.md rule about argon2 long-running ops). The ADR-040 math helper lives at **`convex/refunds/lib.ts`** specifically — matches the project's `approvals/lib.ts` + `transactions/flags.ts` pattern of putting pure-function helpers in a `lib.ts` or named-file sibling of `public.ts`/`internal.ts`. Unit-tested in isolation.
- **New table:** `pos_refunds` (full shape below) with `by_transaction` + `by_settlement_status` indexes.
- **Modified table:** `pos_transaction_lines.refunded_qty: v.number()` (defaults to 0, atomically incremented).
- **Modified module:** `convex/approvals/kinds.ts` gains `refund` kind across all 4 touchpoints (`ApprovalKind` union, `validateContext`, `KIND_AUDIT`, `KIND_TEMPLATE`). `convex/approvals/actions.ts` gains `approveRefund`. (`denyRequest` is already kind-agnostic since v0.4.)
- **Modified module:** `convex/telegram/send.ts` + `convex/lib/telegramHtml.ts` gain `refund` template kind.
- **Modified module:** `convex/receipts/internal.ts` gains `_purgeReceiptCache_internal` (called by `_commitRefund_internal`). `src/lib/receipt-template.ts` gains the refund-block + status-header logic per ADR-039 §4.
- **Modified module:** `convex/inventory/internal.ts` gains `_refundReCredit_internal` (writes positive stock movements per ADR-019).
- **Modified module:** `convex/audit/internal.ts` action verbs gain `refund.requested`, `refund.committed`, `refund.denied`, `refund.settled`. (Requested fires when staff initiates Telegram approval flow, mirroring v0.4 `manual_payment_override.requested`.)
- **New routes:** `src/routes/refund/index.tsx` (today's recent list, Q1=B), `src/routes/refund/[txnId].tsx` (refund form with stepper, Q7=A), `src/routes/mgr/refunds-pending.tsx` (Q2=A).
- **New components/hooks:** `src/components/pos/RefundLineSelector.tsx`, `src/hooks/useRefund.ts`. `src/components/pos/ApprovalPending.tsx` gains a `refund` variant. `src/routes/approve/index.tsx` + `approve/pin.tsx` gain the refund kind UI.
- **New ADR:** `docs/ADR/040-voucher-attribution-partial-refunds.md`. **Drafted on `main` before PR B starts** (single commit, not part of either feature branch) — cleaner provenance, the math helper unit-test references an ADR that already exists in the file tree rather than one landing in the same squash. Cost: one extra commit on main; benefit: the ADR is reviewable as a standalone change and exists as a stable reference for the duration of PR B's work.

**Ship deliverable:** staff initiate refund → mgr approves (inline PIN or Telegram URL+PIN) → receipt auto-reprojects → mgr settles cash out-of-band → mgr clicks settled in `/mgr/refunds-pending`.

## Data model

### `pos_refunds` (new, PR B)

```ts
{
  transaction_id: v.id("pos_transactions"),
  lines: v.array(v.object({
    line_id: v.id("pos_transaction_lines"),
    qty: v.number(),                              // refund qty per line (≤ line.refundable)
    refund_amount: v.number(),                    // per-line, ADR-040 floor-rounded
  })),
  total_refund: v.number(),                       // sum of lines[].refund_amount, in IDR integer
  reason: v.string(),                             // free-text from staff
  requested_by: v.id("staff"),                    // staff who initiated
  approver_id: v.id("staff"),                     // manager whose PIN authorised
  approval_source: v.union(
    v.literal("booth_inline"),
    v.literal("telegram_approval"),
  ),
  approval_request_id: v.optional(v.id("pos_approval_requests")),  // present for telegram path
  settlement_status: v.union(v.literal("pending"), v.literal("settled")),
  settled_by: v.optional(v.id("staff")),
  settled_at: v.optional(v.number()),
  created_at: v.number(),
}
```

**Indexes:** `by_transaction(transaction_id)` for receipt rendering; `by_settlement_status(settlement_status, created_at)` for the `/mgr/refunds-pending` list (oldest-first ordering implicit).

### `pos_receipt_html_cache` (new, PR A)

```ts
{
  token: v.string(),
  html: v.string(),
  expires_at: v.number(),
}
```

**Index:** `by_token(token)`. **No reaper cron** in v0.5.1; expired rows just regenerate on read.

### Field additions

| Table                    | Field            | Type                        | Default            | Notes                                                   |
| ------------------------ | ---------------- | --------------------------- | ------------------ | ------------------------------------------------------- |
| `pos_transactions`       | `receipt_token`  | `v.optional(v.string())`    | mint in `_confirmPaid` | Optional so pre-v0.5.1 rows stay valid. Unique index. |
| `pos_transaction_lines`  | `refunded_qty`   | `v.optional(v.number())`    | undefined ≡ 0      | **MUST be optional** — Convex schemas have no field defaults, so adding a required field would fail validation on every existing v0.3-v0.5.0.1 row. All reads go through helper `lineRefundedQty(line) = line.refunded_qty ?? 0`. Patch writes still set a number, not undefined. |

### ADR-040 source-field mapping

The voucher math helper signature: `computeRefundAmount(line: Doc<"pos_transaction_lines">, txn: Doc<"pos_transactions">, refundQty: number): number`. Source fields are pinned to actual schema:

| ADR-040 variable          | Schema field                              |
| ------------------------- | ----------------------------------------- |
| `paid_total`              | `pos_transactions.total` (post-voucher)   |
| `subtotal_pre_voucher`    | `pos_transactions.subtotal` (pre-voucher) |
| `line_total`              | `pos_transaction_lines.line_subtotal`     |
| `line.qty`                | `pos_transaction_lines.qty`               |
| `refund_qty`              | `pos_refunds.lines[i].qty`                |

### Computed (never stored)

- `line.refundable = line.qty - line.refunded_qty`
- `txn.status = "paid" | "partial_refund" | "refunded"` — computed on read per ADR-008 from sum of refund `lines[].qty` against original line qty.

## Data flow

### Sale-time receipt mint (PR A)

```
Xendit webhook  →  _confirmPaid mutation  →  mint 32-byte url-safe random
                                          →  patch pos_transactions.receipt_token
                                          →  (no cache pre-warm; first /r/ request renders)
```

### Customer loads `/r/<token>` (PR A)

```
GET /r/:token (httpAction in convex/receipts/http.ts)
  1. lookup pos_transactions by_receipt_token index
  2. token missing → return 404 page (Indonesian, no POS branding)
  3. token found:
     a. **Status guard:** if txn.status != "paid", return 404 (defence-in-depth — tokens are only minted at _confirmPaid, so this should never fire; protects against manual DB patches and future bugs)
     b. check pos_receipt_html_cache by_token; if hit and expires_at > now → return cached html
     c. cache miss / expired:
        - call _renderReceipt_internal(txn) → reads pos_settings (cross-module via settings/internal), txn + lines (own module), voucher (cross-module via vouchers/internal), refunds[] (**cross-module via refunds/internal._listForTransaction_internal — NEVER ctx.db.query("pos_refunds") directly; ADR-034 boundary**)
          (in PR A: _listForTransaction_internal is stubbed or absent → render path always sees empty refunds[])
          (in PR B: stubbed helper replaced with real query → ADR-039 refund block + status header)
        - INSERT pos_receipt_html_cache { token, html, expires_at: now + 24h }
        - return html

  **eslint.config.js OWNERSHIP map updates:** PR A adds `pos_receipt_html_cache: "receipts"`. PR B adds `pos_refunds: "refunds"`. Without these, the lint rule `no-cross-module-db-access` won't enforce the boundaries.
```

### Pre-v0.5.1 lazy-mint (PR A, dormant in v0.5.1)

The helper `_lazyMintReceiptToken_internal({ txnId, actor })` exists and is unit-tested but no surface in v0.5.1 invokes it. v0.5.3 history view's "re-send receipt" surface will be the first caller. PR A defines the helper to avoid retro-fitting it later.

**Auth-gate contract:** the helper itself has no session check (it's an internalMutation). The **caller is responsible** for verifying a staff session before invoking. The helper writes an audit row `receipt.token_minted` capturing the `actor` (staff id) so post-hoc auditing can trace which session triggered each lazy mint. The v0.5.3 surface that wires this must require a staff session and pass the resolved `staffId` as `actor`.

### Refund commit (PR B) — atomic, within one mutation

```
_commitRefund_internal(ctx, { transactionId, lines: [{ line_id, qty }], reason, requestedBy, approverId, approvalSource, approvalRequestId? })

  1. Validate:
     - txn exists; status ∈ {"paid", "partial_refund"}
     - every requested line.qty ≤ refundable (line.qty − line.refunded_qty)
  2. Compute per-line refund_amount via ADR-040 helper (single floor at end):
     refund_amount = floor(line_total × paid_total × qty / (subtotal_pre_voucher × line.qty))
     total_refund = sum(refund_amount across all lines being refunded)
  3. INSERT pos_refunds { settlement_status: "pending", ... }
  4. For each refunded line: PATCH pos_transaction_lines { refunded_qty += qty }
  5. For each line's underlying SKU components: INSERT pos_stock_movements
     { kind: "in", source: "refund", sku_id, qty: +N, transaction_line_id }
  6. Call receipts/internal._purgeReceiptCache_internal(txn.receipt_token)
     **Throws PURGE_NO_TOKEN if token undefined.** Pre-v0.5.1 txns can't reach refund flow per Q1=B (recent list cutoff), so this code path is unreachable in normal operation. Assertion-on-violation (rather than silent no-op) surfaces bugs immediately rather than masking them — surfaces include: manual DB patch, future v0.5.3 history-view refund path that bypasses Q1=B, regression in _confirmPaid token-mint. Helper signature: `throw new Error("PURGE_NO_TOKEN — refund commit on txn ${txnId} without receipt_token; investigate _confirmPaid token-mint")`. Unit-tested.
  7. logAudit({
       action: "refund.committed",
       entity_type: "refund", entity_id: refundId,
       actor_id: approverId,
       source: approvalSource,
       metadata: { transaction_id, total_refund, lines_count, reason },
     })

  Wrapped in withIdempotency(authCheck) at every public entry point.
  Convex serialises mutations per-document; OCC handles concurrent refund races.
```

### Two approval paths (mirroring v0.4 `manual_payment_override` exactly)

```
Path 1: Booth-inline (mgr at booth)
  /refund/[txnId] form → mgr taps "Approve" → PIN sheet → submit
  → refunds.actions.commitRefundInline(ctx, { ..., managerPin })
    a. argon2.verify(managerPin, mgr.pin_hash) [in convex action — long-running, can't be mutation]
    b. ctx.runMutation(_commitRefund_internal, { ..., approvalSource: "booth_inline" })

Path 2: Off-booth Telegram
  /refund/[txnId] form → staff taps "Request approval" → submit
  → refunds.actions.requestRefundApproval(ctx, { ..., reason, lines })
    a. **Dedup guard:** check approvals.internal._findPendingRequest_internal({ kind: "refund", transaction_id }); if a pending request already exists for this txn, return its requestId without re-minting (mirrors v0.4 pattern — prevents Telegram spam if staff double-taps)
    b. mint token (32-byte url-safe)
    c. ctx.runMutation(approvals.internal._createRequest_internal, {
         kind: "refund",
         context: { transaction_id, lines, reason, total_refund_preview },
         token_sha256: sha256(token),
         expires_at: now + 60*60*1000
       })
    d. ctx.runAction(telegram.send.sendTemplate, {
         kind: "refund",
         payload: { txnId, total_refund_preview, lines, reason, approveUrl }
       })
    e. **On send failure:** delete the pending request row so next attempt mints fresh (v0.4 pattern); staff sees retry button
    f. ctx.runMutation(approvals.internal._markNotified_internal, { requestId })
    g. logAudit({ action: "refund.requested", actor_id: requestedBy, source: "booth_inline" })

  ── manager taps Telegram URL → /approve/:token → sees refund kind UI → enters PIN ──

  → approvals.actions.approveRefund(ctx, { token, managerPin })
    a. lookup pos_approval_requests by sha256(token); verify expires_at > now; verify pin_attempts < 5
    b. argon2.verify(managerPin, mgr.pin_hash); on fail: increment pin_attempts; on 5 → expire request
    c. ctx.runMutation(_commitRefund_internal, { ..., approvalSource: "telegram_approval", approvalRequestId: requestId })
    d. ctx.runMutation(approvals.internal._markResolved_internal, { requestId, approverId })
```

### Settlement (PR B)

```
refunds.public.markRefundSettled({ refundId, idempotencyKey })
  authCheck: requireManagerSession(sessionId)        [NOT requireManagerPin per ADR-038]
  1. fetch refund row; reject if settlement_status === "settled" (return existing settled_by/settled_at)
  2. PATCH pos_refunds { settlement_status: "settled", settled_by: mgr.staff_id, settled_at: now }
  3. logAudit({ action: "refund.settled", actor_id: mgr.staff_id, entity_id: refundId })

refunds.public.listPendingSettlement()
  authCheck: requireManagerSession
  returns rows where settlement_status === "pending", oldest first (uses by_settlement_status index)
```

## Edge cases + error handling

### Race conditions (all resolve via existing Convex patterns — no manual locking)

| Race | Outcome |
|------|---------|
| Two refund attempts on same line concurrently | Convex OCC rejects the second mutation if `pos_transaction_lines` was patched between read + write. Client retries with fresh data. |
| Off-booth Telegram approval + booth inline approval racing on same refund request | `_markResolved_internal` sees request already resolved → throws `ALREADY_RESOLVED`. UI surfaces "approved by &lt;name&gt; at &lt;time&gt;". |
| Cache purge fires before refund row written | Both step 3 (INSERT refund) and step 6 (purge) are within the same Convex mutation = atomic. No window for stale receipt to render. |
| Customer loads `/r/<token>` mid-refund-commit | Convex reads are snapshot-consistent within a single query. Worst case: customer reloads once to see the updated state. |
| Two managers tap "Mark settled" simultaneously | Second sees `ALREADY_SETTLED { settled_by, settled_at }` error and a toast. |

### Edge cases (all explicit, all locked)

- **Pre-v0.5.1 txn refund attempt:** can't surface in recent list (predates today). Q1=B "contact management" path applies. No special-case code.
- **Cache purge for txn with no `receipt_token`:** `_purgeReceiptCache_internal` no-ops silently when token undefined.
- **No-voucher sale:** `paid_total === subtotal_pre_voucher` → ADR-040 ratio = 1 → `refund_amount = line_total × qty/line.qty` exactly. No special-case code.
- **Full refund (all qty on all lines):** rounding error is zero — each line's `floor(line_total × 1)` is exact.
- **Multiple partial refunds on same sale:** ADR-040 recomputes against ORIGINAL totals each time (stateless). Sum ≤ paid_total guaranteed by floor + refundable_qty cap.
- **Refund of `awaiting_payment` txn:** rejected at validation step 1.
- **Deactivated staff initiating refund:** v0.5.0.1 tightened `_resolveSession_internal`; authCheck rejects upstream.
- **Refund denial:** existing `approvals.denyRequest` (kind-agnostic since v0.4) handles refund denials via the `KIND_AUDIT` mapping. Audit verb: `refund.denied`. No new code needed.

### UX error handling

- **`/refund/[txnId]` validation:** toast for missing reason or zero-qty selection; inline error when stepper hits refundable cap.
- **`/r/<token>` 404:** static Indonesian-language page "Struk tidak ditemukan. Mohon hubungi Frollie." + Instagram CTA. No POS branding (avoids revealing internal system).
- **Telegram send failure:** existing v0.4 pattern — pending request row is deleted so next attempt mints fresh. Staff sees retry button.
- **Session invalid mid-flow:** redirect to `/lock` (this is the v0.5.2 follow-up flagged in v0.5.0.1's CHANGELOG; v0.5.1 inherits the same redirect path).

## ADR-040 outline (drafted as the first commit of PR B)

```
# 040. Voucher attribution on partial refunds: proportional, floor-rounded

Status: Accepted
Group: Pay

Context: Per ADR-024, discount ordering is line → voucher → tax. When a sale
had a voucher and the customer refunds some lines, the refund amount must
honour the proportional discount that line received. Without a locked rule,
implementations could differ (refund list price vs. proportional) and bookkeeping
diverges. The v0.5.1 voucher-math test file needs a spec to reference.

Decision: per-line refund_amount =
  floor(line_total × paid_total × refund_qty / (subtotal_pre_voucher × line.qty))
Single floor at the end — flooring the line total first and multiplying by a
fractional ratio (refund_qty/line.qty) produces non-integer rupiah on partial-
line refunds. Sum across refunded lines = total_refund. Recompute against
ORIGINAL totals each refund (stateless across multiple partial refunds). Single
helper `computeRefundAmount(line, txn, refundQty)` usable from both
_commitRefund_internal and the receipt template's net-retained computation.

Alternatives considered:
- List price (Rp 50.000 in the Q3 example): rejected — customer "consumes" more
  of the voucher than their fair share; bookkeeping-incorrect.
- Manager-discretionary input: rejected — introduces "what did Lucas refund?"
  inconsistency across staff; defeats audit goal.

Consequences:
- Floor favours business by < 1 IDR per refund line; totals stay tidy.
- Multiple partial refunds compose without rounding drift (stateless math against
  original totals).
- Unit test is the spec — the worked example (Rp 130k voucher sale, refund 1 of
  3 cookies → Rp 43.333) is encoded as a test case.
```

### Rollback caveat (PR A)

If PR A merges and is later reverted, tokens already minted on `pos_transactions.receipt_token` become orphans — the `/r/<token>` route 404s for them because the route definition is gone. Tokens have no public meaning without the route, so the leakage is acceptable. CHANGELOG should note: "*Reverting PR A leaves orphan receipt tokens on already-confirmed transactions; the public route 404s for them. To re-enable, re-deploy PR A; tokens remain stable (token field is optional on the row, immutable when set).*"

### PR-A-revisit caveat (PR B)

If PR A needs post-merge fixes mid-flight on PR B, the protocol is: land the fix as a separate hotfix branch + PR off main, merge, then rebase PR B onto the updated main. Avoid cherry-picking the fix into PR B — keeps each PR's scope clean.
