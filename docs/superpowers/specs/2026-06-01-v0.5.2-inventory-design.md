## 1. Context

The original v0.5.2 roadmap line ("stock-in + stock-check + neg-stock reconciliation") was rescoped during the 2026-06-01 brainstorm once the real operating model surfaced:

- **Inventory is Frollie-Pro-driven, FPOS-confirmed.** The mothership ERP (Frollie Pro / FPro) owns inventory *decisions* — replenishment targets, transfer-out orders. FPOS is the **booth-side confirmation + execution** surface. Stock-in and stock-out *requests* originate in FPro and arrive at FPOS via a secure inbound API; staff confirm the physical reality (actual pcs in / out).
- **Off-FPOS sales cause digital drift.** Frollie also sells via food-delivery (Gojek/GrabFood), ~50–80 pcs/day, plus ~50/day in-store. Food-delivery sales decrement physical stock but **not** FPOS's digital count, so digital inventory inflates. Staff need a way to *recount* (absolute set-to-value) regularly.

This is ~5 subsystems. To ship value without blocking on FPro readiness, the work is **split**:

- **v0.5.2 (this spec) — FPOS-internal, zero cross-system dependency.** Inventory view, staff recount, low-stock alerting, neg-stock visibility.
- **v0.5.2b (separate spec, ADR-043) — FPro-driven stock-in/out.** Inbound `convex/api/v1/` request contract (built real), request→confirm state machine, delta-mismatch handling, Telegram notifications. FPro caller is **stubbed** for testing until FPro wires up (the inbound contract is the sanctioned ADR-034 seam; the actual FPro call is the genuinely-v1.1 part).

The **low-stock alert built in v0.5.2 is the trigger** that opens the v0.5.2b stock-in story ("we're down to 20pcs → warning → manager opens FPro"). Building it now means v0.5.2b starts from an already-firing signal.

## 3. Scope

### In scope (v0.5.2)
1. **Inventory / stock-check screen** — all active SKUs with on-hand, status color (negative = red, below reorder point = amber, ok = default); tap a SKU → info block (movement history + threshold). A `pending requests` slot is reserved (empty in v0.5.2; filled by v0.5.2b).
2. **Staff recount** — absolute count entry per SKU; writes a signed `recount` movement (`delta = entered − current_on_hand`); patches `on_hand` to the entered value; always Telegrams the `managers` group; fully audited. Batch-capable (count many SKUs in one session). Hourly in-app nudge.
3. **Low-stock alerting** — per-SKU `reorder_point`; when a decrementing movement (sale or recount-down) pushes `on_hand` below it, Telegram the new `inventory` group. Dedup so it fires once per crossing, re-arms when stock climbs back above.
4. **Neg-stock visibility** — negatives surfaced on the inventory screen; corrected via recount or v0.5.1 refund.
5. **Manager reorder-point config** — manager-gated mutation to set/update `reorder_point` per SKU.

### Out of scope (deferred)
- **FPro stock-in/out request API + confirm state machine + delta handling** → v0.5.2b (ADR-043).
- **Manager `adjustment` public flow** — `_decrementOnHandUnchecked_internal` exists but no public adjustment mutation ships in v0.5.2; recount covers the staff need.
- **Spoilage** → v0.6.
- **Full NEG_STOCK reconciliation dashboard** → v0.5.3.
- **Offline-queued recount** — recount writes are online-only in v0.5.2 (rule 17's "stock-in queues" is a v0.5.2b concern).

## 4. Schema changes (`convex/inventory/schema.ts`)

1. **`pos_stock_movements.source`** — add `v.literal("recount")` to the existing union (`sale | stock_in | spoilage | adjustment | refund` → `+ recount`). No data migration; existing rows unaffected.
2. **Reorder point = the EXISTING `pos_inventory_skus.low_threshold`** (catalog-owned, already a required `v.number()` on every SKU — PLAN-staffreview Issue 1). **No new threshold table.** Inventory reads `low_threshold` via a catalog cross-module internal (ADR-034), the same way `_getActiveSkuIds_internal` is consumed.
3. **New table `pos_low_stock_alerts`** (inventory-owned — the dedup FLAG only, not the threshold value):
   ```ts
   pos_low_stock_alerts: defineTable({
     inventory_sku_id: v.id("pos_inventory_skus"),
     alerted_at: v.number(),                 // set when alert fired; row deleted/cleared when re-armed
     updated_at: v.number(),
   }).index("by_sku", ["inventory_sku_id"])
   ```
4. **New singleton table `pos_recount_state`** (cheap nudge source): `{ last_recount_at: number, updated_by_staff_id?: v.id("staff") }`, no index (single row). Patched by `recordRecount`. *Avoids a full `pos_stock_movements` scan for "latest recount" — no index supports `by_source` across SKUs.*
5. **Audit actions** — `stock.recount`, `stock.low_stock_alerted`, `stock.low_threshold_set`. Plain strings via `logAudit`; **no enum migration** (consistent with existing `stock.sale_movement`, `stock.refund_movement`). Documented in `docs/SCHEMA.md`.

## 5. Backend (`convex/inventory/`)

### Public surface (`public.ts`)
- **`recordRecount`** (mutation) — args `{ idempotencyKey, sessionId, counts: Array<{ skuId, entered }> }`; `withIdempotency(...)` with `authCheck: requireSession` (rule 21); any valid staff session (no manager-PIN). Handler re-calls `requireSession` for the typed `staffId`. For each SKU: read `on_hand`, compute `delta`; if `delta === 0` skip (clean ledger); else write a `recount` movement (`qty = delta`), patch level to `entered`, audit `stock.recount` with `{ sku, before, after, delta }`. After all writes: patch the `last_recount_at` singleton (Improvement 1), `ctx.scheduler.runAfter(0, internal.inventory.internal._dispatchRecountNotice_internal, …)` to send ONE summary Telegram to `managers` (mutation→action bridge), and call `_checkLowStock_internal` once per touched SKU. Reject negative `entered`.
- **`listInventory`** (query) — session-gated; returns active SKUs with `{ skuId, name, on_hand, low_threshold, status }` where `status ∈ {ok, low, negative}`. Reuses catalog's `_getActiveSkuIds_internal`; reads each SKU's `low_threshold` from the catalog rows it already loads for `name`.
- **`getSkuDetail`** (query) — session-gated; movement history (most-recent-N via `by_sku_created`) + `low_threshold` for one SKU's info block.
- **`setLowThreshold`** (mutation) — args `{ idempotencyKey, sessionId, skuId, lowThreshold }`; `withIdempotency` with `authCheck: requireManagerSession`. Updates the catalog-owned `pos_inventory_skus.low_threshold` (single writer stays in catalog: call a new `catalog/internal._setLowThreshold_internal` or place the public mutation in `catalog/public.ts`); audits `stock.low_threshold_set`.

### Internal surface (`internal.ts`)
- **`_checkLowStock_internal`** (internalMutation) — given a SKU id: read `on_hand` (inventory) + `low_threshold` (catalog internal) + the `pos_low_stock_alerts` flag row. If `on_hand < low_threshold` AND no flag row → insert flag row (`alerted_at = now`), audit `stock.low_stock_alerted` (synchronous, transactional), then `ctx.scheduler.runAfter(0, internal.inventory.internal._dispatchLowStockAlert_internal, …)` (mutation→action bridge). If `on_hand >= low_threshold` AND a flag row exists → delete it (re-arm). Reactive, called from both `_recordSaleMovement_internal` and `recordRecount`.
- **`_dispatchLowStockAlert_internal`** / **`_dispatchRecountNotice_internal`** (internal **actions**) — each calls `ctx.runAction(api.telegram.send.sendTemplate, …)` with the respective kind. **Fail-isolated:** if the target role (`inventory` / `managers`) is unbound, no-op-with-audit (mirror `foundersSummary.ts:73-118` skip), never throw. Scheduled (not inline) so a Telegram outage can never roll back a confirmed sale.
- **`_applyLevelDelta_internal`** (helper, optional but recommended) — shared `(ctx, skuId, delta, now)` level read/patch/insert, de-duplicating the three copies now in `_recordSaleMovement_internal` / `_refundReCredit_internal` / `recordRecount`.

#### Hot sale-path integration (CRITICAL — staffreview Issue 2)
`_recordSaleMovement_internal` (called from `transactions/internal.ts:222`) must:
1. Accumulate the **set of SKU ids actually decremented** this call — i.e. skip lines hit by the ADR-026 dedup `continue` (`internal.ts:31`), and collapse multiple lines sharing one SKU.
2. **After** the line loop, call `_checkLowStock_internal` **once per unique touched SKU**.
3. The existing movement-count / on_hand assertions in `inventory.test.ts:28,75` must remain green (injection is additive, not behavioural).

### Idempotency / auth
Both public mutations follow the dual-call authCheck pattern (rule 21, `docs/PATTERNS/idempotency-dual-call-authcheck.md`): `authCheck` runs before the cache lookup; the handler re-calls `require*Session(...)` for the typed session/`staffId`.

## 6. Frontend (`src/routes/`)

- **`stock.tsx`** — inventory list. Status colors; sticky `SpokeLayout` header; tap-through to detail. Uses `listInventory` (reactive).
- **`stock/recount.tsx`** — recount session. Lists active SKUs each with current on-hand + a `NumericKeypad`-driven "actual count" input; shows per-row delta preview; submit calls `recordRecount` with one `idempotencyKey`. Toast on success (Sonner).
- **SKU detail** — info block (can be a sheet/dialog off `stock.tsx`, or `stock/[skuId].tsx`); movement history + threshold; manager-only "set reorder point" control (`setReorderPoint`).
- **Hourly recount nudge** — a dismissible banner shown when `now − lastRecountAt > 60min`. `lastRecountAt` read from the `pos_recount_state` singleton (server-authoritative, survives reload; not a movement scan, not client-only). Non-blocking; informational.

New hook (optional): `useRecount` wrapping the mutation + idempotency lifecycle if the screen complexity warrants it (mirror `useApproval`).

## 7. Telegram

- **New role `inventory`** — add `"inventory"` to `KNOWN_TELEGRAM_ROLES` in `convex/telegram/config.ts`. Chat registry is data-driven, so the new role is assignable from `/mgr/telegram-chats` with no schema change. The `inventory` group is the "FPOS Inventory" chat.
- **Routing:** low-stock alerts → `inventory`; recount notices → `managers`.
- **New `sendTemplate` kinds** in `convex/telegram/send.ts` (`kind` union + payload union) plus matching renderers in `convex/lib/telegramHtml.ts`:
  - `low_stock_alert` — payload `{ sku_name, on_hand, reorder_point }`. Informational (no URL button / token — not an approval).
  - `recount_notice` — payload `{ staff_name, lines: Array<{ sku_name, before, after, delta }>, recorded_at_iso }`. Informational.
- These are **notifications, not approvals** — they do not touch `APPROVAL_KINDS` / `pos_approval_requests` / the `/approve` UI. Simpler than the rule-20 four-touchpoint approval pattern.

## 8. Flows

### Recount (staff)
1. Hourly nudge banner appears when staff are between transactions.
2. Staff opens `stock/recount.tsx`, enters actual counts for SKUs that differ.
3. Submit → `recordRecount`: per SKU writes `recount` movement (`qty = entered − before`), sets `on_hand = entered`, audits; then one summary Telegram to `managers`; re-checks low-stock per SKU.
4. Inventory screen reflects new counts reactively.

### Low-stock alert (system)
1. A sale or recount-down patches `on_hand` below a SKU's `reorder_point`.
2. `_checkLowStock_internal` fires once (dedup via `low_stock_alerted_at`) → Telegram to `inventory` group + audit.
3. When stock later climbs back above `reorder_point` (a future stock-in/recount-up), the flag re-arms so the next crossing alerts again.

## 10. Testing (rule 7 — stock changes REQUIRE tests)

- `recordRecount`: delta math (up, down, zero, first-ever count with no level row), idempotency replay, audit row shape, multi-SKU batch, Telegram dispatch invoked once.
- `_checkLowStock_internal`: fires on crossing, dedups on repeat, re-arms above threshold, no-op when no threshold set.

- `setReorderPoint`: manager-gated rejection for non-managers; upsert semantics.

## 11. Rollback / migration

- Additive only: one new enum literal, one new table, three new audit-action strings, new Telegram kinds. No backfill. Existing `sale`/`refund` movement code paths unchanged except the additive low-stock check call.
- Reverting = remove the new functions/routes/role; the `recount` literal + `pos_stock_thresholds` table can stay (harmless if unused).

## 12. Risks / open items

- **Hot-path cost:** adding `_checkLowStock_internal` to every sale decrement. Mitigated: indexed read + dedup flag; one extra Telegram send only on a genuine crossing. Confirm acceptable in review.
- **Recount concurrency:** two staff recounting the same SKU near-simultaneously. Last-write-wins on `on_hand`; both movements are logged (audit preserves both). Acceptable for a single-device booth (rare); note in ADR-041.
- **Nudge cadence source of truth:** client-derived `lastRecountAt` vs server query. Prefer server (most-recent `recount` movement) so the nudge is consistent across reloads; client timestamp only as a fallback.
