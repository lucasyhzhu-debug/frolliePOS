# v0.5.3a — Reporting: transaction history + manager dashboard (design spec)

**Date:** 2026-06-01
**Status:** Approved (brainstorm complete; ready for `/superpowers:writing-plans`)
**Phase:** v0.5.3a — first slice of v0.5.3 (per `docs/PROGRESS.md` line 903)
**Approach:** One deep `transactions` query module, two read-only projections (history + dashboard)
**Predecessor:** v0.5.1 (refunds + receipts, merged to main). Runs in parallel with in-flight v0.5.2 (inventory); no code dependency on v0.5.2.

---

## Decomposition context — why this is "3a"

v0.5.3 as scoped in PROGRESS.md is six independent subsystems (history, dashboard, staff CRUD, products CRUD, receipt config, settlements). That is the same shape that forced the v0.5 split into four sub-phases. It is decomposed here into three slices, clustered by *nature of work* (coupling by shared substrate, not by screen):

| Slice | Contents | Why grouped | Risk |
| -- | -- | -- | -- |
| **v0.5.3a — Reporting** *(this spec)* | Transaction history + manager dashboard | Both read-only; dashboard reuses the history query layer (PROGRESS line 922). Shared read substrate. | Low |
| **v0.5.3b — In-app admin** | Staff CRUD + products CRUD + receipt config | All manager-portal CRUD over *existing* tables (`staff`, `pos_products`, `pos_settings`). No new financial logic. Includes the v0.2 follow-up: strip `pin_hash` from `listStaff`. | Low–medium |
| **v0.5.3c — Settlements** | `pos_settlements` table + Xendit settlement webhook + nightly recon | Net-new subsystem, new table, new external-event ingress, new cron. The load-bearing v1.0 "settlement timing" risk-register item. | High |

Each slice gets its own spec → plan → implementation cycle. 3a ships first as the read-only foundation; 3c is specced separately given its risk weight.

## Goal

Give staff and managers a read-only window into sales for any given booth day, plus a laptop-first daily-ops dashboard for managers — built as a single deep query module so the two screens are just projections over one role-gated, WIB-day-windowed read. No new financial logic, no mutations (one internal lazy-mint write aside), no rollup table.

**"You'll be able to" outcomes** (carried from PROGRESS.md lines 908–914, scoped to 3a):
- View transaction history (staff: all of today; manager: everything, any day)
- Use the manager dashboard (laptop-first) for daily sales, payment mix, top SKUs, hourly sales curve, voucher usage, flagged transactions, and per-staff activity
- Open any transaction's detail and re-share its customer receipt (`/r/<token>`, lazy-minting the token if absent)

**"Still not yet" out-of-scope reminders** (the design enforces these — not on the build list):
- No staff/product management surfaces (v0.5.3b)
- No receipt template configuration UI (v0.5.3b — template stays hardcoded from v0.5.1)
- No settlements / payout reconciliation, no `pos_settlements` table (v0.5.3c)
- No multi-day trend charts or comparison-over-time analytics (YAGNI for single-booth daily ops)
- No neg-stock reconciliation action (owned by v0.5.2; 3a only *surfaces* flagged txns and deep-links out)
- No inline mutations from the dashboard (Q5 — read-only, deep-link to owning modules)

| Q | Decision | Rationale |
| -- | -- | -- |
| Q1 | **Today-anchored, one WIB-day window, date-picker to page back.** Manager queries one day at a time; staff fixed to today. | Matches how a booth reviews sales; bounds every query. A growing `pos_transactions` table cannot be lazily `.collect()`'d unbounded — the day window is a query-architecture constraint, not just UX. Reuses `lib/time.ts:wibDayWindow`. |
| Q2 | **Core metrics + hourly sales curve + voucher usage.** Core = gross / net / refunds / txn-count / avg-basket · payment mix (QRIS vs BCA VA) · top SKUs · flagged-needs-attention · per-staff tally. | Daily-ops set for a single booth. Voucher usage reads `pos_voucher_redemptions` (apply-at-sale shipped v0.3; data may be thin until v0.6 adds easy voucher creation). Hourly curve = txns grouped by WIB hour over the day window. Deferred: settlement figures (3c), multi-day trend charts (YAGNI). |
| Q3 | **Detail view + re-share receipt with lazy-mint.** Row → detail (lines, totals, payment method, refund status) + "Share receipt" → public `/r/<token>`, lazy-minting via `_lazyMintReceiptToken_internal` if absent. | Activates the seam left dormant in v0.5.1: `_lazyMintReceiptToken_internal` was built explicitly for "the v0.5.3 history view caller." Reuses the v0.5.1 receipt surface + `refunds/lib.ts` refund-status derivation end-to-end. |
| Q4 | **Staff: all of today (any staffer), no prior days. Manager: everything, any day.** | Single shared booth device with overlapping shifts — a staffer often re-opens a colleague's just-made sale to re-share a receipt; "own-only" fights that reality, and same-day visibility is low-sensitivity. Query needs only a `by_<created_at>` index + a role gate (not a `by_staff_and_day` composite). |
| Q5 | **Read-mostly; deep-link out.** "Needs attention" rows are informational and link to where the action already lives (awaiting-payment → charge screen's `cancelAwaitingPayment`; flagged → txn detail). The *only* write in 3a is `shareReceipt` (mint-on-share). | Keeps 3a essentially a read surface — no reconciliation/financial mutations of its own. Neg-stock reconciliation is owned by v0.5.2 (its title is literally "…+ neg-stock reconciliation"); putting a reconcile control here would duplicate that module and cross a phase seam. *(Amended per staffreview Issue 1: receipt re-share requires one mutation since a query cannot mint.)* |
| Q6 | **Deep module (ADR-034).** The read layer is a deep `transactions` query module: narrow public surface (3 queries) over a deep hidden implementation (day-window fetch, WIB resolution, pure aggregators, refund-status derivation, lazy-mint). | The two screens are projections over one substrate. Depth = lots of capability behind a small interface; reviewers recognise the idiom (mirrors `refunds/lib.ts`). |
| Q7 | **Device-clock independence is a hard invariant.** Report output is a pure function of `(WIB calendar date, caller role)`. The phone's clock/timezone are never inputs. | Falls out of business rule #16 / ADR-031 ("server time wins"). The date picker makes the naive mistake (sending `Date.now()` or a `Date`) easy; pinning the invariant closes it. |

## Architecture overview — one deep module, two projections

A single deep `transactions` query module. History and dashboard are two views over one role-gated, WIB-day-windowed fetch. No duplicated fetch, no duplicated math (pure `lib.ts` aggregators), no materialized rollup (YAGNI at single-booth volume — and a rollup would add the same drift/reconciliation burden as the `pos_stock_levels` cache).

```
  PUBLIC SURFACE (narrow — 3 functions, all session-gated)
  ├─ listDayTransactions(day?)      → history list      (scope derived server-side from role)
  ├─ dashboardSummary(day?)         → dashboard metrics (manager-gated)
  └─ getTransactionDetail(txnId)    → drill-down + refund status + receipt token

  HIDDEN INTERNALS (deep — consumers never see these)
  ├─ _fetchDayWindow_internal(window, scope)   one role-gated WIB-day read  (single source)
  ├─ WIB day-boundary resolution               reuses lib/time.ts:wibDayWindow
  ├─ pure aggregators in transactions/lib.ts   gross/net/topSku/hourly/perStaff/voucher/paymentMix
  ├─ refund-status derivation                  reuses refunds/lib.ts
  └─ receipt lazy-mint trigger                 _lazyMintReceiptToken_internal (on detail open)
```

The surface hides *how* days are bounded, *how* access is gated, *how* metrics are aggregated, *how* refund status is derived, and *whether* a receipt token already existed. Consumers (two screens) just call queries.

## Backend (`convex/`)

- **`transactions/public.ts`** — add three queries + one mutation:
  - `listDayTransactions(day?)` — returns the day's transactions (+ a lightweight per-line summary sufficient for the list and detail) for the caller's scope. Staff scope = today only (any staffer); manager scope = the requested day.
  - `dashboardSummary(day?)` — manager-gated; returns the compact summary object produced by `lib.ts` aggregators.
  - `getTransactionDetail(txnId)` — **pure query**: returns lines, totals, payment method, refund status (via the shared `refundStatus` helper), and the *existing* `receipt_token` (may be `null`). Scope-checked against the caller's role. **Does not mint** — a query cannot run a mutation.
  - `shareReceipt({ idempotencyKey, sessionId, txnId })` — **public mutation** invoked on the "Share receipt" tap. Resolves session → `staff_id`, calls `_lazyMintReceiptToken_internal({ transactionId, actor: staffId })`, returns `{ token }`; the frontend then opens `/r/<token>`. Carries `idempotencyKey` + `withIdempotency(...)` + `authCheck` per business rule #21. Idempotent on re-tap (the underlying `_ensureReceiptTokenForPaidTxn_internal` no-ops when a token already exists). Rejects non-paid txns.
- **`transactions/internal.ts`** — `_fetchDayWindow_internal(window, scope)`: the single role-gated day read. Resolves the `[start, end)` ms window via `wibDayWindow(Date.now())` and reads `pos_transactions` over the **existing `by_status_created` index** (`.eq("status","paid").gte("created_at",start).lt("created_at",end)`), then loads the lines for those txns and the refunds referencing them. Cross-module session resolution uses `internal.auth.internal._resolveSession_internal` (ADR-034) — never reads `staff_sessions` directly — degrading to empty/null on an invalid session, exactly as `listRecentAwaitingPayment` (`transactions/public.ts:309`) does.
- **`transactions/lib.ts`** *(new file)* — pure aggregation helpers, no Convex types in the signatures (mirrors `refunds/lib.ts`):
  - `computeDaySummary(txns, lines, refunds)` → `{ gross, net, refundsTotal, count, avgBasket, paymentMix, topSkus, hourlyCurve, perStaff, voucherUsage, needsAttention }`.
  - Smaller pure pieces composed by it: `bucketByWibHour`, `topSkusFromLines`, `perStaffTally`, `paymentMixFromTxns`, `voucherUsageFromRedemptions`.
- **`refunds/lib.ts`** *(extend)* — extract a pure `refundStatus(txn, lines, refunds) → "lunas" | "sebagian" | "dikembalikan"` from where the LUNAS/SEBAGIAN/DIKEMBALIKAN label is currently derived inside the receipt `template.ts`. Both the receipt template **and** the new history badge consume it — no duplicated three-state logic.
- **Reuse (no duplication):** `lib/time.ts:wibDayWindow` (window resolution), `refunds/lib.ts` `lineRefundedQty`/`lineRefundable`/`computeRefundAmount` (per-line refund math) + the new `refundStatus` label helper, `_lazyMintReceiptToken_internal` (re-share via `shareReceipt`), business-rule #1 snapshot fields on lines (never join lines back to `pos_products` for historical price/name).

### Schema
- **No schema changes at all.** No new tables, **no new index** — day-window reads reuse the existing `by_status_created` `["status","created_at"]` index (`transactions/schema.ts:46`), scoped to `status:"paid"`. (A paid-then-fully-refunded txn keeps `status:"paid"` per ADR-008; drafts/cancelled are not sales. Flagged neg-stock txns are `paid` with a flags bit, caught by the same scan. Awaiting-payment "needs attention" rows come from the existing `listRecentAwaitingPayment`.) The settlements table (`pos_settlements`) stays in 3c.
- Audit enum: the only state-changing path is `shareReceipt` → `_lazyMintReceiptToken_internal`, which reuses v0.5.1's existing audit behaviour for token minting (no new audit verb introduced by 3a — `_ensureReceiptTokenForPaidTxn_internal` already emits the audit row).

## Frontend (`src/`)

- **`routes/history.tsx`** — booth-phone-first reverse-chronological list. Defaults to today; date-picker pages back (manager only — staff date control is fixed/absent since staff are today-only). Row → detail. Shows per-row: time, receipt #, total, payment method, refund-status badge (LUNAS / SEBAGIAN DIKEMBALIKAN / DIKEMBALIKAN).
- **`routes/mgr/dashboard.tsx`** — laptop-first (DashA wireframe). Cards driven by `dashboardSummary`: totals block, payment mix, top SKUs, hourly sales curve, voucher usage, "Needs attention" (flagged neg-stock + stuck awaiting-payment), per-staff tally. "Needs attention" rows deep-link out (awaiting-payment → charge screen; flagged → txn detail). Same day-window + date-picker model as history.
- **Transaction detail** — `routes/history/$txnId.tsx` (or a detail component reused by both screens). Shows lines (snapshot names/prices), totals, payment method, refund status; **Share receipt** action opens/links `/r/<token>`, lazy-minting if absent. Reuses the v0.5.1 public receipt surface — 3a adds no new receipt rendering.
- **Layout:** history sits under the existing `SpokeLayout`/`AppHeader` spoke pattern (v0.5.0); dashboard is a laptop-first manager surface.

## Data flow

```
  screen  ──query(day?)──▶  public query  ──▶  _fetchDayWindow_internal(role-gated)
                                                      │
                                          wibDayWindow │ resolves [start,end) in WIB (server time)
                                                      ▼
                                          read pos_transactions (by index) + lines + refunds
                                                      │
                          history ◀── rows ───────────┤
                          dashboard ◀── computeDaySummary(...) (pure lib.ts aggregators)
                          detail ◀── getTransactionDetail (pure query; existing token, may be null)

  "Share receipt" tap ──▶ shareReceipt MUTATION ──▶ _lazyMintReceiptToken_internal (mint if absent) ──▶ open /r/<token>
```

Live-updating via Convex subscriptions: a new sale or refund landing in the current day window reactively updates both the history list and the dashboard cards. At single-booth volume (tens–low-hundreds of txns/day) the recompute-on-tick cost is negligible.

## Access & correctness

- **Q4 boundary** enforced inside `_fetchDayWindow_internal`: staff requests are forced to the server-resolved current WIB day regardless of any `day` argument; manager requests honour the requested day.
- **Device-clock invariant (Q7):**

  ```
  INVARIANT: report output is a pure function of (WIB calendar date, caller role).
             the phone's clock and timezone are NEVER inputs.

    • day param  = optional WIB date label "YYYY-MM-DD" (a calendar label, not a timestamp)
    • omitted    → server resolves "today" from Date.now() (SERVER) + fixed UTC+7, via wibDayWindow
    • date picker emits a date LABEL; server converts to [start,end) ms in WIB
    • the staff "today" access gate (Q4) also uses server-resolved today
    → a staffer changing device date/timezone cannot shift, widen, or narrow any report
  ```

  No client timestamp or `Date` object is ever sent to these queries. The fixed +7 offset (not the device timezone) guarantees a given date label always maps to the same millisecond window.
- **Read-mostly:** three queries + **exactly one** public mutation (`shareReceipt`). The mutation mints a receipt token on demand (idempotent; no-ops when a token already exists). No other writes. (Corrected from the initial "zero mutations" framing per staffreview Issue 1 — a query cannot mint.)
- **Idempotency/auth:** the three queries use the standard session-resolution pattern (`_resolveSession_internal`); manager-only surfaces (`dashboardSummary`, prior-day history) re-check role server-side. `shareReceipt` is a public mutation → `idempotencyKey` + `withIdempotency(...)` + `authCheck` per rules #15/#21.

## Testing

Per "How to add a feature" #7, reporting is not payment/refund/stock-mutating, so tests are encouraged rather than mandatory — but the aggregation math and the clock invariant are exactly the high-value, cheap-to-test units:

- **Unit (`transactions/lib.ts`):** `computeDaySummary` and each sub-aggregator against fixture day arrays — gross/net/refundsTotal, avg-basket, payment mix, top SKUs, hourly curve buckets, per-staff tally, voucher usage. No Convex harness needed (pure functions).
- **Unit (clock invariant):** assert `(day, role) → output` is stable when the simulated client supplies varied/absent date inputs and would-be-divergent device clocks; assert the staff gate always resolves to server "today."
- **Integration (role gating):** staff cannot read prior days (request for a past day collapses to today); manager can; `getTransactionDetail` rejects cross-scope reads.
- **Integration (`shareReceipt`):** tapping share on a txn without a `receipt_token` mints one and yields a working `/r/<token>`; re-tapping returns the same token (idempotent, no second mint); a non-paid txn is rejected; an invalid session is rejected (authCheck).
- **Unit (`refundStatus`):** the extracted helper returns `lunas` / `sebagian` / `dikembalikan` for none/partial/full refund fixtures — asserted from both the receipt-template path and the history-badge path (one helper, two consumers).

## Success criteria & rollback

**Success criteria (minimum bar):**
- `npm run typecheck` + `npm run build` clean.
- New unit tests green: `transactions/lib.ts` aggregators, `refundStatus`, clock-invariant.
- New integration tests green: role gating, `shareReceipt` mint-once/idempotent/auth-reject.
- Behavioural: history renders today's sales (staff) / any picked day (manager); dashboard cards populate from real data and tick live as a sale lands; tapping a row opens detail; "Share receipt" yields a working `/r/<token>`.

**Rollback:** read-mostly + zero schema change → no migration, no data risk. Rollback = revert the slice's commits. Deployment order: backend (`npx convex deploy`) before frontend (Vercel), per repo convention. `shareReceipt` only ever adds a `receipt_token` to a paid txn that lacked one — a forward-compatible, idempotent write that needs no down-migration.

## Out of scope (restated)
