# Spec — v1.2 #12 slice 1: Contextual messaging over toasts (`FieldMessage`)

**Slug:** `v1.2-phase2-inline-messaging`
**Date:** 2026-06-19
**Phase:** v1.2 Phase 2, backlog item #12 (slice 1 of N)
**Effort:** L · **New ADR:** ADR-048 · **Deps:** #2 phthalo-dark (shipped, `cac9616` / PR #84)
**Status:** design approved by Lucas; this spec locks WHAT slice 1 delivers.

> Pure frontend + lint + docs. **No backend, schema, Convex, or validation-logic change.** Only the *delivery* of sync form-validation feedback moves from toast → inline. Validation conditions themselves are untouched.

---

## 1. Problem

Sonner is the app-wide feedback channel — `Toaster` mounted once (`src/main.tsx`). **154 `toast.*` calls across 23 files.** A large share are **synchronous client-side form-validation** errors fired the instant a manager taps a disabled-looking "Continue"/"Save": the toast appears top-of-screen, detached from the field that's wrong, auto-dismisses, and on a booth tablet often fires *behind* the open dialog. The manager has to read a transient banner, then hunt for the offending input. This is the wrong affordance for "this specific field is invalid right now."

There is **no reusable inline-field-message component** in the codebase today.

### Three toast buckets (from roadmap spec `docs/superpowers/specs/2026-06-18-pos-backlog-roadmap-design.md` §#12)

| Bucket | What | Disposition |
|---|---|---|
| **A** | Synchronous client-side **form-validation** toasts | **Convert to inline** (this phase converts the 2 worst files only) |
| **B** | ~10 PIN-flow toasts | **OUT OF SCOPE** — owned by the coordinated **#11 + #7** phase that rewrites `PinEntry`/`login.tsx`. #12 must not touch login/PIN. |
| **C** | ~70 global/async toasts (print, draft saved, cancelled, server-rejection, Telegram, low-stock) | **KEEP as toasts** — correct affordance for transient global events. |

### Grounded bucket-A counts (corrected — verified in code, not the roadmap's round numbers)

The roadmap quoted "~36" (products) and "~18" (vouchers); those counts include `toast.success(...)` and dynamic server-error toasts, which **stay**. The actual **sync-validation literal `toast.error("…")` calls to convert** are:

- **`src/routes/mgr/products.tsx`: 25** literal-arg `toast.error` validation calls (lines 224, 229, 234, 243, 285, 290, 295, 300, 305, 310, 315, 322, 333, 338, 343, 396, 401, 406, 411, 416, 423, 470, 475, 529, 533). The other `toast.*` calls in this file — `toast.success("Saved")`/`toast.success(\`…added\`)` (7) and `toast.error(humanizeCatalogError(err))`/`toast.error(msg)` (4 dynamic) — **stay.**
- **`src/routes/mgr/vouchers.tsx`: 12** literal-arg `toast.error` validation calls (lines 172, 179, 183, 190, 199, 208, 212, 289, 299, 303, 313, 317). The 3 `toast.success(...)` and 3 dynamic `toast.error(humanizeVoucherError(err))`/`toast.error(msg)` **stay.**

So slice 1 converts **37 sync-validation toasts → inline** across the two files. This split — **string-literal first arg = sync validation (convert); dynamic first arg = server/async (keep); `toast.success` = keep** — is the exact heuristic the ESLint guard encodes (§5).

## 3. `FieldMessage` primitive

**File:** `src/components/ui/field-message.tsx` (new), sitting alongside `card.tsx`/`button.tsx`/`badge.tsx`, new-york/stone + Frollie-token conventions.

### API

```tsx
const fieldMessageVariants = cva(
  // base: layout + left-accent rule + quiet type
  "flex items-start gap-1.5 border-l-2 pl-2 text-sm leading-snug",
  {
    variants: {
      tone: {
        error:   "border-error text-error",
        success: "border-success text-success",
      },
    },
    defaultVariants: { tone: "error" },
  }
);

export interface FieldMessageProps
  extends React.HTMLAttributes<HTMLParagraphElement>,
    VariantProps<typeof fieldMessageVariants> {
  id?: string;          // for aria-describedby wiring on the input
}

export function FieldMessage({ className, tone, children, id, ...props }: FieldMessageProps) { … }
export { fieldMessageVariants };
```

- Renders a leading **lucide icon** by tone — `AlertCircle` (error) / `CheckCircle2` (success), ~14px (`className="size-3.5 shrink-0 mt-0.5"`), `aria-hidden`.
- Renders `role="alert"` + `aria-live="polite"` on the wrapper so the message is announced when it appears/changes.
- The tone color carries via the semantic `error`/`success` tokens (made AA-legible in `.dark` per §4). **Never color alone** — icon + left rule + text together satisfy WCAG 1.4.1.
- An optional faint tint (`bg-error/10` rounded right) is permitted but **not** in the locked base — keep the base minimal; the left-rule is the signature. (Validated in the mock; decide tint inclusion at build, default: include the subtle tint — it read well at booth viewport.)
- Honors `prefers-reduced-motion`: any fade-in is a no-op under reduce. (Optional; a static appearance is acceptable — no motion is required for slice 1.)

### Anatomy (validated against the real phthalo tokens — see §4)

```
│⊘  Price must be a whole number of rupiah      ← border-l-2 + icon + text, one tone
```

Text stays quiet (`text-sm`), specific + actionable, interface voice, sentence case. The existing literal messages in the two files are mostly good copy — **reuse/refine, don't regress** (e.g. keep "Price must be a non-negative integer." or tighten to "Price must be a whole number of rupiah.").

### Variant-guard test

`src/components/ui/__tests__/field-message.test.tsx` — mirror `badge.test.tsx`:
- For each tone in `["error","success"]`, `fieldMessageVariants({ tone })` contains the base class (`border-l-2`).
- A `@ts-expect-error` line proving a removed/invalid tone (e.g. `"warning"`) is not in the union (test files are excluded from `npm run typecheck` — the real CI guard is that any *consumer* using a bad tone fails `tsc -b`; mirror badge.test.tsx's documented note verbatim).

> **Tone set decision:** slice 1 ships **`error` + `success` only** (the two tones the two files actually need). `warning`/`info` are deliberately omitted until a consumer needs them — matches the ADR-047 "no dead tokens" ethos. Adding a tone later is a one-line cva + token addition.

## 4. Token support — AA on the dark canvas (the one real systemic decision)

**Finding (grounded in `src/index.css`):** the semantic `--color-error` (`#DC2626`) and `--color-success` (`#059669`) are **fixed `@theme` tokens, NOT `.dark`-lifted.** On the dark card (`#163630`) their colored *body text* fails WCAG-AA:

| Token | vs card #163630 | vs paper #102821 | AA body (4.5:1)? |
|---|---|---|---|
| `#DC2626` error | 2.7:1 | 3.2:1 | ✗ (fails even 3:1 graphical on card) |
| `#E5484D` (`--destructive`, dark-lifted) | 3.3:1 | 4.0:1 | ✗ (passes 3:1 graphical only) |
| `#059669` success | 3.5:1 | 4.1:1 | ✗ |
| **`#F87171`** (red-400) | **4.7:1** | **5.6:1** | **✓** |
| **`#34D399`** (emerald-400) | **6.8:1** | **8.1:1** | **✓** |
| `#F1E9D8` warm ink | ~12:1 | ~14:1 | ✓ (AAA) |

Validated in `.fieldmessage-mock.html` (throwaway sign-off artifact, screenshotted at booth-tablet 1024×768 in `.dark`): three treatments — (A) base tones, (B) lifted tones + 8% tint, (C) colored rule+icon + warm-ink text. **Treatment B won**: fully-colored message, glanceable, AA-passing. Treatment A's success green is visibly murky on the card; C reads like neutral helper text (too quiet for an error at booth speed).

**Locked aesthetic:** Treatment B — lifted tone for text (`#F87171` error / `#34D399` success) + left-rule + icon, optional subtle tint. **Confirmed by Lucas 2026-06-19** ("option B is best") after reviewing the booth-tablet screenshot.

**Recommended token mechanism (confirm at plan staffreview):** lift `--color-error`/`--color-success` for `.dark` via the **var-indirection pattern #2 already uses** (`--error`/`--success` in `:root` + `.dark`, then `--color-error: var(--error)` in `@theme inline`):

- `:root` (light, effectively unused — app is `<html class="dark">`-only): keep `#DC2626` / `#059669`.
- `.dark`: lift to `#F87171` / `#34D399` (red-400 / emerald-400).

This keeps `FieldMessage` on **semantic tokens** (`text-error`/`text-success`) as Lucas required, and as a bonus lifts Badge's `error`/`success` variants in dark (they use `text-error`/`bg-error`) to AA — class names unchanged, so `badge.test.tsx` stays green; only the resolved color shifts. **Consequence to confirm:** error/success badges render slightly brighter red/green in dark. The plan staffreview must (a) confirm no other consumer depends on the exact `#DC2626`/`#059669` value, and (b) eyeball badges in the screenshot.

**Alternative (smaller blast radius, if Lucas rejects the badge shift):** FieldMessage-local lifted tokens `--color-field-error`/`--color-field-success`, leaving global error/success untouched. Trade-off: near-duplicate tokens, against the ADR-047 prune ethos. **Recommend the indirection lift; fall back only if the badge change is unwanted.**

## 5. ESLint regression guard

The migrated files form a **registry** that grows as later slices convert more files. Add a scoped flat-config block to `eslint.config.js`:

```js
{
  // v1.2 #12 — inline-messaging migration registry. Files in this glob have had
  // their sync form-validation toasts converted to <FieldMessage>; this fence
  // stops them regressing back to literal-arg toast.error/toast.warning.
  // Heuristic: string-literal first arg = sync validation (must be inline);
  // dynamic first arg (toast.error(humanizeX(err))) = server/async, stays legal;
  // toast.success(...) stays legal. Append files here as later slices convert.
  files: [
    "src/routes/mgr/products.tsx",
    "src/routes/mgr/vouchers.tsx",
  ],
  rules: {
    "no-restricted-syntax": [
      "error",
      {
        selector: "CallExpression[callee.object.name='toast'][callee.property.name='error'][arguments.0.type='Literal']",
        message: "Sync form-validation must use <FieldMessage>, not toast.error(\"literal\"). Dynamic server errors (toast.error(humanizeX(err))) stay. See ADR-048.",
      },
      {
        selector: "CallExpression[callee.object.name='toast'][callee.property.name='warning'][arguments.0.type='Literal']",
        message: "Sync form-validation must use <FieldMessage>, not toast.warning(\"literal\"). See ADR-048.",
      },
    ],
  },
},
```

- `[arguments.0.type='Literal']` matches a string-literal **first** argument only → bans `toast.error("Price must…")`, allows `toast.error(humanizeCatalogError(err))` (first arg is a `CallExpression`) and all `toast.success(...)`.
- **Verify at plan/exec time:** confirm the esquery `arguments.0.type` attribute selector fires (esquery supports indexed-path attribute selectors) by lint-smoke-testing a scratch `toast.error("x")` line in a migrated file — it must error; `toast.error(fn(e))` must not. If `arguments.0.type` is unsupported by the installed esquery, fall back to `:matches()`/child-combinator form. (Flagged for the plan to lock the exact working selector.)
- This block must come **after** the project-wide blocks so it layers onto the migrated files (ESLint flat config merges matching blocks).

## 6. Per-field error pattern (in the converted files)

Validation **logic is unchanged** — only delivery moves. Pattern:

- Add a per-form error object in component state: `const [errors, setErrors] = useState<Record<string,string>>({})` (one map per dialog/form, keyed by field — e.g. `name`, `price`, `tax`, `sortOrder`, `skuFamily`, `hue`, …).
- Where a handler currently does `toast.error("…"); return;`, instead set `setErrors(e => ({...e, <field>: "…"}))` and `return`. Collect all field errors in one pass where the handler validates several fields, so every invalid field lights up at once (not just the first).
- Render `<FieldMessage tone="error" id={\`<field>-error\`}>{errors.<field>}</FieldMessage>` directly under the relevant `<Input>`/control, only when set.
- Wire a11y on the input: `aria-invalid={!!errors.<field>}` + `aria-describedby={errors.<field> ? "<field>-error" : undefined}`.
- **Clear timing (reward-early / punish-late):** validate on submit; once a field has errored, clear its error on change (`onChange` → drop that key) and re-validate; clear all on successful submit / dialog close/reopen.
- **Focus management:** on submit-with-errors, move focus to the first errored field (`ref.focus()` + `scrollIntoView`) so the manager lands on the problem. (Per-dialog; acceptable to scope focus to the primary Add dialogs if wiring refs to every control is heavy — confirm in plan.)
- Dynamic server errors (`humanizeCatalogError(err)` / `humanizeVoucherError(err)`) **remain `toast.error(...)`** — they're async post-submit failures, bucket-C-like, and the ESLint guard leaves them legal.
- `toast.success(...)` confirmations (saved/added/archived) **remain toasts** — global positive feedback, correct affordance.

### Affected forms

- **products.tsx:** Add-product dialog (`submitAddOpenPin`), Add-SKU dialog (`submitAddSkuOpenPin`), Edit-meta dialog (`commitMetaEdit`), Edit-price dialog (`submitPriceOpenPin`), Components editor (`commitComponents`). Each gets its own error map (or one map namespaced by dialog) reset on open/close.
- **vouchers.tsx:** Add-voucher dialog (`submitAddOpenPin`), Edit-meta dialog (`commitMetaEdit`).
- The Components editor's row-level errors ("Pick an SKU for every row.", "Qty must be a positive integer.") attach to the offending row.

> The `window.confirm(...)` archive flows and the existing `PinSheet error` prop (server PIN failure) are **not** in scope — `PinSheet`'s inline error is already inline and belongs to the PIN-flow surface (#11).

## 8. ADR-048 — Inline messaging policy (outline)

- **Title:** Inline `FieldMessage` for sync validation; toasts for global/async.
- **Context:** 154 toasts; sync form-validation in toasts is the wrong affordance on a booth tablet; no inline primitive existed.
- **Decision:** sync client-side **form-validation** feedback renders inline via the `FieldMessage` design-system primitive, anchored to the field, AA-legible on the phthalo canvas. **Global/async** feedback (print, draft saved, cancelled, **server-rejection / `humanize*Error` results**, Telegram, low-stock) stays as Sonner toasts. PIN-flow feedback is owned by the PIN surface (#11/#7).
- **Heuristic (machine-enforced):** literal-string-first-arg `toast.error`/`toast.warning` = sync validation → inline. Dynamic-arg `toast.error` = server/async → toast OK. `toast.success` = OK. Enforced by `no-restricted-syntax` scoped to a migrated-file registry that grows per slice.
- **Tokens:** `error`/`success` lifted for `.dark` to meet WCAG-AA body text (ADR-047 amend).
- **Rollout:** by-file slices; slice 1 = products + vouchers. New `FieldMessage` consumers must use semantic tones; new files join the ESLint glob as they convert.
- **Consequences:** consistent inline pattern; lint fence prevents regression; one more primitive to maintain; badge error/success render slightly lifted in dark (accepted).
- Cross-ref ADR-047 (phthalo-dark), the roadmap spec §#12, and note #11/#7 own PIN/login.

## 10. Open decisions (recommended defaults — confirm at plan staffreview)

1. **Token mechanism** — lift global `--color-error`/`--color-success` in `.dark` (recommend) vs FieldMessage-local tokens. *Default: global lift; fall back if badge shift unwanted.*
2. **Tint** — include subtle `bg-{tone}/10` band (recommend, read well in mock) vs rule+icon+text only. *Default: include.*
3. **Focus management breadth** — focus first errored field in all dialogs (recommend) vs primary Add dialogs only if ref-wiring is heavy. *Default: all; downgrade if costly.*
4. **Tone set** — `error`+`success` only (recommend) vs also ship `warning`/`info`. *Default: error+success only.*
5. **esquery selector** — `[arguments.0.type='Literal']` (recommend) — confirm it fires on the installed esquery; fall back to child-combinator form if not.
