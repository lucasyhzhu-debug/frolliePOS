# Design Spec — v1.1 Security Hardening

> **Phase:** `v1.1-security-hardening` · **Date:** 2026-06-17
> **Source:** [`docs/reviews/security-audit-2026-06-17.md`](../../reviews/security-audit-2026-06-17.md) — swarm audit, 14 confirmed findings.
> **Scope of THIS phase:** the 3 High + 4 Medium findings (**SEC-01..SEC-07**). The 5 Low + 2 Info findings are explicitly deferred (see *Out of scope*).
> **Decisions locked** (brainstorm, 2026-06-17): one cohesive phase; SEC-03 = env-PIN + soft `must_change_pin`; SEC-04 = throttle + shorter TTL, keep 6-digit code.

## Goal

Close the seven High/Medium security findings with surgical, well-bounded fixes that follow existing codebase patterns, each covered by a regression test that fails before the fix and passes after. No behavioural change to legitimate flows.

## Why one phase (not slices)

Two coupling constraints make a single phase correct:

1. **SEC-01 ⇄ SEC-07 are causally coupled.** SEC-01's fix makes `_recordFailedAttempt_internal` increment the `pos_auth_attempts` counter *reliably*. The Telegram approve path (SEC-07) writes to that **same** table with no lockout pre-check. So fixing SEC-01 alone would *worsen* SEC-07 — every wrong approve-PIN would now reliably lock the booth. They must land together.
2. **SEC-05 ⇄ SEC-06 share one fix** (session-gate two sibling read seams consumed by the same hook, `useXenditPayment`).

The remaining fixes (SEC-02, SEC-03, SEC-04) are independent and small. Shipping all seven as one phase avoids a window where a money/auth fix is unpatched while another lands.

## Fix groups

### Group A — Auth failed-attempt counter (SEC-01 High + SEC-07 Medium)

**Root cause (SEC-01):** `_recordFailedAttempt_internal` (`convex/auth/internal.ts:169-231`) is wrapped in `withIdempotency` under the derived key `${idempotencyKey}:failed`. On a cache hit the handler body never runs, so the counter only increments on the first call per base key. Because the base key is only ever cached on *successful* login (`_loginCommit_internal`), failed attempts never populate it — so reusing one client-controlled `idempotencyKey` across distinct wrong PINs freezes `fail_count` at 1. `verifyPinOrThrow` (`convex/auth/verifyPin.ts:34-44`) *does* perform a lockout pre-check via `_getLockState_internal`, but the lock never trips because the count never advances. The defect reaches `loginWithPin`, `changePin`, and `verifyManagerPinOrThrow` (→ `createStaff`, `resetStaffPin`) — all route through `verifyPinOrThrow`.

**Root cause (SEC-07):** the off-booth Telegram approve actions (`convex/approvals/actions.ts` — `approveStaffPinReset`, `approveManualPayment`, `approveRefund`, `approveSpoilage`, and `denyRequest`) call `argon2Verify` directly (no lockout pre-check, intentionally) but on failure still call `_recordFailedAttempt_internal` with `source: "telegram_approval"`, polluting the booth lockout counter. `args.managerStaffCode` is attacker-supplied, so any holder of a valid pending token can choose which manager to lock. Today the idempotency-dedup masks this; after the SEC-01 fix it becomes a reliable remote DoS of booth login.

**Design:**
1. **Remove the `withIdempotency` wrap** from `_recordFailedAttempt_internal`. Increment `fail_count` unconditionally, keyed on `staff_id` only. Keep the existing Fix-3 (`cleanupAndGetAttempt` dedup of concurrent rows) and Fix-7 (reset to 1 on expired lockout) logic. Drop the `idempotencyKey` arg from the mutation's args.
   - *Rationale for dropping exactly-once:* the original "Fix 10" idempotency wrap guarded against a crash-retry double-incrementing the counter. Over-counting by one is **fail-safe** (locks slightly sooner); the current under-count is **fail-open** (never locks). A security counter must fail safe.
2. **Add a `countTowardLockout: boolean` discriminator** to `_recordFailedAttempt_internal`:
   - `true` (booth paths, via `verifyPinOrThrow`): patch `fail_count` + `locked_until`, fire `notifyStaffLockout` on the newly-locked transition — current behaviour, minus the idempotency wrap.
   - `false` (Telegram approve paths): write the `staff.failed_pin` audit row with `source: "telegram_approval"` (preserves the rule-#10 audit trail) but **do not** patch `fail_count`/`locked_until` and **do not** fire `notifyStaffLockout`. Brute-force bounding on this path remains the per-token cap (`TOKEN_PIN_ATTEMPT_CAP = 5`, already enforced via `_recordTokenPinFailure_internal`).
3. **Sweep ALL 6 call-sites of `_recordFailedAttempt_internal`** (verified complete via grep — single-writer-signature discipline, per the `v0.5.7`/`v1.0` "sweep all writers" lessons):
   - `convex/auth/verifyPin.ts:48` (booth funnel): stop passing `${idempotencyKey}:failed`; pass `countTowardLockout: true`.
   - `convex/approvals/actions.ts:180` (approveStaffPinReset), `:429` (approveManualPayment), `:577` (approveRefund), `:706` (denyRequest), `:992` (approveSpoilage): each currently passes `idempotencyKey: \`${args.idempotencyKey}:failed\`` + `source: "telegram_approval"`. Switch all five to `countTowardLockout: false`, drop the derived key, keep `source`.
4. **Drop the now-unused `idempotencyKey` param** from `verifyPinOrThrow` + `verifyManagerPinOrThrow` and their callers (`loginWithPin`, `changePin`). *Confirmed safe (resolves spec open-Q#5):* `loginWithPin`'s success idempotency is `_loginCommit_internal` (separately `withIdempotency`-wrapped, called from the action body — independent of `verifyPinOrThrow`); the funnel used `idempotencyKey` **only** to derive the `:failed` key. (If the implementer prefers smaller blast radius, the param may be retained one release as an unused arg — state the choice in the plan.)

**Test migration (load-bearing — do NOT just add tests):**
- **Rewrite** `convex/auth/__tests__/auth.test.ts:261` — `describe("Fix 10: _recordFailedAttempt_internal is idempotent")` asserts `fail_count === 1` after two same-key calls, i.e. *the exact vulnerable behavior this fix removes*. Re-frame it as a SEC-01 regression test: two calls → `fail_count === 2`; three → `newly_locked: true` / `locked_until` set. Leaving it as-is gives a false-red CI and tempts a revert toward the bug.
- Grep `convex/auth/__tests__/lockoutScheduler.test.ts` + `auth.test.ts` for any other assertion depending on the derived-key arg and migrate them.

**Invariant:** booth login lockout (ADR-002) is driven solely by `staff_id` + server-side count; no client-supplied value can suppress or amplify it. Off-booth wrong PINs are audited but never touch booth lockout.

**Tests (new):**
- `loginWithPin` 3× with the **same** `idempotencyKey` + different wrong PINs → 3rd throws `LOCKED_OUT`. (Note: a legit user's network-retry of a failed login now over-counts by one — deliberate fail-safe; assert/comment so it isn't "fixed" back.)
- `verifyManagerPinOrThrow` (via `createStaff` or `changePin`) 3× same key + wrong PINs → locks.
- **SEC-07 regression:** Telegram `approveRefund` 3× wrong manager PIN → booth `loginWithPin` for that manager still succeeds with the correct PIN (no booth lock), AND a `staff.failed_pin` audit row with `source: "telegram_approval"` exists, AND `_recordTokenPinFailure_internal` caps the token at 5.

### Group B — Cart quantity integrity (SEC-02 High)

**Root cause:** `commitCart` (`convex/transactions/public.ts:143-300`) declares `lines: v.array({ productId, qty: v.number() })` with no positive-integer guard. `v.number()` admits negatives/zero/floats. A negative qty becomes a *positive* stock movement on confirm (`inventory/internal.ts` inserts `qty: -line.qty`), fabricating inventory; floats break ADR-015 integer-rupiah.

**Design:** add a per-line guard immediately after the `EMPTY_CART` check (`public.ts:181`), before any subtotal math, mirroring the canonical guard already in `_recordSpoilage_internal` and the refund path:
```ts
for (const { qty } of args.lines) {
  if (!Number.isInteger(qty) || qty <= 0) throw new Error("QTY_INVALID");
}
```
No upper-bound cap in v1 (YAGNI — flagged as optional in the audit; a 2-3 staff booth has no realistic huge-cart abuse vector worth the magic number). Reuse the existing `QTY_INVALID` error string for consistency with the sibling guards.

**Tests:** `commitCart` throws `QTY_INVALID` for qty `-1`, `0`, `1.5`; a mixed `[{A,qty:100},{B,qty:-90}]` cart is rejected before any `pos_transactions`/`pos_transaction_lines`/`pos_stock_movements` row is written (assert table counts unchanged).

### Group C — Bootstrap default PIN (SEC-03 High)

**Root cause:** `bootstrap` (`convex/seed/actions.ts:74-85`) hard-codes manager PIN `"1111"`, is prod-runnable (no `assertNotProd`), with no forced rotation. Live prod is bootstrapped on this default (memory `prod-cutover-2026-06-03`).

**Design (env-PIN + soft `must_change_pin`):**
1. `bootstrap` reads `BOOTSTRAP_MANAGER_PIN` from `process.env`. If absent or not `/^\d{4}$/`, **throw** `BOOTSTRAP_PIN_REQUIRED` — no hardcoded default. Hash that PIN instead of `"1111"`.
2. Add `must_change_pin: v.optional(v.boolean())` to the `staff` schema (`convex/auth/schema.ts` or wherever `staff` is defined — plan to confirm exact fragment). `_bootstrapCommit_internal` sets it `true` on the seeded manager.
3. **Soft enforcement:** on a successful manager login where `must_change_pin === true`, surface a forced-rotation prompt in the FE (a non-dismissable "Change your PIN" step after login) — but do **not** hard-block other manager actions. `changePin` clears the flag (`must_change_pin: false` / unset) on success.
4. The flag is read via the existing session/login response. Plan to confirm the minimal field to thread through `getSession`/login return so the FE can branch.

**Limitation — this code change does NOT retroactively protect live prod (I2):** `bootstrap` is idempotent-guarded ("aborts if any staff row exists") and prod is already bootstrapped (`prod-cutover-2026-06-03` memory), so the env-PIN + `must_change_pin` machinery only hardens **future fresh deploys and tests**. The live S-0001 account stays on its current PIN until the operational rotation runs. Do not treat merging this phase as having secured prod.

**Out of band (operational, not code):** rotate the live prod S-0001 PIN off `1111` and set its `must_change_pin` (one-off `npx convex run` or via the new rotation prompt). This is the *only* thing that hardens the live account — listed in the execution handoff pre-flight, not a code task.

**Tests:** `bootstrap` throws `BOOTSTRAP_PIN_REQUIRED` when env unset; with env set, seeded manager has `must_change_pin === true` and authenticates with the env PIN (not `1111`); `changePin` clears the flag.

### Group D — Read-seam IDOR + device-activation throttle (SEC-04/05/06 Medium)

#### D1 — Session-gate `getById` + `getCurrentInvoice` (SEC-05 + SEC-06)

**Root cause:** `getById` (`transactions/public.ts:47-61`) and `getCurrentInvoice` (`payments/public.ts:9-25`) are public queries taking only `{ txnId }` — no session, no day-scope. `getById` `return { ...txn, lines }` spreads the raw Doc including `receipt_token` (ADR-021 capability); `getCurrentInvoice` returns `qr_string`/`va_number`. `getTransactionDetail` (`transactions/public.ts:498-538`) is the established gated pattern.

**Design:**
1. Add `sessionId: v.id("staff_sessions")` to both queries. Resolve via `internal.auth.internal._resolveSessionRole_internal`; return `null` on invalid session (graceful FE degrade, matching `getTransactionDetail`).
2. Non-manager role → enforce `wibDayWindow(Date.now())` on `txn.created_at` (return `null` out of window), identical to `getTransactionDetail:507-516`. For `getCurrentInvoice`, resolve the parent txn and apply the same scope before returning the invoice.
3. **Stop spreading the raw Doc.** `getById` returns an explicit projection (the fields the FE actually consumes — plan to enumerate from `useXenditPayment`/`charge-success` usage) with `receipt_token`, `xendit_*`, `confirmed_*` **omitted**. `getCurrentInvoice` returns only the fields the charge screen renders.
4. **System callers** (`payments/actions.ts:42,100`, `transactions/actions.ts:58`) currently call `api.transactions.public.getById` with the full Doc. Add `internal.transactions.internal._getTxnById_internal` returning the full row for those callers; repoint the public `getById` to the gated/projected version. Plan to confirm each system call-site and migrate it.
5. **FE:** `useXenditPayment(txnId)` → `useXenditPayment(txnId, sessionId)`; thread the active `sessionId` from `useSession`. Only consumed on `sale/charge` + `sale/charge-success`, where a live session exists → no UX regression.
6. **Verify-first (I4):** grep `getCurrentInvoice` to confirm its ONLY caller is `useXenditPayment` (no server-side caller breaks when `sessionId` becomes required). For `getById`, enumerate every system call-site (`payments/actions.ts`, `transactions/actions.ts`, others) and migrate each to `internal._getTxnById_internal` before the public seam changes signature — a missed system caller is a runtime break, not a type error (internal vs public api inference).

**Tests:** anonymous / other-day-staff caller gets `null` from both queries; returned `getById` shape asserts absence of `receipt_token`; `getCurrentInvoice` shape asserts absence of nothing it shouldn't expose beyond render needs; system callers still receive the full row via the internal variant.

#### D2 — Throttle `activateDevice` (SEC-04)

**Root cause:** `activateDevice` (`convex/staff/public.ts:128-244`) is the pre-auth perimeter with a no-op authCheck, validates only `/^\d{6}$/`, and has no attempt counter — a live 6-digit code (~900k space, 1h TTL) is brute-forceable. `pos_auth_attempts` is `staff_id`-keyed and gives zero protection here.

**Design:**
1. New table `pos_device_activation_attempts` (`convex/auth/schema.ts` or `convex/staff/schema.ts` — plan to place with the device-setup tables): `{ device_id: string, fail_count: number, locked_until: number | null, last_attempt_at: number }`, indexed `by_device`. Plus a singleton/global counter row (or a `by_global` sentinel key) for a rolling-window ceiling so spraying fabricated `deviceId`s can't sidestep the per-device cap.
2. In `activateDevice`, **before** the `by_code` lookup: reject if the device's `locked_until > now` (throw `ACTIVATION_LOCKED:<secs>`). On the invalid-code branch (`staff/public.ts:165-171`): increment `fail_count`; set `locked_until` after N misses (mirror auth policy: `MAX_FAILS = 5`, `LOCKOUT_MS = 60_000`, escalating optional but not required v1). On success: clear the device's attempt row.
3. **Global ceiling (load-bearing — I1):** the `deviceId` is an attacker-chosen arg, so a per-device throttle alone is bypassed by spraying fresh `deviceId`s. The global rolling-window ceiling is therefore the real defense. **Concrete v1 design:** a singleton counter row keyed `by_global` tracking failed activations in a rolling 15-min window; cap at **50 failures / 15 min** (a legit booth activates a handful of devices a month, so this never trips in normal use, but it caps a brute-forcer at ~50 guesses/window against a ~900k space — negligible hit probability even across many windows). On breach: **invalidate all live `pending_device_setups`** (force the manager to re-issue) and audit `device.activation_throttled`. The 15-min code TTL (step 4) compounds this — an attacker must land within one 15-min window *and* under the 50-guess global cap. *(Plan to confirm the singleton-row pattern against any existing global-counter precedent; otherwise model it on `pos_auth_attempts` with a sentinel key.)*
4. **Shorten TTL:** `pending_device_setups` code TTL 1h → 15min (`generateSecureSetupCode` / its consumer — plan to locate the TTL constant). Code stays 6-digit.
5. Audit lockout events (`device.activation_locked` or reuse an existing verb — `audit_log.action` is a free `v.string()`, no enum to edit), mirroring the PIN-lockout audit pattern.

**Tests:** 5 wrong codes from one `deviceId` → 6th throws `ACTIVATION_LOCKED`; a correct code after lock-expiry succeeds; global burst invalidates pending setups; expired (>15min) code is rejected.

## Cross-cutting principles

- **Follow existing patterns:** the spoilage/refund `QTY_INVALID` guard (B), `getTransactionDetail`'s resolve→scope→project shape (D1), and the `pos_auth_attempts` lockout shape (D2) are all in-repo templates — match them.
- **Idempotency ≠ rate-limiting.** The SEC-01 root lesson: never key a security counter on a client-controllable idempotency token. Group A enshrines this.
- **Every fix is TDD:** a test that reproduces the exploit (red) precedes each fix (green). Payment/refund/stock/auth changes require tests per CLAUDE.md "How to add a feature" #7.
- **Audit + CHANGELOG:** new audit verbs documented in `docs/SCHEMA.md`; `docs/CHANGELOG.md` updated in the same PR.
