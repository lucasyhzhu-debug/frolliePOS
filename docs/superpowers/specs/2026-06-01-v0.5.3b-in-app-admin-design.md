**Date:** 2026-06-01

**Phase:** v0.5.3b — second slice of v0.5.3 (per `docs/PROGRESS.md` line 903; decomposition table in the v0.5.3a spec)

## Decomposition context — why this is "3b"

v0.5.3 as scoped in PROGRESS.md is six independent subsystems. The v0.5.3a spec clustered them into three slices by *nature of work*:

| Slice | Contents | Risk |
| -- | -- | -- |
| v0.5.3a — Reporting *(shipped)* | Transaction history + manager dashboard | Low |
| **v0.5.3b — In-app admin** *(this spec)* | Staff CRUD + product CRUD + receipt config | Low–medium |
| v0.5.3c — Settlements | `pos_settlements` + Xendit settlement webhook + nightly recon | High |

**3b ships as a single slice** (brainstorm Q-slice): the three surfaces are independent CRUD with no cross-dependencies, they share one tiered-gate helper, and they tell one coherent "the manager no longer needs the Convex dashboard" story. The only net-new infrastructure (Convex file storage for the logo) is contained to the receipt surface.

## Goal

Give managers an in-app portal to run the catalog and the team without the Convex dashboard: create/edit/deactivate staff, create/edit/archive products (including the inventory-SKU linkage that makes a sale decrement stock), and configure the customer receipt's branding (text + uploaded logo). Every write is gated by manager identity at a tier matched to its stakes.

**"You'll be able to" outcomes** (carried from PROGRESS.md lines 911–913, scoped to 3b):
- Add, edit, deactivate staff in-app — the Convex dashboard is no longer required
- Add, edit, archive products in-app, including which inventory SKU(s) a sale decrements
- Configure the receipt template (business name, address, contact, social handle, footer, **uploaded logo image**) from the manager portal

**"Still not yet" out-of-scope reminders** (the design enforces these — not on the build list):
- No new inventory-SKU creation (SKUs are owned by v0.5.2 inventory; 3b only *links* products to existing SKUs)
- No product photo upload (product avatars stay initials/hue; only the **receipt logo** gets image upload)
- No settlements / payout reconciliation, no `pos_settlements` table (v0.5.3c)
- No Telegram-approval path for admin actions — admin CRUD is inline-at-the-portal only (Telegram approval exists for *off-booth* gaps)
- No concurrent-edit conflict resolution beyond last-write-wins (single booth, 1 manager editing at a time)

## Locked brainstorm answers (the source of truth for design decisions)

| Q | Decision | Rationale |
| -- | -- | -- |
| **Q-gate** | **Tiered gating.** PIN-gate the security/financially-sensitive writes; session-only for low-stakes config. | Mirrors the existing split-by-stakes precedent: `setFoundersSummaryEnabled` + `markRefundSettled` (rule #22) shipped **session-only** because they move no money; refunds/PIN-resets are **PIN-gated**. A blanket policy either over-ceremonies config edits or under-protects identity changes. |
| **Q-product** | **Fields + component linkage.** Edit/create products including `pos_product_components` (link to **existing** SKUs + qty). No new-SKU creation, no product photo upload. | A genuinely new product must decrement *some* stock, so component linkage is the line between "shallow rename" and "usable." SKU creation belongs to v0.5.2 (its module owns `pos_inventory_skus`); duplicating it here crosses a phase seam. |
| **Q-newpin** | **Manager sets initial PIN.** Create form collects name + role + a 4-digit initial PIN; the new staffer changes it later via existing `changePin`. | Reuses the `createStaff` action's PIN path. Auto-temp-PIN or first-login-set would each add a new login state + UI for no booth-operational benefit (the manager is standing next to the new hire). |
| **Q-receipt** | **Text fields + footer + uploaded logo image** (Convex file storage). | Un-hardcodes the four business fields already in `receipts/internal.ts`, plus a real branded logo — the receipt is the one customer-facing artifact, so the image polish is worth the storage plumbing here (unlike internal product avatars). |
| **Q-slice** | **One slice.** Staff + products + receipt ship together as v0.5.3b. | Independent CRUD, no cross-deps, shared gate helper. Splitting would create two/three spec→plan→impl cycles for low-coupling screens. |

### Tier assignment (the single source of truth for which gate each mutation uses)

| Write | Gate | Why |
| -- | -- | -- |
| `createStaff` | **PIN** | Creates a new identity that can transact. |
| `updateStaff` — **role change** | **PIN** | Privilege escalation (staff → manager). |
| `updateStaff` — name only | session | Cosmetic. |
| `deactivateStaff` | **PIN** | Revokes access; security-material. |
| `resetStaffPin` | **PIN** *(already built)* | Resets another person's credential. |
| `createProduct` / `updateProduct` — `price_idr`, `tax_rate` | **PIN** | Money fields; mis-set price is financially material. |
| `createProduct` / `updateProduct` — name, `pack_label`, `sku_family`, `initials`, `hue`, `sort_order` | session | Non-money catalog metadata. |
| `setProductComponents` | session | Stock-linkage; wrong qty is reconcilable via v0.5.2, not a money loss at point of edit. |
| `archiveProduct` | session | Reversible (`active` flag); historical lines snapshot-protected (rule #1). |
| `updateReceiptConfig` / logo upload | session | Branding/config; moves no money, changes no ledger. (Same class as `setFoundersSummaryEnabled`.) |

> **A `createProduct` that sets a price counts as a money write → PIN-gated.** The split above is per-*field*, so `updateProduct` is implemented as two callables (a session-gated metadata patch and a PIN-gated price patch) rather than one mutation that conditionally demands a PIN — keeping the gate mechanical per business rule #21 (the `authCheck` slot can't be conditional on payload).

## Architecture overview — three CRUD modules, one shared gate

```
  MANAGER PORTAL (laptop-first; mgr/* routes)
  ├─ /mgr/staff      staff list (no hashes) · create · edit · deactivate · reset-PIN
  ├─ /mgr/products   product list (incl. inactive) · create · edit · archive · component editor
  └─ /mgr/receipt    receipt config form · logo upload · live preview

  BACKEND
  ├─ staff/    listStaff (pin_hash STRIPPED) · createStaff* · updateStaff · deactivateStaff
  ├─ catalog/  listAllProducts · createProduct · updateProduct · setProductComponents · archiveProduct
  ├─ settings/ getReceiptConfig · updateReceiptConfig · generateLogoUploadUrl
  ├─ receipts/ (un-hardcode) read branding from settings · NEW _purgeAllReceiptCache_internal
  └─ auth/     NEW verifyManagerPin (action helper) — shared by every PIN-gated write

  GATE
  ├─ session-only writes → authCheck: requireManagerSession  (on withIdempotency)
  └─ PIN-gated writes    → verifyManagerPin (argon2, action layer) → internal commit
```

`* createStaff` is **extended**, not reused as-is — see §"Staff admin" (it is currently session-gated and gains a `managerPin` arg).

The gate tier is a property of each mutation, declared in the table above and enforced two ways: session-only writes set `authCheck: requireManagerSession` on the `withIdempotency` wrapper (rule #21); PIN-gated writes are **actions** that argon2-verify the manager's own PIN (`verifyManagerPin`) before calling an internal commit mutation — the same two-layer shape as the existing `resetStaffPin`, because argon2 cannot run in a mutation (ADR-004).

## Backend (`convex/`)

### A. Staff admin — `staff/public.ts`, `staff/internal.ts`, `auth/actions.ts`

- **`listStaff` (fix the v0.2 follow-up):** today it `.collect()`s full rows **including `pin_hash`** (`staff/public.ts:30`). Project to a safe shape — `{ _id, name, code, role, active, last_login_at, created_at }`. No hash ever crosses the wire. *(Verify no current caller depends on the hash being present — it should not.)*
- **`createStaff` (extend → PIN-gated):** currently `auth/actions.ts:118` takes `{ idempotencyKey, sessionId, name, role, pin }` and is **session-gated** (only `requireManagerSession` inside `_createStaffCommit_internal`). Add a `managerPin` arg and `verifyManagerPin` before the commit — matching the tier. The new staffer's `pin` continues to hash via `_hashPin_internal`. Audit: `staff.created` (already emitted by the commit).
- **`updateStaff` (new):** split by tier —
  - session-gated metadata patch: `name` (and `code` if surfaced).
  - **PIN-gated** role change: `role: "staff" | "manager"`. Guard: a manager cannot demote **the last active manager** (same `_listActiveManagers_internal` count guard as `deactivateStaff`).
  - Audit: `staff.updated` *(new verb)*, metadata records which fields changed (never PIN/hash).
- **`deactivateStaff` (new, PIN-gated):** set `active = false`. No live-session teardown needed — `requireSession` already rejects inactive staff (`auth/sessions.ts`), so the target's session self-invalidates on its next request. Guards: reject **self-deactivate** (use the lock flow); reject deactivating **the last active manager** — the count guard reuses `auth/internal._listActiveManagers_internal` (the internal behind `staff/public.ts:listActiveManagers`), refusing if removing the target would leave zero active managers, rather than a fresh `staff` scan. Audit: `staff.deactivated` *(new verb)*. *(Reactivation = the inverse session-gated patch; out of scope unless trivial to include — decide at plan time.)*
- **`resetStaffPin` (already built, `auth/actions.ts:236`):** PIN-gated, two-layer, clears lockout, logs `staff.pin_reset`. **Wire the UI only — no backend change.**

### B. Product admin — `catalog/public.ts`, `catalog/internal.ts`

- **`listAllProducts` (new, manager-gated):** the public `catalog` query is `active: true`-only; admin needs inactive rows too. Returns products **+ their components** (so the editor shows current SKU linkage without an N+1) + the active SKU list (for the linkage picker). Reuses `_getComponentsForProducts_internal` (`catalog`'s existing fan-out helper).
- **`createProduct` (PIN-gated — sets a price):** all `pos_products` fields (`sku_family`, `name`, `pack_label`, `price_idr`, `tax_rate`, `initials`, `hue`, `sort_order`; `active: true`, `created_at`/`updated_at` server-stamped per ADR-031). Audit: `product.created` *(new)*.
- **`updateProduct` (split by tier):**
  - session-gated metadata patch: `name`, `pack_label`, `sku_family`, `initials`, `hue`, `sort_order`.
  - **PIN-gated** money patch: `price_idr`, `tax_rate`.
  - Audit: `product.updated` *(new)*, metadata records changed fields + old→new for money fields.
- **`setProductComponents` (session-gated):** replace-set the product's `pos_product_components` rows from a `{ inventory_sku_id, qty }[]` payload (delete existing for the product, insert the new set, in one transaction). Validate each `inventory_sku_id` references an **active** SKU; reject `qty <= 0`. No SKU creation. Audit: `product.updated` (component change is a product edit) with a `components_changed` metadata flag.
- **`archiveProduct` (session-gated):** `active = false`, `updated_at` stamped. Historical `pos_transaction_lines` are unaffected (snapshot rule #1 — never join lines back to products). Audit: `product.archived` *(new)*.

### C. Receipt config — `settings/public.ts`, `settings/schema.ts`, `settings/internal.ts`, `receipts/internal.ts`

- **Schema (`settings/schema.ts`):** add to the `pos_settings` singleton, **all optional**:
  `business_name`, `address`, `contact`, `instagram_handle`, `footer_text` (strings) and `logo_storage_id: v.optional(v.id("_storage"))`.
  Optional + read-time defaults keep the absent-row behavior unchanged (no seed migration).
- **`_getSettings_internal` (extend, `settings/internal.ts:3`):** return the receipt fields with **defaults equal to today's hardcoded constants** (`receipts/internal.ts` `RECEIPT_SETTINGS`): `business_name: "FROLLIE"`, `address: "Pakuwon Mall, Surabaya"`, `contact: "+62 821-xxxx-xxxx · frollie.id"`, `instagram_handle: "@frollie.id"`, `footer_text: <current footer>`, `logo_storage_id: undefined`. Existing `founders_summary_enabled` default unchanged.
- **`getReceiptConfig` (new, session-gated query):** returns the current config + a resolved `logo_url` (`ctx.storage.getUrl(logo_storage_id)` or `null`) for the form + preview.
- **`generateLogoUploadUrl` (new, session-gated mutation):** returns a Convex `ctx.storage.generateUploadUrl()`; the client PUTs the image and gets back a `storageId`.
- **`updateReceiptConfig` (new, session-gated mutation):** patch the text fields + `logo_storage_id` onto `pos_settings` (insert-if-absent), stamp `updated_at`/`updated_by`. **Then invalidate the receipt cache (below).** Validate string lengths (bound footer/contact to keep the receipt layout sane). Audit: `settings.receipt_updated` *(new verb)*, metadata lists changed fields (no full values needed). *(If the logo changes, the prior `_storage` blob may be deleted to avoid orphan accumulation — decide at plan time; low volume, acceptable to leave for now.)*
- **Un-hardcode `receipts/internal.ts`:** the branding is injected in **one place** — the shared `buildVmFromTxnWithLines` helper (`receipts/internal.ts:99`, currently `settings: RECEIPT_SETTINGS`), which is delegated to by **both** `_buildViewModel_internal` (id-keyed) **and** `_renderReceiptByToken_internal` (the public `/r/<token>` render). Replace the const + `🍪` emoji (`template.ts:143`) with `ctx.runQuery(internal.settings.internal._getSettings_internal, {})` + `ctx.storage.getUrl(logo_storage_id)`. Mechanically valid in a query context: `buildVmFromTxnWithLines` already calls `ctx.runQuery` from inside an `internalQuery` (`internal.ts:66,74`), and `ctx.storage.getUrl` is available in `QueryCtx`. The template embeds `<img src="{logo_url}">` when `logo_storage_id` is set, else falls back to the emoji default. One change covers both render paths. (Cost: one extra single-row settings read on the cache-**miss** render path — negligible; noted so the by-token read-count comment at `internal.ts:126` isn't read as a regression.) `convex/lib/` and `receipts/` stay V8-safe (no Node APIs) per the v0.5.1a lesson.

### Cache invalidation — **new `_purgeAllReceiptCache_internal`**

The existing `_purgeReceiptCache_internal` (`receipts/internal.ts:184`) is **keyed by a single `transactionId`** — it purges one receipt (used by refund commit, which knows its txn). A receipt-config change invalidates **every** cached receipt (branding is baked into the stored HTML in `pos_receipt_html_cache`). So `updateReceiptConfig` calls a **new** `_purgeAllReceiptCache_internal`. **Mechanism (resolved):** `pos_receipt_html_cache` (`receipts/schema.ts:9`) is `{ token, html, expires_at }` indexed `by_token`; the purge is `ctx.db.query("pos_receipt_html_cache").collect()` → delete each row — **no token list is needed**, no per-txn lookup, intra-module (receipts owns the table). Cache rebuilds lazily on next `/r/<token>` view. At single-booth volume the table is small; a full scan-delete is correct, atomic, and cheap. *(Do not reuse the per-txn helper in a loop.)*

### Shared gate helper — **new `verifyManagerPin` (`auth/actions.ts`)**

Extract the manager-PIN verify that `resetStaffPin` inlines (`_getStaffPinHash_internal` → assert active + `role === "manager"` → `argon2Verify`) into a reusable action helper `verifyManagerPin(ctx, { sessionId, managerPin }) → { managerId }`, throwing on non-manager / wrong PIN, recording a failed attempt under a derived key (lockout policy, ADR-002), and **never logging the PIN**. Consumed by `createStaff`, `updateStaff` (role), `deactivateStaff`, the PIN-gated product money patches, and refactored back into `resetStaffPin` (so the verify lives in one place). This is the v0.5.0 "helper-extraction follow-through" lesson applied up-front.

## Data flow

```
  session-only write (e.g. updateReceiptConfig)
    UI ──mutation(idempotencyKey, sessionId, …)──▶ withIdempotency{authCheck: requireManagerSession}
                                                        └─ patch + audit (+ purge-all-cache for receipt)

  PIN-gated write (e.g. deactivateStaff, price edit)
    UI ──PinSheet(managerPin)──▶ ACTION ──verifyManagerPin(argon2)──▶ internal commit mutation
                                                                          └─ patch + audit

  logo upload
    UI ──generateLogoUploadUrl──▶ {uploadUrl} ──PUT image──▶ {storageId} ──updateReceiptConfig(logo_storage_id)──▶ purge-all-cache
    /r/<token> next view ──▶ re-render with <img src=getUrl(logo_storage_id)>
```

Live-updating via Convex subscriptions: the staff/product lists reflect edits reactively; a receipt-config change purges the HTML cache so the next public receipt view shows new branding.

## Access & correctness

- **Gate tier is mechanical, not conditional.** Per business rule #21 the `authCheck` slot can't branch on payload, so any write that is *sometimes* money (product edit) is split into a session-gated callable and a PIN-gated callable rather than one mutation that conditionally demands a PIN.
- **Last-active-manager invariant:** deactivate and role-demote both refuse to remove the final active manager (else the booth locks itself out of all PIN-gated actions). Enforced server-side in the commit, not just disabled in UI.
- **Deactivation ⇒ session death** falls out of the existing `requireSession` active-check; no new teardown path (and no new bug surface).
- **Receipt branding is server-resolved:** the public `/r/<token>` render reads branding from `pos_settings` via `_getSettings_internal`; no client supplies branding. Config changes invalidate the cache so a viewer never sees a mix of old HTML + new logo.
- **PIN safety:** PIN values never logged; `verifyManagerPin` records failed attempts under a derived key (lockout, ADR-002); `listStaff` never returns `pin_hash`.
- **Idempotency:** every public mutation carries `idempotencyKey` + `withIdempotency` + `authCheck` (rules #15/#21). PIN-gated actions add an action-level idempotency pre-check (like `resetStaffPin`) so a double-tap doesn't double-commit.

## Testing

Staff/product/receipt admin touches identity + money fields, so per "How to add a feature" #7 the gated paths warrant tests; the pure validators are cheap high-value units.

- **Integration (gate tiers):** each PIN-gated mutation rejects a wrong/absent manager PIN and a non-manager session; each session-only mutation succeeds with a manager session and rejects a staff session. Assert `createStaff` now requires `managerPin`.
- **Integration (invariants):** cannot deactivate/demote the last active manager; cannot self-deactivate; deactivated staff's existing session fails its next `requireSession`.
- **Integration (`listStaff`):** response contains **no** `pin_hash` field.
- **Integration (component editor):** `setProductComponents` replace-set produces exactly the new rows; rejects inactive SKU / `qty <= 0`.
- **Integration (receipt cache):** `updateReceiptConfig` purges `pos_receipt_html_cache`; a subsequent `/r/<token>` render reflects the new branding; `_getSettings_internal` returns the hardcoded defaults when fields are absent.
- **Unit (validators):** PIN format (4 digits), product field bounds, receipt string-length bounds.
- **Behavioural:** add a staff member and log in as them; add a product with a component, sell it, confirm the right SKU decrements (reuses v0.3 sale path); upload a logo and confirm it renders on a fresh receipt.

## Success criteria & rollback

**Success criteria (minimum bar):**
- `npm run typecheck` + `npm run build` + `npm run lint` clean.
- New integration tests green: gate tiers, last-manager invariant, `listStaff` hash-strip, component replace-set, receipt cache purge + default fallback.
- Behavioural: a manager can run staff + catalog + receipt branding entirely in-app; a new product sells and decrements correctly; an uploaded logo appears on a freshly-rendered receipt.

**Rollback:** additive schema (6 optional fields) + new mutations → no destructive migration. Rollback = revert the slice's commits; absent receipt fields fall back to the hardcoded defaults, so a half-rolled-back state still renders receipts. Deployment order: backend (`npx convex deploy`) before frontend (Vercel), per repo convention.
