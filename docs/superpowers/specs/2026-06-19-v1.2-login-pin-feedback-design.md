## 1. Goal

The booth login (`/login`) gives a staffer **no feedback** between tapping the 4th PIN digit and either landing on home or seeing a toast: the keypad stays live during the long argon2 verify, wrong-PIN errors fire as transient toasts, and keys have no pressed state (hover does nothing on touch). One toast ("PIN reset declined") also fires seemingly at random. This phase makes PIN entry feel responsive and legible.

**You'll be able to:**
- See a **pressed state** on every keypad key and staff-list row you tap (touch-friendly).
- See a **spinner** the instant you complete the PIN, with the keypad locked, so you know it's verifying (no double-submit, no "did it register?").
- Read a **wrong-PIN** error **inline** under the dots (red) instead of a toast that vanishes; a **locked-out** message stays put with its countdown.
- See a brief **green confirmation** on success before home loads.
- Stop seeing the **spurious "PIN reset declined"** toast that re-fired on remount.

**Not in scope:** route-slide / View Transitions (motion kept functional-only per Lucas); no copy/i18n work ‚Äî new strings stay **English** to match the existing login screen ("Who's working?", "Wrong PIN.") and the mega-spec's English-default decision (#3); the EN/ID toggle is #1; no `PinSheet` rewrite (it's the *pattern source*, not a target); the booth-state login fork (`shifts.public.boothState`) is the **Shift SOP Flow** spec (#6), not this one.

## 2. Architecture ‚Äî a phase state machine in `login.tsx`

Today `PinEntry` owns its buffer and has **no notion of the async result** ‚Äî that's the root of the missing spinner and inline error. The fix lifts result-state to the parent and makes `PinEntry`/`NumericKeypad` **prop-driven** for feedback (mirroring how `PinSheet` already threads `pending`/`error`). No async logic moves *into* the leaf components.

`login.tsx` gains:

```ts
type Phase =
  | { kind: "idle" }
  | { kind: "pending" }
  | { kind: "error"; message: string; sticky: boolean }   // sticky = LOCKED_OUT
  | { kind: "success" };
```

`onPinSubmit` drives it:
1. set `pending` ‚Üí keypad disabled, dots row swaps to spinner.
2. `await login(...)` (long argon2 action).
3. **resolve** ‚Üí `success` ‚Üí render green dots + `FieldMessage` success ~**200ms** ‚Üí `navigate("/", { replace: true })`. The 200ms `setTimeout` is **cleared on unmount** (avoids a state-set-after-unmount warning if the tree unmounts mid-tick).
4. **reject** ‚Üí map the error (¬ß4) ‚Üí `error` ‚Üí bump `pinReset` (clears the buffer via the existing counter) ‚Üí red dots + inline message.

`pinReset` already exists and already mints a fresh `idempotencyKey` per retry (the F2/lockout fix at `login.tsx:46-54`) ‚Äî that contract is preserved; we only add phase transitions around it.

**Projecting the parent machine onto leaf props:** the parent's richer `Phase` flattens to `PinEntry`'s presentational props ‚Äî `pending` ‚Üí `pending={true}`; `error` ‚Üí `phase="error"` + `message` + **`persist`** (= the `sticky` flag, true only for LOCKED_OUT); `success` ‚Üí `phase="success"` + green `message`. `PinEntry` never sees the union ‚Äî only the flattened props (see ¬ß3B for the `persist` clear-rule).

## 3. Component changes

### A. `NumericKeypad` (shared ‚Äî also used by `PinSheet`)
- **New prop** `disabled?: boolean`. When set: digit/clear/backspace buttons get the native `disabled` attribute (the `Button` base already carries `disabled:pointer-events-none disabled:opacity-50`, so pointer clicks are blocked + dimmed for free) **and** the `document` keydown handler early-returns (a physically-held hardware key bypasses `pointer-events` and must not slip a digit past the spinner ‚Äî this guard is the real new logic).
- **Pressed feedback:** add `active:bg-accent` to the digit keys for a pressed-background cue. The press-**scale already exists** ‚Äî the `Button` base class carries `active:scale-[0.97] disabled:active:scale-100` (shipped with #2), so a disabled key auto-suppresses the scale and we **reuse** it rather than re-adding a (conflicting) one.
- `PinSheet` may pass `disabled={pending}` to inherit the lock (its handler already guards on `pending`), but no PinSheet behavior change is required by this phase.

### B. `PinEntry` (presentational rewrite)
- **New props:** `pending: boolean`, `phase: "idle" | "error" | "success"`, `message?: string`, **`persist?: boolean`**. Keeps the existing `onSubmit` + `reset` buffer contract.
- **Dots** tinted by phase: neutral (`border-foreground`) normally ‚Üí **red** (`border-error`, error fill) on error ‚Üí **green** (`border-success`) on success. (`border-error`/`border-success` are real utilities backed by the `--color-error`/`--color-success` tokens ‚Äî the same ones `FieldMessage` rides.)
- **Pending** ‚Üí replace the dots row with an inline `Loader2` spinner + a `role="status"` / `aria-live="polite"` "Verifying‚Ä¶" label; pass `disabled` to `NumericKeypad`.
- **Message** ‚Üí the shipped **`FieldMessage`** primitive (ADR-048), centered under the dots: `tone="error"` for wrong-PIN/locked, `tone="success"` for the green confirmation ("Welcome"). `FieldMessage` already carries `role=alert` (error) / `role=status` (success) for screen readers.
- **Error-clear rule (`persist`):** a **non-persistent** error message (INVALID_PIN) auto-hides once the buffer becomes non-empty ‚Äî `PinEntry` already owns the buffer, so it derives this with **no new parent callback** (show `message` only while `phase!=="error" || persist || buffer.length===0`). A **persistent** message (`persist=true`, LOCKED_OUT) stays regardless of typing until the parent clears `message`/phase on the next transition. This is the mechanism behind ¬ß4's "clears on next digit" vs "persists until next successful transition."

### C. `StaffListItem`
- `StaffListItem` is a **raw `<button>`** (not the `Button` component), so it has **no** active state today. Add `active:bg-accent motion-safe:active:scale-[.98]` (touch has no hover). The `motion-safe:` gate skips the scale under reduced-motion while keeping the background cue. Cosmetic-only; no prop change.

## 4. Error taxonomy ‚Üí channel

The single most important behavior change of #11: **stop routing sync PIN errors through Sonner.** Mapping is done in `login.tsx`'s catch, keyed off the thrown `Error.message`:

| Server result | Channel | Behavior |
|---|---|---|
| `INVALID_PIN` | **inline** `FieldMessage` (red) | "Wrong PIN" under the dots. Clears on the next digit (buffer reset). **Removes the current `toast.error("Wrong PIN.")`.** |
| `LOCKED_OUT:<n>` | **inline persistent banner** (`FieldMessage` red, `persist: true`) | **Static** text "Locked out ‚Äî wait `<n>`s. A manager has been notified to reset your PIN." (no live tick ‚Äî matches current behavior; `useCountdown` reuse is an optional QA-driven follow-up). Keypad stays live (server enforces the lockout). A toast would vanish before the staffer reads it. Persists until the next successful transition. |
| PIN-reset **denied** (async ‚Äî a manager declined elsewhere) | **toast** (kept) | Legitimately global/async per ADR-048; **not** a field error. Channel unchanged ‚Äî only the **re-fire bug is fixed** (¬ß5). |
| any other thrown message | **inline** `FieldMessage` (red) | Show the message verbatim (already the fallback today). |
| success | **inline** `FieldMessage` (green) ~200ms | "Welcome" + green dots, then navigate. |

Sonner stays mounted for the denial toast and for any genuinely global/async events ‚Äî consistent with ADR-048's "inline for sync validation; toast for global/async."

## 5. The "random" toast ‚Äî evidence-first (#11, part 2)

**Prime suspect (from the mega-spec investigation):** the PIN-reset-denial effect (`login.tsx:66-81`). Its fired-once guard is `shownDenialRef = useRef<Set<string>>(new Set())`. React **reinitializes refs on every mount**, so any remount within the 10-minute denial window (re-entering the PIN stage, a post-login remount) **re-fires** "PIN reset declined" though the staffer did nothing. The `useSession` debounce some half-remember is a **refuted red herring** (issue #44 postmortem) ‚Äî do not chase it.

**Per the #44 rule, the work order is fixed:**

1. **Repro first (no fix yet).** Add a regression test: mount the PIN stage with `getRecentPinResetForStaff` returning `status: "denied"` for a fixed `requestId`; assert the toast fires once; **unmount + remount** within the window; assert ‚Äî *with the current code* ‚Äî it fires **again**. This proves the remount-reset mechanism is the cause before anything changes.
2. **Then fix.** Persist the shown-denial `requestId`s **across remounts**. Concrete shape (mirror `useLastStaff.ts`, *not* `storeSession` ‚Äî that's session-id storage in `useSession`; `storage-keys.ts` is a constants-only file): declare `SHOWN_PIN_RESET_DENIALS_KEY = "frollie-shown-pin-reset-denials"` in `src/lib/storage-keys.ts`, and add a tiny helper module exposing `hasShownDenial(requestId)` / `markDenialShown(requestId)` over a JSON string-array in `localStorage`. The effect calls `hasShownDenial` instead of the in-memory `useRef<Set>`. The toast still fires exactly once per genuine denial, and never re-fires on remount. **Channel stays a toast** (correct per ADR-048); only the dedup is hardened.
3. The same regression test, after the fix, asserts **one** fire across the remount.

(If step 1 somehow fails to reproduce, stop and re-investigate before touching the guard ‚Äî that is the postmortem rule.)

## 6. Motion & accessibility

- Functional-only: pressed-state, spinner, inline red/green, 200ms green tick. **No route animation** (View Transitions deferred).
- The new scale we add (`StaffListItem`) uses the `motion-safe:` Tailwind variant ‚Üí reduced-motion users get the color/state change without movement. The keypad's press-scale is inherited from the existing `Button` base (`active:scale-[0.97]`, a pre-existing global behavior we don't re-gate); we only add `active:bg-accent` there.
- The 200ms success delay is the **only** added latency; it is feedback, not decoration, and is bypassed under no special condition (it's short enough not to feel like lag, long enough to register green).
- `FieldMessage` supplies the ARIA roles; the spinner state also exposes an accessible "Verifying‚Ä¶" label (`role="status"`).

## 7. Constraints & collision warnings

- **One PR with #7 + #11** ‚Äî they rewrite the same `PinEntry`/`login.tsx` render; splitting them causes a merge collision (mega-spec warning).
- **Preserve the `pinReset` ‚Üí fresh-`idempotencyKey` contract** (`login.tsx:46-54`). Reusing a key across retries makes the server's `_recordFailedAttempt_internal` dedupe attempts and silently freeze the lockout counter ‚Äî do not collapse it.
- **Evidence-before-mitigation** (#44 postmortem): the toast fix is labeled "fixed" only after the repro test goes red-then-green.
- **`FieldMessage` reuse, not reinvention** ‚Äî use the ADR-048 primitive; do not add a bespoke login banner component.
- **English copy** ‚Äî new strings ("Wrong PIN", "Verifying‚Ä¶", "Welcome", the lockout banner) stay English to match the existing login screen and the English-default decision (#3); keep them extraction-ready (no concatenation ‚Äî interpolate the lockout seconds as a single template) for i18n #1.
- **No `PinSheet` behavior change** ‚Äî it is the pattern source. Adding the shared `disabled` prop to `NumericKeypad` is allowed; refactoring `PinSheet` is out of scope.
- **Rollback = revert the PR.** FE-only ‚Äî no backend function-signature change (no mutation‚Üîaction rename) and no schema, so there is **no deploy-skew / atomic-build concern** (unlike the mutation‚Üîaction hazard in CLAUDE.md's deploy notes).

## 9. Tests

- Keypad **disabled while pending** (no digit registers; held-key keydown ignored).
- `INVALID_PIN` renders **inline** red `FieldMessage`, **no `toast.error`** fired.
- `LOCKED_OUT:<n>` renders the **persistent banner** with the wait-seconds; survives until next transition.
- **Success** path shows green, then `navigate("/")` after the 200ms tick.
- **Denial toast fires once across a remount** ‚Äî the ¬ß5 regression. Grounded in the existing `login.test.tsx` harness (`useQuery` call-slot 0 = `getActiveStaff`, slot 1 = `getRecentPinResetForStaff`): set `LAST_STAFF_KEY` to enter the PIN stage via pre-stage, make slot 1 return `{ status: "denied", requestId, denied_by_manager_name, ‚Ä¶ }`, then `unmount()` + re-`render()` **within one test** (no `localStorage.clear()` between) and assert `toast.error` fired **once** (red on current code, green after the dedup).
- Pressed-state CSS is presence-only (class assertion); not behavior-tested.
