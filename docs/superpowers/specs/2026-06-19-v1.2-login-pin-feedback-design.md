## 1. Goal

The booth login (`/login`) gives a staffer **no feedback** between tapping the 4th PIN digit and either landing on home or seeing a toast: the keypad stays live during the long argon2 verify, wrong-PIN errors fire as transient toasts, and keys have no pressed state (hover does nothing on touch). One toast ("PIN reset declined") also fires seemingly at random. This phase makes PIN entry feel responsive and legible.

**You'll be able to:**
- See a **pressed state** on every keypad key and staff-list row you tap (touch-friendly).
- See a **spinner** the instant you complete the PIN, with the keypad locked, so you know it's verifying (no double-submit, no "did it register?").
- Read a **wrong-PIN** error **inline** under the dots (red) instead of a toast that vanishes; a **locked-out** message stays put with its countdown.
- See a brief **green confirmation** on success before home loads.
- Stop seeing the **spurious "PIN reset declined"** toast that re-fired on remount.

**Not in scope:** route-slide / View Transitions (motion kept functional-only per Lucas); copy stays Bahasa for staff-facing strings (i18n #1 later); no `PinSheet` rewrite (it's the *pattern source*, not a target); the booth-state login fork (`shifts.public.boothState`) is the **Shift SOP Flow** spec (#6), not this one.

## 2. Architecture — a phase state machine in `login.tsx`

Today `PinEntry` owns its buffer and has **no notion of the async result** — that's the root of the missing spinner and inline error. The fix lifts result-state to the parent and makes `PinEntry`/`NumericKeypad` **prop-driven** for feedback (mirroring how `PinSheet` already threads `pending`/`error`). No async logic moves *into* the leaf components.

`login.tsx` gains:

```ts
type Phase =
  | { kind: "idle" }
  | { kind: "pending" }
  | { kind: "error"; message: string; sticky: boolean }   // sticky = LOCKED_OUT
  | { kind: "success" };
```

`onPinSubmit` drives it:
1. set `pending` → keypad disabled, dots row swaps to spinner.
2. `await login(...)` (long argon2 action).
3. **resolve** → `success` → render green dots + `FieldMessage` success ~**200ms** → `navigate("/", { replace: true })`.
4. **reject** → map the error (§4) → `error` → bump `pinReset` (clears the buffer via the existing counter) → red dots + inline message.

`pinReset` already exists and already mints a fresh `idempotencyKey` per retry (the F2/lockout fix at `login.tsx:46-54`) — that contract is preserved; we only add phase transitions around it.

## 3. Component changes

### A. `NumericKeypad` (shared — also used by `PinSheet`)
- **New prop** `disabled?: boolean`. When set: digit/clear/backspace buttons get the `disabled` attribute **and** the `document` keydown handler early-returns (a physically-held hardware key must not slip a digit past the spinner).
- **Pressed feedback:** `active:bg-accent motion-safe:active:scale-[.97]` on the digit keys. The `motion-safe:` variant means reduced-motion users still get the color change but skip the scale — no `useReducedMotion` hook needed for a CSS pseudo-state.
- `PinSheet` may pass `disabled={pending}` to inherit the lock (its handler already guards on `pending`), but no PinSheet behavior change is required by this phase.

### B. `PinEntry` (presentational rewrite)
- **New props:** `pending: boolean`, `phase: "idle" | "error" | "success"`, `message?: string`. Keeps the existing `onSubmit` + `reset` buffer contract.
- **Dots** tinted by phase: neutral (`border-foreground`) normally → **red** (`border-error`, error fill) on error → **green** (`border-success`) on success.
- **Pending** → replace the dots row with an inline `Loader2` spinner + "Memverifikasi…"; pass `disabled` to `NumericKeypad`.
- **Message** → the shipped **`FieldMessage`** primitive (ADR-048), centered under the dots: `tone="error"` for wrong-PIN/locked, `tone="success"` for the green confirmation ("Selamat datang"). `FieldMessage` already carries `role=alert` (error) / `role=status` (success) for screen readers.

### C. `StaffListItem`
- Add `active:bg-accent motion-safe:active:scale-[.98]` pressed state (touch has no hover). Cosmetic-only; no prop change.

## 4. Error taxonomy → channel

The single most important behavior change of #11: **stop routing sync PIN errors through Sonner.** Mapping is done in `login.tsx`'s catch, keyed off the thrown `Error.message`:

| Server result | Channel | Behavior |
|---|---|---|
| `INVALID_PIN` | **inline** `FieldMessage` (red) | "PIN salah" under the dots. Clears on the next digit (buffer reset). **Removes the current `toast.error("Wrong PIN.")`.** |
| `LOCKED_OUT:<n>` | **inline persistent banner** (`FieldMessage` red, `sticky: true`) | Shows the wait-seconds + the manager-notified note. A toast would vanish before the staffer reads the countdown. Persists until the next successful transition. |
| PIN-reset **denied** (async — a manager declined elsewhere) | **toast** (kept) | Legitimately global/async per ADR-048; **not** a field error. Channel unchanged — only the **re-fire bug is fixed** (§5). |
| any other thrown message | **inline** `FieldMessage` (red) | Show the message verbatim (already the fallback today). |
| success | **inline** `FieldMessage` (green) ~200ms | "Selamat datang" + green dots, then navigate. |

Sonner stays mounted for the denial toast and for any genuinely global/async events — consistent with ADR-048's "inline for sync validation; toast for global/async."

## 5. The "random" toast — evidence-first (#11, part 2)

**Prime suspect (from the mega-spec investigation):** the PIN-reset-denial effect (`login.tsx:66-81`). Its fired-once guard is `shownDenialRef = useRef<Set<string>>(new Set())`. React **reinitializes refs on every mount**, so any remount within the 10-minute denial window (re-entering the PIN stage, a post-login remount) **re-fires** "PIN reset declined" though the staffer did nothing. The `useSession` debounce some half-remember is a **refuted red herring** (issue #44 postmortem) — do not chase it.

**Per the #44 rule, the work order is fixed:**

1. **Repro first (no fix yet).** Add a regression test: mount the PIN stage with `getRecentPinResetForStaff` returning `status: "denied"` for a fixed `requestId`; assert the toast fires once; **unmount + remount** within the window; assert — *with the current code* — it fires **again**. This proves the remount-reset mechanism is the cause before anything changes.
2. **Then fix.** Persist the shown-denial `requestId`s **across remounts**: replace the in-memory `useRef<Set>` with a small localStorage-backed set keyed by `requestId` (via the `src/lib/storage-keys.ts` namespace — `storeSession`-style helper). The toast still fires exactly once per genuine denial, and never re-fires on remount. **Channel stays a toast** (correct per ADR-048); only the dedup is hardened.
3. The same regression test, after the fix, asserts **one** fire across the remount.

(If step 1 somehow fails to reproduce, stop and re-investigate before touching the guard — that is the postmortem rule.)

## 6. Motion & accessibility

- Functional-only: pressed-state, spinner, inline red/green, 200ms green tick. **No route animation** (View Transitions deferred).
- All scale effects use the `motion-safe:` Tailwind variant → reduced-motion users get color/state changes without movement.
- The 200ms success delay is the **only** added latency; it is feedback, not decoration, and is bypassed under no special condition (it's short enough not to feel like lag, long enough to register green).
- `FieldMessage` supplies the ARIA roles; the spinner state should also expose an accessible "Memverifikasi…" label.

## 7. Constraints & collision warnings

- **One PR with #7 + #11** — they rewrite the same `PinEntry`/`login.tsx` render; splitting them causes a merge collision (mega-spec warning).
- **Preserve the `pinReset` → fresh-`idempotencyKey` contract** (`login.tsx:46-54`). Reusing a key across retries makes the server's `_recordFailedAttempt_internal` dedupe attempts and silently freeze the lockout counter — do not collapse it.
- **Evidence-before-mitigation** (#44 postmortem): the toast fix is labeled "fixed" only after the repro test goes red-then-green.
- **`FieldMessage` reuse, not reinvention** — use the ADR-048 primitive; do not add a bespoke login banner component.
- **Bahasa copy** — staff-facing strings ("PIN salah", "Memverifikasi…", "Selamat datang") hardcode Bahasa now; keep them extraction-ready (no concatenation) for i18n #1.
- **No `PinSheet` behavior change** — it is the pattern source. Adding the shared `disabled` prop to `NumericKeypad` is allowed; refactoring `PinSheet` is out of scope.

## 9. Tests

- Keypad **disabled while pending** (no digit registers; held-key keydown ignored).
- `INVALID_PIN` renders **inline** red `FieldMessage`, **no `toast.error`** fired.
- `LOCKED_OUT:<n>` renders the **persistent banner** with the wait-seconds; survives until next transition.
- **Success** path shows green, then `navigate("/")` after the 200ms tick.
- **Denial toast fires once across a remount** — the §5 regression (red on current code, green after the localStorage dedup).
- Pressed-state CSS is presence-only (class assertion); not behavior-tested.

## 10. Open decisions for the plan / staffreview

- **Denial-dedup storage shape** — a single localStorage Set of `requestId`s vs per-request keys with a TTL sweep. (Lean: a small Set under one namespaced key; the 10-min server window already bounds relevance, so no client TTL needed.)
- **Spinner copy** — "Memverifikasi…" vs a non-verbal spinner only. (Lean: labeled, for the a11y status role.)
- **Success-tick timing** — 200ms (locked by Lucas). Confirm it doesn't read as lag on the real tablet during staffreview/QA.
- **Cart/other keypads** — whether to also pass `disabled` from `PinSheet` now (it already guards in its handler). (Lean: pass it for visual consistency; zero behavior risk.)
