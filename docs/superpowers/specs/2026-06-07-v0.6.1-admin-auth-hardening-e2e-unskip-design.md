## 1. Motivation

### 1a. Wave A — the action-cache leaks cached results to unauthenticated callers

`convex/idempotency/action.ts::withActionCache` is the action-layer idempotency
helper (v0.5.3b extraction). Its current shape:

```ts
export async function withActionCache<T>(
  ctx: ActionCtx,
  params: { key: string; mutationName: string },
  fn: () => Promise<T>,
): Promise<T> {
  const cached = await ctx.runQuery(internal.idempotency.internal._lookup_internal, { key: params.key });
  if (cached) return JSON.parse(cached) as T;          // ← returns BEFORE any auth
  const result = await fn();
  await ctx.runMutation(internal.idempotency.internal._writeCache_internal, { ... });
  return result;
}
```

Every PIN-gated admin action delegates to it and performs auth (`verifyManagerPinOrThrow`)
**inside `fn()`** — i.e. *after* the cache lookup. Consequence: a caller who knows a
previously-used `idempotencyKey` can call the action with that key (and any garbage
session/PIN) and receive the cached return value — an opaque document id
(`Id<"pos_vouchers">`, `{ event_id }`, `{ ok: true }`) — with **no valid credentials**.

This violates the principle codified for **mutations** in CLAUDE.md rule #20 /
[ADR-013](../../ADR/013-idempotency-keys.md) and
[`docs/PATTERNS/idempotency-dual-call-authcheck.md`](../../PATTERNS/idempotency-dual-call-authcheck.md):
*authCheck must run BEFORE the cache lookup*. The mutation HOF `withIdempotency`
already implements this — it takes an `authCheck?` option that runs first
(`convex/idempotency/internal.ts:57,61`). The **action** helper simply never grew
the equivalent.

**Scope of the gap (7 call sites, all PIN-gated admin actions):**

| Module | Action | Cached return |
|---|---|---|
| `staff/actions.ts` | `setStaffRole` | `{ ok: true }` |
| `staff/actions.ts` | `deactivateStaff` | `{ ok: true }` |
| `catalog/actions.ts` | `createProduct` | `Id<"pos_products">` |
| `catalog/actions.ts` | `createInventorySku` | `Id<"pos_inventory_skus">` |
| `catalog/actions.ts` | `updateProductPricing` | `{ ok: true }` |
| `vouchers/actions.ts` | `createVoucher` | `Id<"pos_vouchers">` |
| `inventory/actions.ts` | `recordSpoilage` | `{ event_id, ... }` |

This is **systemic** (a v0.5.3b pattern), **not a v0.6 regression** — v0.6's two
actions faithfully followed prior art. Real-world severity is low (the
`idempotencyKey` is an unguessable client UUID and the payload is an opaque id),
but it is a genuine divergence from a documented invariant and should be closed
once, at the helper, so it cannot recur.

### 1b. Wave B — 6 Playwright specs are quarantined on an unverified root cause

The v0.6 merge shipped a Playwright scaffold, but 6 of 7 functional specs are
`test.skip`'d (only `auth` sign-in runs):

`sale-qris`, `sale-bca-va`, `voucher-online`, `voucher-offline`, `refund`, `spoilage`.

Issue #43 (MERGED) attributes the skips to *"session lost during `page.goto`
post-login … `useSession.isDead`'s reaction to transient null Convex query results."*

**That attribution is unverified and suspect.** The same "useSession transient
null" hypothesis was the headline diagnosis for issue #44 — and the
[#44 misdiagnosis postmortem](../../postmortems/2026-06-issue-44-misdiagnosis.md)
found the *actual* cause was catalog "Add"-button selector drift, not a Convex
client race. The useSession fix plans were given REFUTED banners. Writing a fix
for the e2e skips on the #43 hypothesis would risk repeating that exact mistake.

**Therefore Wave B is investigation-first:** prove the cause with captured
evidence (trace/console/network) before any fix is designed.

## 2. Goals / Non-goals

**Goals**
- G1: Close the action-cache auth-before-lookup gap at the helper, with all 7
  call sites wired and a regression test per behaviour.
- G2: Make it structurally impossible to add a cached admin action without auth
  (required param, not optional).
- G3: Document the decision as ADR-046, extending rule #20 to the action layer.
- G4: Establish the **true** root cause of the e2e hard-nav failures with evidence.
- G5: Un-skip the 6 quarantined specs once they pass, one at a time.
- G6: Close out the 2 already-landed follow-ups on the board.

**Non-goals**
- N1: No change to the mutation-side `withIdempotency` (already correct).
- N2: No change to which actions are PIN-gated vs session-gated (rule #22 tiering
  is unchanged).
- N3: No predetermined e2e fix — the fix is whatever B1's evidence dictates.
- N4: Writing the empty lockout-test stub (`auth.spec.ts:24`) is out of scope
  (it's an unwritten stub, not a quarantined spec); note as a future follow-up.
- N5: No broadening of e2e coverage beyond un-skipping the existing 6 specs.

## 3. Wave A — Action-cache auth-before-lookup

### 3.1 Helper change

`withActionCache` gains a **required** `authCheck` parameter that runs before the
cache lookup, mirroring `withIdempotency`'s `authCheck` option:

```ts
export async function withActionCache<T>(
  ctx: ActionCtx,
  params: { key: string; mutationName: string },
  authCheck: () => Promise<void>,   // NEW — runs BEFORE _lookup_internal; throws on bad auth
  fn: () => Promise<T>,
): Promise<T> {
  await authCheck();                                    // fail-cheap gate, pre-cache
  const cached = await ctx.runQuery(internal.idempotency.internal._lookup_internal, { key: params.key });
  if (cached) return JSON.parse(cached) as T;
  const result = await fn();
  await ctx.runMutation(internal.idempotency.internal._writeCache_internal, { ... });
  return result;
}
```

`authCheck` is a zero-arg closure (symmetric with `fn`); each caller captures
`ctx` + `args` and asserts a live manager session. The expensive argon2 PIN
verify (`verifyManagerPinOrThrow`) **stays inside `fn()`** — so the existing
"cache hit skips PIN on a legit retry" optimisation (documented at
`catalog/actions.ts:88`) is preserved. Only the *unauthenticated-replay* hole closes.

**Why required, not optional:** every current `withActionCache` caller is a
PIN-gated admin action that needs auth. `loginWithPin`/`changePin` deliberately
do *not* use the helper (custom post-cache logic). Making the param required
closes the hole *by construction* — you cannot add a cached admin action that
skips auth — which is stronger than an optional param + a lint rule to remember.

### 3.2 Shared action-side manager assert

To avoid 7 copies of the manager check, add a small action-context helper
(home: `convex/auth/verifyPin.ts`, already the action-side manager funnel):

```ts
/** Action-context fail-cheap manager-session assert. Throws if the session is
 *  ended/absent or the staff is not an active manager. Does NOT verify PIN. */
export async function assertManagerSessionInAction(
  ctx: ActionCtx,
  sessionId: Id<"staff_sessions">,
): Promise<void> {
  const resolved = await ctx.runQuery(
    internal.auth.internal._resolveSessionRole_internal,  // existing, returns {staffId,deviceId,role}|null
    { sessionId },
  );
  if (!resolved || resolved.role !== "manager") {
    throw new Error("MANAGER_SESSION_REQUIRED");
  }
}
```

This reuses the existing non-throwing `_resolveSessionRole_internal`
(`convex/auth/internal.ts:340`).

### 3.4 ADR-046

`docs/ADR/046-action-cache-auth-before-lookup.md` — Accepted. Extends ADR-013 /
rule #20 to the action layer: every action-level idempotency cache must gate a
live-session authCheck before the cache lookup; PIN verify stays inside the
cached body. References the threat (cached-result replay) and the
required-param enforcement.

### 3.5 Tests (per action behaviour)

For each of the 7 actions (or a representative + a shared parametrised test):
- **New:** calling with an *ended/absent session* or a *staff-role* session
  throws `MANAGER_SESSION_REQUIRED` and returns **no cached value**, even when a
  cache row exists for the key (assert `_lookup_internal` is not the source of
  the return). This is the regression guard for the closed hole.
- **Unchanged-green:** the existing happy-path + idempotent-replay (same key,
  same valid manager session → cache hit, PIN not re-verified) tests still pass.

## 4. Wave B — e2e root-cause + un-skip (evidence-gated)

### 4.1 B1 — Investigation gate (produces the fix decision; no fix yet)

Pick ONE representative quarantined spec (`sale-qris` — exercises the full
cart→charge→paid golden path) and reproduce the failure under Playwright with
full instrumentation:
- `trace: "on"`, capture console + network, screenshot at failure.
- Determine **exactly** what happens after the post-login `page.goto`: does the
  app redirect to `/login`? If so, is it `useSession.isDead` firing on a
  transient-null Convex query (the #43 hypothesis), or a selector/timing/seed
  race (the #44 trap), or session state genuinely not surviving a hard reload?

**Deliverable:** a short findings note (append to
`docs/postmortems/2026-06-issue-43-e2e-hardnav.md`) stating the **proven** cause
with evidence pointers, and the chosen fix approach. This is a hard gate: no
B2 task starts until B1's cause is established.

**Candidate causes and their fixes** (decision matrix for B1 — not a commitment):

| If evidence shows… | Fix direction |
|---|---|
| Fixtures use hard `page.goto` and session is in localStorage but not rehydrated before the route's session-gate runs | Fixture navigates via in-app links (SPA nav) instead of `page.goto`; or an explicit "wait for session ready" signal before asserting |
| `useSession.isDead` truly flips on a transient null Convex result on reload | Harden `useSession` null-trust (the v0.5.7.1 "Option B" direction) — *only if confirmed*, with its own evidence |
| Selector drift / element renamed (the #44 pattern) | Fix the selectors; no session logic touched |
| Seed reset races the test (`execSync` in fixture) | Sequence the seed before navigation deterministically |

### 4.2 B2+ — Fix (depends on B1 finding)

Apply the fix B1's evidence dictates. Keep it minimal and targeted to the proven
cause. If the fix touches shared fixture code (`e2e/fixtures.ts`), re-confirm the
already-passing `auth` spec stays green.

### 4.3 B-final — Incremental un-skip

Un-skip the 6 specs **one at a time**, confirming each goes green locally before
removing the next `test.skip`. Order by golden-path centrality:
`sale-qris` → `sale-bca-va` → `refund` → `voucher-online` → `voucher-offline`
→ `spoilage`. Confirm `.github/workflows/e2e.yml` runs the now-active set; update
the v0.6 PROGRESS "Partial" note once the suite proves the golden path.

> If B1 reveals the fix is larger than a single targeted change (e.g. a real
> `useSession` rework), B-final may un-skip only the specs the fix unblocks and
> leave a residual tracked task — surfaced explicitly, never a silent partial.

## 7. Risks & open questions

- **R1 (Wave A):** Adding a required 4th param to `withActionCache` is a breaking
  signature change touching 7 call sites in one commit. Mitigation: mechanical,
  covered by typecheck + per-site tests; all sites change identically.
- **R2 (Wave B):** B1 may reveal the cause is a deeper `useSession`/PWA issue, not
  a fixture tweak — expanding scope. Mitigation: the spec treats B-final as
  possibly-partial and requires surfacing any residual, not silently capping.
- **OQ1:** Should an ESLint rule also enforce "every `withActionCache` site passes
  a real manager authCheck"? With a *required* param the type system already
  forces *an* authCheck; a lint rule would additionally assert it's a
  `assertManagerSessionInAction` call. **Proposed:** defer — required param +
  ADR-046 is sufficient for v0.6.1; revisit if a non-conforming authCheck appears.
- **OQ2:** Does `assertManagerSessionInAction` belong in `verifyPin.ts` or a new
  `auth/actionAuth.ts`? **Proposed:** `verifyPin.ts` (already the action-side
  manager funnel) to keep the auth surface in one file.
