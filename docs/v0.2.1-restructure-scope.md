## 2. Cross-module dependencies in current code

These are what the restructure inherits. Walking the imports surfaced one **backwards dependency** that needs untangling and several **legitimate foundational dependencies** that stay.

### 2.1 Backwards dependency (must fix)

**`convex/audit.ts` line 5: `import { requireManagerSession } from "./staff";`**

This is wrong directionally — `audit/` is foundational (every state-changing mutation calls into it), so it can't depend on `staff/`. The `list` query in `audit.ts` uses `requireManagerSession` to gate the manager-only audit log read.

**Fix:** extract `requireSession` + `requireManagerSession` (currently exported from `convex/staff.ts` lines 10-28) into a shared location. Two options:

- **A. `convex/lib/sessions.ts`** — fully neutral shared lib. Imported by both `staff/`, `audit/`, and any future module needing session gates.
- **B. `convex/auth/sessions.ts`** — lives in auth module. Other modules import from `auth/`. Conceptually aligns with "auth owns sessions."

**My pick:** **B**. `auth/` is already foundational (everyone imports `withIdempotency` and `logAudit`), so `auth/` owning sessions matches the existing dependency graph. `staff/` becomes a thin module that uses `auth/`'s session helpers, rather than `staff/` being foundational.

### 2.2 Legitimate foundational dependencies (stay)

`auth/` (sessions, PIN), `idempotency/`, and `audit/` are foundational. Every domain module depends on them. This is fine — they form a dependency triangle at the bottom of the graph and are stable.

```
                  [domain modules: cart, checkout, transactions, ...]
                  /          |               \
              auth/      idempotency/      audit/
                ^              ^             ^
                |______________|_____________|
                       (no upward deps)
```

The module-boundary lint rule has explicit allow-list entries for these three (the ADR §"Cross-module patterns" already names `audit/` as the singleton allow; this scope doc extends that to all three foundational modules).

### 2.3 Same-module direct table reads (not violations)

`convex/products.ts catalog` query reaches 4 tables: `pos_products`, `pos_inventory_skus`, `pos_product_components`, `pos_stock_levels`. All four are catalog-owned per §1 — not a violation. Discussed further in §3.4.

`convex/seed.ts _reset_internal` writes across 11 tables — inherently cross-cutting. Handled per §3.7.

## 3. In-scope for v0.2.1

Twelve workstreams. Estimated 1-2 days mechanical + verification. Each is a natural commit boundary.

### 3.1 Module-boundary lint rule + CI gate (MUST LAND FIRST)

**Mechanism (recommended):** custom ESLint rule under `tools/eslint-rules/no-cross-module-db-access.js`. Alternative: a CI shell script using `grep`.

**What it does:**
- Builds a table-ownership map from `convex/*/schema.ts` exports at lint time.
- For every `.ts` file under `convex/<moduleA>/`, flags any `ctx.db.query("<table>")` / `ctx.db.insert("<table>")` / `ctx.db.patch(...)` where the table is owned by `convex/<moduleB>/` and `moduleA !== moduleB`.
- Allow-list: foundational modules `auth/`, `idempotency/`, `audit/` (their tables can be touched cross-module by their owners' helpers, but in practice they're only touched through the helper functions anyway).
- Allow-list: `seed/` module (inherently cross-cutting, see §3.7).
- Allow-list: `_generated/` (Convex codegen, doesn't count).

**CI:** runs in `npm run lint`; PR fails on violation. Hard gate per ADR §"Verification".

**Why first:** if this lands after the restructure, mid-restructure violations are invisible. Lint-first guarantees every module move is verified the moment it lands.

### 3.2 Schema composition

`convex/schema.ts` currently has all 11 tables in one `defineSchema({ ... })`. v0.2.1 splits each table into its owning module's `<module>/schema.ts` fragment and composes:

```ts
// convex/schema.ts (after restructure)
import { defineSchema } from "convex/server";
import { authTables } from "./auth/schema";
import { catalogTables } from "./catalog/schema";
import { idempotencyTables } from "./idempotency/schema";
import { auditTables } from "./audit/schema";
import { telegramTables } from "./telegram/schema";  // POC stays for now

export default defineSchema({
  ...authTables,
  ...catalogTables,
  ...idempotencyTables,
  ...auditTables,
  ...telegramTables,
});
```

**Validation:** CI runs `npx convex dev --once` after the restructure to verify the composed schema parses, deploys cleanly, and indexes build. Catches any accidental name collision.

### 3.4 `pos_stock_levels` ownership decision

**The boundary case.** The `catalog` query needs it. Future `inventory/` module (when stock movements ship in v0.3) writes to it.

**Decision:** `pos_stock_levels` lives in `catalog/schema.ts`. Rationale: it's a denormalised cache for catalog reads ([ADR-017](./ADR/017-available-qty-computed-clientside.md) — `available_qty` is computed client-side from stock levels). The `inventory/` module (v0.3) will write to it via `ctx.runMutation(internal.catalog.updateStockLevel, {...})` — clean cross-module call, no direct table access.

`pos_stock_movements` (when introduced in v0.3) is `inventory/`-owned — that's the append-only ledger; `pos_stock_levels` is the projection. Classic event-source / read-model split.

### 3.5 Audit pattern: `logAudit` as plain helper, not `internalMutation`

**ADR-034 §"Cross-module patterns — Audit logging" says:** *"`logAudit` is an `internalMutation` (NOT an `internalAction`). Convex `internalMutation` calls invoked from inside another `mutation` run **in the same transaction**."*

**Current code:** `logAudit` is a **plain async function**:
```ts
export async function logAudit(ctx: MutationCtx, args: {...}): Promise<void> {
  await ctx.db.insert("audit_log", {...});
}
```

Called from inside mutations as `await logAudit(ctx, {...})` — direct TypeScript import, no Convex runtime boundary.

**Why the current code is right and the ADR text needs tweaking:**
- A plain function called inside a mutation runs in the same transaction trivially. No serialization at the boundary.
- Promoting to `internalMutation` would require call sites to use `await ctx.runMutation(internal.audit.logAudit, {...})` — same transactional guarantee, but **adds JSON serialization** of args + return value at the runtime boundary. Pure overhead.
- The ADR's intent ("modules talk through public/internal APIs") is satisfied by importing the helper from `audit/internal.ts` rather than letting modules touch `audit_log` directly. The import boundary IS the API boundary; whether the helper happens to be a Convex `internalMutation` or a plain function is an implementation detail.

**Resolution (small ADR-034 amendment, captured here):**

The `audit/` module exports `logAudit` as a **plain async function** from `audit/internal.ts`. The contract:
- Only callable from inside a `mutation` / `internalMutation` (typed `MutationCtx`).
- Atomicity inherited from the caller's transaction (Convex mutation semantics).
- Cross-module imports allowed (foundational module per §2.2).
- Direct `ctx.db.insert("audit_log", ...)` from outside `audit/internal.ts` is a CI lint block.

If a future need arises to call `logAudit` from an `internalAction` or from cross-mutation orchestration (rare), a thin `internalMutation` wrapper can be added at that time. Don't pre-build it.

**Action:** update ADR-034 §"Cross-module patterns — Audit logging" with this clarification. One-paragraph amendment, doesn't change the architectural commitment.

### 3.6 Stable string identifiers

Per ADR-034 §"Implementation notes — Stable string identifiers", POS owns 5 stable IDs. Inventory:

| Identifier | Current state | v0.2.1 action |
|---|---|---|
| `receiptNumber` (`R-YYYY-NNNN`) | Doesn't exist yet — `pos_transactions` is a v0.3 table | NO ACTION in v0.2.1 (defer to v0.3) |
| `productCode` (UPPERCASE_SNAKE, e.g. `DUBAI_8PC`) | Doesn't exist — `pos_products` has no `code` field | **ADD `code: v.string()` to `pos_products`, backfill seed** |
| `componentCode` (UPPERCASE_SNAKE, e.g. `DUBAI`) | `pos_inventory_skus.sku` is lowercase slug (`"dubai"`) | **ADD `code: v.string()` alongside existing `sku`; populate from seed** |
| `voucherCode` (UPPERCASE) | Doesn't exist — vouchers ship later | NO ACTION (defer) |
| `staffCode` (`S-NNNN`) | Doesn't exist — `staff` table has `_id` only | **ADD `code: v.string()` to `staff` table + `by_code` index, backfill seed** |

**Migration approach for the 3 fields that need adding (productCode, componentCode, staffCode):**
- Add field as optional in schema first (`code: v.optional(v.string())`).
- Update seed (`convex/seed.ts _reset_internal`) to populate codes:
  - Staff: `S-0001` (Bayu), `S-0002` (Citra), `S-0003` (Dewi), `S-0004` (Eka), `S-0005` (Lucas)
  - Components: SKU slug uppercased — `DUBAI`, `CHOCO`, `MATCHA`, `LOTUS`, `BROWNIE`
  - Products: `<family>_<pack>PC` — `DUBAI_1PC`, `DUBAI_3PC`, `DUBAI_8PC`, `CHOCO_1PC`, `MATCHA_1PC`, `LOTUS_1PC`, `MIXED_4PC`
- Re-seed dev DB. Verify uniqueness + format.
- Flip field to required (`v.string()` not optional) in a follow-up commit once dev seed is verified.
- Add `by_code` indexes on each table.

**Why keep `sku` alongside new `code` on `pos_inventory_skus`?** The `sku` field is the unique slug (kebab) used in code as `sku_family`. Renaming it to `code` would touch every call site. Safer: add `code` as the **stable external identifier** (uppercase), keep `sku` as the **internal slug** (lowercase) for now, deprecate `sku` in v0.3 when the API endpoints start emitting `code`.

### 3.7 Seed module

`convex/seed.ts` writes across 11 tables. Inherently cross-cutting. Options:

- **A. `seed/` is its own module, exempt from the lint** — single allow-list entry, no other code goes through it.
- **B. Seed delegates to per-module `internal.seed*` helpers** — e.g., `auth/internal.seedStaff`, `catalog/internal.seedProducts`. Seed orchestrates.

**My pick: A.** Option B sounds clean but adds 5 thin wrapper functions, each doing a trivial insert loop. Not worth the indirection. Seed is a known exception; document it in the lint rule's allow-list.

## 4. Open decisions for the planning phase

Three calls the planning phase should resolve explicitly. My recommended picks below.

### 4.1 `staff` table — `auth/schema.ts` or `staff/schema.ts`?

`auth/` already owns sessions, devices, lockout. `staff/` is the manager-CRUD module.

**Pick:** `staff` table in `auth/schema.ts`. Rationale: a `staff` row IS the authentication identity. `staff/` module is the **management surface** for that identity (CRUD, device codes) but doesn't own the table itself. Symmetric to how `auth/` owns `staff_sessions` even though no module is named "sessions."

### 4.2 `logAudit` — plain helper or `internalMutation`?

**Pick:** plain helper (current pattern). Document in ADR-034 amendment per §3.5.

### 4.3 `pos_inventory_skus.sku` — keep or rename to `code`?

**Pick:** keep `sku` as internal slug, add `code` as stable external identifier. Drop `sku` field in v0.3 when API endpoints standardise on `code`. Avoids a churn migration during the restructure.

## 6. Risk + reversal

**Risk:** low. The codebase is small (~1,100 LOC across 10 backend files + 5 frontend hooks/components touching Convex). Tests are comprehensive (110 passing per CHANGELOG). The restructure is mechanical — file moves + import path updates + one schema split. No business logic changes.

**Reversal cost:** ~1 day. Flattening the module structure back to today's layout is the inverse mechanical operation. Lint rule removal is trivial. The `staffCode` / `productCode` / `componentCode` schema additions can stay as optional fields permanently if reversed (no harm).

**What could go wrong:**

| Failure | Likelihood | Mitigation |
|---|---|---|
| Schema composition has a name collision | Low | CI smoke (`npx convex dev --once`) catches it on the restructure PR |
| `_generated/api` regen breaks every frontend call site | High (expected — that's the migration) | Mechanical sed; 8 call sites; verified by `npm run typecheck` |
| Lint rule has false positives | Medium | Soft-deploy lint as warning for 1 PR cycle, harden to error in next |
| `logAudit` plain-helper decision conflicts with future tooling | Low | Documented in ADR amendment; can promote to `internalMutation` later if needed |
| `staff` table moves to `auth/schema.ts` breaks `_seedHashedStaff_internal` import | Low | Test catches it; helper imports update mechanically |
