## 1. Summary

**Overall: Revise → Approved after fixes.** Architecture is sound and directly resolves the recurring locked-booth `BOOTH_NOT_OPEN` class by separating stored outlet status (Level 1) from a single shift holder (Level 2). Two Criticals found by grounding against the real codebase; both fixed inline.

## 2. Critical Issues (fixed inline)

### C1 — No mutation for the "open outlet, no holder → next staffer starts" transition
The login gate's case 4 and the transitions table said *"create shift"* for the post-handover incoming staffer, but `openBooth` rejects an already-open outlet (`BOOTH_ALREADY_OPEN`) and `handover` requires being the holder. The most common shift-change path (incoming after a handover) had **no mutation**.
**Fix:** introduced a distinct **`startShift`** mutation (Level-2-only: open outlet, no holder, `started_via:"handover"`, `prev_shift_id` link, `open_count` incoming confirm). Spec §4 case 4 + §7 transitions table now name it, with a rationale box on why it's separate from `openBooth`. *(Plan adds a corresponding task.)*

### C2 — Backfill depends on `deriveBoothState`, which the same phase deletes
The migration derived status via `deriveBoothState`, but that function is removed in this phase's retire step. Since everything ships in one atomic deploy, by the time the backfill is *run* on prod (post-deploy) the function is gone → the backfill throws.
**Fix:** spec §9 now requires the derivation logic to be **inlined into the migration** (latest event type → open/closed), explicitly forbidding the `deriveBoothState` import. *(Plan's migration task updated to match.)*

## 3. Improvements (addressed)

- **Deploy atomicity** — this phase renames public functions (`lockShift→lock`, etc.), which is deploy-skew-fatal per CLAUDE.md. Added a load-bearing note to §9 requiring the single atomic Vercel prod build; the PR #143 stopgap is replaced wholesale.
- **`loginContext` unbound-device** — it's pre-login/session-less; `_getDeviceOutletId_internal` throws `DEVICE_HAS_NO_OUTLET` on an unbound device. §4 now requires graceful degradation (`{outletOpen:false}`) so an unactivated device still renders login.
- **`_sendSignoffSummary.eventId`** typed `Id<"pos_shift_events">` — must widen to accept `Id<"pos_shifts">`/`v.string()` since the per-shift summary now keys off a `pos_shifts` id. (Tracked in the plan, Task 5.)
