01 · the product

what is it?

a production point-of-sale for our cookie booth at Pakuwon Mall. real money, real QRIS, on one Android phone.
sell
cart to commit to charge. QRIS QR or BCA VA rendered in the app. price and name frozen onto each line at the moment of sale.
confirm
a signature-verified webhook is primary. a manager PIN is the only fallback. polling is gone.
prove
every paid sale mints a signed-URL receipt and prints to a 58mm thermal printer over Web Bluetooth, in one tap.
govern
refunds are new rows, never status edits. an append-only audit log is the system of record.
reach
off-booth, a gated action posts a Telegram card with an approve link. the token authorizes the view, the PIN authorizes the act.

04 · the patterns

the patterns I actually run

six patterns in the playbook. I lean on three, hard. (the diagram on the last slide was all three at once.)
fan-out + synthesize backbone
3 reviewers, 3 lenses, different files. 4 simplify agents. I never let them merge notes early.
adversarial verify signature move
double staffreview gate before code. then triple-review the diff against 22 ADR rules.
loop until done convergence
review, fix, re-review until zero criticals. then run the actual command and confirm green.
classify + act plumbing
/progress routes tasks by lane. approval kinds route by validator. deterministic routers, not reasoning agents.
generate + filter supporting
/simplify generates every cleanup. the filter is brutal: does it change behavior? can I verify it without a live API?
tournament not used
my agents collaborate by division of labor. they don't compete on the same task. no winner to crown.

06 · what changed under me

the rules rewrote themselves

five decisions I wrote down, then tore up mid-build. the ADRs are the diary.
share one Convex project with our other product ▸ own dev + prod deployments. talk over a versioned HTTP API. "integrate by sharing the database" is the decision you regret first. (ADR-034)
QRIS needs status polling plus reconcile-on-reload ▸ webhook confirms, manager PIN is the only fallback, polling gone. the reconcile hook is a no-op shell I left in on purpose. (ADR-036)
approvals over WhatsApp share-intent links ▸ Telegram URL buttons carry the auth token in the link, no bot-side state. the wa.me literal sits in the schema like a fossil.
settlement arrives on a webhook ▸ the webhook does not exist. settlement is poll-only: GET the transactions list, match on my own reference_id.
the Xendit invoice returns a scannable QR at creation ▸ it returns a URL to a hosted page. one wrong sentence meant zero payments could complete. fixed with the dedicated QR Codes + FVA APIs.
the fix was not being smarter up front. it was making decisions cheap to reverse, in writing, where a reviewer could catch them.

07 · what it actually cost

the build was the cheap part

features took an afternoon. the boring, load-bearing wiring took the days.
SHORT · a whole slice in a session
Telegram as the real approval channel: 444 tests, 29 commits. receipts, reporting, admin CRUD: each one slice. the design system: one baseline commit.
LONG · the unglamorous wiring
which order auth and the idempotency cache run in. threading one source argument through every call site. keeping two divergent mutations in parity.
WORST BUG-TO-FIX RATIO
one HTTP header: api-version: 2022-07-31. leave it off and the payment webhook never fires. no error. the QR works. the money never confirms. now asserted by a test.
tests, version by version
the surprises that lied to me
a green suite lied: convex-test matched absent rows on undefined, not prod reality. 288 passing, would have returned null in prod.
tsc --noEmit reported clean while checking zero files.
the Vite "Network:" line is a lie. Windows Defender silently drops the inbound TCP.

08 · the takeaway

force is free now.
direction is the whole job.
